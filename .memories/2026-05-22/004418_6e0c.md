## Fixed T2 regression: titles_for_files including import-kind symbols

### WHAT
Fixed `titles_for_files` in `src/database/symbols/queries.rs` to filter out non-definition symbols (imports, references, etc.) from the SQL query. Added regression test `titles_for_files_excludes_import_kind_symbols`.

Also resolved 5 merge conflicts that were blocking compilation (gdscript/identifiers.rs, tests/powershell/mod.rs, tests/vue/script_setup.rs, vue/script_setup.rs, fixtures/extraction/vue/basic/expected.json).

### WHY
`titles_for_files` was returning ALL symbol rows including `kind='import'`. A file like `tracing.rs` that has `use crate::database::SymbolDatabase` stores an import symbol with `name='SymbolDatabase'`, `kind='import'`. This caused `tracing.rs` to receive `EXACT_TITLE_BOOST` (+100) when searching for "SymbolDatabase", beating the actual definition file in search rankings. `test_ranking_source_over_tests` was failing as a result.

### HOW
Added `AND kind IN ('class','struct','interface','trait','enum','function','method','constructor','module','namespace','type','constant','delegate')` to the SQL query. The kind set mirrors `DEFINITION_KINDS` in `src/search/scoring.rs` (which is `const` not `pub`, so it's mirrored as a SQL literal with a sync comment rather than imported).

**Approach**: whitelist (narrow), not blacklist (wide). Rationale: new symbol kinds default to excluded from boost eligibility unless explicitly added, which is the safer behavior.

**Merge conflicts**: All 5 were from a stash pop. GDScript and PowerShell were pure whitespace/duplicate-attribute conflicts. Vue conflicts involved a `name_node` vs `node` change in `script_setup.rs` (took the `name_node` version as more precise). Vue fixture `expected.json` had stash values with `start_byte: 0, end_byte: 0` (incomplete), took upstream values.

### IMPACT
