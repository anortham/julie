## T3: Simplify tokenizer + delete env-var ablations

### WHAT
Implemented T3 of FTS Phase 2 unified-schema rollout:
- Added `SimpleCodeTokenizer` (lowercase + ASCII-fold + 80-char cap, registered as `"simple_code"`)
- Added `pretokenize_code(text: &str) -> String` free function (original + camel splits + snake splits per whitespace token)
- Removed `ablate_stemming` / `ablate_camel_emit` fields from `CodeTokenizer` and `TokenizerCompatibilitySignature`
- Removed env-var reads (`JULIE_ABLATE_STEMMING`, `JULIE_ABLATE_CAMEL_EMIT`) from `CodeTokenizer::new()`
- Removed stemmer invocation from `tokenize_code`; removed `ablate_camel_emit` gate (camel splitting now unconditional)
- Deleted `Ablation::NoStemming`, `NoCamel`, `Both` variants from `xtask/src/cli.rs`; `apply_env` now returns a no-op `EnvGuard::capture(&[])`
- `EnvGuard` kept in `xtask/src/cli.rs` â€” still returned by `apply_env`, still called from `search_matrix.rs:run_baseline_async`
- Deleted `src/tests/tools/search/tokenizer_ablation_tests.rs` and `xtask/tests/search_matrix_ablation_tests.rs`
- Updated `src/tests/tools/search/mod.rs`: removed `tokenizer_ablation_tests`, added `tokenizer_simple_test` and `pretokenized_emit_test`
- Registered `"simple_code"` tokenizer in `SearchIndex::register_tokenizer` (wiring for T4/T5)

### WHY
Phase 2 unified schema moves code-aware splitting to a dedicated `pretokenized_code` index field. The new fields use `"simple_code"` tokenizer so no further splitting happens at Tantivy level. `CodeTokenizer` gets cleaned of stemming (which hurts precision on code) and the env-var ablation scaffolding no longer needed after Phase 1 bakeoff completed.
