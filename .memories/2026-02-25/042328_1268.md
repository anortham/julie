## Fixed FILE_CONTENT migration check using wrong DB for reference workspaces

`filter_changed_files` had a FILE_CONTENT migration check (lines 222-253) that queried `handler.get_workspace()` (primary DB) to check if JSON/TOML/markdown files had symbols. For reference workspaces, those files don't exist in the primary DB → `get_file_symbol_count` returns 0 → files always flagged as "modified".

- **Root cause:** Third instance of "reference workspace using primary DB" pattern
- **Fix:** Thread `workspace_db` from the earlier DB-routing block and use it for the FILE_CONTENT check instead of `handler.get_workspace()`
- **Result:** Back-to-back non-force refreshes should now show "Already up-to-date" on second run
- **Also:** Cleaned TODO.md — removed resolved bug entries
