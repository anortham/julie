## Fixed reference workspace Tantivy search tokenizer mismatch

### Root cause
Reference workspace search opened Tantivy index with `SearchIndex::open()` which uses `CodeTokenizer::with_default_patterns()`, but the index was **created** with `SearchIndex::open_or_create_with_language_configs()` which uses `CodeTokenizer::from_language_configs()`. Despite sharing the same `tokenize_code()` core, the different configurations produce different Tantivy term dictionaries. `TermQuery` does exact term matching, so the mismatch causes search to return nothing.

### Evidence
- Wrote `test_tokenizer_mismatch_reproduces_ref_workspace_bug` — FAILS, confirming the bug
- Wrote `test_same_tokenizer_search_works` — PASSES, confirming same tokenizer works
- Wrote `test_ref_workspace_search_with_matching_tokenizer` — PASSES with the fix

### Fix
Changed 3 reference workspace search paths from `SearchIndex::open()` to `SearchIndex::open_with_language_configs()`:
- `src/tools/search/text_search.rs:92`
- `src/tools/search/line_mode.rs:171`
- `src/tools/get_context/pipeline.rs:401`
