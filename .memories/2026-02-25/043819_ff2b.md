## Fixed infinite re-indexing loop for parser-less files in reference workspaces

### Root cause
`filter_changed_files` had a `needs_file_content` migration check that flagged files with `symbol_count == 0` for re-indexing. But `create_file_info` always initializes `symbol_count = 0`, and `process_file_without_parser` never updates it (only `process_file_with_parser` does at line 635). This created an infinite loop for text/json/markdown/toml files without parsers:

1. Hash matches (file unchanged) ✓
2. `needs_file_content` check sees `symbol_count == 0` → flag for re-indexing
3. File processed → `symbol_count` stays 0 → stored as 0
4. Next refresh → same check → same result → forever

### Evidence
- 25 files with `symbol_count = 0` (21 text, 2 json, 1 markdown, 1 dockerfile)
- 24 of those match `needs_file_content` language filter (dockerfile excluded)
- Exactly matches the "24 modified" count from back-to-back refreshes

### Fix
Removed the `needs_file_content` migration check entirely. It was a one-time migration concern that:
- Can't distinguish "never processed" from "processed, no symbols"
- Creates permanent re-indexing loops for legitimate zero-symbol files
- Is handled by force-refresh for any workspace that genuinely needs migration
