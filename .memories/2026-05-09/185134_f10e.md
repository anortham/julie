## Sensitive cwd workspace-root guard

Expanded the cwd-only workspace-root rejection from "filesystem root only" to a full sensitive-paths list, after discovering `murphy_00b4139d` (path=/Users/murphy) had been auto-registered May 7 from a `cwd=/Users/murphy` MCP session. v7.8.3's `reject_cwd_filesystem_root_startup_hint` only blocked `path.parent().is_none()` (i.e. `/`), so home dirs slipped through.

### What
- Renamed `reject_cwd_filesystem_root_startup_hint` â†’ `reject_sensitive_cwd_startup_hint` in `src/handler.rs`
- New free fn `is_sensitive_workspace_root(&Path)` returning true for: `/`, drive root, `dirs::home_dir()`, plus platform list:
  - macOS: `/Users`, `/var/root`
  - Linux: `/home`, `/root`
  - Windows: `C:\Users`, `C:\Windows`, `C:\Windows\System32`, `C:\Program Files`, `C:\Program Files (x86)`, `C:\ProgramData`
- `normalize_sensitive_root_for_compare` strips trailing slashes; lowercases on Windows for case-insensitive compare
- New error: "Refusing to use sensitive system path {} as Julie primary workspace from process cwd. Set JULIE_WORKSPACE or launch the MCP server from a project directory."
- Both call sites in handler.rs (`reconcile_primary_workspace_to_startup_hint`, `ensure_primary_workspace_for_request`) updated
