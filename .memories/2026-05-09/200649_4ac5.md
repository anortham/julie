## Handler split shipped — bucket selection now precise per tool

Test feedback loop fix: any edit to `src/handler.rs` previously forced `cargo xtask test changed` to fall back to the entire dev tier (~22 buckets, 10+ min compile + run). After this commit, a tool-only edit maps to its specific bucket(s).

- **Approach:** Each of 12 MCP tools moved into its own file under `src/handler/tools/<name>.rs` with a dedicated `#[tool_router(router = tool_router_<name>, vis = "pub(crate)")]` impl block. `handler.rs` has a hand-written `tool_router()` composer adding them with `+`. Required rmcp 1.6's multi-router composition feature.
- **handler.rs:** 3254 → 2500 lines (-23%)
- **xtask change:** removed `src/handler/` from `DEV_FALLBACK_PREFIXES`; added `handler_tool_buckets_for_path()` with 1:1 tool→bucket mapping; added `session_workspace.rs`, `tool_metrics.rs`, `tool_targets.rs`, `tools/mod.rs` to `DEV_FALLBACK_FILES` to preserve dev-tier fallback for cross-cutting changes.
- **Cargo.toml:** `[profile.dev.package."*"] opt-level = 3` so heavy deps (Tantivy, tree-sitter, sqlite) compile at release speed in dev profile.
- **Live verification:** edit to `src/handler/tools/fast_search.rs` selected `mode=Buckets` with the 7 `tools-search-*` buckets only — confirmed end-to-end after rebuild + restart.
- **Test evidence:** 380 search tool tests, 96 editing tool tests, 30 xtask `changed_tests` (5 new) all pass.
- **Commit:** 8ce66ac0
- **Caveats:** Gemini's original analysis had two false premises (298GB target, opt-level=2 dev) — actual values were 1.3GB and 0. The user's pre-session `cargo clean` had already reclaimed the bloat. The handler.rs split was directionally right despite the bad framing.

## What still triggers dev fallback (intentional)
- `src/handler.rs` itself (the workspace plumbing + ServerHandler trait impl)
- `src/handler/session_workspace.rs`, `tool_metrics.rs`, `tool_targets.rs` (cross-cutting helpers used by every tool)
- `src/handler/tools/mod.rs` (organizational; touching it means adding/removing a tool)

## Next
- After more dogfooding, consider further splitting handler.rs's workspace-plumbing chunk (lifecycle, snapshots, routing) into submodules so handler.rs itself can leave the fallback list. Not urgent.
