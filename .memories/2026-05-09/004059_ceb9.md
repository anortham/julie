## Implemented `cargo xtask dev-link` and `dev-restart` for maintainer dev loop

Goal: replace the painful "rebuild → daemon force-kill → MCP session dies" cycle with a one-command graceful path that works across all three harnesses (Claude Code, Codex CLI, OpenCode).

- **`dev-link`**: scans `~/.claude/plugins/cache/julie-plugin/julie/<v>/bin/<arch>/julie-server`, replaces real binaries with symlinks to `target/release/julie-server`. Idempotent. `--dry-run` and `--cache-root <path>` flags. Codex CLI and OpenCode already point their MCP `command` at `target/release/julie-server` directly per their config conventions, so no symlink needed for them — `cargo build --release` updates everything in one shot now.
- **`dev-restart`**: graceful SIGTERM via existing `lifecycle::stop_daemon()`. No drain timeout (unlike stale-binary auto-restart), no force-kill. Adapter respawns daemon on next MCP request, picks up new binary via the symlink.
- **Code**: new `xtask/src/dev_workflow.rs` (4 unit tests, all green: link replaces real binary / idempotent / dry-run preserves filesystem / missing cache subtree handled). Wired into `xtask/src/cli.rs` (parsing), `xtask/src/lib.rs` (module), `xtask/src/main.rs` (dispatch). CLAUDE.md + AGENTS.md updated with Quick Reference rows and a new "Maintainer dev loop" section in the Dogfooding Strategy.
- **Verified end-to-end**: linked the live v7.8.1 + v7.8.2 plugin caches, confirmed symlinks point at the dev binary. Re-running reports `already-linked`. After user reinstalled v7.8.2 plugin from GitHub (real binary), one `dev-link` call relinked the new cache entry. `~/source/julie-plugin` divergence resolved separately via `git reset --hard origin/main` (the local doc commit was already in remote at `4d914fd`).
- **Tests**: `cargo nextest run -p xtask --lib` 19/19, `cargo check --workspace` clean. No new test buckets in tier files since the suite is xtask-only.
