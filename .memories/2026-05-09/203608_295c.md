## Test bucket selection refinement: 14 buckets → 22 buckets, dev tier from 22 to 32 entries

### What
Implemented all suggestions from the analysis to reduce per-edit test wall-clock time:

**Tier 1 — Eliminate unnecessary `dev` fallbacks:**
- Removed from DEV_FALLBACK_FILES: `src/handler/{session_workspace,tool_metrics,tool_targets,tools/mod}.rs`, `src/migration.rs`, `src/tests/migration.rs`, `src/startup.rs`
- Removed from DEV_FALLBACK_PREFIXES: `src/extractors/`, `src/analysis/`
- Each routed to a narrow bucket set in `xtask/src/changed.rs::buckets_for_path`

**Tier 2 — Restored orphaned coverage:**
- Added `analysis` bucket. `src/tests/analysis/` (5 files, ~24K lines: change_risk, early_warnings, test_linkage, test_quality, test_roles) was previously not in any tier. Now in dev + full.

**Tier 3 — Split oversized buckets:**
| Old bucket | Old time | New buckets |
|---|---|---|
| tools-search-line-file | 80s | tools-search-line / tools-search-file-mode / tools-search-zero-hit |
| tools-search-ranking-format | 40s | tools-search-promotion / tools-search-format-quality |
| tools-navigation | 35s | tools-deep-dive / tools-call-path / tools-fast-refs / tools-blast-spillover |
| tools-get-context | 35s | tools-get-context-pipeline / tools-get-context-format / tools-get-context-graph |
| tools-workspace | 40s | tools-workspace + tools-workspace-targeting (carved out global_targeting + refresh_routing) |

### Why
"We HAVE to get the testing times down in this project before we can make any real engineering progress." Per-edit `cargo xtask test changed` was running the full dev tier (~5min) for any handler subfile or migration/startup edit, and was running 80s buckets when one feature touched.

### How
- All four files updated: `xtask/src/changed.rs`, `xtask/test_tiers.toml`, `xtask/tests/changed_tests.rs`, `xtask/tests/manifest_contract_tests.rs`
- New helper `get_context_test_buckets_for_path` for per-file get_context routing
- Source-level routing for src/tools/get_context, src/tools/navigation/{call_path,fast_refs,target_workspace}.rs, src/tools/{deep_dive,impact,spillover,workspace}/, src/handler/tools/{deep_dive,call_path,fast_refs,blast_radius,spillover_get,get_context,manage_workspace}.rs each map to their precise bucket(s)
- Heavy targeting (global_targeting.rs ~100KB + refresh_routing.rs ~28KB) carved out
- xtask test inventory + 143/144 xtask tests pass; pre-existing docs marketing failure on main is unrelated

### Impact
- **Edit one tool, run `cargo xtask test changed`:** today 35-80s focused-bucket time → 8-30s per-feature time
- **Edit a handler subfile / migration / startup:** today ~5min full dev → 10-60s narrow bucket
- **Restored ~24K lines of analysis test coverage** that previously ran nowhere
- dev tier grew from 22 → 32 bucket entries (test count unchanged, just routed more precisely)

### Verification
- `cargo nextest run -p xtask` passes 143/144 (1 pre-existing docs/site marketing failure on main, unrelated)
- `cargo nextest list --lib <new-filter>` confirms all new buckets find real tests:
  - tests::analysis → many tests
  - tests::tools::search::line_ → 60 lines listed
  - tests::tools::search::definition_ → 24
  - tests::tools::deep_dive_tests → 66
  - tests::tools::workspace::global_targeting → 35
  - tests::tools::get_context_pipeline_tests → 10
- `cargo xtask test changed` correctly identifies xtask-runner + daemon for the current diff
- `cargo xtask test list` shows all new buckets with metadata
