## Adapter Mode Connection Flow - Julie IPC Architecture

### WHAT - The Issue

The error "MCP serve failed: connection closed: initialize request" indicates that:
1. The adapter successfully connects to the daemon's IPC socket
2. The daemon reads the workspace header (`WORKSPACE:/path\n`)
3. The daemon starts the MCP handshake (rmcp's `serve()` method)
4. The connection closes before `initialize` completes, causing rmcp's service to fail

### HOW - The Connection Flow

#### 1. Adapter Initialization (`src/adapter/mod.rs` - `run_adapter()`)

- **Lines 32-55**: Main adapter function
  - Spawns daemon if not running (via `DaemonLauncher::ensure_daemon_running()`)
  - Connects to IPC socket: `IpcConnector::connect(&ipc_addr)`
  - Sends workspace header: `WORKSPACE:/path\n`
  - Enters byte forwarding loop: `forward_bytes(stream)`

#### 2. Daemon Launcher (`src/adapter/launcher.rs`)

- **Lines 49-98**: `ensure_daemon_running()`
  - Uses advisory lock (`daemon.lock`) to serialize startup across adapters
  - Spawns daemon process: `julie daemon` command
  - Waits for IPC socket to appear (45-second timeout with exponential backoff)
  - Critical: The lock is released **before** waiting for the socket

#### 3. IPC Transport Layer (`src/daemon/ipc.rs`)

**Unix (Lines 8-100)**:
- `IpcListener::bind()`: Creates a Unix domain socket, removes stale sockets if present
- `IpcConnector::connect()`: Simple `UnixStream::connect(path)`
- Both implement `AsyncRead + AsyncWrite` directly (no wrapper needed)

**Windows (Lines 106-237)**:
- Uses named pipes: `ServerOptions::new().create(path)`
- `accept()` pre-creates the next instance before waiting for a connection
- `connect()` implements retry logic (10 attempts) for ERROR_PIPE_BUSY (231)
  - Backoff: 50ms, 100ms, 200ms, 400ms, 500ms (capped)

#### 4. Daemon Session Handler (`src/daemon/mod.rs` - `handle_ipc_session()`)

Lines 676-821: Per-session handler for each IPC connection

**Key sequence**:
1. **Lines 687-693**: Read workspace header with 5-second timeout
   - `read_workspace_header()` reads byte-by-byte until newline
   - Timeout: "Workspace header read timed out (5s)" if header doesn't arrive
   - Failure: "Failed to read workspace header" wraps underlying I/O error

2. **Lines 701-719**: Initialize workspace from path
   - Compute workspace ID using `generate_workspace_id()`
   - Get/create shared workspace from `WorkspacePool`

3. **Lines 725-808**: MCP serve + session blocking
   - **Lines 785-788**: `handler.serve(stream).await` wraps error as "MCP serve failed: {}"
   - The `serve()` method is from `rmcp::ServiceExt` trait, which:
     - Takes ownership of the stream
     - Runs the MCP initialize handshake
     - Returns a service for further interaction
   - **Lines 791-800**: `service.waiting().await` blocks until client disconnects or MCP error
     - This is where "connection closed: initialize request" would originate (from rmcp)

### THE CRITICAL ISSUE - Byte Forwarding Race

The adapter's `forward_bytes()` function (lines 84-126) creates a subtle race condition:

```rust
let (mut ipc_read, mut ipc_write) = tokio::io::split(stream);
let mut stdin = tokio::io::stdin();
let mut stdout = tokio::io::stdout();

tokio::select! {
    result = copy(&mut stdin, &mut ipc_write) => { ... }
    result = copy(&mut ipc_read, &mut stdout) => { ... }
}
```

**The Problem**:
1. When either direction finishes (stdin EOF or daemon disconnects), **both `copy()` tasks are cancelled**
2. The `tokio::select!` macro cancels all other branches when one completes
3. If the MCP client sends all its initialize request to stdin, `copy(&mut stdin, ...)` completes
4. This cancels `copy(&mut ipc_read, ...)`, which stops reading from the daemon
5. Meanwhile, the daemon's `serve()` method is waiting for a response from the client
6. The daemon writes the response but the adapter isn't reading it (read task cancelled)
7. When the daemon's write buffer fills, it gets `BrokenPipe` or connection drops

**Detailed scenario**:
- Client (e.g., Cline) sends complete initialize request and closes stdin (EOF)
- Adapter's `copy(&mut stdin, ...)` task completes, returns `Ok(bytes)`
- `tokio::select!` cancels the opposing `copy(&mut ipc_read, ...)` task
- Daemon's rmcp service sends initialize response back through IPC
- Daemon's write succeeds initially but adapter isn't reading (task cancelled)
- Eventually daemon's write buffer fills or connection times out
- rmcp detects closed connection during initialize: "connection closed: initialize request"

### IMPACT

This explains why:
- The daemon sees "Adapter connected" and socket accepts successfully
- The workspace header is read fine (5-second timeout not hit)
- The MCP handshake starts (`serve()` called)
- But then fails with "connection closed: initialize request"
- The adapter may see no error on its end (stdin already EOF'd)

### ROOT CAUSE

`tokio::select!` is **not** the right mechanism for bidirectional forwarding. When one direction finishes (stdin EOF), it shouldn't cancel the opposite direction's read loop. The proper pattern is:
- **Option A**: Use `tokio::join!` instead, so both tasks run to completion
- **Option B**: Graceful shutdown: stdin EOF → close write half, keep read half open
- **Option C**: Half-close semantics with `shutdown(Write)` on socket

Current code assumes unidirectional forwarding (client sends, then exits), but MCP is bidirectional.
