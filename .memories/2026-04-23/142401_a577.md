## Plan A Complete: CLI Interface for Julie

All 5 tasks implemented and reviewed on `feat/cli-tool-interface`.

### What was built

- **A1: Clap surface** - 7 top-level Command variants (search, refs, symbols, context, blast-radius, workspace, tool) plus global flags (--json, --format, --standalone). 26 parsing tests.
- **A2: Execution core** - `run_cli_tool()` entry point with daemon mode (IPC reusing adapter handshake) and standalone mode (production `JulieServerHandler::new()` constructor). DaemonFallback when IPC unavailable. 28 tests.
- **A3: Tool wiring** - All 12 MCP tools reachable via generic `tool <name> --params` path. 6 named wrappers map CLI args to tool structs. Found and fixed 2 daemon-mode JSON field name bugs. 18 deserialization tests.
- **A4: Output formatters** - Text (as-is tool output), JSON (pretty-printed CallToolResult), Markdown (header + fenced block). Diagnostics to stderr. Exit code 0/1. 18 tests.
- **A5: Integration tests** - 13 end-to-end tests via `std::process::Command`. Covers named wrappers, generic path, all 3 output formats, exit codes, stderr isolation. Uses `#[ignore]` with xtask prebuild.

### Key architecture decisions

- Hybrid CLI: named subcommands for common tools, generic fallback for all 12
- `CliToolCommand` trait with `tool_name()`, `to_tool_args()`, `call_standalone()` unifies both paths
- Daemon mode reuses adapter IPC (build_ipc_header, read_daemon_ready), not a parallel transport
- Standalone mode uses production constructor, not `new_for_test()`

### Known limitations

- `FastRefsTool` lacks `file_path`/`file_pattern` fields; CLI args exist but tool struct doesn't support them
- `BlastRadiusTool --rev` needs a revision resolution layer for standalone mode (git ref to DB revision ID)
- `Box::leak` in `GenericToolArgs::tool_name()` for trait lifetime; acceptable for single-shot CLI
