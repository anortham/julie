## Implemented HTTP stdio shim

### WHAT
- Added `src/adapter/http_stdio.rs` to bridge client stdin/stdout JSON-RPC to the daemon's canonical Streamable HTTP MCP endpoint.
- Updated `run_adapter` to use HTTP discovery when available while preserving legacy IPC fallback.
- Enabled the `rmcp` reqwest Streamable HTTP client feature in `Cargo.toml` and `Cargo.lock`.
- Added focused adapter HTTP shim tests and recorded Task 4 verification evidence in `docs/plans/2026-05-03-http-daemon-transport.md`.

### WHY
The daemon now publishes and serves a real HTTP MCP endpoint, but the spawned adapter still used IPC. This left the canonical transport incomplete and prevented dogfooding the HTTP path through normal stdio MCP clients.

### HOW
- Built a message-level stdio shim on top of `rmcp::StreamableHttpClientTransport`, including bearer-token loading, `x-julie-*` headers, retry replay for immediate restart handoff, and newline-delimited stdout framing.
- Kept the pump bidirectional enough for server requests like `roots/list` while a client request is in flight, avoiding a half-duplex deadlock.
- Split the implementation out of `src/adapter/mod.rs` to keep implementation files under the project file-size rule.

### IMPACT
- No-argument `julie-server` can now proxy stdio clients to the daemon HTTP endpoint when discovery reports Streamable HTTP.
- Adapter compatibility is covered by focused tests plus the transport bucket.
- The changed gate still fell back to the expensive dev tier because `Cargo.toml` and `Cargo.lock` changed, which is useful evidence for the ongoing test-runtime cleanup work.
