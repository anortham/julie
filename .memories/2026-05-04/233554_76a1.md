## Completed tree-sitter plan tasks 1-4

### WHAT
Implemented the first verified batch of the tree-sitter best-in-class plan:

- Added a production-path `extract_canonical` golden fixture harness for every registry language entry and variant.
- Added `fixtures/extraction/capabilities.json` plus one golden fixture and expected snapshot per language entry.
- Added `extractors` and `parser-upgrade` xtask buckets with changed-path routing and manifest contract tests.
- Centralized indexing candidate policy for watcher, startup repair scans, and initial discovery.
- Fixed watcher incremental indexing to run extraction on the blocking pool and resolve pending relationships after updates.
- Preserved vendor-directory discovery breadth while keeping final indexing strict for unknown binary extensions.

### WHY
The tree-sitter extraction data is the base layer for Julie search, navigation, and editing. Before parser upgrades or deeper extractor refactors, we need whole-inventory golden gates and consistent indexing behavior so language entries do not quietly fall through the cracks.

### HOW
The golden harness normalizes IDs into stable fixture-local keys while retaining meaningful extraction contract fields: symbols, spans, parent links, relationships, pending relationships, structured pending relationships, identifiers, types, doc comments, and annotations. The shared file policy now accepts supported parser-backed extensions, accepts unknown/extensionless text as text-only, rejects blacklisted names/extensions, and is reused by watcher filtering, startup scans, and discovery. Watcher handlers now use `spawn_blocking` for extraction and reuse pending relationship resolution logic from batch indexing.

### IMPACT
Tasks 1-4 now have executable gates. Parser upgrades have a whole-language tripwire, and watcher/repair/batch paths agree on text fallback and blacklists. This reduces the chance of silent parser drift and stale or missing relationship data from incremental updates.
