## Removed WorkspacePool Migration Fossils

### WHAT
- Simplified `WorkspacePool::new` to accept only `indexes_dir` and optional `DaemonDatabase`.
- Removed the unused watcher and embedding constructor parameters that were left behind after session lifecycle ownership moved to `WorkspaceSessionAttachment`.
- Deleted `WorkspacePool::active_count` and rewrote tests to assert observable cache behavior via `get`/`get_or_init` and `Arc::ptr_eq`.
- Updated xtask search-matrix harnesses and cleanup-plan notes.

### WHY
The extra constructor parameters were migration baggage, and `active_count` existed only because tests reached into a private implementation detail. Keeping both made WorkspacePool look responsible for watcher/session lifecycle even though that responsibility had already moved.

### HOW
- Used Julie `deep_dive`/`fast_refs` and raw `rg` to verify references.
- Applied a mechanical constructor call-site rewrite, then fixed the xtask calls caught by the changed gate.
- Replaced map-length tests with cache reuse and distinct-workspace assertions.

### IMPACT
WorkspacePool has a smaller API and clearer ownership: shared workspace cache plus daemon registry persistence only. The changed gate also exposed another testing pain point: this small constructor cleanup selected 22 buckets and took 379.1 seconds.
