## Wrote Toolkit Metrics / Blast Radius / Edit Telemetry Plan

### WHAT
Created `docs/plans/2026-05-04-toolkit-metrics-blast-radius-edit-telemetry.md`, a full implementation plan covering:
- failure-aware tool metrics
- bounded and faster `blast_radius`
- edit-tool efficiency/friction telemetry
- dashboard/query updates

### WHY
Local tool telemetry showed unpopular tools, but the investigation found concrete issues instead of simple popularity problems:
- failed tool calls are not recorded because instrumentation only runs after success and writes `success=true`
- `blast_radius` is slow because depth-2 graph traversal expands through common identifier names and broad SQL query shapes
- edit tools need input/outcome metrics to prove token savings across Claude Code and Codex-style harnesses

### HOW
Used Razorback brainstorming and writing-plans workflows. Delegated three read-only investigation lanes for metrics, blast radius, and edit-tool UX. Saved a plan with exact files, TDD test names, verification gates, model routing, and hard metric probes.

### IMPACT
The plan is ready for user review before execution. It keeps `manage_workspace` out of scope, treats metrics honesty as the first task, bounds `blast_radius` traversal before ranking, and adds edit telemetry before making instruction/removal decisions.
