## Tauri Tray App v4.2.0 — Complete Implementation, Code Review, and Release

### WHAT
Full Tauri 2.0 system tray app for Julie daemon management, shipped as v4.2.0 across 7 phases:

1. **Skeleton** — Tray-only app (`"windows": []`), workspace member
2. **Daemon lifecycle** — Start/stop/restart via subprocess + HTTP, startup lock, health polling with backoff
3. **Dynamic menu** — Live project list, embeddings summary, icon state machine (green/yellow/red)
4. **Autostart + Updates** — Login launch via `tauri-plugin-autostart`, GitHub releases checker (6h)
5. **Diagnostic bundle** — Tray-side zip export with path sanitization
6. **Dashboard enhancements** — `GET /api/diagnostics/report` endpoint + "Export Diagnostic Bundle" button
7. **CI/CD** — Parallel `build-tray` job in release workflow (.dmg, .nsis, .AppImage, .deb)

### Code Review Fixes Applied
- **Quit no longer stops daemon** — tray is optional overlay, not lifecycle owner
- **Shared `reqwest::Client` via `OnceLock`** — was creating new client every 10s health check
- **Edition 2024 + version synced to 4.2.0** — was 2021/0.1.0
- **Stuck daemon detection** — `STUCK_THRESHOLD` (90s) prevents eternal "Starting..." state
- **Windows path sanitization** — handles forward-slash variants of home path
- **@2x tray icons** — HiDPI support
- **Removed duplicate tray icon** — `tauri.conf.json` `trayIcon` + programmatic `TrayIconBuilder` created two icons

### Release Workflow Bug
- First release attempt failed: `find` commands looked for bundles at `tauri-app/src-tauri/target/` but workspace members output to root `target/`
- Fixed paths and force-pushed tag to re-trigger

### Key Architecture Decisions
- **No `julie` crate dependency** — subprocess + HTTP keeps tray at ~5-8MB
- **Startup lock** reuses `daemon.startup.lock` to prevent races with `julie connect`
- **JSON (not zip) for server endpoint** — dashboard triggers browser download
- **NSIS (not MSI) for Windows** — better per-user install support
