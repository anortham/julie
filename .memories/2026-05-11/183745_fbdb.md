## Removed Language-Tier Carveouts From Extractor Plans

- **WHAT:** Updated the Julie extractor data-model plan and Eros V1 plan docs so body-span/body-hash and future structured richness are all-language capability contracts, not Python/Rust/TypeScript or priority-language features.
- **WHY:** User clarified the standing product rule: all Julie languages are equally important, and public functionality must not be better for favored languages.
- **HOW:** Julie plan now adds `body_span`, requires fixture-backed capability-matrix coverage with `0 open_gaps` before downstream support, removes priority-language milestone wording, and tells Eros to consume Julie's capability matrix instead of hardcoded language allowlists. Eros plan now blocks `replace_symbol_body` unless Julie's all-language body-span/body-hash contract is globally certified.
- **IMPACT:** Next extractor work should start by defining body-span/body-hash semantics and certification across all applicable languages. Eros should not expose language-gated body replacement.
