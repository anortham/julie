## Implemented all-language body_span/body_hash contract

### WHAT
Added `body_span`/`body_hash` as a certified all-language extractor capability in Julie, persisted it through SQLite/search surfaces, and updated Eros to consume the capability matrix instead of a language allowlist.

### WHY
Eros needs reliable symbol body coordinates for commercial-grade code modification. The user explicitly rejected language-tier support, so the capability needed one global contract across all applicable languages with no per-language functional carveouts.

### HOW
- Defined the contract in Julie docs: normalized body span semantics, deterministic body hash semantics, `supported`/`not_applicable`/`open_gaps`, and no public support while applicable languages have open gaps.
- Implemented shared extractor body-span/hash inference in `crates/julie-extractors/src/base/body.rs` and wired it into symbol creation/manual symbol paths.
- Extended golden fixtures and `fixtures/extraction/capabilities.json` with `kind_coverage.body_spans`; current capability matrix has zero body-span open gaps.
- Added SQLite schema/migration/storage/query support for nullable body span/hash fields and fixed symbol queries to use the shared column list to avoid select-list drift.
- Updated xtask certification report output to include body spans.
- Updated Eros PyO3 bindings, provider capabilities, canonical symbol model, normalizer, SQLite store migration, and `replace_symbol_body` to use body spans without a language allowlist.

### IMPACT
Julie now treats body extraction as a global certified capability, and Eros can gate body-editing on the certified capability matrix instead of granting functionality to selected languages only.
