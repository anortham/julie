## Fixed certification report self-reference gate

### WHAT
- Changed `docs/LANGUAGE_CERTIFICATION_REPORT.md` generation so the checked-in report no longer embeds the current git `HEAD`.
- Added an xtask regression test proving rendered certification markdown is stable when only the git head metadata changes.
- Updated active tree-sitter docs/rubric wording to use checked-in evidence state plus verification ledger provenance instead of an impossible committed self-hash requirement.

### WHY
A generated file that contains `git rev-parse HEAD` cannot be committed in a clean state where it also contains its own final commit SHA. This made `cargo xtask certify tree-sitter --check` pass only in a dirty post-commit tree.

### HOW
- Replaced the report's `Current HEAD` line with a stable scope line.
- Renamed current real-world evidence labels to checked-in evidence labels.
- Left real-world JSON head as generation provenance, but removed it from the certification report's self-hash gate semantics.

### IMPACT
- `cargo xtask certify tree-sitter --check` can remain meaningful after a commit without requiring generated docs to contain their own commit hash.
- Release claims still need ledger rows tied to exact HEAD, so commit provenance moves to the verification ledger where it is actually satisfiable.
