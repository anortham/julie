## Fixed daemon/dashboard lifecycle flakiness

### WHAT
- Added dashboard action gating so workspace actions are rejected while daemon is stopping/draining or restart-pending.
- Preserved tool-call error severity in dashboard project notices instead of rendering failures as neutral/success output.
- Kept HTTP MCP sessions active until async DELETE cleanup finishes, so stale-binary restart/drain logic sees the real active session count.
- Changed stale-binary restart-pending policy to reject new sessions and changed adapter readiness so `draining` is treated as stopping, not ready.
- Added sensitive-root guards to standalone CLI bootstrap and handler workspace initialization paths.
- Hardened dashboard anchor cleanup with canonical containment checks and stopped synthetic dashboard handlers from creating project logs under process cwd.

### WHY
Dashboard clicks could appear inert because lifecycle-blocked actions still tried to use daemon resources and tool errors lost danger semantics. Daemon restart/drain behavior could race because HTTP sessions were removed from active tracking before DELETE cleanup finished, and adapters accepted `draining` as ready.

### HOW
Added explicit dashboard phase helpers, action short-circuiting, async cleanup ordering in `HttpJulieService::drop`, stricter stale-binary accept policy, adapter draining reclassification, canonical path containment checks, and regression tests for each failure mode.

### IMPACT
Verified with exact red/green tests plus broad gates: `cargo check`, `cargo xtask test changed`, `cargo xtask test dev`, and `cargo xtask test system`. The dashboard, daemon session lifecycle, adapter readiness, and bad-path safety surfaces now have targeted regression coverage.
