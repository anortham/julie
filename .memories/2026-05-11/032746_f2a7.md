## Fixed live dogfood health degradation

### WHAT
- Found live daemon health could report DEGRADED/SQLite BUSY on a fresh session even when the workspace was up to date.
- Added regression coverage: `test_startup_noop_repair_does_not_mark_catchup_active_while_planning`.
- Changed startup repair runtime state so no-op freshness planning does not mark catch-up active or watcher-paused.
- Scoped SQLite locking in startup repair planning to DB reads only; filesystem scans now run after releasing the SQLite mutex.

### WHY
The live dogfood handoff requires `manage_workspace health` to report READY. The first daemon-mode health request created a session, immediately started background startup repair planning, and then health raced with that planner. Because the planner held the SQLite mutex while scanning files, health saw SQLite BUSY and reported DEGRADED even though logs later said the index was up to date.

### HOW
- Moved `set_catchup_active(true)` / `set_watcher_paused(true)` until after an actual repair plan exists.
- Captured DB-derived repair inputs while holding the mutex, then released the lock before `get_max_file_mtime_in_workspace` and `scan_workspace_files`.
- Rebuilt release binary, restarted matching 7.8.4 daemon, and reran daemon-mode tool checks.

### IMPACT
- First-session daemon-mode health now reaches FULLY READY after embedding init instead of false SQLite BUSY/catch-up degradation.
- Daemon-mode call-path, fast_refs, SQLite engine-version, and refresh no-op checks all pass via `julie-server tool` HTTP daemon path.
- The Codex MCP connector `mcp__julie__` remains transport-closed in this harness, so direct MCP tool invocation is still not recovered.
