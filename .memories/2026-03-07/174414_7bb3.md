# Julie Platform Vision - Brainstorming Complete

## What
Brainstormed and designed a vision for transforming Julie from a per-session MCP code intelligence server into a **persistent personal developer intelligence platform**.

## Why
Julie as a per-session tool has reached its ceiling. The daemon model unlocks cross-project intelligence, integrated developer memory, a management web UI, and agent dispatch — capabilities that aren't possible with the current per-session architecture.

## Key Decisions
- **Daemon + HTTP from the start** — this is the foundational pivot; without it, Julie stays as-is
- **Federated per-project indexes** (not shared) — 95% of queries are single-project, cross-project via parallel query + RRF merge
- **MCP over Streamable HTTP** — same MCP protocol, daemon-compatible transport, same port as web UI
- **Dual-level memory** — per-project `.memories/` (git-tracked, shareable) + user-level `~/.julie/memories/` (personal)
- **Single embedding model** for all content types (code, memories, docs) — shared embedding space enables cross-content queries
- **Agent dispatch via CLI backends** — `claude -p`, `codex -q`, API keys, Ollama — no credential management needed
- **Cross-platform** — no platform-specific code in core daemon logic
- **Dynamic embedding dimensions** — learned from Miller's LanceDB approach

## Roadmap (5 phases)
1. Daemon + HTTP Foundation (the pivot)
2. Cross-Project Intelligence (federated search, auto-reference discovery)
3. Memory Integration (Goldfish reunion, Rust implementation)
4. Management UI + Agent Dispatch (command center, feedback loop)
5. Search Enhancement (dynamic embeddings, weighted RRF per tool)
