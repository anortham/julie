## Closed cold-start "phantom daemon" race in adapter launcher + daemon startup

Two-part fix for the bug where concurrent adapters spawned duplicate `julie-daemon`
processes during slow cold-start, surfacing as phantom processes in Task Manager /
`ps`. Confirmed via Codex `gpt-5.5 high` second opinion (verdict at
/tmp/codex-daemon-verdict.txt).

### Root cause

- New daemon acquires kernel `daemon.lock` in `acquire_or_yield_to_existing_daemon`
  before binding any listener.
- New daemon does NOT write `daemon.pid` (pinned by `test_daemon_app_does_not_write_legacy_artifacts`).
- `discovery.json` was published LATE in `DaemonApp::serve` (app.rs:386), after
  DB migrations + workspace-ID migration + path normalization + vector-count
  backfill — easily >5s on cold start with many workspaces.
- During that window, `daemon_readiness()` saw no discovery + no PID → returned
  `Dead`. Adapter `spawn_under_lock` then fork+exec'd a duplicate daemon, which
  the kernel singleton immediately refused.

### Fix 1: Lock probe in adapter (`src/adapter/launcher.rs`)

Added `kernel_lock_held()` helper using `DaemonLockGuard::try_acquire`. Called
from the `PidFileStatus::Dead` branch of `daemon_readiness()` before declaring
the daemon dead. AlreadyHeld → `DaemonReadiness::Starting`, else fall through
to `Dead`.

### Fix 2: Early discovery publish in daemon (`src/daemon/mod.rs` + `helpers.rs`)

Added `publish_starting_discovery(paths, host, port)` in `app/helpers.rs`.
Builds a `DiscoveryRecord` with `phase="starting"` and `paths.token_file()` as
the placeholder token path. Called from `run_daemon` between
`bind_mcp_listener_with_fallback` and `DaemonApp::new`. The existing publish
at app.rs:386 atomically overwrites with `phase="running"` once `serve()`
reaches startup_complete.
