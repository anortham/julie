## Codex adversarial review caught 3 real material bugs; reverted lock probe + added phase-starting guard

After my initial Fix 1 (kernel lock probe) + Fix 2 (early discovery publish), Codex
`gpt-5.5 high` adversarial review identified 3 high/medium-confidence material
issues. All verified real. Took Codex's recommended fix path.

### Codex findings (all verified)

1. **Lock probe race** (HIGH, 0.9): `kernel_lock_held()` briefly acquires the
   kernel lock during the check. A real daemon's `acquire_or_yield_to_existing_daemon`
   racing with the probe sees `AlreadyHeld` → returns `Ok(None)` → silent exit.
   My defensive probe could KILL the daemon it's waiting for.

2. **phase=starting can return Ready** (HIGH, 0.88): `daemon_readiness()` only
   short-circuited on `stopping`/`draining`. A Live record with `phase=starting`
   fell through to `endpoint.probe_readiness()`. The kernel TCP stack can
   complete handshakes on a bound-but-not-routing listener — so the probe could
   return Ready before the late `phase=running` flip in `DaemonApp::serve`.
   Adapters could open sessions against a daemon that hadn't finished init.

3. **Kernel-lock Starting falls into broken poll** (MEDIUM, 0.86): My lock probe
   made `daemon_readiness()` return Starting, but `poll_for_state_change` calls
   `is_daemon_running()` which checks discovery + daemon.pid (NOT the kernel
   lock). Returns false during the lock-only window → BrokenPipe → propagates
   as Err. My probe's "Starting" verdict was broken end-to-end.

### Fix applied

- **Removed** `kernel_lock_held()` helper and its call from `daemon_readiness()`
- **Removed** `AcquireError`/`DaemonLockGuard` imports from launcher.rs
- **Added** explicit `phase=="starting"` short-circuit in `daemon_readiness()`
  before the endpoint probe (closes Codex finding #2)
- **Kept** the early discovery publish in `run_daemon` — this is the
  architectural fix that closes the cold-start race window from seconds to
  microseconds
- **Rewrote** tests: deleted the two lock-probe tests (their scenario "kernel
  lock alone is sufficient" is no longer the contract); added
  `test_readiness_starting_when_discovery_phase_is_starting_even_with_reachable_endpoint`
  which sets up phase=starting with a real reachable HTTP server, asserts
  Starting, then flips phase=running and asserts Ready (proves endpoint was
  genuinely reachable)

### Verification

- 30/30 launcher tests pass
- 12/12 daemon::app_test tests pass (including the 10s `test_daemon_app_serve_and_shutdown` lifecycle test)
- `cargo xtask test changed`: transport (3.2s), lifecycle (70.4s), daemon (61.4s) — all green, total 135.0s

### Residual risk

With the lock probe removed, there is a microsecond-wide window between
`acquire_or_yield_to_existing_daemon` (line 309) and `publish_starting_discovery`
(after listener bind) where `daemon_readiness()` returns Dead. If a concurrent
adapter hits this window, it spawns a duplicate daemon → kernel singleton
refuses → adapter respawns. This is the original phantom-daemon mechanism, but
the window is now ~microseconds instead of >5s. In practice it should be
unobservable. Accepted trade-off vs. the broken lock probe.
