## Added lazy activation for known ready workspaces in workspace-scoped tool routing

### WHAT
- Updated `resolve_workspace_filter` to auto-activate known daemon workspaces when they are `ready` but inactive in the current session.
- Preserved strict failure paths for unknown workspace IDs and non-ready statuses (`pending`, etc.).
- Added a swap-safety guard so resolver does not mutate session workspace state during a primary swap window.
- Reworked global-targeting tests to reflect lazy activation semantics and added a pending-status regression test.

### WHY
- Agents running in one project were hitting repeated `known but not active in this session` errors when explicitly targeting another already-indexed workspace.
- This created noisy failures in normal cross-workspace flows (`fast_search`, `get_context`, and any tool using shared workspace resolution).

### HOW
- In daemon mode, resolver now:
  - Accepts active/startup-attached workspaces as before.
  - Rejects non-ready workspaces with a status-aware error that still points to `manage_workspace(open)`.
  - Calls `handler.activate_workspace_with_root(...)` for known ready workspaces to attach/activate on first use.
  - Returns a contextual error if auto-activation fails.
- Tests updated in `src/tests/tools/workspace/global_targeting.rs`:
  - `test_known_ready_workspace_auto_activates_for_fast_search`
  - `test_known_pending_workspace_requires_open_before_fast_search`
  - `test_persisted_pairing_metadata_does_not_preactivate_known_workspace`

### IMPACT
- Cross-workspace tool calls now succeed on first explicit use when the target workspace is indexed and ready.
- Error messaging remains strict and informative for non-ready targets.
- Dev regression tier remains green after the behavior shift.
