## Operational Metrics Design + Implementation Plan Complete

Brainstormed, designed, and planned a full operational metrics feature for Julie.

### What
- **Design spec** committed: `docs/superpowers/specs/2026-03-19-operational-metrics-design.md`
- **Implementation plan** written: `docs/superpowers/plans/2026-03-19-operational-metrics.md`
- 10-task plan covering: types, migration, CRUD, handler instrumentation, query expansion, skill

### Key Decisions
- **Hybrid storage**: in-memory `Arc<SessionMetrics>` with atomic counters for session stats, SQLite `tool_calls` table for history
- **Bytes-based efficiency**: `source_bytes` vs `output_bytes` gives apples-to-apples compression ratios
- **Headline metric**: "X MB of source code was NOT injected into your context" (factual arithmetic, not counterfactual)
- **query_metrics category expansion**: `code_health` (existing), `session` (atomics), `history` (SQLite aggregation)
- **No new tools**: extends existing `query_metrics` with `category` param

### Architecture
- `ToolCallReport` returned from inside `call_tool_with_metrics` (has access to both inputs and outputs)
- `SessionMetrics` uses fixed `[ToolCounters; 8]` array indexed by `ToolKind` enum (no HashMap, no Clone issues)
- Duration stored as microseconds in `AtomicU64` for sub-ms precision
- `session_id` UUID column enables session counting in history queries
- Per-call SQLite write via `tokio::spawn_blocking` (~0.02ms, non-blocking)
