## Batch Pending Relationship Resolution — ~140x speedup

### What
Replaced O(N) per-relationship DB queries with batched resolution that groups by callee_name and uses `WHERE name IN (...)` chunks.

### Why
On Guava (434K pending relationships), the resolver was burning ~320s (82% of total indexing time) doing individual `find_symbols_by_name` queries. On Julie itself (56K pendings), it was ~40s.

### How
- **`find_symbols_by_names_batch`** in `database/symbols/queries.rs` — accepts a slice of names, deduplicates, chunks into groups of 500 (SQLite param limit), uses `IN` clause, returns `HashMap<String, Vec<Symbol>>`
- **`resolve_batch`** in `resolver.rs` — groups pendings by callee_name, single batch query for all unique names, then runs existing `select_best_candidate` per pending against cached results
- **Processor wiring** — replaced 28-line inline loop in `processor.rs` with 2-line `resolve_batch` call + timing log

### Impact
- Julie workspace: 56,528 pendings → 1,487 unique names → **0.28s** (was ~40s)
- Guava estimate: 434K pendings → ~50K unique names → likely **2-5s** (was ~320s)
- Resolution phase no longer dominates indexing time
