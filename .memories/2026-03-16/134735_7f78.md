## Fixed 4 GPT-reviewed bugs in security risk + get_context

### What
Validated and fixed 4 remaining bugs from GPT's code health intelligence review:

1. **Removed `filter` from `DATABASE_SINKS`** — bare `"filter"` matched iterator `.filter()` across all languages (Rust, JS, C#, etc.), causing false-positive security risk flags
2. **Fixed `protected` → `public` mislabel** — `exposure_score()` returns 0.5 for `protected`, and formatting printed "public" for `>= 0.5`. Fix: store raw visibility string in `security_risk.signals.visibility` metadata, use it for display
3. **Surfaced `test_quality` on `get_context` pivots** — added `test_quality_label` field to `PivotEntry`, extracted from metadata, rendered in both compact (`quality=thorough`) and readable (`[thorough quality]`) formats
4. **Boosted exact test-name matches** — added `select_pivots_for_query` with 5x exact-name boost to overcome `TEST_FILE_PENALTY` (0.3) when query matches symbol name exactly

### Why
GPT's static review found real code-level issues in the code health intelligence system. The security signals were producing misleading output (false sinks, wrong visibility labels), and `get_context` was actively hiding test results even when users explicitly searched for them.

### How
- TDD for all fixes (red → green)
- Subagent-driven: Chunk 1 (security) and Chunk 2 (get_context) implemented by separate agents
- All 8 xtask dev buckets green, no regressions

### Impact
- Security risk output is now accurate: no more `filter` false positives, `protected` methods show correct visibility
- `get_context` now surfaces test quality metadata on pivots and correctly prioritizes exact test-name matches
