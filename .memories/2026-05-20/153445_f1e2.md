## Fixed fast_search review findings

### WHAT
- Fixed acronym CamelCase line matching by reusing the tokenizer's acronym-aware `split_camel_case` logic in `src/tools/search/query.rs`.
- Fixed FileLevel density dedupe so query terms differing only by case do not inflate line density in `src/tools/search/line_mode.rs`.
- Added regression coverage for acronym CamelCase, mixed acronym identifiers, case-insensitive density dedupe, and public `SearchIndex::search_files` behavior for Makefile, `.gitignore`, and `.env.local` edge cases.
- Updated the fast-search improvements plan to reflect that `FileSearchResult.match_kind` is exposed.

### WHY
Recent fast_search improvements had two correctness gaps: acronym-style identifiers such as `HTTPResponse` were not treated as compounds consistently, and FileLevel density sorting counted case variants of the same query term as distinct. File-mode edge cases also needed caller-facing retrieval coverage instead of only direct classifier checks.

### HOW
- Replaced the local CamelCase splitter path with `split_camel_case()` plus a fallback for digit-to-uppercase boundaries.
- Kept original term text for matching while deduping density keys via lowercase normalization.
- Added tests in `src/tests/tools/search/line_match_strategy_tests.rs` and `src/tests/tools/search/file_mode_index_tests.rs`.
- Ran `cargo fmt`, focused exact RED/GREEN tests, `cargo nextest run --lib tests::tools::search`, `cargo xtask test changed`, and `cargo xtask test dogfood`.

### IMPACT
All accepted review findings are fixed. Search behavior now handles acronym identifiers without broad subtoken false positives, density ranking no longer overweights duplicated case variants, and file search edge cases are protected through the public API.
