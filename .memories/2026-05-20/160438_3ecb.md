## Dogfooded fast_search and fixed Cargo.lock indexing gap

### WHAT
Dogfooded live `fast_search` after the fast_search quality tweaks and found a real file-mode symptom: `fast_search(search_target="files", query="Cargo.lock")` missed the repo's real `Cargo.lock` and returned relaxed partial matches instead.

Added a regression test and fixed workspace discovery policy so `Cargo.lock` is accepted despite the generic `.lock` extension blacklist.

### WHY
`detect_language_for_indexing` already treats `Cargo.lock` as TOML, but discovery rejected it earlier because `.lock` is blacklisted. That meant the file never reached SQLite/Tantivy and file search could not find it.

### HOW
- Added `test_should_index_file_accepts_cargo_lock` in `src/tests/tools/workspace/discovery.rs`.
- Verified RED against the old behavior.
- Added shared `allows_blacklisted_extension()` in `src/tools/workspace/indexing/file_policy.rs`.
- Used that helper in both `should_index_path_candidate` and `ManageWorkspaceTool::should_index_file` so the discovery path and shared policy agree.
- Preserved existing lockfile exclusions such as `pnpm-lock.yaml`, `package-lock.json`, `composer.lock`, `Pipfile.lock`, and `poetry.lock`.

### IMPACT
Future rebuild/restart/reindex will allow Julie to index Rust `Cargo.lock` files while continuing to exclude high-noise dependency lockfiles from other ecosystems.
