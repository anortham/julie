## T1: Profile and fix NL symbol-query latency (8.6s → 130ms in standalone)

### WHAT
Fixed a 9-second latency spike on NL multi-token `definitions` queries (`"function display template"`) in `--standalone` mode. Added 4 tests proving the invariant. Documented findings in `docs/SEARCH_FLOW.md`.

### WHY
Phase 1 bakeoff recorded 9.5s standalone vs 1ms for lancedb-fts on the same query. Unacceptable p95 regardless of ranking.

### HOW — Root Cause (not what the plan suspected)

**Three suspects from the plan:**
- (a) `expand_query_terms` combinatorial blowup — **RULED OUT**: runs in 340µs, produces 3 original terms, 0 alias terms, 0 normalized terms for "function display template"
- (b) AND/OR double-run in `search_symbols` — **RULED OUT**: AND finds 30 hits in 800µs, OR fallback never fires
- (c) Cold-start cost — **PARTIALLY CORRECT**: the actual culprit is a specific cold-start code path

**Actual root cause:**
`maybe_initialize_embeddings_for_nl_definitions` in `src/tools/search/nl_embeddings.rs` fires for every NL `definitions` query. In standalone mode (no daemon `EmbeddingService`), when `workspace.embedding_runtime_status.is_none()`, it calls `spawn_blocking(workspace.initialize_embedding_provider())` → `create_embedding_provider()` which probes and launches the Python sidecar. Cost: **~8.6s** of the 9s total.

The `bootstrap_standalone_handler` comment already said "skip embeddings to keep startup responsive" — but the NL search path was ignoring that intent.

**Fix:**
Added `JulieServerHandler::mark_standalone_embedding_skipped()` (handler.rs) which sets `workspace.embedding_runtime_status = Some(...)` sentinel immediately after indexing in `bootstrap_standalone_handler`. The guard `if workspace.embedding_runtime_status.is_none()` then short-circuits — sidecar probe never runs. Keyword-only search is the correct degraded mode for single-shot CLI.

### IMPACT

| Scenario | Before | After |
|----------|--------|-------|
| Standalone, warm OS cache, NL query | 9s | 130ms |
| Daemon mode, NL query | 100ms | 100ms (unchanged) |
| Single-token queries | 170ms | 170ms (unchanged) |

### Tests added
`src/tests/tools/search/nl_symbol_query_latency_tests.rs` — 4 tests:
1. `nl_symbol_query_does_not_trigger_combinatorial_expansion` — proves expand_query_terms is O(k)
2. `expansion_cap_applies_to_high_alias_nl_queries` — proves MAX_ADDED_TERMS cap holds
3. `nl_three_token_definition_search_completes_within_latency_bound` — latency bound 1500ms
4. `and_or_fallback_does_not_blow_up_latency` — AND/OR fallback both within 1500ms
