## Centralized Language Metadata And Pending Storage

### WHAT
Task 7 is complete. Added a shared `LanguageSpec` table in `crates/julie-extractors/src/language_spec.rs` and changed `language.rs` to expose parser lookup, extension detection, supported language lists, and doc comment policy from that table. Registry entries now derive ordering and capabilities from the same specs.

Moved pending relationship storage into `BaseExtractor` and changed all relationship-capable language modules to delegate their existing pending accessors/adders through `base`. Elixir remains legacy pending only, but storage is centralized.

### WHY
The old implementation duplicated language metadata across parser lookup, registry capability rows, extension lists, capability fixtures, and doc comment logic. That is exactly how `jsx`, `tsx`, `vue`, and `vbnet` drift. Pending storage was also copied across many extractor structs.

### HOW
- Introduced `LanguageSpec`, `LanguageCapabilities`, and `DocCommentStyle`.
- Updated capability matrix extension/parser validation to compare against specs.
- Fixed JavaScript/JSX accidentally accepting Java `///` doc comments by replacing substring language checks with explicit doc comment policies.
- Added `BaseExtractor::{add_pending_relationship, add_structured_pending_relationship, get_pending_relationships, get_structured_pending_relationships, clear_pending_relationships}`.
- Removed duplicated pending vectors from language modules while preserving wrapper methods and ordering.

### IMPACT
Language metadata is now table-driven for all 36 registry entries, including JSX and TSX variants. Pending relationship behavior is locked by API parity and all-language golden tests while reducing duplicated storage code.
