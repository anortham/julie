## Completed Task 6 Relationship Precision Batch

### WHAT

Added scope-aware local relationship resolution and changed legacy name maps so duplicate names no longer overwrite each other across relationship-capable extractors.

### WHY

The old `HashMap<name, symbol>` pattern silently picked the last symbol with a duplicate name. That creates confident wrong call edges, which is worse than an unresolved or pending edge.

### HOW

- Added `ScopedSymbolIndex` and `LocalTargetResolution` to `base::relationship_resolution`.
- Added resolver tests for self receiver resolution, foreign receiver rejection, duplicate unqualified ambiguity, and duplicate-safe legacy maps.
- Updated Rust, Python, TypeScript, and JavaScript call extraction to resolve self/this receiver calls only with parent-scope proof.
- Changed legacy map construction across relationship extractors to use duplicate-dropping maps instead of overwrite maps.
- Updated Python method-call expectations so receiver-qualified calls without type-flow evidence become structured pending relationships rather than false local edges.
- Restored and pinned the JavaScript exported-class local helper call in golden output after the scoped resolver stabilized.

### IMPACT

Relationship extraction now fails closed on duplicate names by default. Languages with explicit receiver evidence can still create precise local edges, while ambiguous or foreign receiver calls move toward pending relationship data.
