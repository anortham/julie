## Closed semantic index invalidation gap

### WHAT
Added a persisted semantic index engine version for workspace indexes and wired it into startup repair and non-force indexing.

- Added `index_engine_state` via migration 024 and fresh schema setup.
- Added `SEMANTIC_INDEX_ENGINE_VERSION` / `SEMANTIC_INDEX_ENGINE_COMPONENT`.
- Startup repair now reports `semantic_version_changed` when existing derived index data has a missing or stale stamp.
- Non-force indexing now treats semantic drift as an effective full re-index, including relationship rebuilds and Tantivy clear, even when file hashes are unchanged.
- Updated the gap-closure plan and master best-in-class plan with closure notes and verification ledger rows.

### WHY
Live MCP dogfood showed new resolver/extractor code running against old persisted daemon graph data after rebuild/restart. Source mtimes and hashes were unchanged, so incremental indexing skipped the pipeline and `call_path(extract_symbols_static -> extract_canonical)` stayed broken until a forced refresh rebuilt relationships.

### HOW
Used TDD around the exact stale-derived-data failure: seed an old semantic version, delete relationship rows while keeping file hashes unchanged, run non-force indexing, and assert the canonical revision advances, relationships are restored, and the current semantic version is stored.

The implementation uses a deliberate code stamp instead of package version or binary mtime so UI-only releases do not force every workspace to rebuild.

### IMPACT
Binary upgrades that alter extractor, resolver, or indexing semantics now self-heal stale derived graph data. Startup health has an explicit repair reason instead of silently skipping. Verification passed exact regressions, changed tier, system tier, and dogfood tier.
