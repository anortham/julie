## Fixed Startup Stale-File Embedding Catch-Up

- **WHAT:** Updated startup repair so repairs that mutate the symbol corpus (`stale_files`, `new_files`, `deleted_files`, `extractor_failure`, `watcher_overflow`, plus existing full rebuild cases) run the embedding pipeline. Updated the atomic incremental replacement path to delete embeddings for files being replaced before deleting/reinserting symbols. Added a regression test for a stable-ID symbol whose file changes while the watcher is offline.
- **WHY:** Startup catch-up already detected stale files and refreshed SQLite/Tantivy, but embeddings remained stale. Since embeddings are a functional part of the index, startup repair was incomplete.
- **HOW:** The regression uses a batch-marker embedding provider: initial indexing stores marker `1.0`; after modifying the same `alpha` function on the same line, startup repair must produce marker `2.0`. Opening the embedding gate alone failed because stable symbol IDs were considered already embedded. Deleting file embeddings during `incremental_update_atomic` made the workspace embedding pipeline re-fill the changed file vectors.
- **IMPACT:** Offline file changes are now caught on next Julie startup for both text indexes and vector indexes. New builds are marked `7.7.2` in Cargo metadata, docs, and dashboard site footer.
