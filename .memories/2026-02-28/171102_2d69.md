## Addressed Task 6 code-quality blockers for sidecar provider

### WHAT
- Upgraded `src/embeddings/sidecar_provider.rs` with startup readiness probing, bounded response timeout behavior, safer PYTHONPATH merging, and hardened drop cleanup.
- Expanded `src/tests/core/embedding_sidecar_provider.rs` with protocol/error/concurrency/drop/readiness coverage.

### WHY
Review feedback identified real reliability risks:
- constructor could succeed even if sidecar was unhealthy,
- request path could block forever on stdout read,
- module launch clobbered PYTHONPATH behavior,
- poisoned mutex in Drop could leak child process.

### HOW
- Added constructor readiness probe via `health` request and validation:
  - fails if `ready=false` or health dims mismatch 384 contract.
- Replaced direct blocking `read_line` flow with a background stdout reader thread + `mpsc::Receiver` + `recv_timeout`.
  - request path now fails fast on timeout/disconnect/stream errors.
  - timeout is configurable by `JULIE_EMBEDDING_SIDECAR_TIMEOUT_MS` (default 5000ms).
- In module launch mode, PYTHONPATH now merges sidecar default path with existing entries using `split_paths/join_paths`.
- Drop hardened:
  - normal path sends best-effort `shutdown` (short timeout) then terminates child,
  - poisoned lock path uses `poisoned.into_inner()` and still performs child termination.
- Tests expanded:
  - sidecar error envelope maps to `anyhow` error text,
  - request_id mismatch rejected,
  - concurrent multi-threaded calls succeed via serialized mutex access,
  - drop writes deterministic shutdown marker,
  - constructor fails when health probe reports not ready.

### IMPACT
Sidecar provider now has bounded liveness semantics and deterministic startup/shutdown behavior, with tests guarding key transport failure modes.
