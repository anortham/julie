## Task 1 code-quality blockers resolved for sidecar resolver/fallback safety

### WHAT
- Updated `BackendResolverCapabilities::current()` in `src/embeddings/factory.rs` to keep `sidecar_available` disabled at runtime for now (`false`).
- Extended `fallback_backend_after_init_failure()` to handle `Auto + resolved Sidecar` failures with ordered fallback:
  - ORT first when available
  - Candle second when ORT unavailable and Candle available
- Renamed misleading resolver test to `test_resolver_auto_prefers_ort_when_sidecar_unavailable`.
- Added sidecar-specific fallback tests:
  - `test_auto_fallback_target_is_ort_when_sidecar_init_fails_and_ort_is_available`
  - `test_auto_fallback_target_is_candle_when_sidecar_init_fails_and_ort_unavailable`

### WHY
- Prevent `auto` from selecting an unconstructable sidecar backend in current builds.
- Avoid dead-end behavior when sidecar is selected in capability-driven contexts (tests/future wiring) but initialization fails.
- Align test naming with actual precondition (sidecar unavailable).

### HOW
- Followed TDD for fallback behavior:
  - Added new sidecar fallback tests first.
  - Verified red state (`None` returned instead of expected fallback backend).
  - Implemented minimal fallback logic in resolver factory helper.
  - Re-ran requested targeted tests to verify green state.

### IMPACT
- Task 1 behavior remains testable with `sidecar_available` field while production `current()` stays safe.
- Auto fallback path is now resilient for sidecar failure scenarios.
- Test suite wording now accurately communicates resolver conditions.
