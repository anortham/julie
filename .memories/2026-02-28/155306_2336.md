## Brainstorm kickoff for PyTorch embeddings sidecar

Captured product direction for replacing in-process embedding execution path with a Python sidecar while preserving Julie search reliability.

- **What:** Investigated current embedding architecture and runtime wiring in `src/workspace/mod.rs`, `src/embeddings/factory.rs`, `src/embeddings/pipeline.rs`, and `src/search/hybrid.rs`
- **Why:** Need a grounded design for PyTorch-based acceleration without breaking existing semantic search contracts
- **How:** Reviewed existing design docs/commits and gathered constraints via one-by-one clarifying questions

### Confirmed constraints
- Sidecar should be **default**, with **automatic fallback** to existing provider path
- Keep **384-dim** embedding contract for compatibility with existing `sqlite-vec` schema
- Packaging target is **Julie-managed local Python venv** (not user-managed service)
- Primary success metric is **indexing speedup**
- Rollout preference is **immediate default**, not experimental flag first

### Impact
- We can design around low-risk data compatibility while still pursuing aggressive acceleration goals.
- The key architecture decision now is transport/lifecycle model for sidecar and batching strategy tuned for indexing throughput.
