## Embedding Fallback Fix & Semantic Search Observability (v3.5.1 Release)

**Problem:** Embeddings failed to generate in runtime. ORT backend initialization failed, but Candle (available as fallback) was never tried. Logs showed `fallback_used=false`.

**Root Cause:** Asymmetric backend fallback logic in `src/embeddings/factory.rs`:
- `resolve_backend_preference()` returns ORT as primary on all platforms
- `fallback_backend_after_init_failure()` only handled Candle→ORT, never ORT→Candle
- When ORT failed, fallback returned `None` instead of trying Candle
- Classic bug: preference order changed, fallback path never updated

**Solution Implemented:**
1. Added symmetric ORT→Candle fallback case (lines 85-92 in factory.rs)
2. Added `test_fallback_ort_to_candle()` to match existing pattern
3. Added info-level logging to hybrid search pipeline for observability:
   - Entry point logs when hybrid mode activates (text_search.rs:288)
   - Merge stats log keyword vs semantic result counts (hybrid.rs)
   - Failure cases upgraded to `warn!` for production visibility

**Testing:**
- Embedding fallback tests: all 3 pass
- Runtime verification: 19,489 vectors generated via ORT on macOS
- Hybrid search logs confirmed: queries like "gracefully handle failures when backend is unavailable" correctly trigger semantic search, find 10 semantic + 10 keyword results, merge via RRF, and land on relevant symbols (factory.rs fallback function)

**Outcome:** v3.5.1 tagged and pushed to origin/main. Two critical improvements:
1. Embeddings now survive backend failures gracefully
2. Semantic search pipeline has production-grade observability

**Key Learning:** The fallback was originally designed as Candle (primary) → ORT (safety net). When this was flipped in v3.5.0 (ORT → Candle), the recovery path wasn't updated. Symmetric fallbacks are non-obvious — must maintain both directions when preference order changes.
