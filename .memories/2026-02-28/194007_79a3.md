## Fixed atomic save and real deletion bugs in file watcher

Two related bugs in the file watcher's DELETE event handling:

### Bug 1: Atomic saves destroy valid data
Editors (VS Code, vim, IntelliJ) do atomic saves: write-temp → delete-original → rename-temp. The DELETE event fires, and by the time it's processed, the file already exists again. The handler was nuking all symbols and embeddings for a file that was merely edited.

**Fix:** Added `path.exists()` guard at three levels:
- `handle_file_deleted_static()` in `handlers.rs` — early return if file exists
- Both `FileChangeType::Deleted` match arms in `mod.rs` — skip embedding + symbol deletion
- Clear the 1s dedup entry when skipping, so the follow-up Create/Modify event is NOT deduped

### Bug 2: Real deletions silently dropped
`should_index_file()` in `events.rs` calls `path.is_file()` — which returns `false` for deleted files. Real Remove events were never queued, leaving stale symbols/embeddings in DB forever.

**Fix:** Added `should_process_deletion()` that checks extension + ignore patterns without requiring the file to exist on disk.
