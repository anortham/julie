## Embeddings + Semantic Search: Fully Operational

### What Was Done
Fixed asymmetric embedding fallback logic and verified end-to-end semantic search.

**Fallback Fix:**
- Added ORT→Candle reverse fallback in `src/embeddings/factory.rs` (~lines 85-92)
- When `Auto` resolves to ORT and ORT init fails, system now tries Candle as backup
- Added `test_fallback_ort_to_candle()` to match existing `test_fallback_candle_to_ort()`
- Reason: fallback logic was written when Candle was primary (ORT as safety net). When priorities flipped in commit d700f9b to make ORT primary everywhere, the reverse fallback path was never added — classic asymmetry bug

**Observability Improvements:**
- Added `info!` logs to `text_search.rs:292` when hybrid search activates (shows query)
- Added `info!` logs to `hybrid.rs:189` showing keyword vs semantic merge results (how many of each, RRF limit)
- Upgraded semantic search failure to `warn!` for production visibility

### Verification
- ✅ **Embeddings generating:** 19,489 of 24,925 symbols embedded via ORT (39 batches, ~61 seconds)
- ✅ **ORT backend:** Initialized successfully on rebuild. Previous session's failure was transient.
- ✅ **Semantic search working:** Two test queries show keyword + semantic hybrid results:
  - Query: "how to handle when primary option fails and fall back to backup" → 10 keyword + 10 semantic → RRF merge
  - Query: "how do I index code and extract symbols efficiently" → 20 keyword + 20 semantic → RRF merge
- ✅ **Logs confirm:** Both queries show `🔍 Hybrid search` entry and `Hybrid merge` stats with exact result counts

### Architecture Insight
Tantivy keyword search + ORT embeddings → RRF merge produces genuinely better results. The semantic dimension catches intent-based matches that keyword search misses, and RRF balances both signals without requiring threshold tuning.
