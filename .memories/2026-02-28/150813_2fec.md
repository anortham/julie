## Fixed asymmetric embedding backend fallback logic

Embeddings failed to generate on macOS because ORT couldn't initialize the ONNX model. The fallback system only handled Candle → ORT, never the reverse.

### Root Cause
`resolve_backend_preference(Auto)` defaults to ORT. When ORT init fails, `fallback_backend_after_init_failure` returned `None` because it only covers the Candle → ORT case, not ORT → Candle. Candle (which works on this machine with MPS) was never tried.

### Fix
Added symmetric fallback in `src/embeddings/runtime.rs`:
- When `Auto` resolves to ORT and ORT init fails → try Candle as fallback
- Mirrors the existing Candle → ORT fallback but reversed
- Maintains priority (ORT preferred) while guaranteeing a working backend when available

### Verification
- Added `test_fallback_ort_to_candle_when_ort_unavailable` — confirms ORT failure triggers Candle fallback
- All 3 fallback tests pass
- ORT-specific tests still fail (pre-existing issue — ORT can't init on this machine)
- Embedding integration tests all pass
