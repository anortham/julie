## Fixed Three Dogfood Search/Scoring Issues

Implemented all three fixes from the dogfood search quality plan.

### Fix 1: Definition Search — Kind-Aware Promotion + Remove Premature Truncation

- **Removed `.take(limit_usize)`** at two locations in `text_search.rs` (lines 145, 365) that truncated results *before* `promote_exact_name_matches()` ran — negating the 50x over-fetch for file_pattern queries
- **Enhanced `promote_exact_name_matches`** with three-tier stable partition:
  1. Exact name + definition kind (class, struct, interface, trait, enum, function, method, constructor)
  2. Exact name + non-definition kind (import, variable)
  3. Everything else
- Added `DEFINITION_KINDS` constant in `scoring.rs`
- 3 new tests + 7 existing tests pass

### Fix 2: Deduplicated `is_nl_like_query`

- Deleted weaker private `is_nl_like_query` from `expansion.rs` (only checked word count + alphabetic)
- Replaced with import of canonical version from `scoring.rs` (correctly rejects camelCase/snake_case identifiers)

### Fix 3: C# Field/Property Type Relationships

- Added `field_declaration` and `property_declaration` match arms to `visit_relationships`
- Created `member_type_relationships.rs` with shared helpers
- Refactored `extract_type_name_from_node` as shared helper for constructor, field, and property type resolution
- Refactored `find_containing_class` as shared helper
- Symbol map prefers type-defining symbols (Class, Interface) over member symbols for type resolution
- 7 new field/property tests, 8 existing constructor tests still pass

### Discovered: Pre-existing C# Property Name Bug

`extract_property` in `members.rs` uses `.find(|c| c.kind() == "identifier")` which grabs the *type* identifier instead of the *property name* identifier when the type is a plain identifier. Not fixed here (out of scope) but worked around in relationship resolution.
