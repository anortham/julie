## Fixed asymmetric embedding fallback logic

Fixed critical bug where ORT initialization failure was not falling back to Candle backend.

### Root Cause
- `resolve_backend_preference(Auto)` prioritizes ORT over Candle on all platforms
- ORT ONNX model failed to initialize (fastembed loading issue)
- `fallback_backend_after_init_failure()` only handled Candle→ORT fallback, never ORT→Candle
- Result: semantic search/embeddings completely offline; Candle (which works) never tried

### Fix
Added reverse fallback case in `src/embedding/backend/fallback.rs`:
- When ORT fails to init and Candle is available, try Candle
- Maintains original priority (ORT first) while ensuring recovery path exists
- Symmetric fallback: Candle→ORT and ORT→Candle both work now

### Testing
- Added `test_fallback_ort_to_candle()` to verify reverse fallback
- All 3 fallback tests pass (new + existing)
- 7 ORT runtime tests fail (pre-existing—ORT can't init on this machine anyway)
- Change is isolated to selection logic, zero behavioral impact on search

### Impact
Embeddings should now initialize successfully on machines where ORT fails but Candle works (like this Mac with MPS acceleration available).
