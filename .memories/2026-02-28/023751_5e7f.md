## Embeddings Fallback Logic: Fixed Asymmetric Backend Recovery

Runtime showed embeddings initialization failed (ORT backend couldn't initialize ONNX model) with `fallback_used=false`, meaning Candle was never attempted as a fallback despite being available.

### Root Cause
Traced `src/embeddings/factory.rs` and found asymmetric fallback logic:
- `fallback_backend_after_init_failure()` only handled Candle→ORT
- Never handled ORT→Candle

Since `resolve_backend_preference()` now returns ORT as primary on all platforms (flipped in commit d700f9b), when ORT init failed, fallback returned `None` instead of trying Candle.

### Fix Applied
Modified `src/embeddings/factory.rs` (~lines 85-92) to add symmetric reverse case: when `Auto` resolves to ORT and ORT init fails, try Candle. Added `test_fallback_ort_to_candle()` to match existing `test_fallback_candle_to_ort()`.

### Verification
Ran embedding test suite: all 3 fallback tests pass (newly added test green). 8 ORT-specific test failures are pre-existing environment issues (ORT can't initialize on this machine).

### Impact
On next restart, Auto backend selection will now properly fall back from ORT→Candle when ORT fails. Embeddings should generate via Candle (CPU-based, functional but slower than ORT would be).
