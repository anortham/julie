## Embeddings Semantic Search Diagnostic Session

Verified embeddings generate successfully via ORT fallback fix (19,488 vectors, BGE-small-en-v1.5, CPU). Added diagnostic logging to hybrid search pipeline to determine if semantic search is actually being used in `fast_search` queries.

### Investigation
- **Embeddings status:** ✅ Working. ORT backend initialized successfully after rebuild, 19,488 vectors embedded via CPU in ~61s
- **Vector storage:** ✅ Working. Vectors stored in `sqlite-vec` virtual table (verified via logs)
- **Deep_dive semantic matching:** ✅ Working. Tool output shows "Semantically Similar (5)" with cosine scores — vector queries work at the application level
- **Fast_search semantic mode:** ❓ Unclear. Test queries returned "Relaxed search" (keyword-only fallback) with no evidence of hybrid search activating

### Decisions
Added three `info!`-level logs to diagnose hybrid search behavior:
1. `text_search.rs` entry point: log when hybrid mode activates
2. `hybrid.rs` merge stage: log keyword vs semantic result counts
3. `hybrid.rs` semantic failure: upgraded to `warn!` to surface any errors

### Problem Encountered
Changes written to disk but not staged. User rebuilt without re-staging, so binary compiled old code without new logging. Logging won't appear in the next run unless changes are committed first.

### What's Uncertain
- Is `fast_search` with `search_target="definitions"` actually hitting the hybrid path, or is NL query detection rejecting all test queries?
- Is the query detection too strict (filtering out natural language queries that contain code-like tokens)?
- Is the embedding provider actually being passed to `text_search_impl` at runtime?

### Next Steps
1. Stage + commit the logging changes in `text_search.rs` and `hybrid.rs`
2. Rebuild Julie

4. Check logs to see if hybrid search is firing and what the merge statistics show
