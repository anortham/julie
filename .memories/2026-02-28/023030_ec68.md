## Embeddings Fallback Logic Fix — Symmetric ORT↔Candle Recovery

**What was done:** Fixed asymmetric backend fallback logic in embeddings provider initialization that prevented Candle from being tried when ORT failed.

**Root cause:** The fallback function `fallback_backend_after_init_failure()` only handled Candle→ORT, never ORT→Candle. When `resolve_backend_preference()` returns ORT as primary on all platforms, an ORT init failure would return `None` instead of falling back to Candle.

**The bug:** Designed during a phase when Candle was primary (ORT as safety net). When priorities flipped in commit d700f9b to make ORT primary everywhere, the fallback path was never updated — classic asymmetry where recovery logic didn't match the new preference order.

**Fix applied:**
- Modified `src/embeddings/factory.rs` (~lines 85-92)
- Added reverse fallback case: if `Auto` resolves to ORT and ORT init fails → try Candle
- Added `test_fallback_ort_to_candle()` test matching existing `test_fallback_candle_to_ort()`
- All 14 hybrid search unit tests pass; all 3 fallback tests pass

**Result:** Embeddings successfully generated on rebuild: 19,488 of 24,925 symbols embedded via ORT CPU in ~61 seconds (BGE-small-en-v1.5, 384d). ORT recovered on its own this time, but fallback is now insurance for environments where ORT genuinely can't initialize.

**Hybrid search status:** All unit tests pass (RRF merge, KNN conversion, orchestrator, NL query detection). Integration dogfood test is running but taking extended time (~110s expected).

**Open:** Root cause of ORT init failure in previous session (transient vs. environment-specific) not yet diagnosed."
