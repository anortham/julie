## Embeddings Fallback Fix + Observability Logging

Fixed asymmetric fallback logic in embedding backend selection and added info-level observability to hybrid search path.

### What Was Wrong
- `fallback_backend_after_init_failure()` only handled Candle→ORT, never ORT→Candle
- When `resolve_backend_preference()` returned ORT as primary, ORT failures went unrecovered
- Result: previous session's ORT init failure meant no embeddings generated, fallback never attempted

### Fixes Applied

**1. Symmetric Fallback (src/embeddings/factory.rs)**
- Added ORT→Candle fallback when Auto resolves to ORT and ORT init fails
- Added `test_fallback_ort_to_candle()` test
- All 3 fallback tests pass; 8 pre-existing ORT test failures unrelated

**2. Observability Logging (src/tools/search/hybrid.rs, text_search.rs)**
- Added info-level log when hybrid search is triggered (shows query + semantically_similar_threshold)
- Added info-level log after vector search (shows how many semantic results returned)
- Upgraded semantic failure to warn-level for visibility
- Reason: debug logs are filtered at INFO level in production; can't verify hybrid search firing without this

### Current Status
- Rebuild #1: ORT initialized successfully this time, 19,488 symbols embedded (39 batches × 500)
- Embeddings verified working: `deep_dive` shows "Semantically Similar" results with cosine scores
- Initial semantic search tests showed mixed results — unclear if hybrid path triggering or just keyword matches
- Added logging to confirm hypothesis

### Next Step
User rebuilding + restarting. Will run semantic queries again and check logs for new `[hybrid]` entries. This will definitively confirm whether hybrid search is taking the path or if there's a gating issue.

### Open Questions
- Why did ORT fail in previous session but succeed after rebuild? Transient vs environment issue?
- Is hybrid search actually being invoked despite successful embedding generation?
