## Task 1 implemented: sidecar backend resolver support with TDD

### WHAT
- Added two tests in `src/tests/core/embedding_provider.rs`:
  - `test_parse_provider_preference_accepts_sidecar`
  - `test_resolver_auto_prefers_sidecar_when_available`
- Implemented minimal resolver changes in:
  - `src/embeddings/mod.rs` (`EmbeddingBackend::Sidecar`, `as_str()` mapping)
  - `src/embeddings/factory.rs` (parse support for `sidecar`, `BackendResolverCapabilities.sidecar_available`, and auto-resolution preference)
- Updated existing resolver capability fixtures in tests to include `sidecar_available: false` where needed.

### WHY
- Task 1 required adding sidecar as a first-class backend option and making `auto` prefer sidecar when capability says it is available.

### HOW
- Followed TDD:
  1. Added failing tests first.
  2. Verified failure from targeted command (`E0599` missing `Sidecar`, `E0560` missing `sidecar_available`).
  3. Implemented smallest changes to enum + resolver + parser.
  4. Re-ran targeted command and then explicit test-name runs for both new tests.
- Kept fallback logic unchanged (`fallback_backend_after_init_failure`) to preserve existing semantics.
- Added a defensive `EmbeddingBackend::Sidecar` branch in provider factory that currently returns a clear unavailable error until sidecar provider implementation lands in later tasks.

### IMPACT
- Resolver now supports `sidecar` preference and `auto` can prioritize sidecar when declared available.
- Existing ORT/Candle fallback and strict-acceleration behavior remain intact for current capabilities.
- Task 1 implementation is unblocked for subsequent sidecar feature/provider tasks.
