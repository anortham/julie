## Fixed asymmetric embedding backend fallback bug

Identified and fixed a bug where ORT embedding backend failures were never falling back to Candle, leaving embeddings permanently unavailable.

### Root Cause
The fallback logic only handled one direction:
- Candle fails → try ORT ✓
- ORT fails → try Candle ✗ (was missing)

When `auto` backend resolved to ORT first and ORT initialization failed (ONNX model load), the system hit a dead end instead of trying Candle.

### What Changed
- Updated `fallback_backend_after_init_failure()` in `src/embeddings/provider.rs` to add reverse fallback case
- Added test `test_fallback_ort_to_candle_when_ort_fails` to cover the previously-missing path
- Renamed existing test to `test_fallback_candle_to_ort_when_candle_fails` for clarity

### Impact
- Embeddings will now generate via Candle after ORT fails (CPU path, slower but works)
- Fixes the permanent unavailable state that was occurring on this machine
- All fallback tests pass; ORT runtime test failures are pre-existing (ONNX model load issues on Mac)

Restart server for fix to take effect. Embeddings should then backfill from SQLite fixture.
