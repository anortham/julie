## Fixed asymmetric embedding backend fallback

Discovered and fixed a one-directional fallback bug in `src/embeddings/runtime.rs` that prevented embeddings from generating on macOS.

### Root Cause
- `resolve_backend_preference()` returns ORT as primary on all platforms
- `fallback_backend_after_init_failure()` only handled Candle → ORT fallback
- When ORT failed to initialize ("Failed to initialize fastembed ONNX model"), Candle was never tried
- Result: degraded_reason=unavailable, embeddings never generated

### The Fix
Added reverse fallback case in `fallback_backend_after_init_failure()`:
```rust
// Auto resolves to ORT, ORT fails → try Candle
if requested_backend == EmbeddingBackend::Auto
    && resolved_backend == EmbeddingBackend::Ort
    && capabilities.candle_available
{
    return Some(EmbeddingBackend::Candle);
}
```

Added test `test_fallback_auto_ort_to_candle` to prevent regression.

### Impact
- Embeddings will now fall back to Candle when ORT fails (fixes the "degraded" status on current machine)
- Candle CPU fallback is slower than ORT but provides vector search functionality
- All fallback tests pass (3/3); ORT test failures are pre-existing (ORT can't init on this machine)
- Requires rebuild + Julie restart to take effect

### Why This Happened
Backend preference priority was flipped (ORT → Candle) but fallback path wasn't updated to match the new direction. Classic asymmetry from refactoring without full coverage.
