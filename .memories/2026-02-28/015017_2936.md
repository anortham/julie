## Fixed asymmetric fallback logic preventing Candle embedding fallback

Embeddings were failing silently: ORT failed to init fastembed ONNX model, but `fallback_backend_after_init_failure` never tried Candle as a fallback.

### Root cause
The fallback logic was one-directional:
- When `Auto` → Candle → fails: falls back to ORT ✅
- When `Auto` → ORT → fails: returns `None`, never tries Candle ❌

This happened because the logic was designed when Candle was primary + ORT was the safety net. When priorities flipped (ORT preferred globally in `resolve_backend_preference`), the fallback path was never updated.

### Fix
Added reverse fallback case in `src/embeddings/embedding_provider.rs`:
```rust
// After ORT init fails, try Candle as fallback
if requested_backend == EmbeddingBackend::Auto
    && resolved_backend == EmbeddingBackend::Ort
    && capabilities.candle_available
{
    return Some(EmbeddingBackend::Candle);
}
```

Also updated test naming (`test_ort_falls_back_to_candle_on_init_failure` → clearer intent).

### Verification
- 3 fallback tests pass (both directions)
- 8 ORT-specific test failures are pre-existing (ORT can't init on this machine—the original bug)
- No regressions

### Impact
Next rebuild + restart: `Auto` → ORT fails → Candle fallback → embeddings generate via Candle CPU (will unblock the incremental pipeline + performance work).
