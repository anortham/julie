## Fixed Embeddings Backend Fallback Logic

**Problem:** Runtime logs showed ORT embedding backend initialization failed, but Candle was never attempted as fallback despite being available. Flag `fallback_used=false` confirmed fallback logic wasn't working.

**Root Cause:** Asymmetric fallback design in `src/embeddings/factory.rs`. The `fallback_backend_after_init_failure()` function only handled Candle→ORT failure recovery, never ORT→Candle. When `resolve_backend_preference()` returns ORT as primary (for all platforms), and ORT init fails, the function returned `None` instead of trying Candle.

**Historical Context:** Fallback was written when Candle was primary and ORT was safety net. Later, commit d700f9b flipped priorities to ORT-first everywhere, but fallback path was never updated to match. Classic asymmetry bug.

**Solution:** Added reverse fallback case at `factory.rs` ~lines 85-92:
- If Auto resolved to ORT and ORT fails, try Candle
- Added test `test_fallback_ort_to_candle()` to mirror existing `test_fallback_candle_to_ort()`

**Verification:** All 3 fallback tests pass (including new reverse case). 8 pre-existing ORT test failures are environment issues (ORT can't init on this Mac), not regressions from this change.

**Impact:** Next restart, embeddings will now: Auto→ORT→fails→Candle (CPU-based, functional). Unblocks semantic search on this machine.
