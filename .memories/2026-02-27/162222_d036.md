## Completed Task 2 (deterministic backend resolver with auto policy)

### WHAT
- Added deterministic resolver logic for embedding backend selection and integrated it into factory + workspace init path.
- Added resolver capability model and tests for platform/capability matrix behavior.
- Fixed resolver fallback behavior so `auto` on Apple Silicon falls back to ORT when Candle is unavailable.
- Added workspace-level regression tests for explicit-candle-unavailable and auto-fallback status behavior.

### WHY
- Needed to enforce the approved platform policy:
  - Apple Silicon prefers Candle by default
  - explicit provider override remains strict
  - auto mode should degrade safely instead of disabling embeddings when Candle is unavailable.

### HOW
- `src/embeddings/factory.rs`:
  - added `BackendResolverCapabilities`
  - added `resolve_backend_preference(...)`
  - wired `EmbeddingProviderFactory::create` through resolver
  - ensured explicit unavailable backend errors are clear
- `src/workspace/mod.rs`:
  - status path now uses resolver outcome to set truthful requested/resolved runtime status
- `src/tests/core/embedding_provider.rs`:
  - resolver matrix tests (auto on Apple Silicon, auto elsewhere, explicit override, unavailable backend cases)
  - workspace-level tests for explicit candle unavailable and auto fallback behavior

### IMPACT
- Backend selection is now deterministic, test-covered, and aligned with the design.
- Prevents accidental semantic-search disablement on Apple Silicon when Candle isn't available.
- Establishes a robust base for Task 3 (Windows DirectML policy) and Task 4 (strict acceleration mode).
