## Task 3 runtime signaling fixes completed (truthful ORT + health visibility)

### WHAT
Implemented a focused fix for remaining Task 3 quality issues in the DirectML branch:
- Replaced ORT runtime inference-by-device-label with explicit runtime signaling.
- Prevented overclaiming DirectML activation when only provider policy path is known.
- Added embedding runtime status reporting to `manage_workspace health` output.
- Added/updated tests for ORT runtime signaling and workspace health rendering.

### WHY
Previous behavior could claim DirectML activation based on initialization policy path and detect degraded state via fuzzy device-string matching, which is brittle and potentially misleading for operators.

### HOW
- Added provider-level diagnostics hooks on `EmbeddingProvider`:
  - `accelerated() -> Option<bool>`
  - `degraded_reason() -> Option<String>`
- In `OrtEmbeddingProvider`, introduced explicit `OrtRuntimeSignal` + `ort_runtime_signal(target_os, directml_fallback_to_cpu)` to encode truthful runtime outcomes:
  - Windows policy success without provider-confirmed EP => `device=Unknown`, `accelerated=false`, `degraded=None`.
  - Windows DirectML init failure + CPU fallback => `device=CPU`, `accelerated=false`, degraded reason set.
- Updated `JulieWorkspace::initialize_embedding_provider` to consume provider-reported status directly (no fuzzy matching helper).
- Extended `manage_workspace health` to include an **Embedding Runtime** section with backend/device/accelerated/degraded fields.

### IMPACT
- Runtime status is now conservative and truthful instead of optimistic inference.
- Degraded detection no longer depends on fuzzy string contains checks against device labels.
- Operators can directly see fallback/degraded embedding state via health output.
- New targeted tests validate these behaviors.
