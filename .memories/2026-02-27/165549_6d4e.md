## Fixed Task 4 strict-acceleration and provider parsing quality gaps

### What
- Updated strict acceleration gating to disable embeddings whenever runtime reports `accelerated=false`, even if backend resolved and `degraded_reason` is `None`.
- Updated provider preference parsing to trim surrounding whitespace before matching provider name.
- Added/updated focused tests in `src/tests/core/embedding_provider.rs` for both behaviors.

### Why
- Strict mode must enforce acceleration truthfully based on runtime status, not only unresolved/degraded metadata.
- Provider strings from env/config may include incidental whitespace and should still parse correctly.

### How
- Extended `should_disable_for_strict_acceleration(...)` in `src/embeddings/factory.rs` to accept `accelerated: bool` and include `!accelerated` in disable conditions.
- Passed runtime `accelerated` flag at call site in `src/workspace/mod.rs`.
- Changed `parse_provider_preference(...)` to use `provider.trim().to_ascii_lowercase()`.
- Added test coverage:
  - `test_parse_provider_preference_accepts_known_values` now validates whitespace/case input (`"  ORT\t"`).
  - New `test_should_disable_for_strict_acceleration_when_not_accelerated` validates strict-mode disable on `accelerated=false` with no degraded reason.
  - Updated existing strict-helper tests for the new helper signature.

### Impact
- Strict mode now behaves correctly for runtime non-accelerated states that previously slipped through.
- Provider selection is more robust to real-world env/config formatting.
- Changes are minimal and aligned with existing strict-mode helper design.
