## Task 7 implemented: runtime status surfaced in `manage_workspace stats`

### WHAT
- Added embedding runtime status rendering to stats output path in `src/tools/workspace/commands/registry/refresh_stats.rs`.
- Added a dedicated formatter (`format_runtime_status_for_stats`) that surfaces:
  - `Backend`
  - `Device`
  - `Accelerated`
  - `Degraded`
- Added workspace tool tests in new file `src/tests/tools/workspace/runtime_status_stats.rs` for stats output behavior.
- Registered the new test module in `src/tests/mod.rs`.

### WHY
- Health output already surfaced embedding runtime details, but stats output did not.
- Requirement was to ensure both `manage_workspace health` and `manage_workspace stats` surface backend/device/accelerated/degraded information when runtime metadata exists, while remaining truthful when metadata/provider is missing.

### HOW
- Implemented stats runtime formatting based on workspace type + in-memory runtime/provider state:
  - Primary + runtime metadata + provider: show resolved backend/device/acceleration/degraded reason.
  - Primary + runtime metadata but missing provider: keep backend/acceleration and report `Device: unavailable` + provider-aware degraded text.
  - Primary without runtime metadata: report unresolved/uninitialized runtime truthfully.
  - Non-primary stats requests: report runtime metadata as unavailable/unknown (not tracked for unloaded reference workspaces).
- TDD cycle followed:
  1. Added stats runtime tests first.
  2. Ran targeted tests and captured failures (missing embedding runtime section in stats output).
  3. Implemented formatter + stats output integration.
  4. Re-ran targeted stats + health tests to green.

### IMPACT
- `manage_workspace stats` now exposes embedding runtime health information parallel to `manage_workspace health` when available.
- Missing metadata/provider states are explicit and truthful instead of silently implying initialization.
- Regression coverage now exists specifically for stats runtime output behavior.
