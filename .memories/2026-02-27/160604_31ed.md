## Applied Task 1 embedding runtime-quality fixes

Updated embedding backend diagnostics to be truthful and consistent with runtime behavior.

- **WHAT**
  - Added `EmbeddingBackend::Unresolved` and used it when provider preference parsing fails.
  - Switched acceleration status from hardcoded `true` to `DeviceInfo`-derived heuristic via `DeviceInfo::is_accelerated()`.
  - Reworked `EmbeddingProviderFactory::create()` unreachable invalid-backend handling to an explicit `unreachable!` guard after parse-time validation.
  - Added focused tests in `src/tests/core/embedding_provider.rs` for unresolved backend status, acceleration heuristic (CPU vs GPU hints), and invalid-provider workspace runtime status.

- **WHY**
  - Prevents CPU fallback from being incorrectly reported as accelerated.
  - Prevents misleading `resolved_backend=Auto` diagnostics for invalid provider input.
  - Aligns factory control flow with the parse contract to avoid dead-code semantic drift.

- **HOW**
  - Introduced small enum/method updates in `src/embeddings/mod.rs`.
  - Updated `initialize_embedding_provider()` status population in `src/workspace/mod.rs`.
  - Simplified invalid branch in `src/embeddings/factory.rs` to a justified unreachable branch.
  - Followed TDD by adding failing tests first, then implementing minimal fixes.

- **IMPACT**
  - Runtime status now more accurately reflects real backend/device state.
  - Diagnostics are safer for MPS/DirectML rollout analysis and CPU fallback detection.
  - Verified behavior with targeted tests and fast-tier suite.
