## Planned Candle + DirectML rollout design and implementation plan

### WHAT
- Recalled prior embedding hardening work and confirmed existing backend seam (`EmbeddingProviderFactory`, feature flags, workspace init wiring).
- Added approved design document:
  - `docs/plans/2026-02-27-candle-mps-and-directml-design.md`
- Added detailed TDD implementation plan:
  - `docs/plans/2026-02-27-candle-mps-and-directml-implementation.md`

### WHY
- User wants Apple Silicon MPS acceleration via Candle, plus a concrete fix for Windows currently running CPU instead of DirectML.
- The project needed a written, approved design + execution-ready task breakdown before implementation.

### HOW
- Used brainstorming workflow: clarified fallback policy, compared approaches, selected unified resolver approach.
- Verified current technical constraints in code and dependencies:
  - ORT provider currently CPU-only behavior in `src/embeddings/ort_provider.rs`
  - Existing feature seam in `Cargo.toml` and `src/embeddings/factory.rs`
  - Fixed 384-dim vector schema in sqlite-vec migration
- Captured architecture decisions, fallback semantics, diagnostics contract, and a TDD-first task plan with file-level scope and commands.

### IMPACT
- Team now has a concrete, implementation-ready plan that includes:
  - Auto Candle selection on Apple Silicon
  - DirectML-first ORT policy on Windows with degraded fallback visibility
  - Strict acceleration mode behavior
  - Diagnostics updates in workspace stats/health
  - Targeted verification strategy and commit boundaries
- Unblocks immediate execution in a follow-up implementation session without re-discovery work.
