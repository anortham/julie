## Fixed Task 5 embedding quality issues in Candle/ORT runtime path

- **WHAT**
  - Feature-gated ORT-only imports/tests in `src/tests/core/embedding_provider.rs` so candle-only compile paths no longer pull ORT symbols from that module.
  - Added safe non-strict auto fallback logic: when auto resolves to Candle but Candle init fails, runtime now attempts ORT fallback when ORT is available.
  - Improved Candle scaffold diagnostics by changing dim guard input to `Option<usize>` and emitting explicit scaffold/unavailable error when effective dim is unknown (instead of fake `got 0`).
  - Added tests for fallback policy, strict-mode no-fallback, workspace runtime fallback (macOS+aarch64+candle+ort), and new scaffold dim-guard semantics.

- **WHY**
  - Candle scaffold failures in auto mode were disabling embeddings even when ORT could recover.
  - Existing scaffold diagnostics were misleading and looked like a real mismatch.
  - ORT references inside core embedding tests broke candle-only test build paths.

- **HOW**
  - Added `fallback_backend_after_init_failure(...)` in `src/embeddings/factory.rs` and exported it via `src/embeddings/mod.rs`.
  - Updated `JulieWorkspace::initialize_embedding_provider` to reuse resolver capabilities and execute fallback attempt before declaring embeddings unavailable.
  - Updated Candle dimension guard in `src/embeddings/candle_provider.rs` to distinguish unavailable effective dimension from real mismatch.
  - Added/updated tests in `src/tests/core/embedding_provider.rs` with feature gates around ORT-only tests and new assertions for fallback + scaffold messaging.

- **IMPACT**
  - Auto mode now degrades gracefully to ORT for Candle scaffold/init failures in non-strict mode.
  - Strict acceleration behavior remains explicit and unchanged (no silent fallback).
  - Candle scaffold errors are now truthful and actionable.
  - The requested candle-only feature matrix check passes for library compile (`cargo check --lib --no-default-features --features embeddings-candle`).
