## Implemented Task 4: strict acceleration mode via `JULIE_EMBEDDING_STRICT_ACCEL`

### WHAT
- Added strict-acceleration env parsing helper: `strict_acceleration_enabled_from_env_value` (`1/true/on` => enabled).
- Added policy helper: `should_disable_for_strict_acceleration(strict, resolved_backend, degraded_reason)`.
- Updated `JulieWorkspace::initialize_embedding_provider` to:
  - Read `JULIE_EMBEDDING_STRICT_ACCEL`.
  - Keep current best-effort fallback behavior in non-strict mode.
  - Disable provider (`embedding_provider = None`) in strict mode when initialization is degraded/unresolved.
  - Attach a clear runtime status reason prefixed with strict-mode context including `JULIE_EMBEDDING_STRICT_ACCEL`.

### WHY
- Task 4 requires opt-in strict behavior so users can avoid silently running embeddings without acceleration when preferred acceleration is unavailable/degraded.

### HOW
- Followed TDD: wrote failing tests first (missing strict parser symbol / strict reason expectation), then implemented minimal changes to pass.
- Extended `src/tests/core/embedding_provider.rs` with strict parsing tests, strict policy tests, and strict-vs-non-strict workspace init behavior tests.

### IMPACT
- Runtime status remains coherent with existing design (`requested_backend`, `resolved_backend`, `accelerated`, `degraded_reason`) and now explicitly communicates strict-mode disable decisions.
