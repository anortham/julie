## Added structured embedding runtime status logs for startup visibility

### WHAT
- Implemented a structured logging snapshot for embedding initialization outcomes in `JulieWorkspace::initialize_embedding_provider`.
- Added reusable log-field builder in `src/workspace/mod.rs`:
  - `EmbeddingRuntimeLogFields`
  - `build_embedding_runtime_log_fields(...)`
- Emitted a consistent `info!` log event for all initialization paths (success, strict-disable, fallback, unresolved/error), including:
  - requested backend
  - resolved backend
  - runtime
  - device
  - accelerated
  - degraded reason
  - strict mode
  - fallback used
- Added tests in `src/tests/core/embedding_provider.rs` validating field construction for provider-present and provider-missing cases.

### WHY
- User asked to ensure logs make it obvious what is working vs failing during backend selection/runtime initialization.
- Existing logs had warnings/errors and partial success messages, but not one canonical structured status line for every outcome.

### HOW
- Introduced a closure inside `initialize_embedding_provider` to emit the structured status from persisted runtime metadata + provider device info.
- Invoked that logger immediately after each branch sets `embedding_runtime_status`.
- TDD cycle used:
  - Added failing tests referencing missing helper
  - Implemented helper + logging calls
  - Re-ran focused and fast-tier tests.

### IMPACT
- Operators can now diagnose backend resolution and fallback behavior directly from startup logs without inferring state from multiple messages.
- Log shape is consistent across success/degradation/failure paths.
