## Re-review of Task 3 fixes (DirectML/degraded/health)

Validated the latest changes in the feature worktree against the three previously raised review items.

- **Checked code paths:** `src/embeddings/ort_provider.rs`, `src/workspace/mod.rs`, `src/tools/workspace/commands/registry/health.rs`
- **Ran targeted tests:**
  - `cargo test runtime_signal`
  - `cargo test test_manage_workspace_health_surfaces_embedding_runtime_status`
- **Observed:** runtime signal tests and health-surface test pass.

### Finding retained
- `check_embedding_runtime_health()` treats any `Some(embedding_runtime_status)` as `Embedding Status: INITIALIZED`, including cases where provider initialization failed and `embedding_provider` is `None`.
- This can misreport failed/unavailable embedding runtime as initialized.

### Why this matters
Health output is meant to be operator-trustworthy; labeling failed init as initialized is a correctness issue in diagnostics.

### Suggested next step
Derive initialization state from both `embedding_runtime_status` and `embedding_provider` presence (or add explicit `initialized` boolean to runtime status).
