## Implemented Task 3: ORT Windows EP policy DirectML -> CPU with fallback signaling

### WHAT
- Added explicit ORT execution-provider policy behavior in `OrtEmbeddingProvider` for Windows:
  - primary attempt requests `DirectML` first, then `CPU`
  - if the DirectML attempt fails, provider retries with CPU-only policy (non-fatal)
- Added helper APIs in ORT path:
  - `ort_execution_provider_policy_kinds(target_os)`
  - `ort_runtime_degraded_reason(target_os, device)`
- Updated runtime status construction in workspace init to set degraded reason when ORT on Windows is running without DirectML.
- Added tests in `src/tests/core/embedding_provider.rs` for:
  - Windows EP ordering assumptions (`DirectML`, then `CPU`)
  - non-Windows default-safe behavior
  - degraded-reason assumptions for Windows CPU fallback

### WHY
- Windows needed deterministic ORT EP policy instead of implicit defaults.
- CPU fallback must stay non-fatal so semantic search keeps working.
- Runtime status should surface that acceleration degraded when DirectML is not active.

### HOW
- Introduced explicit EP policy logic in `src/embeddings/ort_provider.rs` and wired it into `TextEmbedding::try_new(...)` init options.
- Kept non-Windows behavior CPU/default-safe by not forcing provider policy there.
- Used device/runtime info at workspace init to derive degraded status for the ORT Windows path.

### IMPACT
- Windows startup path now actively requests DirectML before CPU.
- When DirectML is unavailable, embeddings continue with CPU instead of disabling provider.
- Runtime status can now truthfully report degraded acceleration expectations on Windows fallback.
