## Completed Task 1 (backend preference + runtime status)

### WHAT
- Implemented foundational runtime selection/status plumbing in worktree branch `feature/candle-mps-directml`.
- Changed embedding config default provider from `ort` to `auto`.
- Added provider preference parser with explicit validation (`auto|ort|candle`).
- Added runtime status model for embedding initialization.
- Added workspace field to persist embedding runtime status and wired initialization/clone paths.
- Added tests for default config, parser behavior, runtime status, and invalid-provider diagnostics.

### WHY
- This is the prerequisite for auto-selecting Candle on Apple Silicon and for truthful degraded-state reporting on Windows DirectML fallback.

### HOW
- `src/embeddings/mod.rs`: added `EmbeddingBackend`, `EmbeddingRuntimeStatus`, plus `DeviceInfo::is_accelerated()` heuristic.
- `src/embeddings/factory.rs`: added `parse_provider_preference()` and switched default config + factory create path to parsed backend.
- `src/workspace/mod.rs`: added `embedding_runtime_status` field and set status in `initialize_embedding_provider()` success/failure paths, including `resolved_backend = Unresolved` for invalid config.
- `src/tests/core/embedding_provider.rs`: added focused tests for defaults, parser validation, runtime status semantics, and acceleration heuristic.
- `src/tests/tools/search_quality/helpers.rs`: added required struct field initialization to keep tests compiling after workspace struct expansion.

### IMPACT
- Runtime diagnostics now have enough structure to distinguish requested vs resolved backend and whether acceleration is actually active.
- Enables next tasks (deterministic backend resolver and Windows EP ordering) without refactoring churn.
