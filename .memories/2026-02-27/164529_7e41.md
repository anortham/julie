## Completed Task 3 (Windows ORT DirectML policy + truthful degraded signaling)

### WHAT
- Implemented explicit Windows ORT execution-provider policy behavior and fallback semantics.
- Added/updated ORT runtime diagnostics to avoid overclaiming acceleration.
- Added health output surfacing for embedding runtime status and fixed status classification logic.
- Added targeted tests for ORT policy behavior and workspace health reporting.

### WHY
- User reported Windows was silently using CPU instead of DirectML.
- Needed explicit DirectML-first policy with clear degraded/fallback visibility.

### HOW
- `src/embeddings/ort_provider.rs`:
  - encoded explicit Windows policy path (DirectML-first, CPU fallback path)
  - added truthful runtime diagnostics for policy vs fallback outcomes
- `src/embeddings/mod.rs` + `src/workspace/mod.rs`:
  - wired explicit degraded reason metadata instead of brittle string inference
- `src/tools/workspace/commands/registry/health.rs`:
  - added embedding runtime section (backend/device/accelerated/degraded)
  - fixed status logic to avoid reporting INITIALIZED when provider is missing
- `src/tests/core/embedding_provider.rs` + `src/tests/tools/workspace/mod_tests.rs`:
  - added policy/order and degraded-status tests
  - added health-report regression tests for DEGRADED/UNAVAILABLE states

### IMPACT
- Windows fallback behavior is now observable and diagnosable.
- Health reporting is materially more trustworthy for embedding acceleration status.
- Sets up strict-mode and Candle-provider tasks on top of solid runtime status plumbing.
