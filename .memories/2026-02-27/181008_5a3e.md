## Task execution reached blocker: Candle path remains scaffold-only

### CONTEXT
- Executed Task 1-8 from the implementation plan in subagent-driven flow.
- Added resolver, strict mode, Windows DirectML policy, diagnostics surfacing, feature-matrix gating, and verification suite.

### EVIDENCE
- Final holistic review flagged that `CandleEmbeddingProvider::try_new` still returns an explicit init error (scaffold behavior) and does not provide real embedding inference yet.
- `JULIE_EMBEDDING_PROVIDER=candle` currently cannot initialize a functioning provider.
- Auto mode can fall back to ORT (non-strict), but this does not deliver MPS acceleration.

### WHAT HAPPENED
- The current Candle integration step was implemented as architecture + safety guardrails first (feature-gated module, selection path, dim guard, diagnostics), but actual model runtime was not completed.
- Main technical constraint: existing sqlite-vec schema and search assumptions require 384-dim vectors, while readily available fastembed candle paths are not known to provide compatible 384 output in this integration.

### IMPACT
- Windows DirectML policy and runtime diagnostics are in place and verified.
- Apple Silicon Candle default policy is wired, but real Candle inference is not yet functional; acceleration goal is only partially met.

### NEXT
- Need product/engineering decision on Candle implementation path:
  1) Keep current scaffold + ORT fallback (safe, no real MPS gains yet)
  2) Implement full 384-dim Candle model integration (larger effort)
  3) Expand dimension/storage strategy to support non-384 Candle models (schema/search changes)
