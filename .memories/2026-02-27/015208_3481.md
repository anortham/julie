## Dogfood Testing: Semantic Embedding Features

Tested all three semantic integration points live against Julie's own codebase (19,019 embedded symbols).

### Findings

#### 1. `deep_dive` Semantic Similarity — Works Great
- Consistently surfaces relevant types/helpers (e.g., `TokenizerConfig` for `CodeTokenizer` at 0.48)
- Score range 0.37–0.48, no obvious false positives
- **Verdict: Ship-quality**

#### 2. `get_context` Hybrid Search — Good for Technical, Weak for Vague NL
- "KNN nearest neighbor vector embedding search" → perfect (`knn_search` as pivot)
- "find symbols that are similar to each other using vectors" → missed (`find_symbols_by_name` as noise pivot)
- "how does the server start up and initialize" → 2.5/3 relevant pivots
- **Verdict: Technical queries good, vague NL needs work**

#### 3. `fast_search` Semantic Fallback — Effectively Dead
- Fallback condition: `is_nl_like_query(query) && keyword_result_count < 3`
- Problem: Tantivy keyword search always returns 3+ results for NL queries (matches words in docs/fixtures)
- "how are search results ranked and scored" → only docs
- "how to trace execution flow through async functions" → garbage from metadata.json
- **Verdict: Threshold `< 3` is too strict; fallback almost never triggers**

#### 4. Julie Bug: `fast_refs` Missed Fully-Qualified Call
- `should_use_semantic_fallback` called at `text_search.rs:309` via `crate::search::hybrid::should_use_semantic_fallback`
- `fast_refs` only found test import, missed the real call site
- Likely because fully-qualified path doesn't match bare identifier pattern
