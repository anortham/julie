## Dogfooding Code Health Intelligence Against C# Reference Workspace

### What
Added `~/source/LabHandbookV2` (C#/.NET + TypeScript/Vue full-stack app, ~45K LOC, ~713 tests) as a reference workspace and systematically dogfooded `deep_dive`, `get_context`, `fast_search`, and `fast_refs` to evaluate the quality of test intelligence, change risk, and security risk metadata.

### Why
Code Health Intelligence was developed and tested against Julie's own Rust codebase. Needed to verify cross-language correctness for the 30 other supported languages, especially class-based, DI-heavy, ORM-heavy ecosystems like C#.

### Key Findings

**Rust metadata: excellent.** Test quality tiers, test locations, change risk, security risk — all accurate and useful. `compute_reference_scores` showed 6 tests with quality tiers, correct security risk for DB operations.

**C# metadata: six concrete issues identified:**

1. **Class shows "untested" even when methods are well-tested** — `compute_test_coverage` links tests to individual methods but never aggregates up to the parent class. `LabTestService` class shows "untested" while its methods have 15-28 covering tests each.

2. **Cross-service false positive test linking** — Name-match fallback produces ties when multiple services share method names (`ListAsync` exists in 6 definitions). All production files are in `src/LabHandbook.Api/Services/`, giving identical directory proximity scores. Tests from `MediaServiceTests` get attributed to `LabTestService.ListAsync`.

3. **`[SetUp]`/`[TearDown]` not recognized as test context** — NUnit lifecycle methods aren't in `test_attrs`, so `from_symbol_id` from SetUp (where SUT instantiation happens) doesn't have `is_test = true`. Relationships from SetUp don't count toward coverage.

4. **Class-level centrality always 0** — C# DI patterns inject via constructor parameters, so all references flow to the constructor (26 refs for `LabHandbookDbContext` constructor) while the class itself gets 0 incoming refs. Misleading change risk at class level.

5. **EF Core LINQ patterns not detected as security sinks** — Sink patterns (`execute`, `eval`, `spawn`) miss ORM patterns like `SaveChangesAsync()`, `ExecuteDeleteAsync()`, `_db.LabTests.Include(...)`. Entire ORM-based database access invisible to security risk.

6. **`RequestDelegate` false positive on input_handling** — Middleware constructors flagged for "accepts string params" when `RequestDelegate` and `ILogger<T>` are DI framework types, not user input.

### How
- Added reference workspace via `manage_workspace(add)`
- Ran 20+ deep_dive/get_context/fast_search/fast_refs queries across both workspaces
- Traced test coverage failure to database level with direct SQLite queries
- Found Step 2b query DOES produce matches but disambiguation picks wrong candidates
- Confirmed identifiers correctly linked to test methods with `is_test = 1`

### Root Cause Analysis
The issues are **framework-aware, not language-aware** problems. Julie's extractors handle C# syntax correctly. The analysis layer assumes a uniform "function has is_test → it's a test" model that doesn't account for:
- DI constructor injection patterns (centrality flows to constructors, not classes)
- Test lifecycle methods (SetUp/TearDown/BeforeEach aren't test functions)
- ORM abstractions (LINQ replaces raw SQL)
- Common interface method names across services (ListAsync, DeleteAsync)

### Categorization of Fixes
**Systemic (fix once, helps all languages):**
- Coverage aggregation from methods to parent class
- Name-match disambiguation improvement
- Lifecycle method recognition
- ORM sink pattern expansion
- Input handling false positive tightening
- Constructor-to-class centrality propagation

**Then spot-check:** Java (Spring/JUnit), Python (pytest), TypeScript (Jest) — each has its own variant of these patterns.
