## Security Risk Signals — Complete

### What
Implemented structural security risk analysis for Julie — the second spec of Code Health Intelligence:
- **`compute_security_risk()`** in `src/analysis/security_risk.rs` — scores symbols based on 5 signals: exposure (visibility × kind), input handling (signature parameter type detection), sink calls (one-hop batch detection of exec/query/eval patterns), blast radius (P95-normalized centrality), and untested (binary from test_coverage)
- **`deep_dive` integration** — full security risk breakdown section with signal details
- **`get_context` integration** — `security=HIGH` labels on pivot lines in both compact and readable formats
- **DB query** — `get_call_identifiers_grouped()` for batch sink detection

### How
- Case-insensitive final-segment matching for sink patterns (e.g., `db.Execute` → matches `execute`)
- Batch pre-load of identifiers + relationship callees into HashMaps (avoids O(N) per-symbol queries)
- Scoring gate: only symbols with exposure >= 0.5, input handling, or sink calls get scored (no metadata noise on irrelevant symbols)
- 31 tests (27 unit + 4 integration) — all pass
- Separate from change_risk (different question, different signals, coexists in metadata)

### Impact
Agents now see both change risk AND security risk in tool output. A public function that calls `db.execute` with string params shows up as security=HIGH, giving agents immediate awareness of security-sensitive code without requiring security expertise or extra tool calls.
