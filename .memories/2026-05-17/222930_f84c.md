## Task 1: mark_restart_pending always signals restart channel (daemon-fragility T1)

### WHAT
Implemented Task 1 of `docs/plans/2026-05-17-daemon-restart-listener-fix.md`. `DaemonLifecycleController::mark_restart_pending` now calls `self.notify_restart()` on the first transition (gated by the existing `first_request` flag). Two redundant explicit `notify_restart()` call sites in `mcp_session.rs` removed.

### WHY
Production bug: daemon enters `restart_pending=true` then rejects every subsequent session init forever. Root cause was the `restart_notify` channel had no `.notified()` consumer in src/ — calls fired into a void. T1 ensures every `mark_restart_pending` transition signals; T2 (parallel implementer) wires the listener bridge in `DaemonApp::serve` that bridges restart_notify -> stop_notify; T3 is the integration test.

### HOW
- Added `if first_request { self.notify_restart(); }` inside `mark_restart_pending`
- Added doc comment to `restart_pending` field explaining the one-way commit semantics
- Deleted explicit `admission.lifecycle.notify_restart()` at `mcp_session.rs:411` (disconnect arm) and `:452` (ShutdownForRestart arm) — both call `mark_restart_pending` immediately prior so the notify is now redundant
- Wrote two RED tests in `src/tests/daemon/lifecycle.rs`:
  - `test_mark_restart_pending_signals_listener_on_first_transition` — register `.notified()` waiter, call mark_restart_pending, assert wake within 100ms
  - `test_mark_restart_pending_does_not_double_signal` — first call consumed by waiter, fresh waiter then second call, assert fresh waiter does NOT wake within 50ms (proves first_request gate)

### IMPACT
- RED → GREEN confirmed: both tests fail on prior code, pass after the fix
- 5 existing lifecycle controller tests still pass unchanged
- Field is correctly type-checked; no production behavior change for callers that already mark+notify together
- Unblocks T3 integration test once T2 (listener bridge) lands
