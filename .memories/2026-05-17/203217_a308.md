## P1.2 complete: capability_flags removed from Tantivy schema

**WHAT:** Dropped the `capability_flags` field from the C.3-enriched Tantivy schema, `SchemaFields`, `SymbolSearchResult`, and every site that constructed the struct. The on-disk schema signature now hashes 18 fields instead of 19.

**WHY:** Codex review B2: julie indexed an empty string into the field for every symbol, the reranker did not read it, and no dashboard/MCP consumer touched it. One of 19 schema fields earning nothing. Q3 (any external consumer?) answered "no" via fast_search across src/ — the only references were the schema declaration, the in-process SymbolSearchResult construction sites, and test fixtures.

**HOW:**
- `src/search/schema.rs`: removed the `CAPABILITY_FLAGS` constant, the `add_text_field` call, the `Field` member, and the `SchemaFields::new` init.
- `src/search/index.rs`: removed the `SymbolSearchResult.capability_flags` field, the `add_symbol` write site that wrote empty, the two read sites in `search_symbols` / `search_symbols_relaxed`.
- `src/search/hybrid.rs`, `src/tools/search/text_search.rs`, `src/tools/get_context/task_signals.rs`: removed three more construction sites (semantic-search → SymbolSearchResult conversion, DB rescue, task-signals stub).
- 12 test files: removed `capability_flags: String::new()` initializers via `edit_file` with `occurrence=all`.
- `src/tests/tools/search/c3_enriched_schema_tests.rs`: removed the assertion that the schema must define the field.
- `docs/OPERATIONS.md`: updated the schema-evolution operator note.

**IMPACT:**
- Schema compatibility signature changes automatically; on first daemon start after this lands, every workspace's Tantivy directory will auto-rebuild from canonical SQLite (the same recovery path documented in OPERATIONS.md). No user action.
- 19 → 18 schema fields. Inserts marginally faster, storage marginally smaller. Real win is one fewer dead field to maintain across SymbolSearchResult construction sites.
