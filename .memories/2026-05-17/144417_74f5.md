## Fix concurrent-adapter daemon spawn cascade

### What
Hold `daemon-startup.lock` across `cmd.spawn()` + a short wait for the new daemon to publish liveness (`discovery.json` or `daemon.pid`), instead of releasing immediately after the spawn syscall. Default timeout: 5s; on timeout, `warn!` then release.

### Why
At 2026-05-17T09:24:33.190 production daemon.log showed 6 daemons starting in the same millisecond (and 12 earlier at 07:24:01). Root cause: the launcher released `daemon-startup.lock` between `cmd.spawn()` returning and the child writing any liveness file. The next waiting adapter would re-check `daemon_readiness()` under the lock, still see `Dead`, and spawn another daemon. The daemon-side singleton lock (`daemon.lock`, kernel-held) killed the losers silently, but each loser burned a fork+exec.

### How
- Extracted `spawn_under_startup_lock_with<F>(spawn_fn, liveness_timeout)` from `spawn_under_lock`.
- Added `wait_for_spawned_liveness` that polls `daemon_readiness()` until non-`Dead` or timeout.
- The stale comment claiming the lock couldn't wrap the wait was wrong — that constraint referred to an older version that used `daemon.lock`; the current code uses `daemon-startup.lock` which only the launcher contends on.

### Codex Review Findings (all addressed)
1. **Test signal**: original test simulated only the legacy PID-file path; new daemons publish via `discovery.json`. Added second test variant via `write_live_discovery`.
2. **Stale comments**: rewrote `ensure_daemon_ready` and `spawn_under_lock` doc comments to describe the new locking strategy.
3. **Timeout logging**: upgraded `debug!` → `warn!` on liveness timeout (it means cascade protection failed).

### Impact
- Eliminates the visible thundering-herd spawn cascade on every daemon restart with concurrent MCP sessions (3+ in this user's setup).
- Single fork+exec per restart instead of N=6-12.
- No behavioral change to single-adapter restarts.

### Verification
- RED→GREEN cycle: temporarily disabled wait → test failed with count=6 (matches production). Restored wait → count=1.
- Both PID and discovery.json variants pass.
- `cargo xtask test changed` (transport bucket): 4/4 PASS in 3.1s.
