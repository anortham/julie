## Test-intent override across search pipeline (Bug 1 + Bug 1b)

Eros benchmark vs julie showed julie scoring 0.21 MRR on test-intent lookups vs eros 1.0 â€” the worst category by far. Investigation:

### Root cause
- `apply_nl_path_prior` in `src/search/scoring.rs` unconditionally penalized test paths (0.85x) and boosted source (1.08x). For queries like "test synthetic extraction eval writes report artifact" â€” clearly asking for a test â€” this pushed the matching test below its production counterpart.
- `sort_file_hits` in `src/tools/search/execution.rs` had the same bug via `file_path_priority_bucket`: test paths in bucket 1, source in bucket 0, so source ranked first.

### Fix
1. New `has_test_intent(query)` in scoring.rs â€” detects bare tokens (`test`, `tests`, `spec`, `specs`, `fixture`, `fixtures`, `conftest`) and identifier shapes (`test_*`, `*_test`, `*_spec`).
2. `apply_nl_path_prior` skips both the test-path penalty AND the source boost when test intent is detected â€” natural BM25 takes over and the test function name (which matches the query verbatim) wins.
3. `sort_file_hits` now takes a `test_intent: bool` arg; when true, swaps testâ†”source buckets so tests rank first. Docs/fixtures still demoted.
