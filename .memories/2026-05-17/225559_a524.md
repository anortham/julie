## Daemon restart_pending zombie state fixed — listener wired

Julie's MCP daemon entered `restart_pending` indefinitely after binary rebuild, requiring SIGKILL to recover. 22-minute outages in production. Blocked v7.10.0.

### Root cause (NOT what we initially guessed)

**The `restart_notify: Arc<Notify>` channel at `src/daemon/lifecycle.rs:152` had no `.notified()` consumer anywhere in src/.** `notify_restart()` calls fired into a void. The daemon's only exit triggers were SIGTERM/SIGINT and the Windows named-event waker.

Initial hypothesis (H1: counter leak) was wrong. Counter accounting is correct. The bug was the missing listener — built and exposed but never consumed.

Additionally, `RejectForRestart` and `AcceptWithRestartPending` admission arms called `mark_restart_pending` but not `notify_restart`. Even with a listener wired, the dominant failure mode (active sessions never disconnecting) would still hang. Codex caught this in adversarial review.

### Fix (5 commits + plan doc)

- `219ce80e` T2: spawn_restart_bridge helper in `src/daemon/app/helpers.rs` bridges `restart_notify → stop_notify` with `tracing::info!` between
- `6a0ead78` T1: `mark_restart_pending` now calls `notify_restart()` on first transition (gated by `first_request`); deleted redundant explicit `notify_restart()` calls at `mcp_session.rs:411` and `:452`
- `2fee56aa`, `c1e0abf5`: cleanup commits from cartographer review
- `781884b7` T3: integration test for bounded recovery + `DaemonConfig::current_binary_mtime` testing seam
- `240ea0d0`: plan doc + verification ledger

Bounded recovery: worst-case user-visible outage drops from "indefinite, requires SIGKILL" to ~95-100s (31s adapter retry + 60s drain + ~5s respawn).

### Key insight for testing

"The seam is bug-orthogonal" — `DaemonConfig::current_binary_mtime` injection changes WHERE mtime comes from but not WHETHER the bug fix code path is exercised. So cherry-picking the seam onto pre-fix commits + the integration test isolates the actual fix as the variable under test. RED proven this way.
