## P1.1 complete: NL path prior single-application contract

**WHAT:** Removed `apply_nl_path_prior` from `SearchIndex::search_symbols` (index.rs:606) and `search_symbols_relaxed` (index.rs:701). The assembly layer (`text_search::definition_search_with_index`) is now the sole owner.

**WHY:** Codex review (B1) flagged double-application: the same candidate was multiplied by the prior twice on the keyword path. Although the reranker would clobber the first application in the default-on case, the second-class apply was wasted compute and would resurface as a real double-application bug whenever the reranker is disabled (which Phase 3 ablation may end up doing).

**HOW:**
- Added 3 RED tests in `src/tests/tools/search/nl_path_prior_pipeline_tests.rs`: two contract tests (search_symbols / search_symbols_relaxed return raw BM25, no prior baked in) and one pipeline test (assembly-layer application produces exactly the single-application multiplier ratio).
- Removed both internal calls and the now-unused `apply_nl_path_prior` import.
- Updated two existing tests that asserted the now-inverted contract:
  - Deleted `test_search_symbols_relaxed_applies_nl_path_prior` (redundant with new contract tests).
  - Converted `test_nl_path_prior_can_change_top1_with_small_limit` and `test_nl_query_prefers_code_over_docs` to run through `definition_search_with_index_for_test`. Renamed `router.rs` → `routing.rs` in fixtures so the reranker's per-term path boost is symmetric across docs and src candidates, leaving the path prior as the only differentiator.

**IMPACT:** Phase 1 first bug closed. All 11 search buckets pass; full dev tier (32 buckets) passes.
