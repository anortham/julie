## #50 Dart 3 class-modifier signature drop — FIXED (2/2 green)

Pre-existing bug: Dart 3 class modifiers (sealed/base/final/interface/mixin) dropped from class signatures. Uncommitted, lead batches.

### Root cause + fix
`dart/signatures.rs::extract_class_signature` only handled `abstract`, via a fragile `get_node_text(node).contains("abstract")` (matched the word anywhere in the body). Verified vs tree-sitter-dart 0.2.0 grammar: modifiers are anonymous token children inlined before the `class` keyword (hidden `_class_modifiers`/`_mixin_class_modifiers` rules: sealed/abstract/base/interface/final/inline/mixin).
- Added `extract_class_modifier_prefix(node)` — collects leading modifier tokens in source order, stops at `class`. Replaces the abstract-only logic (also fixes the latent body-contains-"abstract" bug).
- Removed now-unused `is_abstract_class` from `dart/helpers.rs` (only caller was signatures.rs).

### Test-assertion change (flagged to lead)
The riverpod test asserted `final class AsyncData extends AsyncValue` produces a PENDING relationship. Mis-specified: AsyncValue is same-file, so `emit_type_relationship_or_pending` resolves it locally to a RESOLVED relationship (pending empty) — the convention the existing passing `test_infer_types_and_extract_relationships` relies on. Changing emit to pending-always would break that test. Aligned the assertion to check the resolved Extends edge (from AsyncData → to AsyncValue), preserving intent (final modifier doesn't break inheritance). Offered to revert if lead prefers an emit-logic change.
