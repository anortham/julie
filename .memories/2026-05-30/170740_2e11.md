## Workspace-root boundary consistency: stop the discovery walk at all VCS roots, not just `.git`

Hardened both `find_workspace_root` resolvers so workspace-root detection recognizes every VCS repository root as a project boundary — closing the follow-up to issue #33 (a stray ancestor `.julie/` being captured because there was no boundary to stop the upward walk).

### What
- New `pub const VCS_ROOT_MARKERS: &[&str] = &[".git", ".hg", ".svn", ".jj", ".bzr", "_darcs"]` in `src/paths.rs` (authoritative list).
- `JulieWorkspace::find_workspace_root` (discovery walk, `src/workspace/mod.rs`): the STOP-boundary check, previously `.git`-only, now iterates `VCS_ROOT_MARKERS`. Made `pub(crate)` for direct unit testing (matches its sibling). Per-level precedence unchanged: `.julie` capture → VCS boundary → climb.
- `ManageWorkspaceTool::find_workspace_root` (explicit-path resolver, `src/tools/workspace/paths.rs`): ancestor marker set now `VCS_ROOT_MARKERS.chain(build_ide_markers)` — gains `.hg`/`.svn`/`.jj`/`.bzr`/`_darcs` for cross-VCS breadth.
- `BLACKLISTED_DIRECTORIES` (`src/tools/shared.rs`): added `.jj`/`_darcs` (it already had `.git`/`.svn`/`.hg`/`.bzr`) for indexing-exclusion parity.

### Why the literal "copy paths.rs marker set" framing was REJECTED
A design-verification workflow proved that adding nesting build manifests (`Cargo.toml`/`package.json`) as boundaries to the discovery walk would BREAK monorepos: starting from a Cargo-workspace member crate (its own `Cargo.toml`), the walk would halt at the member and never reach the real workspace-root `.julie/`, splitting the index. *This very repo* is a Cargo workspace (`members = [".", "crates/julie-extractors", "xtask"]`), so the regression was self-demonstrating. VCS-root markers do NOT nest → safe boundaries.

### How verified (TDD)
- 4 new tests in `root_detection.rs`: genuine RED `.hg`-boundary-stops-walk + Function-#1 `.hg`-root-recognition (both confirmed failing pre-fix); regression guards for member-crate resolution and home-`.julie` rejection.
- Adversarial 4-reviewer workflow → `ship_with_nits`, zero blockers. Acted on both `should_fix` items: added a `VCS_ROOT_MARKERS ⊆ BLACKLISTED_DIRECTORIES` parity test + extended the blacklist test with the VCS dirs; corrected a docstring that falsely claimed universal non-nesting (qualified the pre-1.7-SVN caveat). Reworded a misleading "VCS-first wins" comment and restored the `.git`-file-or-directory worktree note.

### Impact
- `cargo xtask test dev` → 35 buckets green (553.8s). `system` tier → 7 buckets green.
- Honest residual: a project with NO VCS at all + a stray ancestor `.julie/` still has no boundary; the `~/.julie` home guard still covers the worst case.
