## Fixed daemon hang + version flap from 05/29 incident (daemon.log.2026-05-29)

Diagnosed an all-night Claude Code hang to two distinct root causes and fixed both, TDD.

### Root cause (from log analysis)
A v7.12.1 adapter and a v7.12.2 daemon shared one daemon. The version gate restarted the daemon on ANY mismatch, but the restarting adapter respawns its OWN binary, so the daemon flapped 7.12.2→7.12.1→7.12.2 with no convergence. During a flap the client's in-flight request was severed and never answered → CC hung with no timeout.

### #2 — Adapter surfaces transport death as a JSON-RPC error
- **Files:** `src/adapter/forwarder.rs`, `src/adapter/http_stdio.rs`
- **Invariant:** every client request id gets exactly one response (real or synthesized error) before the adapter gives up.
- Three orphan paths now answer the client: daemon EOF after output (forwarder errors in-flight ids), send failure after output (in-flight + failed line), retries exhausted (run loop errors queued `pending_lines` + final unsent line).
- Made retry backoff injectable (`run_http_adapter_inner_with_backoff`) so the exhaustion test runs instantly.

### #1 — Version gate converges (no flap)
- **Files:** `src/daemon/lifecycle.rs`, `src/daemon/mcp_session.rs`
- `version_gate_action` compares semver: adapter **newer** → restart (upgrade); adapter **older** → new `RejectStaleAdapter` (reject, daemon does NOT restart); equal/missing → accept.
- `RejectStaleAdapter` returns `invalid_request` (-32600, no "restart" word) so the stale client fails fast instead of retry-looping.
- Repurposed 2 existing integration tests that encoded the old flap behavior; fixed short-circuit + dashboard tests.

### Environment
- Installed cargo-nextest 0.9.137 (prebuilt) → xtask runner now usable.
- Killed two zombie daemons (PIDs 23276, 42108) left from the incident night.
