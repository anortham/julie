### Reference workspace storage bug (dogfooding find)
While adding reference workspaces for token measurements, discovered that `workspace_db_path`, `workspace_tantivy_path`, and `workspace_index_path` in daemon mode wrote all reference workspace data to the primary workspace's database.

**Root cause:** When `index_root_override` is set (daemon mode), these methods ignored the `workspace_id` parameter entirely. The override pointed to the primary workspace's specific index dir (e.g., `~/.julie/indexes/julie_528d4264/`), and the override branch returned that path regardless of which workspace_id was requested.

**Fix:** Introduced `shared_indexes_dir()` that resolves the parent (`~/.julie/indexes/`) and always appends workspace_id. Added regression test. Verified fix live: Zod reference workspace now gets its own isolated directory.
