## Fixed CRLF line ending bug in Goldfish checkpoint/plan parsing

### Root Cause
`parseCheckpointFile` and `parsePlanFile` use regex with `\n` to split YAML frontmatter. On Windows with `core.autocrlf=true`, git converts line endings to `\r\n` on checkout, breaking the regex. The `catch` block in `getCheckpointsForDay` silently swallows the parse error, making checkpoints vanish without any error message.

### Fix
- Added `content.replace(/\r\n/g, '\n')` normalization at the top of both `parseCheckpointFile` (`checkpoints.ts:87`) and `parsePlanFile` (`plans.ts:40`)
- Added 3 test cases: CRLF checkpoint, mixed LF/CRLF checkpoint, CRLF plan

### Evidence
- **Before fix**: `recall()` found 0 checkpoints in julie workspace (all `.md` files had CRLF from git checkout)
- **Goldfish workspace**: Found 2/12 checkpoints — only the ones created fresh today (LF), missed the 10 committed+checked-out ones (CRLF)
- **After fix**: 234 tests pass (231 → 234), including new CRLF tests

### Also Noted
- Julie workspace old checkpoints use Unix timestamps (`timestamp: 1771089853`) instead of ISO 8601 — separate compatibility issue
