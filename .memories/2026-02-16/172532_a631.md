## Fix: Default recall uses last-N mode instead of date window

### Problem
Default `recall()` used a 2-day rolling time window, which meant:
- Projects with intermittent activity returned 0-1 checkpoints
- Checkpoints near the 48-hour boundary flickered in/out depending on time of day
- 281 old JSON checkpoints + 89 newer MD checkpoints were invisible at default settings

### Root Cause
`getDateRange()` defaulted to `days: 2` when no date params were specified. Combined with `limit: 10`, most of the project history was silently hidden.

### Fix (in `c:\source\goldfish`)
- **New `getAllCheckpoints()`** in `checkpoints.ts` — scans all date directories (newest-first), returns all checkpoints sorted by timestamp
- **`recallFromWorkspace()`** now checks `hasDateParams()` — if no date params (`since`, `days`, `from`, `to`), uses `getAllCheckpoints` instead of date-range query
- **Default limit changed from 10 to 5** — "last 5 checkpoints" is the new default behavior
- **Updated**: `tools.ts` (schema descriptions), `types.ts` (comments), `skills/recall/SKILL.md`

### Key Design Decision
"Last N checkpoints" is inherently correct regardless of activity cadence. Whether you checkpoint daily or once a month, you always get your recent context.

### Tests
- 3 new tests: finds old checkpoints without date params, excludes them with `days` param, newest-first ordering
- Updated default limit test (10 → 5)
- **106/106** recall + checkpoint tests pass
- Pre-existing flaky registry tests unrelated (concurrent file write races)
