## Fixed Windows Daemon Connection (3 Bugs)

Julie v6.0.1's Windows IPC was completely broken. The adapter could never connect to the daemon.

### Root Cause: `is_process_alive()` was a TODO stub on Windows

`pid.rs:is_process_alive()` returned `false` unconditionally on Windows. This caused a cascade:
1. Adapter thinks daemon is dead, deletes PID file, spawns duplicate daemon
2. New daemon can't bind the named pipe (old daemon still has it)
3. `wait_for_socket` probe connects to OLD daemon's pipe, consuming the only server instance
4. Real adapter connection gets `ERROR_PIPE_BUSY` (os error 231)

### Three Fixes

1. **`pid.rs`**: Implemented `is_process_alive` using `OpenProcess(PROCESS_QUERY_LIMITED_INFORMATION)` Win32 API, with `ERROR_ACCESS_DENIED` handling (same pattern as Unix `kill(pid, 0)` / `EPERM`)

2. **`ipc.rs`**: `IpcConnector::connect` now retries up to 10 times with exponential backoff (50ms-500ms) on `ERROR_PIPE_BUSY`, which is a transient condition when the daemon is creating a new pipe instance after accepting a connection

3. **`launcher.rs`**: `wait_for_socket` treats `ERROR_PIPE_BUSY` as success (pipe exists = daemon is up). Timeout increased from 10s to 45s to cover cold-start ORT model loading on DirectML (~26s)

### Verified
- All 14 PID tests pass (including `test_is_process_alive_for_current_process` which was certainly failing before)
- Debug binary adapter test: full connection flow works (workspace header read, SQLite migrations, Tantivy init, file watcher started)
- Release binary needs rebuild (binary lock prevents building while Claude Code has Julie configured)
