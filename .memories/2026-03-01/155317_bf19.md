## Cross-File Inheritance Fix (5 Languages) + fast_refs Import Classification

### WHAT
Added `PendingRelationship` fallback to 5 OOP language extractors (Java, TypeScript, JavaScript, Kotlin, Swift) when base types aren't found in local symbols during inheritance extraction. Also added import/definition separation in `fast_refs` output formatting.

### WHY
These 5 languages had the same bug as C# (fixed 2026-02-28): their `extract_inheritance_relationships` only resolved against same-file symbols, silently dropping all cross-file extends/implements. This meant `get_context` pivot selection couldn't find inheritance chains across files.

### HOW
- **Java**: Added `else` branches to both `extends` and `implements` code paths (already had `&mut JavaExtractor`)
- **TypeScript**: Changed signature to `&mut`, added `implements_clause` support, refactored to two-phase pattern with `collect_heritage_data` helper
- **JavaScript**: Same as TypeScript but without `implements` (JS has no `implements` keyword)
- **Kotlin**: Changed from `&BaseExtractor` to `&mut KotlinExtractor`, extracted `collect_base_type_names` helper for immutable phase
- **Swift**: Changed `&self` to `&mut self` on both `extract_inheritance_relationships` and `add_inheritance_relationship`
- **fast_refs**: Partitioned definitions into real definitions vs imports, added separate "Imports" section

### IMPACT
- **1347 extractors tests pass** (0 failures), **860 main crate tests pass** (0 failures)
- 10 new inheritance-specific tests added across 5 languages
- 2 new formatting tests for import separation
- Fixed Java `mod.rs` missing `cross_file_relationships` module registration
- Cross-file inheritance now creates `PendingRelationship` records that get resolved during workspace indexing, enabling proper centrality propagation and `get_context` pivot selection
