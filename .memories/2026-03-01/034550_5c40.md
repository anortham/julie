## Fixed sidecar crash at batch 34/38 — root cause was missing exception handler

The Python sidecar kept crashing at embedding batch 34/38 with "sidecar stdout closed". After multiple wrong hypotheses (GPU memory leak, `convert_to_numpy`, CUDA vs DirectML), the actual root cause was found:

- **Root cause:** `protocol.py`'s `run_stdio_loop` only caught `json.JSONDecodeError`, not general exceptions from `dispatch_request`. When `model.encode()` threw a `RuntimeError` (from DirectML), the exception propagated out of the protocol loop, the Python process started exiting, stdout closed, and the Rust side saw EOF.
- **Evidence:** "process still running" in the error message — the process was still alive during exit teardown when we checked, confirming it wasn't a segfault.

### Three fixes applied:
1. **`protocol.py`:** Added `except Exception` handler around `dispatch_request` — returns proper error response instead of crashing the sidecar
2. **`pipeline.rs`:** Changed from `?` (abort) to `break` on batch failure — successful batches are already persisted, incremental filter picks up missed symbols next run
3. **`sidecar_provider.rs`:** Added 50ms delay before checking process exit status — avoids Windows race condition between pipe close and process handle update
