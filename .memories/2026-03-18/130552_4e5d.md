## Import-Constrained Call-Edge Filtering

### What
Added **identifier-based parent reference disambiguation** to Julie's relationship resolver. When resolving ambiguous callee names (e.g., `Success` appearing on both `AuthenticateResult` and `ApiResponse`), the resolver now checks whether the caller file's identifiers reference the candidate's parent type, giving a +200 score boost that dominates all other heuristics.

### Why
The LabHandbookV2 C# workspace had a bug where `HandleAuthenticateAsync` was incorrectly shown as calling `ApiResponse.Success` instead of `AuthenticateResult.Success`. The existing resolver heuristics (language +100, directory +50/+25, kind +10) couldn't disambiguate methods with the same name on different types.

### How
- Added `ParentReferenceContext` struct to resolver — pre-computes parent type names and caller file identifier references
- Added `get_identifier_presence()` batch query on identifiers table — checks which (file_path, name) pairs exist
- `build_parent_reference_context()` wires it together: collect parent_ids from candidates → resolve to names → query identifiers → build context
- `score_candidate()` now accepts parent context and adds +200 for confirmed parent reference
- Soft boost (not hard filter) — resolution still works without identifier data

### Impact
- Fixes the "common method names" bug for any codebase with overloaded method names
- Works across all 33 languages (language-agnostic — uses identifier data, not import parsing)
- 2 extra batch queries per resolve_batch call (<1ms each), short-circuits when no candidates have parent_id
- Also confirmed Scala `type_usage` identifiers are already fixed
- Also checked off the Sentrux "sound call-edge filtering" idea as implemented
