## Daemon Lifecycle Redesign: Brainstorm Complete, Writing Spec

### What
Diagnosed a race condition in the daemon startup/restart lifecycle and designed a fix using an explicit state file (`daemon.state`).

### Problem
When the daemon shuts down for stale-binary restart, it removes the IPC socket BEFORE the PID file. This creates a window where the adapter thinks the daemon is alive (PID exists) but can't connect (socket gone). The adapter's 3-retry x 2s budget (6s total) can be exhausted if daemon shutdown takes longer.

### Root Cause
PID file is used as a readiness signal when it's only a liveness signal. Three compounding issues:
1. Cleanup ordering: socket removed before PID, creating "ghost daemon" window
2. `ensure_daemon_running` only checks PID, never probes socket health
3. Retry budget too small for slow shutdowns (embedding sidecar cleanup)

### Design Decision: State File (Approach A)
- Add `~/.julie/daemon.state` with explicit phases: `starting`, `ready`, `stopping`
- Daemon writes state at each phase transition
- Adapter reads state + PID to make informed decisions (wait vs spawn vs connect)
- `ensure_daemon_ready()` holds `daemon.lock` through full readiness confirmation
- Also fixes `stop_daemon()` bug (unconditionally deletes files after 5s even if process alive)

### Codex CLI Analysis
Got independent review from Codex (GPT-5.4). Key agreements: state file is right approach, don't swap PID/socket cleanup order, lock releases too early. Codex also caught `stop_daemon()` split-brain bug.

### Windows Considerations
- `std::fs::write`/`read_to_string` open-close immediately, no sharing violations
- Named pipes are kernel-managed (no socket file cleanup needed), state file works identically
- No atomic rename needed for 7-byte file writes
