## Cold-start measurement validates parallel extraction; projection split markers misplaced

### What
Measured hermes-agent cold-start with parallelized tree-sitter extraction in place. Wall clock dropped from **148s baseline → 50.05s** (66% reduction, more than double the expected ~30%). Discovered the projection sub-phase timing markers (added in the housekeeping batch) were in dead code from cold-start's perspective, then moved them to the actual call sites.

### Stage breakdown (49.64s)
- Workspace init/scan: 0.57s
- **Parallel extraction: 12.91s** (was ~111s serial → 8.6x speedup)
- Canonical persistence: 3.51s
- **Projecting (Tantivy): 29.68s** (≈ unchanged from prior 29.3s — now 59% of cold-start)
- Resolving: 0.62s, Analyzing: 2.12s, Final commit: 0.13s

### Why the markers were in dead code
`src/search/projection.rs:236-259` has timing markers inside `SearchProjection::project_documents`. But the cold-start path is `src/tools/workspace/indexing/pipeline.rs:529 project_batch`, which calls `load_symbol_contexts_from_database` (line 631) and `apply_documents_with_context` (line 648) directly, bypassing the wrapper.

### How (fix applied)
Added `info!` markers around the actual call sites in pipeline.rs:
- `pipeline.rs:637` — `⏱️  projection.load_contexts: {:.2}s ({} symbols)`
- `pipeline.rs:665` — `⏱️  projection.apply_documents: {:.2}s ({} symbols, {} files, {} cleaned)`

Left the orphaned markers in projection.rs alone since `project_documents` may run on refresh/repair paths.

### Impact
- Validated the parallel extraction win is real and bigger than expected.
- Tantivy projection is now unambiguously the dominant phase (59% of cold-start).
- Built release binary (Apr 25 19:26 local). Daemon will detect stale mtime when this session disconnects.
