## Tantivy IndexWriter heap bumped 50MB → 256MB

### What
Doubled-and-then-some the IndexWriter memory budget in `src/search/index.rs:37` to unlock parallelism Tantivy was already willing to give us.

### Why this works (Tantivy 0.26 internals)
`Index::writer(memory_budget)` is multi-threaded by default — it picks `min(available_parallelism(), MAX_NUM_THREAD=8)` worker threads, but auto-clamps thread count if per-thread budget falls below `MEMORY_BUDGET_NUM_BYTES_MIN = 15MB`.

- **Before (50MB):** 50MB / 8 threads = 6.25MB/thread → below 15MB floor → Tantivy reduced to **3 threads at ~16.67MB each**
- **After (256MB):** 256MB / 8 threads = 32MB/thread → above floor → **8 threads at 32MB each**

Combined: 2.7x more threads × 1.9x more per-thread budget = projected 4–6x apply_documents speedup.

Bottleneck found: `apply_documents` is 96% of projection and 57% of total cold-start.

### Verification

### Risk to watch
Each open IndexWriter now holds 256MB. If the daemon ever has many active workspaces simultaneously, RAM scales accordingly (30 workspaces × 256MB = ~7.7GB). Fine on a dev box, worth a follow-up if julie ever runs in a memory-constrained environment.
