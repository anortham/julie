## Fix: Embedding Pipeline Wipes Vectors on Model Name Change

Previously, switching embedding models with the same dimensionality (e.g., between two 384d models) would only update the config label but leave the old model's incompatible vectors in place. KNN results would be silently wrong.

- **Root cause:** The pipeline had separate branches for dimension changes (wipe + re-embed) and model name changes (update config only). The name-change branch didn't clear stale vectors.
- **Fix:** Merged both branches into a single condition: any mismatch (name OR dimensions) wipes the vector table and re-embeds from scratch.
- **TDD:** Verified RED without fix, GREEN with fix. Test pre-stores embeddings, changes stored model name, re-runs pipeline, asserts all symbols are re-embedded (not skipped).
- **Docs:** Updated README.md and `docs/operations/embedding-sidecar.md` to document that model switches are handled automatically.

### Per-Language Config Benchmark Results

Also ran the full embedding benchmark (19 workspaces) after re-embedding everything with BGE-small + per-language TOML configs from last session:

- **Coverage increased dramatically**: Zod +136%, JS +58%, C++ +52%, Ruby +41%, Python +34%
- **Quality held steady**: Avg top similarity 0.929 (unchanged), diversity +0.002, namespace overlap -0.002
- **Same-kind ratio dropped 1.4%**: Expected and good; constructors and more variables make results more diverse in kind
