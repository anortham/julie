## Designed daemon lazy embedding init to fix cold-start MCP failures

After a reboot, Claude Code marks Julie MCP as `failed` on first connect; manual `/mcp reconnect` works instantly. Investigated and designed the fix.

### Diagnosis
- Daemon cold start takes **~39 seconds** from PID creation to `ready` state
- Dominant cost: `EmbeddingService::initialize()` at `src/daemon/mod.rs:330` (Python sidecar + PyTorch CUDA + CodeRankEmbed 768d model load)
- **Claude Code MCP startup timeout is 30 seconds** (verified in `claude.exe v2.1.98`: `MCP_TIMEOUT || 30000`)
- The adapter at `src/adapter/mod.rs:47` calls `ensure_daemon_ready` BEFORE reading stdin, so Claude Code kills it before it can forward the `initialize` handshake
- Daemon keeps running (stdio detached), reaches ready ~8 seconds after the adapter was killed, user's manual reconnect hits the fast path at `src/adapter/launcher.rs:107`
- Cold vs warm delta is only ~3 seconds (log evidence from 2026-04-08 vs 2026-04-09), so model load time dominates, not disk I/O — every restart faces this problem

### Codex second opinion
Codex validated the diagnosis AND caught a regression in my initial "spawn and forget" fix proposal:
- `EmbeddingService` at `src/daemon/embedding_service.rs:15` is immutable after construction — needs a real state machine
- `spawn_workspace_embedding` at `src/tools/workspace/indexing/embeddings.rs:30` has `} else if handler.embedding_service.is_some() { return 0; }` — would silently skip embedding for workspaces auto-indexed during warmup
- `nl_embeddings.rs:52` falls through to stdio-style per-workspace init if daemon provider returns None — would spawn a SECOND Python sidecar during warmup
- Watchers in `WorkspacePool::shared_embedding_provider` read the service live (good) but need audit
- Dashboard `embedding_available` is a static `bool` at `src/dashboard/state.rs` — would permanently show stale `false`

### Design (committed to docs/plans/2026-04-09-daemon-lazy-embedding-init-design.md)
1. **EmbeddingService state machine** using `tokio::sync::watch` (not `RwLock + Notify` — `Notify::notify_waiters()` has a TOCTOU race where notifications fire before waiters enroll)
2. **Background init task in `run_daemon`** with explicit panic/cancel handling (publish `Unavailable` on JoinError, not silent hang)
3. **Move `opener::open(dashboard_url)` off critical path** via `tokio::spawn` (bonus fix)
4. **`spawn_workspace_embedding` waits 120s** for service settlement before falling through
5. **`nl_embeddings` splits daemon-mode (3s wait) vs stdio-mode paths** — prevents duplicate sidecar in daemon warmup
6. **Dashboard reads `EmbeddingService` live** via Arc reference instead of static bool

### Acceptance target
Daemon reaches `ready` in **<2 seconds** from PID creation (down from 39.35s).

### Self-review caught
- `Notify` TOCTOU race → switched to `watch::channel`
- Missing JoinError handling in background task → added explicit `publish_unavailable` on panic/cancel
