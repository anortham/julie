## Fixed dashboard workspace status stuck at "pending" on macOS

Two bugs combined to keep workspaces permanently "pending" on the dashboard after daemon restart:

### Bug 1: Status restoration was Windows-only
`normalize_workspace_paths()` in `daemon/database.rs` had an early return `if !cfg!(windows) { return Ok(0); }` that skipped the status restoration logic on macOS/Linux. The function combined two independent fixes: Windows path normalization (forward slashes to backslashes) and cross-platform status restoration ("pending" with `symbol_count > 0` back to "ready"). The Windows gate killed the status fix as collateral damage.

- **Fix:** Removed the early return, made path normalization conditional (`if cfg!(windows)`) while keeping status restoration unconditional

### Bug 2: Stale binary auto-restart race condition
`accept_loop` in `daemon/mod.rs` only checked binary mtime on **new connections** (line 494), not on **disconnections** (line 538). If a binary was rebuilt during an active session, the disconnect handler found `restart_pending = false` and didn't shut down. The adapter reconnected before the next mtime check could fire.

- **Fix:** Added binary mtime check at disconnect time (before the `remaining == 0` guard) so the daemon detects stale binaries immediately when the last session ends

### Why the CASE guard in upsert wasn't enough
The SQL `ON CONFLICT(path)` CASE statement correctly prevents future "ready" to "pending" downgrades. But workspaces were ALREADY "pending" from the old code (before the CASE was added), so there was no "ready" to protect.
