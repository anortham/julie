## Fixed repo-local `.julie/indexes` pollution and stale duplicate cleanup

### WHAT
Patched Julie so repo-local index directories stop being created by test helpers and stale per-project duplicates are cleaned when the daemon pool touches a workspace.

### WHY
The repo had accumulated `.julie/indexes/*` directories even though daemon-backed indexing is supposed to live under `~/.julie/indexes`. Root cause split into two paths:
- test helpers anchored handlers at `current_dir()`, which let tests dirty the repo tree
- migration logic existed but production never invoked it from the pool, and duplicate cleanup stopped when the destination already existed

### HOW
- Changed `JulieServerHandler::new_for_test()` to root itself in an isolated temp directory instead of `current_dir()`.
- Added `create_isolated_storage_handler()` in test helpers so tests that intentionally index the Julie repo can store indexes in a temp daemon home instead of the repo tree.
- Updated fixture generation and `get_symbols_target_filtering` tests to use the isolated-storage helper.
- Updated `WorkspacePool::get_or_init()` to run per-project index migration before pool initialization.
- Changed migration behavior so an existing valid centralized index causes the stale per-project duplicate to be deleted, and so migrated-state does not skip cleanup when the source still exists.
- Added regression coverage for duplicate cleanup, migrated-state cleanup, pool-triggered migration, and isolated `new_for_test()` roots.
- Swept the existing `/Users/murphy/source/julie/.julie/indexes/*` directories after verifying nothing had them open.

### IMPACT
- New tests stop repopulating the repo’s `.julie/indexes` tree by default.
- Historical split-brain duplicates now get cleaned when the shared pool initializes a workspace.
- The current workspace no longer has stale local index directories sitting under `.julie/indexes`.
- Verification stayed focused instead of rerunning the full dev tier.
