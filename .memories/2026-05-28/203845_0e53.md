## Root cause: stale-binary daemon restart force-aborts LIVE sessions (recovery-marker storm)

**Symptom:** `~/.julie/daemon.log` shows repeated `Session drain timed out ... active_sessions=1` → `force-aborting in-flight requests` → recovery markers climbed to **count=18**. A live MCP session got killed mid-work.

**Root cause (confirmed):** `mark_restart_pending` (src/daemon/lifecycle.rs:206) fired `notify_restart()` on the **first arming** (gated on `first_request`), regardless of whether sessions were still connected. Both `AcceptWithRestartPending` (new connection, sessions active) and `MarkRestartPending` (a session disconnects while others remain) arm with `active_sessions > 0` (src/daemon/mcp_session.rs:432, 396), so arming immediately bridged into `shutdown()`'s **60s bounded drain**. `drain_sessions` (src/daemon/mod.rs:57) waits for the session count to reach **zero** (clients to disconnect), not for in-flight requests to finish. Long-lived MCP sessions don't disconnect during a daemon-initiated restart → drain always times out → `mark_aborted()` force-abort → recovery marker + killed session. The intended trigger (`TriggerShutdown` when `remaining==0`) was dead code because the latch was already consumed (`first_request==false`).

The 15:04 `Received SIGTERM` was a SEPARATE, deliberate stop via `stop_daemon` (lifecycle.rs:386) = `dev-restart --force` / `julie stop` — documented `--force` caveat, not this bug.

## Fix applied (lifecycle unit contract, tests green)
Gated the restart-channel signal on `active_sessions == 0` in `mark_restart_pending` (decouple arm-the-latch from fire-the-restart). Arming with live sessions now ONLY sets the flag; the disconnect handler re-invokes with `active_sessions==0` when the last session leaves, which is when we fire.

- RED→GREEN: new test `test_restart_channel_fires_only_after_last_session_disconnects` (src/tests/daemon/lifecycle.rs).
- Rewrote two tests that encoded the OLD buggy contract: `test_mark_restart_pending_signals_listener_when_no_active_sessions` (was `..._on_first_transition`), `test_mark_restart_pending_with_active_sessions_never_signals` (was `..._does_not_double_signal`).
- Updated stale doc comments in lifecycle.rs + mcp_session.rs.
- `cargo nextest run --lib tests::daemon::lifecycle` → 26/26 pass.

## BLOCKER / design fork (needs user decision)
My fix REVERSES the prior Codex-reviewed fix `docs/plans/2026-05-17-daemon-restart-listener-fix.md`. Its load-bearing end-to-end test `daemon_exits_within_drain_when_active_session_never_disconnects` (src/tests/daemon/restart_listener.rs:189) asserts the OPPOSITE: a stale-binary `AcceptWithRestartPending` with an active session that never disconnects MUST force-abort it within the drain timeout, to avoid hanging in `restart_pending` forever. My fix makes that test hang (RED).

Sessions are removed ONLY via `HttpJulieService::drop` (mcp_session.rs:572) on connection close/DELETE — there is **no idle session-reaper**. In production, clients exiting drop the connection → session removed → count decrements → my fix restarts cleanly when the last session leaves. The "never disconnects" case is a genuine leak (half-open TCP) — rare. Prior fix conflated "user still has a session open" (intended non-disruptive defer) with "leaked session."
