## Reviewed 2026-05-27 daemon cold-start fix and found a new liveness risk

### WHAT
Reviewed commit `1c45455a` (`fix(daemon): close cold-start race via early discovery + phase guard`) with lead analysis plus two subagent reviewers. The commit fixes the long lock-held/no-liveness cold-start window by publishing `discovery.json` with `phase="starting"` before `DaemonApp::new`, then making the launcher treat `phase="starting"` as `DaemonReadiness::Starting`.

### WHY
The user reported the daemon remains a constant problem. The review needed to verify whether today's fix actually closes the phantom-daemon race without introducing another stuck state.

### HOW
Key verified finding: `DaemonApp::serve` calls `startup_complete()` and writes `daemon.state=ready`, then attempts to publish `discovery.json` with `phase="running"`. That running publish is warn-and-continue on failure. Because `DaemonLauncher::daemon_readiness` now unconditionally returns `Starting` for live `phase="starting"`, a daemon can be HTTP-ready but permanently invisible to adapters if the running discovery overwrite fails. Also verified the commit's comment about bare TCP readiness is misleading: `TransportEndpoint::probe_readiness` sends `GET /mcp/ready`, reads a response, and requires HTTP 2xx/3xx, so TCP accept alone is not enough to return Ready.

### IMPACT
Recommended immediate fix: make the startup discovery state machine mandatory. If early `phase=starting` publish succeeds, the transition to `phase=running` must succeed before `startup_complete`; otherwise startup must tear down transport/resources and remove stale discovery rather than continuing as a headless daemon. Follow-on architecture: make `discovery.json` the authoritative adapter lifecycle source and remove legacy PID/state fallbacks from the new daemon path.
