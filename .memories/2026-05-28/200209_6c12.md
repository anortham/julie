## Sidecar GPU bootstrap: add Linux ROCm + Intel XPU, Windows `py -3.X` launcher probing

Cleared two TODO.md enhancement items (plus Intel XPU, explicitly requested alongside ROCm).

### WHAT
- **AMD ROCm (Linux)**: `detect_amd_rocm()` (probes `rocminfo`, falls back to `/opt/rocm`) + `rocm_torch_index_url()` → `whl/rocm6.4`.
- **Intel XPU (Linux)**: `detect_intel_xpu()` (probes `xpu-smi discovery`, the nvidia-smi analog) + `xpu_torch_index_url()` → `whl/xpu`. Native in stable torch 2.5+, no intel-extension-for-pytorch needed (web-verified).
- **GPU selection refactor**: extracted `detect_gpu_torch_index()` (priority CUDA→ROCm→XPU on Linux; CUDA-only on Windows since torch-directml covers AMD/Intel there; nothing on macOS/MPS) + `reinstall_torch_from_index()` DRY helper replacing the old inline CUDA-only block.
- **Windows launcher**: `python_interpreter_candidates()` now returns `PythonCommand { program, prefix_args }` instead of `Vec<OsString>`, probing `py -3.12/-3.13/-3.11/-3.10` before bare names. Threaded prefix args through `detect_bootstrap_python_interpreter` + `ensure_venv_exists`.
- Supervisor: added `python_version_from_command(program, prefix_args)`; `python_version_from_program` delegates.
- Bumped `INSTALL_MARKER_VERSION` v10-cuda-torch → v11-gpu-torch so existing Linux AMD/Intel installs re-run the GPU swap on upgrade.

### WHY
Linux AMD-GPU users silently got CPU-only embeddings (no detect/install path). Windows `py` launcher couldn't request a specific Python version (bare binary names rarely exist on Windows).

### HOW
TDD: 6 new tests (index URLs, idempotent detection, versioned-launcher candidates, unix no-prefix-args) RED→GREEN. `python_interpreter_candidates_for(target_windows: bool)` split out so both OS branches are host-independent unit-testable.

### IMPACT
AMD/Intel Linux GPUs now get accelerated torch; Windows venv creation uses correct version selection.

### CAVEATS
- No GPU hardware here — real end-to-end ROCm/XPU Linux install unverified (logic verified via unit tests + fake-binary serialization harness).
- `rocm6.4` pin tracks stable torch like `cu124`; mismatch degrades gracefully to CPU + warning, needs periodic bumping.
