## Fixed watcher dropping delete/rename events under symlinked workspace roots

Three watcher event-queue tests were failing on clean `main` (HEAD `95d6c2f3`); root-caused to a real production bug and fixed.

- **Symptom:** `test_remove_event_queued_for_deleted_file`, `test_rename_any_nonexistent_path_emits_deleted`, `test_queue_atomic_save_coalesces_delete_then_modify` asserted a delete/rename event would be queued, but the queue was empty (`left: 0, right: 1`).
- **Root cause:** watcher filtering (`contains_blacklisted_directory_relative`, `is_gitignored`, `should_process_deletion`) relativized event paths with a naive `path.strip_prefix(workspace_root)`. On macOS, `notify`/FSEvents emits the *canonical* event path (`/private/tmp/...`) while the workspace root may be a *symlink* (`/tmp/...`). `strip_prefix` fails, and the blacklist check fell back to the *absolute* path whose ancestor component `tmp` matches `BLACKLISTED_DIRECTORIES` → event silently dropped → orphaned symbols/embeddings. Bites any workspace under a symlinked path, not just tests.
- **Fix:** added `crate::utils::paths::relative_within_workspace(path, root) -> Option<PathBuf>` (direct strip → canonicalize root and retry → canonicalize candidate and retry). The canonicalize-root step recovers deleted leaves since only the root must exist. Wired into all three filtering sites. Lifted duplicated `strip_unc_prefix` to module level (Windows UNC correctness preserved).
- **TDD:** RED unit tests in `src/tests/core/paths.rs` use an explicit symlink (deterministic cross-platform); the 3 integration tests are the end-to-end regression guard.

This is a separate fix from the daemon stale-binary restart force-abort fix (`fb8d5797`) committed earlier this session.
