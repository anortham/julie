## Hardened Daemon Discovery Readiness

### WHAT
- Removed legacy `daemon.pid` / `daemon.state` fallback from `DaemonLauncher::daemon_readiness`, `is_daemon_running`, and readiness wait loops.
- Made `discovery.json.phase` the adapter-facing lifecycle source for the split daemon.
- Kept legacy daemon compatibility only through the explicit `legacy_migration::detect_and_attach` path.
- Removed stale direct `daemon-mcp-transport.json` fallback unless a live legacy PID is detected.
- Made early `phase=starting` discovery publish mandatory in `run_daemon`; startup now aborts if adapters would be unable to see liveness.

### WHY
The adapter had two competing lifecycle contracts: new `discovery.json.phase` and old PID/state files. That kept old cascade/stuck-startup failure modes alive and made stale legacy files capable of influencing modern daemon readiness.

### HOW
- Reworked `DaemonLauncher::daemon_readiness` to classify only validated discovery records.
- Reworked polling to wait on discovery readiness/exit instead of reading `daemon.state`.
- Updated adapter tests to assert PID/state files are ignored and not mutated by new-daemon readiness.
- Added regression coverage for stale `daemon-mcp-transport.json` refusal without live legacy PID.
- Added daemon startup regression coverage proving early discovery publish failure aborts before HTTP transport starts.

### IMPACT
Adapters now have one lifecycle source for new daemons. A daemon cannot continue running if the early discovery record cannot be written, and stale legacy files no longer create false readiness or endpoint selection.
