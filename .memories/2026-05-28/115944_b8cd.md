## Fixed mandatory daemon running discovery handoff

### WHAT
Added a regression test for `DaemonApp::serve` aborting when the late `phase="running"` `discovery.json` publish fails after early `phase="starting"` discovery exists. Changed `DaemonApp::serve` so the running discovery publish is no longer warn-and-continue: after lifecycle reaches ready, the publish must succeed or startup tears down partial artifacts and returns an error.

### WHY
The 2026-05-27 cold-start fix made launcher readiness unconditionally return `Starting` for live `phase="starting"` records. If the later running overwrite failed, adapters would wait until timeout even though the HTTP transport was up. That replaced the phantom-daemon race with a possible headless/stuck-starting daemon.

### HOW
Used TDD. RED: `cargo nextest run --lib test_daemon_app_serve_aborts_when_running_discovery_publish_fails` failed because `serve()` returned a handle. GREEN: moved the running discovery publish into the required startup path, call `startup_complete()` before writing `phase="running"`, and on publish failure remove `discovery.json`, `daemon.state`, `daemon.port`, shut down HTTP transport, and abort startup sweep tasks before returning contextual error. Also rustfmt-cleaned the touched daemon helper line.

### IMPACT
Prevents a daemon from continuing live while adapters only see stale `phase="starting"`. Startup now has one explicit handoff: ready lifecycle plus successfully published running discovery, or no advertised daemon artifacts remain.
