## Enforced Windows ORT-only embedding policy and added DirectML adapter selection

### WHAT
Implemented the Windows embedding policy so `auto` resolves to ORT and explicit `sidecar` is rejected on Windows. Added a new `windows_directml` helper with adapter ranking heuristics and updated ORT runtime state handling so query/batch CPU fallback updates telemetry truthfully.

### WHY
The repo had split-brain Windows behavior: release builds were ORT-only, but source builds still preferred the Python sidecar. Windows also lacked explicit adapter choice, so DirectML could pick the wrong GPU or a remote/virtual adapter, and runtime fallback telemetry could lie after GPU failures.

### HOW
- Updated `src/embeddings/factory.rs` to force Windows `auto -> ort` and reject Windows `sidecar`
- Added `src/embeddings/windows_directml.rs` with adapter metadata, filtering, and ranking (discrete preferred, remote/virtual/software skipped)
- Refactored `src/embeddings/ort_provider.rs` to keep mutable runtime state in `OrtRuntimeState` and reuse `run_with_cpu_fallback()` for both query and batch fallback paths
- Updated `README.md` and `docs/operations/embedding-sidecar.md` so Windows is documented as ORT + DirectML rather than sidecar-backed
- Added focused tests in `src/tests/core/windows_embedding_policy.rs`

### IMPACT
- Source builds now match release policy on Windows
- DirectML selection logic is centralized in the ORT path
- Query-time GPU failures no longer leave stale DirectML telemetry behind after CPU fallback
- Remaining uncertainty is Windows-only compile verification because the local environment lacks the Windows Rust target
