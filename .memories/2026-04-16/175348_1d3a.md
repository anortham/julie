## Completed Track 2 Task 4 indexing health visibility

### WHAT
Implemented shared indexing runtime health and surfaced it through Julie health reporting and the dashboard.

- Added runtime indexing state to capture active operation, stage, catch-up activity, watcher pause/rescan state, dirty projection counts, and repair reasons.
- Threaded that runtime state through primary workspace snapshots, indexing routes, pipeline execution, watcher runtime, workspace initialization, and startup repair.
- Extended health output and dashboard JSON/UI to show indexing lag and repair-needed state.
- Added focused tests for health snapshots, `manage_workspace health`, dashboard JSON exposure, and dashboard HTML rendering.

### WHY
Track 2 needed live visibility into indexing degradation so repair-needed states stop hiding in logs. Without this, users and the dashboard could not tell whether indexing was idle, catching up, paused, or carrying dirty projections.

### HOW
- Introduced `IndexingOperation`, `IndexingRuntimeState`, and `IndexingRuntimeSnapshot` in the indexing state module.
- Updated startup repair and watcher runtime to publish indexing runtime signals instead of only logging them.
- Extended `HealthChecker` and dashboard state/route/template code to expose a consistent indexing health contract.
- Verified with targeted red-green tests plus `cargo xtask test system` and `cargo xtask test dev`.

### IMPACT
- Dashboard and health output now expose indexing lag, repair reasons, and dirty projection work without log inspection.
- Track 2 now has an observable health seam that later indexing hardening tasks can build on.
- The watcher follow-up items remain recorded in the active plan so they do not disappear during later tasks.
