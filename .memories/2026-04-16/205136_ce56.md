## Completed Track 4 Task 2: sidecar capability probe and load-policy contract

### WHAT
Implemented the Task 2 split between low-level sidecar capability reporting and runtime load policy.

- Added structured sidecar health metadata in the Python sidecar:
  - `capabilities` now reports backend availability for `cpu`, `cuda`, `directml`, and `mps`
  - `load_policy` now reports requested vs resolved device backend, acceleration, and degraded reason
- Extended the Rust sidecar protocol contract with matching `HealthResult` fields and validation helpers
- Added fake-sidecar integration coverage so Rust provider startup is tested against the richer health payload
- Kept legacy top-level health fields in place for compatibility while moving new truth into structured blocks

### WHY
Task 2 in the embedding runtime plan called for explicit capability and degradation reporting so Julie can understand sidecar state without inferring fallback behavior from stderr.

Before this change:
- Python runtime fallback decisions lived inside `build_runtime()` control flow and stderr logging
- Rust only validated a thin health payload and had no structured view of capability data
- The cross-language contract could drift because no integration test exercised provider startup with richer health metadata

### HOW
- Refactored the Python runtime to probe backend capabilities and capture requested vs resolved device backend during runtime construction
- Updated `handle_health()` to validate structured metadata before returning it
- Extended Rust protocol types with `DeviceBackendCapability`, `DeviceBackendCapabilities`, and `DeviceLoadPolicy`
- Added Rust validation for inconsistent health responses, including missing degraded reason on backend fallback and blank load-policy fields
- Extended the fake sidecar helper to inject custom health payloads and added integration tests for healthy and degraded startup paths
- Reviewed and reconciled a cross-lane contract mismatch: Python emitted nested capability objects, so the Rust contract and tests were updated to match that better shape

### IMPACT
- Julie can now ask the sidecar what hardware backends are available and how runtime fallback resolved, instead of scraping stderr folklore
- The sidecar protocol has stronger cross-language validation and better regression coverage
- Track 4 Task 3 can now consume settled runtime/device state from a cleaner contract
- A brittle contract mismatch between Rust and Python was caught and fixed before it escaped into broader testing
