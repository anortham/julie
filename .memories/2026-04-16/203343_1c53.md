## Track 4 Task 1: normalized embedding runtime status and settled service reporting

### WHAT
Completed the Task 1 runtime-contract pass for embeddings.

- Preserved richer sidecar runtime health through a shared `HealthResult` protocol type
- Extended `SidecarEmbeddingProvider` to surface acceleration and degraded-reason hints through the provider trait
- Refactored `EmbeddingServiceSettled` into structured variants that carry `EmbeddingRuntimeStatus`
- Verified daemon and stdio embedding paths still share the same runtime status model

### WHY
The old service settlement boundary threw away runtime detail and forced downstream callers to guess why the embedding service was unavailable or degraded. That made cross-platform failures, especially CPU fallback and Windows sidecar issues, harder to reason about.

### HOW
- Added `HealthResult` plus `validate_health_response` in the Rust protocol layer
- Threaded resolved backend, acceleration, and degraded reason through the Python sidecar runtime metadata and the Rust sidecar provider
- Updated daemon embedding-service tests to assert runtime metadata survives `Ready` and `Unavailable` outcomes
- Ran focused Rust and Python protocol tests plus `cargo xtask test dev`

### IMPACT
Track 4 now has a real runtime contract instead of a hand-wave. The next embedding tasks can consume settled runtime truth without inventing their own side stories, and later health/dashboard work has a stronger base.
