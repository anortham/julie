## Completed Track 4 Task 4: shared embedding health surfaced through dashboard and health command

### WHAT
Finished the last planned implementation lane by surfacing richer embedding runtime details through both `manage_workspace health` and the dashboard.

- Added a shared embedding-runtime projection in `src/health/embedding.rs`
- Extended `EmbeddingRuntimeHealth` with:
  - `requested_backend`
  - `query_fallback`
- Reused that shared projection in both:
  - `HealthChecker::build_runtime_plane()`
  - `DashboardState::health_snapshot()`
- Extended the health report to print requested backend and query fallback
- Extended the dashboard runtime section to show embedding state, runtime, requested backend, resolved backend, device, acceleration, degraded reason, and query fallback
- Kept the existing dashboard `runtime_status` snapshot in place for compatibility while adding the shared `embeddings` payload

### WHY
Task 4 required the health command and dashboard to say the same truthful thing about embedding availability and degradation.

Before this change:
- `manage_workspace health` already printed runtime/backend/device/acceleration/degraded fields, but had no explicit query fallback state
- the dashboard only exposed coarse booleans (`configured`, `embedding_available`, `embedding_initializing`) and hid the real runtime details from the UI
- health and dashboard built their embedding story independently instead of sharing a common projection

### HOW
- Wrote red tests for dashboard snapshot structure, `/status/live`, and health command output
- Introduced `project_embedding_runtime()` as the shared mapping from runtime status + provider state to `EmbeddingRuntimeHealth`
- Updated the health report renderer to print the richer fields
- Updated dashboard templates and polling JS to render the new runtime rows
- Adjusted the not-initialized health test to model the real scenario: no provider and no runtime metadata

### IMPACT
- The final planned track is complete
- The health command and dashboard now surface the same embedding-runtime truth, including query fallback behavior
- Operators and dogfooding sessions can see whether embeddings are semantic-capable, keyword-only, or pending without reading logs
- The branch is ready for the final convergence pass before merge and release-binary dogfooding
