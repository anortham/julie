## Track 3 Task 3: Added durable Tantivy projection state and repair path

### WHAT
Implemented a revision-aware `SearchProjection` layer so Tantivy is treated as a durable projection of canonical SQLite state instead of an incidental side effect.

- Added `projection_states` storage and migration support in the database layer
- Added `src/search/projection.rs` with rebuild and incremental projection entry points
- Routed indexing pipeline projection writes and backfill repair through `SearchProjection`
- Added integration coverage for empty-index rebuild, projection write failure, and rebuild after canonical revision advance

### WHY
The old model relied on restart folklore and partial in-memory state when Tantivy drifted from SQLite. That made it hard to answer a basic question: did canonical data commit and, if so, which projection revision is search serving?

### HOW
- Created durable projection metadata with status, canonical revision, detail, and timestamp
- Replaced ad hoc backfill behavior with `SearchProjection::ensure_current_from_database()`
- Replaced direct pipeline-side Tantivy writes with `SearchProjection::project_documents()`
- On projection failure, persist `stale` state tied to the target canonical revision so startup, health, and repair can see drift without reopening Tantivy first
- Extended integration tests to prove canonical revision survives projection failure and search can rebuild forward to the newer revision

### IMPACT
Track 3 now has an explicit canonical-to-projection seam. SQLite commit state and Tantivy freshness are observable and testable, which sets up the next slice for health reporting and repair policy. It also gives us a stronger footing for the separate false deleted-file auto-indexing bug, because search drift can now be distinguished from bogus workspace staleness signals.
