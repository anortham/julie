## Track 3 Task 4: Surfaced projection freshness through health and dashboard

### WHAT
Extended the canonical-to-projection seam so health and dashboard surfaces now report projection freshness, canonical revision, projected revision, lag, and repair-needed state.

- Added durable `projected_revision` tracking to `projection_states`
- Added migration 018 to backfill ready rows with projected revision metadata
- Extended shared health types with projection freshness and revision fields
- Routed system health, `manage_workspace health`, dashboard snapshot, `/status/live`, and the status page through the same projection-health model
- Added targeted tests for lagging projection health, dashboard JSON, dashboard render, and health report output

### WHY
Task 3 could now persist projection state, but operators still had a binary story: search was either present or missing. That hid the important cases, lagging projection, stale metadata, and repair-needed state, which are the cases that matter when Julie feels slow or wrong.

### HOW
- Stored both canonical target revision and last known projected revision in SQLite
- Taught `SearchProjection` to preserve the last served revision during rebuild and failure paths
- Added `ProjectionFreshness` plus revision fields to `SearchProjectionHealth`
- Built shared projection-health derivation in `HealthChecker` and reused it from the dashboard
- Updated reports and templates to render projection workspace, revisions, lag, and repair-needed state
- Fixed a stale test expectation that previously treated an unprojected rebound workspace as search-ready

### IMPACT
Julie can now tell us whether Tantivy is current, lagging, or in repair, and it can do that without restart folklore. The health command and dashboard are now aligned on the same truth, which is the footing we need before moving into the next runtime lane.
