## Track 3 Task 2: canonical/projection seam threaded through indexing

### WHAT
- Threaded canonical revision out of the persist phase and into the indexing pipeline result.
- Changed `persist_batch()` to return a canonical-write outcome instead of hiding SQLite commit state.
- Changed `project_batch()` to accept the canonical revision it is trying to project.
- Propagated canonical revision through `IndexingPipelineResult` and `IndexResult` into command status output.
- Strengthened projection-failure tests so they prove SQLite state and canonical revision survive a failed Tantivy pass.

### WHY
- Task 2 needs canonical SQLite commit to stand on its own before projection repair and revisioned Tantivy can be built cleanly.
- Without carrying revision through the pipeline, projection failure still leaves later code guessing which canonical state exists.
- The existing projection failure test only proved search stayed unready; it did not prove canonical state survived.

### HOW
- Added `canonical_revision` to `IndexingPipelineResult`.
- Had `persist_batch()` query the committed canonical revision immediately after canonical write completion.
- Passed that revision into `project_batch()` and included it in projection-repair details.
- Propagated the same value through `ManageWorkspaceTool::index_workspace_files()` into command-path status strings.
- Verified with targeted indexing-pipeline tests and a fresh `cargo xtask test dev` run.

### IMPACT
- Canonical commit and projection failure are now distinct, observable phases.
- Projection code can target a known canonical revision instead of operating by side effect and hope.
- Search failure no longer implies ambiguity about whether SQLite committed.
- Track 3 can move into revisioned Tantivy projection work without re-litigating the canonical handoff seam.
