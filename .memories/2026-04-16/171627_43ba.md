## Completed Track 2 Task 3 shared repair lane

### WHAT
- Added `src/watcher/runtime.rs` as the shared queue and repair runtime for watcher ticks, manual `process_pending_changes()`, dirty Tantivy retries, overflow repair scans, and shutdown drain.
- Refactored `src/watcher/mod.rs` to route background queue processing and manual pending-change processing through the same runtime instead of keeping a separate manual drain path.
- Added shared repair vocabulary in startup and watcher flows so cold-start planning and watcher repair speak the same `IndexingRepairReason` language.
- Centralized primary catch-up execution in `src/startup.rs` and made `JulieServerHandler::run_auto_indexing()` a thin orchestrator over that shared repair helper.
- Hardened shutdown and routing after review: watcher shutdown now keeps processing buffered notify events, shutdown drain retries dirty Tantivy projections, and `IndexRoute::for_current_primary()` anchors `db_path` from the live snapshot database instead of recomputing it against a later binding.
- Added watcher regression coverage for stale-plus-new repair during manual pending-change processing.

### WHY
- Before this work, the live watcher queue had one repair policy while `process_pending_changes()` had a weaker one. Manual drain skipped dirty projection retry, overflow repair, and dedup behavior.
- Catch-up repair and cold-start planning were also spread across `handler.rs` and `startup.rs`, which made pause/resume policy and repair reasoning drift over time.
- Review surfaced two shutdown holes and one route-anchor race that could leave stale search state or mismatched primary routing after rebinds.

### HOW
- Built a dedicated watcher runtime that owns queue draining, dedup, dirty retry, overflow repair scans, and Tantivy commit behavior.
- Reused `startup::scan_workspace_files()` inside the watcher overflow repair path so cold-start freshness checks and watcher recovery enumerate files through the same logic instead of separate extension-only scans.
- Added `run_primary_workspace_repair()` in `startup.rs` to own watcher pause/resume and repair execution around `ManageWorkspaceTool::index`.
- Switched current-primary route anchoring to derive `db_path` from the snapshot database handle and derive the sibling Tantivy path from that same anchor.

### IMPACT
- Cold-start repair, catch-up indexing, and watcher/manual repair now share one policy lane instead of drifting implementations.
- Manual queue repair can now recover stale tracked files and newly discovered files after overflow or paused watcher gaps.
- Shutdown is safer: buffered notify events and dirty Tantivy projections survive the drain path instead of falling on the floor.
- `src/watcher/mod.rs` is back under the repository file-size target after moving the queue runtime into its own file and removing duplicate inline tests.
