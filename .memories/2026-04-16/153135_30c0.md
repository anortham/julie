## Completed Track 2 Task 2 pipeline/state refactor

- **WHAT:** Added explicit indexing pipeline state and moved cross-stage orchestration out of `src/tools/workspace/indexing/processor.rs` into a dedicated `pipeline.rs`, with stage metadata in `state.rs` and new direct tests for parser-backed, text-only, repair-needed, and projection-failure paths.
- **WHY:** `process_files_optimized` was still a monolith that mixed grouping, extraction, SQLite persistence, Tantivy projection, relationship resolution, analysis, and readiness publication. That made health visibility weak and hid correctness bugs like search readiness being published after Tantivy projection failure.
- **HOW:**
  - Created `src/tools/workspace/indexing/state.rs` for explicit stage history and repair-needed tracking.
  - Created `src/tools/workspace/indexing/pipeline.rs` to orchestrate `Grouped -> Extracting -> Persisting -> Projecting -> Resolving -> Analyzing -> Completed` with route-aware storage and projection behavior.
  - Shrunk `src/tools/workspace/indexing/processor.rs` down to file-level helpers and shared Tantivy population.
  - Added direct tests in `src/tests/integration/indexing_pipeline.rs` and `src/tests/tools/workspace/processor.rs`.
  - Fixed the review-found correctness bug where a dead Tantivy writer still resulted in `search_ready = true`.
- **IMPACT:**
  - `processor.rs` is back under the repo size limit.
  - Indexing stages are now explicit and testable.
  - Repair-needed state covers missing-file extraction failures and broken projection writers.
  - Search readiness no longer lies when projection fails, which is the sort of bug that makes dashboards look smart while the tool is quietly broken.
