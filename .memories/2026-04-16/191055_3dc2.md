## Track 3 Task 1: canonical revision ledger landed

### WHAT
- Added a database-level canonical revision ledger in `src/database/revisions.rs`.
- Canonical writes now record workspace revision metadata from `incremental_update_atomic()` and `bulk_store_fresh_atomic()`.
- Workspace cleanup now deletes revision metadata, and workspace usage stats now expose the current canonical revision.
- Added regression tests for incremental writes, fresh writes, cleanup behavior, no-op writes, and skipped-row count accuracy.

### WHY
- Track 3 needs SQLite to be the canonical source of truth before Tantivy and other projections can become revisioned, rebuildable followers.
- Without a durable revision seam, later projection freshness and repair work would have to infer state from side effects.
- The first implementation exposed two hidden bugs: fake revisions on no-op writes and inflated revision counts when relationships or types were skipped.

### HOW
- Added `canonical_revisions` schema creation and migration 016.
- Exposed `get_latest_canonical_revision()` and `get_current_canonical_revision()` on `SymbolDatabase`.
- Recorded revisions inside the same canonical write transaction, not in a later side channel.
- Changed revision recording to use persisted row counts and to skip revision writes when the canonical store did not change.
- Verified with targeted red/green tests, then `cargo fmt` and `cargo xtask test dev`.

### IMPACT
- Julie now has a durable canonical revision contract for later storage/projection split work.
- Health, repair, and search code can query canonical revision without poking indexing internals.
- No-op indexing no longer invents freshness movement, which would have poisoned projection lag tracking.
- The storage track can move into Task 2 with a clean handoff target.
