## Post-review checkpoint for search-matrix branch

### WHAT
Completed the requested Claude pre-merge review and handled the only material finding.

- Claude surfaced one high-severity product-path issue: the line-mode `total_matches` change removed the early file-loop break and forced content search to fetch and scan every candidate file in the Tantivy window.
- Replaced that approach with an honest lower-bound signal in the baseline harness.
- Content executions in the matrix report now mark saturated counts with `hit_count_is_lower_bound = true` and render as `10+` in Markdown.
- Re-ran verification after the fix.

### WHY
The earlier exact-total implementation improved baseline precision but at the cost of turning ordinary `fast_search(..., search_target="content")` calls into expensive blob-fetch and full-file scan work across the whole candidate window. That is the wrong trade-off on a hot path.

### HOW
- Reverted the line-mode scan changes in `src/tools/search/line_mode.rs`, `src/tools/search/execution.rs`, and `src/tools/search/mod.rs`
- Kept the baseline harness honest by adding lower-bound metadata in `xtask/src/search_matrix.rs` and `xtask/src/search_matrix_report.rs`
- Updated the xtask contract test so dense content queries assert the lower-bound behavior instead of an exact count
- Re-ran `cargo nextest run -p xtask search_matrix_contract_tests`, `cargo xtask test changed`, and `cargo xtask test dogfood`

### IMPACT
- The harness still distinguishes saturated content result sets from exact counts without regressing the product search path
- The Claude review produced one verified fix and no remaining material blockers
- Branch is ready for finish-up work (commit, push/PR or local handoff depending on integration choice)
