## Added Tantivy file-mode indexing lane

Implemented the Task 2 core for `fast_search(search_target="files")` in the worktree.

- **Schema:** added `basename` and `path_text` fields to the Tantivy schema and wired them through `SchemaFields`.
- **Index:** added `SearchIndex::search_files()` with exact-path and exact-basename boosts, path-token matching, glob-like query filtering, and deterministic match-kind ordering.
- **Compat:** bumped the search compatibility marker version so stale indexes recreate instead of limping along with the old file-only schema.
- **Projection:** stopped Tantivy backfill from skipping `NULL`-content file rows by adding a projection-only database reader that selects all file rows and coerces missing content to `""`.

## Why

The old index could only do exact `file_path` identity lookups and content search. It had no path-token field for basename or fragment queries, and projection could drift whenever legacy `files.content` rows were `NULL`.

## How

Kept the change narrow:

- reused the existing `FileDocument` ingestion path instead of inventing a new document type;
- added a focused `build_file_query()` builder instead of overloading content search;
- used glob post-filtering only after Tantivy candidate retrieval, so file mode still stays index-led;
- added a projection-specific DB reader rather than weakening the existing content-only readers.

## Impact

Task 2 acceptance is green.

- Targeted tests now cover exact basename ranking, exact relative path ranking, glob-like queries, stale compat-marker recreation, and contentless projection repair.
- `cargo xtask test changed` passed after the schema/index/projection batch.
- Task 3 can now focus on execution plumbing and lean file-shaped output instead of index mechanics.
