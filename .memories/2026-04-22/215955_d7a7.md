## Completed Task 1 for fast_search file mode

### WHAT
Implemented the API-edge slice for `fast_search` file mode in the worktree:

- Added `SearchTarget` parsing in `src/tools/search/target.rs`
- Canonicalized `search_target="paths"` to `files`
- Rejected unknown search targets
- Added file-mode validation that rejects `context_lines`
- Normalized telemetry metadata to canonical `files`
- Added `file_lookup` intent classification
- Added focused regression tests in `src/tests/tools/search/file_mode_tests.rs`
- Added telemetry regressions in `src/tests/core/handler_telemetry.rs`

### WHY
The current search-target split was inconsistent across `mod.rs`, `execution.rs`, and `text_search.rs`, and telemetry would have leaked the alias `paths` as a separate bucket. There was also a subtle deserialization trap where omitted `context_lines` would default to `Some(1)`, which would make a naive file-mode rejection fire on valid input.

### HOW
- Kept `FastSearchTool.search_target` as a canonicalized `String` for compatibility with existing struct construction sites.
- Added enum-backed parsing and validation instead of changing the public field type.
- Implemented a custom deserialization path for `FastSearchTool` so file-mode requests can distinguish omitted `context_lines` from provided values.
- Added a dedicated `SearchTarget::Files` branch in internal routing, with a temporary clean failure until the Tantivy lane lands.
- Updated telemetry to normalize the target and classify all file-mode calls as `file_lookup`.

### IMPACT
This closes the split-brain target parsing bug and gives the feature a stable API edge before the search backend work lands. It also makes telemetry trustworthy for file-mode traffic and prevents the default `context_lines` behavior from sabotaging valid file-mode calls.
