## WHAT

Pruned `codex/search-hardening-rescue` into an observability-first search hardening branch.

- Kept file-pattern parser fixes, empty-pattern normalization, zero-hit attribution, trace propagation, telemetry persistence, and replay harnesses.
- Removed promotion-only execution plumbing from the rescue branch, including `SearchExecutionKind::Promoted`, `PromotionInfo`, promoted trace fields, and tests that existed only to support that path.
- Updated replay reporting to track multi-token hints instead of promotion rescues.
- Verified the branch with targeted search tests, `cargo xtask test dev`, and `cargo xtask test dogfood`.

## WHY

The original hardening branch mixed useful diagnostics with speculative behavior changes. That made it hard to tell what was helping search quality versus what was adding surface area. Main needs the measuring tools first.

## HOW

- Edited the search trace types and execution/result plumbing to remove promotion state while preserving zero-hit reason and hint attribution.
- Simplified telemetry and dashboard code so they no longer depend on the removed promotion variant.
- Trimmed handler and integration tests to assert the surviving observability fields.
- Re-ran focused search tests, the standard dev bucket, and the dogfood bucket to catch regressions in the search stack.

## IMPACT

- The rescue branch is now coherent: it improves search observability without pretending to solve search quality.
- This gives `main` a safer path forward, because we can merge instrumentation and parser fixes first, collect better live data, then attack file-pattern failures with evidence instead of hope.
