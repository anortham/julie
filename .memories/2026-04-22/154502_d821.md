## Search matrix harness implemented in worktree

### WHAT
Built the first pass of the `cargo xtask search-matrix` harness in the `codex/search-matrix` worktree.

- Added top-level xtask command parsing for `search-matrix mine` and `search-matrix baseline`
- Added committed case and corpus fixtures under `fixtures/search-quality/`
- Implemented telemetry mining into JSON + Markdown seed reports
- Implemented baseline execution against daemon workspaces using the real `FastSearchTool` path
- Documented the harness in `docs/TESTING_GUIDE.md`
- Added `.gitignore` coverage for generated `artifacts/search-matrix/` reports
- Fixed a root-resolution bug in the baseline runner so workspace matching now uses configured corpus roots plus exact resolved repo paths, not basename-only matching

### WHY
We needed an investigative harness for search quality that uses real Julie telemetry and real repos under `~/source`, without pretending Julie-only telemetry can measure cross-tool recourse.

The root-resolution fix mattered because basename-only workspace matching could silently run the wrong repo when two different roots contain the same repo name.

### HOW
- Extended `xtask` with a new `CliCommand` split so `test` behavior stays intact
- Added `search_matrix`, `search_matrix_mine`, and `search_matrix_report` modules
- Wired xtask to the main `julie` crate, then reused `DaemonDatabase`, `WorkspacePool`, `JulieServerHandler`, and `FastSearchTool::execute_with_trace`
- Exposed the narrow search trace surface needed by xtask from `src/tools/search/mod.rs`
- Added contract tests covering parser shape, fixture loading, telemetry mining, ready-vs-pending baseline behavior, and configured-root workspace matching

### IMPACT
- We can now mine live `fast_search` rows into candidate families and run a curated baseline against committed fixtures
- The harness writes machine-readable and human-readable reports under `artifacts/search-matrix/`
- The new regression test locks in root-aware workspace matching, which closes a correctness hole before pre-merge review
- Remaining work in this phase is batch verification, dogfood verification, Claude pre-merge review, and branch finish-up
