## Completed fast_search file mode end to end

Finished the file-mode plan in the worktree.

- **Tool path:** `fast_search(search_target="files")` now executes through a dedicated file lane instead of bailing or pretending the query was a definition/content search.
- **Hit shape:** file results now carry `kind="file"`, no line numbers, no symbol ids, and a real file backing in the search trace.
- **Formatting:** added lean file output for `return_format="full"` and `return_format="locations"`.
  - `locations` prints one relative path per line.
  - `full` prints one path per line with terse annotations like language, match kind, and `test` when applicable.
- **Ranking:** exact path and exact basename stay first, with test/docs/fixture demotion layered in at execution time.
- **Telemetry:** file-mode traces serialize cleanly and keep the canonical `file_lookup` intent.
- **Matrix harness:** smoke and breadth now include file-mode cases, and file hit counts are reported as lower bounds when the result list saturates the limit.

## Why

The mined traffic showed agents were using `fast_search` to find files. Before this work, Julie had indexed file paths but no first-class path search target, so users were shoving filename lookups through content or definition search and getting avoidable misses.

## How

Built it in five slices:

- Search-target parsing and alias normalization
- Tantivy schema/index/projection support for file lookups
- Execution + formatting path with real file hits
- Telemetry and matrix wiring
- Batch validation + fresh smoke/breadth baselines
