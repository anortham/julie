## Fixed daemon lifecycle deadlock: Draining vs Stopping state collapse

Diagnosed and fixed a liveness deadlock in the daemon's stale-binary restart flow.

### Root Cause

The daemon has 4 internal lifecycle phases (Starting, Ready, Draining, Stopping), but `state_file_value()` collapsed Draining and Stopping into the same string `"stopping"`. The adapter reads this file to decide readiness:
- `"stopping"` → wait for PID exit
- But in Draining, the daemon is still accepting sessions and won't exit while sessions remain

This created a deadlock: adapter waits for PID death, daemon won't die because sessions are active, no new sessions can connect because all adapters are waiting.

### The Fix (3 lines of production code)

- **`lifecycle.rs`**: `Draining` now writes `"draining"` instead of `"stopping"` to the state file
- **`launcher.rs`**: Adapter treats `"draining"` as `Ready` in `daemon_readiness()`
- **`launcher.rs`**: `poll_for_state_change("ready", ...)` treats `"draining"` as success (Codex review caught this gap)

### Codex Review

Sent to Codex CLI for external review. Three findings:
1. **(High)** Draining during normal shutdown path (after accept loop exits) — valid but benign, adapter retry loop handles it
2. **(Medium)** `poll_for_state_change` gap — valid, fixed
3. **(Medium)** Test coverage gap for poll path — valid, added test
