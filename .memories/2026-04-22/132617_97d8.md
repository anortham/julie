## Implemented search quality recovery batch

### WHAT
Implemented the next recovery batch from the search-quality plan:
- Task 3: adaptive scoped widening in `src/tools/search/line_mode.rs`
- Task 5: token-aware line verification in `src/tools/search/query.rs`
- Added regression coverage for scoped starvation recovery, scoped no-in-scope diagnostics, tokenized file-level verification, tokenized exclusion handling, and live `LineMatchMiss` avoidance.

### WHY
The old scoped content path could zero-hit when the first ranked window was saturated by out-of-scope files, and the line verifier still used raw lowercase substring checks that disagreed with Tantivy tokenization. That left real hits stranded behind `CandidateStarvation` and `LineMatchMiss`.

### HOW
- Extracted shared helpers in `line_mode.rs` so both primary and target workspace branches use the same adaptive fetch loop and per-file filtering path.
- Added bounded scoped widening with a smaller initial scoped window and a larger hard cap.
- Kept widening limited to the case where the current window is saturated, zero-hit attribution is `FilePatternFiltered`, and no in-scope candidate path has surfaced yet.
- Reworked `line_matches` so `Tokens` and `FileLevel` strategies compare tokenized query terms against a tokenized line using `CodeTokenizer::with_default_patterns()`, while `Substring` stays literal substring matching.

### IMPACT
- Scoped candidate starvation now returns the in-scope hit instead of a diagnostic-only zero-hit.
- Scoped no-in-scope misses still classify as `NoInScopeCandidates`.
- Tokenized/stemmed file-level hits now survive line verification instead of collapsing into `LineMatchMiss`.
- Batch validation is green: narrow tests, `cargo xtask test changed`, and `cargo xtask test dogfood` all passed.
