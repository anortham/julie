## Phase 3d.2b-ii: de-pool broke storage-resolution read-path (root cause + validated fix)

The 3d.2b-ii `WorkspacePool` deletion was assumed "behavior-preserving" but was NOT. The pool used to resolve every workspace's storage as a sibling under the shared `~/.julie/indexes/` root, even during a rebind window (current≠loaded). The de-pool (worker-b2) wired `in_process_index_root` into the **write** paths (`handler.rs:1550, 1668`) but left the **read-path resolver** `workspace_index_dir_for` falling through to project-local `.julie`.

- **Symptom:** rebound-primary + secondary tests assert storage stays under shared `indexes_dir`; they resolved project-local → FAIL.
- **Root cause:** `workspace_storage_anchor()` early-returns `(current_workspace_root(), None)` for the rebound window (drops the override), and `workspace_index_dir_for` never consulted `in_process_index_root`.
- **Fix (PRODUCTION, validated):** `workspace_index_dir_for` now consults `in_process_index_root` FIRST — `index_root.parent().join(workspace_id)` (mirrors the existing `anchor_override.parent().join(...)` resolution; `in_process_index_root` == `<shared_indexes>/{primary_id}`). Single chokepoint — `workspace_storage_anchor` has only one prod caller.
- **Test-side pattern:** make handler `let mut`, set `handler.in_process_index_root = Some(indexes_dir.join(&primary_id))`.
