## T6: In-process embedding acquisition via resident 3b host

Created `src/server_in_process.rs` with `acquire_in_process_embedding_provider()` as the T6 deliverable.

### What
- New module `src/server_in_process.rs` with `acquire_in_process_embedding_provider(paths: &DaemonPaths) -> Option<Arc<dyn EmbeddingProvider>>`
- Registered as `pub mod server_in_process;` in `src/lib.rs`
- Test file `src/tests/daemon/inprocess_embedding.rs` registered in `src/tests/daemon/mod.rs`

### Why
In-process MCP sessions (Phase 3c) need embeddings from the **shared resident host** — not per-session `create_embedding_provider()` which spawns a Python sidecar per session and causes OOM under N concurrent sessions.

### How (key decisions)
- **Source:** `connect_or_spawn_host(&paths)` (shared host) — NOT `create_embedding_provider()` (per-session sidecar)
- **HARD GATE (F2 lesson from 3b):** `rpc.ensure_ready()` is the readiness gate — NOT `device_info()` / `accelerated()` which silently fall back to defaults on an unhealthy host
- **Force-disable:** `JULIE_EMBEDDING_PROVIDER=none/disabled/off` → returns `None` immediately, reusing same env var as `create_embedding_provider()`
- **Thread safety:** All blocking I/O in `tokio::task::spawn_blocking` — never on async runtime thread
- **Degradation:** Any error (connect fail, health fail, panic) → `None` (keyword-only), never hangs startup
- **handler.rs untouched** — T4 already wired `new_in_process(hint, provider, leader)`; T8 calls `acquire_in_process_embedding_provider` and passes result in

### Gate invariant
In-process embedding acquisition routes through the shared 3b host; readiness is gated on the fallible `ensure_ready()` (not silent getters), so an unhealthy host degrades to unavailable rather than false-Ready; host-down does not hang startup; N sessions share ONE resident model.
