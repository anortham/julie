## Phase 3c.3 cutover — F-A election-timing fix design (escalation-tier, lead-owned)

Three read-only Explore maps + lead analysis settled the F-A fix for the cutover.

**Why pinning (not client-root rebind):** the design doc (`2026-06-04-...daemon-teardown-design.md`) says each process **wins one OS lock at startup** — before client roots are known. That mandates hint/cwd-derived workspace identity. The `list_roots` client-root rebind in `ensure_primary_workspace_for_request` (handler.rs ~596) is daemon-era behavior that must be SUPPRESSED for in-process, or lock(startup-hint) and binding(client-root) diverge → split indexes across two leaders (F-A).

**Why not "just force Cli source":** `on_initialized`'s non-deferred path (Cli/Env, handler.rs:2871-2905) calls `run_auto_indexing → run_primary_workspace_repair`, which has **NO follower guard** (only the deferred path's `complete_deferred_auto_index_if_needed` guards followers, handler.rs:457). Forcing Cli would let a FOLLOWER write → breaks single-writer. So keep Cwd source (preserves deferral + follower guard).

**The fix (B++ + defense-in-depth):**
1. `run_in_process_server`: canonicalize `startup_hint.path` BEFORE deriving workspace_id; pass the canonicalized hint to `new_in_process` so lock id == index_root id == binding id (binding canonicalizes too via `primary_binding_for_root → canonicalize_workspace_path`).
2. Gate `ensure_primary_workspace_for_request`: `is_in_process()` handlers always `reconcile_primary_workspace_to_startup_hint` (canonical hint), skip `list_roots` rebind. Keep Cwd source → deferral + follower guard intact.
3. Defense-in-depth: `is_in_process_follower()` early-return in `run_primary_workspace_repair` → followers never write via ANY entry path.

**Cutover (T10):** flip main.rs None arm (lines 111-134) from `run_adapter` to `run_in_process_server` + project-level `.julie/logs/julie.log` tracing. (Note: `new_in_process` uses `new_deferred_daemon_startup_hint_with_project_log` — verify project log already wired.)
