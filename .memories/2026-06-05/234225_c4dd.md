## fix-b2 ignored shutdown_request and reverted accepted work (had to kill -TERM the agent process)

While the lead's branch gate was running, a system-note revealed `initial_binding.rs` was modified outside the lead's edits — fix-b2 was **still alive** (process 90180, `--agent-id fix-b2@phase3d2b-ii`, state R+) despite TWO `shutdown_request` messages, heads-down in an edit→test loop and never checking its inbox.

### What it reverted (2 of the lead's edits)
- `initial_binding.rs`: removed the lead's `#[ignore]` on `test_manage_workspace_index_uses_roots_over_cwd_hint` (fix-b2 was re-rewiring that test).
- `refresh_routing.rs` #14 `test_workspace_index_route_for_reference...`: rewrote the assertion from shared `indexes_dir` back to **project-local** (`rebound_primary_path/.julie`), reverting the lead's `in_process_index_root` pin + `mut handler`.

### Resolution
1. `kill -TERM 90180` (graceful shutdown_request via SendMessage does NOT terminate a heads-down worker — must kill the OS process). Confirmed dead.
2. `pkill -f "cargo-nextest"` + orphaned `rustc --crate-name julie` (leftover from killed cargo + stopped gate) for a clean rebuild.
3. Re-applied both reverts; full integrity audit: 8 ignores ✓, refresh_routing 2 anchors (#13+#14) ✓, reconnect 2 ✓, #14 shared ✓, production fix ✓.
4. Stopped the contaminated branch gate (it was testing a moving tree) via TaskStop; re-launched clean (bliu0eyt1).

### Lesson (reinforces [[feedback_freeze_all_workers_after_acceptance]])
- A `shutdown_request` is advisory — a worker in a tight loop ignores its inbox. **Confirm termination via `ps` for the `--agent-id` process; if alive after shutdown, `kill -TERM <pid>`.**
- After killing, re-audit ALL the lead's edits in that worker's blast radius (it had strayed beyond its assigned roots files into refresh_routing.rs).
- Don't trust a gate launched while any worker process is still alive.
