## T9 Complete: Leader-gated handoff recovery + F1 bounded in-process read envelope

Final task of PR 3c.2 (branch `julie-rescue-phase3c2`). All 7 new tests green.

### What

- **Part A — Leader-gated handoff recovery**: `complete_deferred_auto_index_if_needed` now early-returns `Ok(())` for in-process followers. Followers never trigger the writing SQLite/Tantivy repair (prevents Race Risk #2).
- **Part B — F1 bounded read envelope**: `call_tool` now wraps the entire in-process read path (`ensure_primary_workspace_for_request` + `tool_router.call`) in a single `tokio::time::timeout`. On leader: deferred auto-index is spawned as a non-cancellable `tokio::spawn` task before the timeout. Daemon/stdio (`LeadershipState::none()`) take the existing path unchanged.
- **`is_in_process()` added**: `LeadershipState::is_in_process()` + `JulieServerHandler::is_in_process()` — gate condition for the F1 envelope.

### Key design points

- `is_in_process_follower()` is the skip condition for Part A (not `!is_leader()` — that would break daemon/stdio which use `none()`)
- In-process writes (`is_write_exempt == true`) fall through to existing path — no timeout on writes
- The spawned repair task holds its own `deferred_auto_index_gate` mutex; cancelling the timeout future never cancels it
- Deadline `None` (env var "0") → no timeout → stalling future runs to completion (existing escape hatch preserved)
