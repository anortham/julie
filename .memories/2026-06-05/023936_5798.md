## T5: Gate file watcher start behind leadership (PR 3c.2)

### WHAT
Gated file-watcher startup behind `is_leader()` in two handler call sites + added defense-in-depth parameter to `start_file_watching` in julie-runtime.

### Changed files
- `crates/julie-runtime/src/workspace/mod.rs`: `start_file_watching` now takes `is_leader: bool`; early-returns for non-leaders
- `src/handler.rs` (`restore`): added `&& handler.is_leader()` to outer guard so both `initialize_file_watcher` AND `start_file_watching` are gated
- `src/handler.rs` (`initialize_workspace_with_force`): passes `self.is_leader()` to `start_file_watching`
- `src/tests/core/handler/leader_watcher.rs`: new TDD test proving invariant
- `src/tests/core/handler.rs`: registered `mod leader_watcher`

### WHY
In-process MCP server phase (3c): leader = sole watcher+writer; losers are read-only. Without the gate, ANY in-process handler would start an OS notify watcher and eventually acquire a Tantivy IndexWriter, violating cross-process correctness.

### HOW
- Two gating layers: call-site (`is_leader()`) + runtime function parameter (defense-in-depth)
- `IncrementalIndexer::new` does NOT acquire OS locks — only `start_watching()` does. So blocking `start_watching` is sufficient to prevent writer-lock acquisition.
- `restore()` path also gated: both `initialize_file_watcher` (constructs IncrementalIndexer) and `start_file_watching` (starts OS watcher) gated behind leader check

### IMPACT
Gate invariant proven by test: leader watcher running=true, loser watcher running=false, loser workspace still loaded for reads. RED→GREEN: test failed at loser assertion before implementation, passes after.

### Caller audit findings
- `initialize_file_watcher`: 2 callers (initialize_all_components, restore) — all paths covered
- `start_file_watching`: 2 callers (restore, initialize_workspace_with_force) — both updated
- `initialize_all_components`: 2 callers (JulieWorkspace::initialize, detect_and_load) — these construct IncrementalIndexer but DO NOT start it; harmless
- `start_watching`: 1 production caller (start_file_watching) — gated
- No double-start risk for leader: initialize_all_components creates IncrementalIndexer (idempotent), start_file_watching starts it (called once per initialize_workspace_with_force)
