## Cross-client readiness review (Claude Code vs Codex/OpenCode)

### WHAT
Completed a read-only audit focused on whether Julie remains effective when used from Codex/OpenCode/GPT-oriented workflows rather than Claude Code.

### WHY
User is considering switching agent/runtime and wants to avoid hidden Claude-specific assumptions that degrade onboarding or runtime behavior.

### HOW
- Reviewed top-level docs and setup instructions:
  - `README.md`
  - `docs/README.md`
  - `CLAUDE.md`
  - `JULIE_AGENT_INSTRUCTIONS.md`
- Searched code/docs for client-coupled terms (`claude`, `Claude Code`, `JULIE_WORKSPACE`, `superpowers`, `mcpServers`).
- Inspected runtime paths and server metadata behavior in:
  - `src/main.rs` (workspace detection priority)
  - `src/tools/workspace/paths.rs` and `src/tools/workspace/commands/index.rs`
  - `src/handler.rs` (`ServerInfo.instructions` loading)

### IMPACT
Identified likely migration risks without changing code:
- Docs/onboarding are heavily Claude-biased and lack explicit Codex/OpenCode quickstart.
- Server instructions are loaded from a relative file path, creating client/CWD-dependent behavior.
- Workspace root fallback to current directory can misplace `.julie` for clients that launch from non-project CWD when env vars are absent.
- Internal planning docs include Claude-specific sub-skill directives, potentially reducing cross-agent reuse.

Next step is to deliver a prioritized issue list + concrete remediation plan (docs and small runtime hardening).
