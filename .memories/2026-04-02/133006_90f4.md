## Fix: Daemon version mismatch detection + embedding model sync

Two bugs fixed in the daemon:

### Bug 1: Daemon never restarts on plugin version mismatch

**Root cause:** The VERSION header check happened inside `handle_ipc_session`, AFTER `accept_loop` had already called `sessions.add_session()`. So the daemon detected the mismatch and set `restart_pending`, but then served the entire session with the old binary. It could only restart after the session ended.

**Fix:** Moved IPC header reading and version check into `accept_loop`, BEFORE `sessions.add_session()`. Now follows the same pattern as the binary mtime check: if a version mismatch is detected and the daemon is idle (0 sessions), it shuts down immediately. The adapter sees the broken pipe, exits, and the MCP client restarts it with the new binary.

- `handle_ipc_session` now takes `workspace_path: PathBuf` instead of reading headers internally
- Header reading has a 5s timeout with graceful fallback (continue loop, don't crash)

### Bug 2: Dashboard shows stale embedding model name (jina-code instead of CodeRankEmbed)

**Root cause:** `daemon.db`'s `embedding_model` column was only set during backfill when the value was `None`. Once set to "jina-code", neither the daemon startup backfill nor the handler backfill would ever update it, even after the embedding pipeline re-embedded with a different model.

**Fix (two locations):**
1. `spawn_workspace_embedding`: captures provider model name before `spawn_blocking`, writes it to daemon.db after pipeline completion (alongside existing vector_count update)
2. Daemon startup backfill: changed from `is_none()` check to `!= current_model` comparison, so stale model names are corrected on restart
