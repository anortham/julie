## Fixed call_path review findings

### WHAT
- Added receiver-aware Rust extraction so non-self receiver calls such as `params.call_tool(self)` enter structured pending resolution instead of binding to a same-file method with the same name.
- Added resolver scoring from scoped identifiers plus caller signature/code-context type mentions, with token-set matching to avoid full-repo indexing blowups.
- Hardened `call_path` output, hop limits, deterministic BFS ordering, hop target identity, schema descriptions, and JSON Schema bounds.
- Added a named `call-path` CLI wrapper and avoided `--workspace` collision by using `--target-workspace` for tool workspace targeting.

### WHY
The review found that `call_path` could not trace from the MCP handler into `CallPathTool::call_tool`, returned plain text diagnostics, accepted unbounded hops, produced row-order-dependent paths, hid target identity, lacked schema detail, and lacked a direct CLI path.

### HOW
- Rust extractor now keeps `self.foo()` local but sends non-self receiver calls through structured resolution.
- Resolver uses caller scope identifiers and caller signature/code-context tokens to prefer the receiver type parent, then keeps the token matching linear in text size.
- Call path response/error handling and CLI conversions gained regression tests.
- Dogfood with a tiny standalone Rust workspace verifies `call-path call_path call_tool --from-file ... --to-file ...` returns a one-hop path to the cross-file tool method.

### IMPACT
Agents get parseable `call_path` responses, bounded traversal, stable path selection, richer hop locations, a usable named CLI wrapper, and the original handler-to-tool route now resolves through the intended tool implementation.
