## Claude review follow-up applied

### WHAT
Closed the only actionable low-severity issue from Claude CLI's review of the cleanup plan/fix slice:
- `MultiFileTransaction::commit_all` now acquires the same normalized per-path edit locks as `EditingTransaction` before validation/temp writes/renames.
- Added a deterministic race test proving a paused multi-file commit serializes with a concurrent checked single-file writer.
- Added handler assertions pinning `applied=false` on prepare-failure metadata, which was already true via base metadata.
- Added a warning when `.h` language detection cannot initialize C/C++ parser comparison before using the conservative fallback.
- Updated the cleanup plan ledger with Claude review and follow-up verification.

### WHY
Claude's independent review approved the branch but identified that the new same-process edit lock initially covered only single-file transactions. Multi-file transactions share the same source-file mutation surface and should not bypass that serialization.

### HOW
Implemented sorted lock acquisition by normalized path in `MultiFileTransaction::commit_all` to avoid deadlocks across multi-file operations. Used test-only hooks to reproduce the previous interleaving and verify the checked single-file writer now rejects stale content.

### IMPACT
Same-process Julie-originated writes now serialize consistently across single-file and multi-file edit transaction classes. The plan has external verification evidence and follow-up rows.
