## Candidate 10 resolver language metadata

### WHAT
Completed the remaining Candidate 10 resolver slice.

- Added `SymbolDatabase::get_file_languages_by_paths(&[&str])`
- Batch-loaded caller file languages in `resolve_structured_batch`
- Threaded caller language into same-language scoring, namespace scoring, and Rust re-export selection
- Preserved extension fallback when caller file metadata is absent or empty
- Added DB coverage for batch language lookup
- Added resolver coverage for source-aware C++ `.h` callers and missing-row fallback
- Updated cleanup plan progress and verification ledger

### WHY
Source-aware detection fixed indexing/rewrite paths, but relationship resolution still used extension-only caller language. C++ `.h` callers could still be scored as C and prefer a same-directory C decoy over the correct C++ target.

### HOW
Kept resolver behavior local: one DB lookup per batch, one `caller_language` value passed through existing scoring seams, extension fallback only when metadata is unavailable. Fresh Claude review approved the slice; follow-up documented the file-row-before-resolution dependency and pinned fallback behavior.

### IMPACT
Relationship resolution now honors stored source-aware file language. The remaining cleanup work moves to Candidate 12 harness helpers and later Candidate 11 structural splits.
