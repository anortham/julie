## Candidate 10 First Slice: Source-aware .h language detection

### WHAT
- Exposed public `julie_extractors::language::detect_language_for_source(file_path, content)`.
- Moved the existing private `.h` C++ heuristic out of the pipeline into shared extractor language detection.
- Added app-side `detect_language_for_indexing_with_content(path, content)` and wired it into watcher, indexing-core extraction, and `rewrite_symbol` live parsing.
- Added targeted tests for public API, app file policy, indexing metadata, watcher storage, and rewrite body replacement on C++ `.h`.

### WHY
Path-only `.h` detection defaulted to C while the extractor pipeline could parse C++ headers, causing stored file language and live parser selection to diverge from extracted symbols.

### HOW
- Preserved `detect_language_from_extension` for path-only policy.
- Reused one public source-aware detector from extractor pipeline and app wrappers, avoiding duplicated `.h` heuristics.
- Recomputed source-aware language after file content is read in indexing and watcher paths.
- `rewrite_symbol` now selects the live tree-sitter parser using source content.

### IMPACT
- C++ `.h` files with C++ syntax now store `cpp` language consistently in batch indexing and watcher updates.
- `rewrite_symbol replace_body` on inline C++ header methods succeeds with the C++ parser.
- Public MCP JSON/output shape unchanged except language/parser correctness.
