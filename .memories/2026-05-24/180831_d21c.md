## Candidate 9 Prepare-Once Editing Flow

### WHAT
Implemented prepare-once flow for `edit_file` and `rewrite_symbol` handler invocations.

- Added `PreparedEdit` and `PreparedRewrite` state objects.
- Added metadata/render/apply helpers that consume prepared state.
- Rewired handler wrappers to prepare once, derive success/failure metrics from that state, then execute from the same state.
- Preserved direct `call_tool` and `success_metrics_metadata` wrappers for non-handler callers.
- Added focused tests for prepared-state metrics, dry-run output parity, stale target rejection, and edit failure metadata fields.

### WHY
Handler wrappers were preparing editing work once for metrics and then `call_tool` prepared again for execution. That duplicated resolution, reads, parsing/apply planning, and created a possible gap between measured state and executed state.

### HOW
`prepare_edit`/`prepare_rewrite` now centralize resolve/read/apply planning. Handlers call them once and pass the prepared object to `success_metrics_metadata_from_prepared` and `call_prepared`. Direct tool APIs continue to prepare internally.

### IMPACT
Metrics and execution now share one prepared state per MCP handler invocation. Dry-run/apply output remains covered by existing suites plus new parity tests. Changed-file-before-commit behavior remains guarded for both edit and rewrite prepared apply paths.
