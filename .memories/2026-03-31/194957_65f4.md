## Standardized embedding pipeline on Python sidecar, removed ORT backend

Investigated a DirectML GPU acceleration failure that was bogging down the machine during embedding. Root cause: DirectML's fused `LayerNormFusion` operator broken at the system/driver level on this Windows machine (worked fine on 3/26-3/27, broke by 3/31). Both ORT and PyTorch hit the same `E_INVALIDARG` through different code paths, confirming it was a DirectML runtime issue, not a code bug.

### What was done

- **Diagnosed DirectML failure**: Added warmup probe to `init.rs` that catches GPU failures at init instead of mid-pipeline. Confirmed failure is system-level, not code.
- **Switched to CUDA via Python sidecar**: Installed `torch+cu124` in the sidecar venv. CodeRankEmbed on CUDA runs at ~2s/batch vs ~40s/batch on CPU fallback.
- **Removed ORT backend entirely** (6 tasks, 10 commits, net -2,601 lines):
  - Deleted `ort_provider.rs`, `windows_directml.rs`, 5 ORT-only test files
  - Removed `fastembed` and `ort` crate dependencies (-1,576 lines of lockfile)
  - Added CUDA auto-detection to sidecar bootstrap (`detect_nvidia_cuda()` via nvidia-smi)
  - Simplified factory/init to sidecar-only resolution
  - Ported 8 integration tests from ORT to sidecar provider
  - Updated CI, README, CLAUDE.md

### GPU support matrix (sidecar-only)

| Platform | GPU Backend | How |
|----------|------------|-----|
| Windows/Linux + NVIDIA | CUDA | Auto-detected, torch+cu124 installed at bootstrap |
| Windows + AMD/Intel | DirectML | via torch-directml (already in pyproject.toml) |
| macOS Apple Silicon | MPS | Native PyTorch support |
| No GPU / broken GPU | CPU | Automatic fallback |
