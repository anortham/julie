## Sidecar Embedding Bootstrap Pipeline - Complete Architecture Analysis

### WHAT
Comprehensive investigation of Julie's embedding provider architecture for planning CUDA torch support on Windows. Examined the full pipeline from Rust bootstrap through Python sidecar runtime detection.

### KEY FINDINGS

#### 1. **Sidecar Bootstrap (sidecar_bootstrap.rs)**
**ensure_sidecar_package_installed()** flow:
- Validates venv exists and has marker file (v9-vram-aware-batching)
- Installs via `uv pip install --editable .[runtime]` (preferred) or `pip install`
- Marker stored in venv_path/.julie-sidecar-install-root
- **Important**: No CUDA detection happens here — pure venv setup

Current bootstrap steps:
1. Check if marker exists and matches expected hash
2. If not, run pip install with editable pyproject.toml extras
3. Write marker file

**For CUDA support, would need to**:
- Add CUDA probe AFTER venv creation but BEFORE package install
- Conditionally inject `--index-url` or `--extra-index-url` for PyTorch CUDA wheels
- Detect nvidia-smi or check CUDA environment variables
- Pass device info to Python via env vars

**Current env vars available**:
- `JULIE_EMBEDDING_SIDECAR_BOOTSTRAP_PYTHON` - override bootstrap Python
- `JULIE_EMBEDDING_SIDECAR_VENV` - custom venv path
- `JULIE_EMBEDDING_SIDECAR_PROGRAM` - custom sidecar program
- `JULIE_EMBEDDING_SIDECAR_RAW_PROGRAM_ENV` - bypass module/script args
- `JULIE_EMBEDDING_SIDECAR_MODULE` - custom module name
- `JULIE_EMBEDDING_CACHE_DIR` - cache base dir

#### 2. **Sidecar Supervisor (sidecar_supervisor.rs)**
**build_sidecar_launch_config()** construction:
- Calls ensure_venv_exists() → ensure_sidecar_package_installed()
- Returns SidecarLaunchConfig with:
  - `program`: venv Python executable path
  - `args`: ["-m", "sidecar.main"] or script override
  - `env`: Empty vec (no env vars passed to sidecar!)

**Important gap**: Launch config has no way to pass device preference to Python sidecar. This would need to be added.

Proposed extension:
```rust
// In SidecarLaunchConfig or env setup:
env.push((
    OsString::from("JULIE_EMBEDDING_DEVICE"),
    OsString::from("cuda"), // or "cpu", "directml"
));
```

#### 3. **Python Dependencies (pyproject.toml)**
Current [runtime] optional deps:
```
torch>=2.0.0
sentence-transformers>=3.0.0
einops
torch-directml; sys_platform == 'win32'  ← Windows GPU only!
wmi; sys_platform == 'win32'  ← For VRAM detection via WMI
```

**For CUDA Windows support**:
- Remove windows-only torch-directml conditional
- Add conditional: `torch-cuda` or use PyTorch index URL selection
- torch-directml can co-exist with CUDA torch if needed
- Need to add `nvidia-smi` probing capability

#### 4. **Runtime Device Selection (runtime.py)**

**_select_device() chain (lines 22-41)**:
```
1. torch.cuda.is_available() → "cuda"
2. dml_module.is_available() → str(dml_device())
3. torch.backends.mps.is_available() → "mps"
4. fallback → "cpu"
```

**This already supports CUDA!** The issue is:
- Needs CUDA-enabled torch (torch+cu124) not installed
- DirectML is fallback but won't provide CUDA even if GPU present
- MPS (macOS) comes before checking DirectML

**For Windows CUDA**:
- Python-side code is ready if torch[cuda] is installed
- Bootstrap needs to detect CUDA and install right torch variant

**_detect_gpu_vram_bytes() (lines 123-186)** already has CUDA support:
```python
if telemetry_device == "cuda":
    return torch_module.cuda.get_device_properties(0).total_memory
```

And Windows CUDA fallback to nvidia-smi:
```python
subprocess.run(["nvidia-smi", "--query-gpu=memory.total", ...])
```

**Probe/fallback chain (lines 410-463)**:
1. Probe on selected device (GPU or CPU)
2. If GPU fails: try patching DirectML ops (.contiguous() for RoPE)
3. If patch fails: rebuild model on CPU
4. If CPU fails: FATAL error

**This already handles GPU→CPU fallback gracefully!**

#### 5. **Factory Resolution (factory.rs)**

**resolve_backend_preference()** (lines 109-162):
- Windows Auto mode: prefers ORT, falls back to sidecar
- Other platforms: prefers sidecar, falls back to ORT
- Both checked against `BackendResolverCapabilities`

**For ORT removal with sidecar-only approach**:
```rust
// Current Windows logic:
if cfg!(target_os = "windows") {
    if capabilities.ort_available { EmbeddingBackend::Ort }
    else if capabilities.sidecar_available { EmbeddingBackend::Sidecar }

// New sidecar-primary Windows:
if cfg!(target_os = "windows") {
    if capabilities.sidecar_available { EmbeddingBackend::Sidecar }
    else if capabilities.ort_available { EmbeddingBackend::Ort } // optional feature flag
}
```

**BackendResolverCapabilities** structure:
- `sidecar_available: bool` - cfg!(feature = "embeddings-sidecar")
- `ort_available: bool` - cfg!(feature = "embeddings-ort")
- `target_os`, `target_arch` - platform detection

#### 6. **Initialization Flow (init.rs)**

**create_embedding_provider()** full pipeline:
1. Read JULIE_EMBEDDING_PROVIDER env var (default "auto")
2. Parse preference → EmbeddingBackend enum
3. Resolve backend with platform capabilities
4. Try EmbeddingProviderFactory::create()
5. Warmup probe on GPU (catches DirectML op failures early)
6. Check degraded_reason and accelerated flags
7. If strict acceleration enabled: fail if not GPU or has reason
8. Fallback attempt (sidecar→ORT) only if auto mode and init failed
9. Return (provider, status) tuple

**Fallback chain** (lines 175-240):
- Only triggered if auto mode AND init failure
- Sidecar fails → tries ORT
- ORT fails → returns None
- Both failures still return status with degraded_reason

#### 7. **Sidecar Provider (sidecar_provider.rs)**

**SidecarEmbeddingProvider** lifecycle:
- Spawns Python child process via spawn_process()
- Health check via `health` RPC (timeout: 120s default)
- Consecutive fatal failure circuit breaker (threshold: 3)
- Process respawned on connection-fatal errors

**Launch command construction**:
```
[venv_python, "-m", "sidecar.main"]
```
With env vars from launch_config.env (currently empty!)

**Important**: If we add CUDA detection, must pass device hint via env vars here.

#### 8. **ORT Provider Integration (ort_provider.rs)**

**resolve_ort_model()** (lines 31-52):
- Windows default: jina-code-v2 (768d)
- Other platforms: bge-small-en-v1.5 (384d)
- Override via JULIE_EMBEDDING_ORT_MODEL_ID

**ExecutionProviderPolicy** for Windows:
```rust
#[cfg(target_os = "windows")]
use ort::execution_providers::DirectMLExecutionProvider;
```

**run_with_cpu_fallback()** (lines 113-160):
- Tries GPU (DirectML on Windows)
- On failure: rebuilds model on CPU
- Marks as degraded with reason
- Subsequent calls skip expensive rebuild

### IMPACT & MIGRATION PATH

#### What simplifies:
- No DirectML/Windows-specific ORT setup needed
- Python-side code already handles CUDA
- Sidecar has unified device selection logic
- Fallback chain still works (GPU→CPU)

#### What breaks:
- Windows users relying on ORT/DirectML backend
- Any code checking for ORT feature flag
- Existing ORT model ID configs (JULIE_EMBEDDING_ORT_MODEL_ID)

#### Backward compat approach:
1. Keep embeddings-ort feature flag, but make sidecar-first on Windows
2. Fallback to ORT still available if sidecar init fails
3. Document: "CUDA torch on Windows preferred, ORT available as fallback"
4. Version bump with deprecation notice

#### Test coverage needed:
- CUDA detection in bootstrap (mock nvidia-smi)
- Correct torch[cuda] installed via --index-url
- Device reporting matches actual GPU
- Fallback still works (GPU fail → CPU)
- Non-CUDA systems still work (falls back to CPU)

### CRITICAL DISCOVERY

The Python sidecar's device selection and fallback logic is **already production-grade and handles CUDA**. The gap is purely in the **bootstrap phase**: we need to:

1. Detect CUDA at venv creation time
2. Install torch with CUDA wheels (not DirectML)
3. Pass device preference to Python runtime

Everything else (probe, fallback, batching, VRAM detection) is ready.
