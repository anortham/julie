## Commit 4: swap-in-progress guards on `open` and `refresh` workspace operations

Closed Findings #28 and #29 from `ROOTS_IMPL_REVIEW_NOTES.md`: both commands mutated `handler.workspace` / `session_workspace.primary_binding` outside `PrimarySwapRollback` machinery, creating a window where concurrent swap and open/refresh could clobber half-applied state.

### What
- **`handle_open_command`:** entry-point guard — returns `Err` when `is_primary_workspace_swap_in_progress()` is set. Simplest, most conservative fix (secondary opens during the brief swap window are refused too, but that's acceptable for correctness).
- **`handle_refresh_command`:** unconditional entry-point guard plus defensive re-check before the post-refresh `initialize_workspace_with_force` rebind. Async code yields between those two points; if a swap started during `refresh_workspace_internal`, the re-check catches it.
- Guards intentionally do NOT condition on "target == current primary" — `current_workspace_id()` returns `None` during a swap, so such a check would be a no-op.

### Why
Review notes said "concurrent `ensure_primary_workspace_for_request` mid-swap races and can partially clobber swap state". Those code paths are the exact bug shape the swap machinery was built to prevent. Must close on this branch.

### How
TDD cycle:
1. Wrote 2 regression tests (`test_manage_workspace_open_refuses_while_primary_swap_in_progress`, `test_manage_workspace_refresh_refuses_primary_mutation_while_swap_in_progress`) — both RED.
2. First attempt conditioned refresh guard on `targets_current_primary` — failed because `current_workspace_id()` returns `None` during swap. Simplified to unconditional guard.
3. Both tests GREEN.
