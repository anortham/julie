## v6 Phase 1 Implementation Complete

### What
Implemented the full daemon + adapter architecture for Julie v6 Phase 1. 12 commits, 33 files changed, 3,382 lines added across 13 tasks.

### Components Built
- **DaemonPaths**: Centralized path resolution for `~/.julie/` layout (14 tests)
- **PID file lifecycle**: Atomic create, validate, stale cleanup (11 tests)
- **WorkspacePool**: Shared workspace state with tokio RwLock, double-checked locking (8 tests)
- **Handler refactoring**: `new_with_shared_workspace()` constructor, auto-indexing guard (5 tests)
- **CLI subcommands**: daemon/stop/status/restart via clap (12 tests)
- **IPC transport**: Unix domain socket listener/connector (6 tests)
- **Daemon server**: IPC accept loop, workspace header protocol, MCP serving over UnixStream (10 tests)
- **Adapter**: Auto-start daemon (advisory file lock), byte forwarding, reconnect logic (6 tests)
- **Stop/status/restart**: Lifecycle commands with SIGTERM (4 tests)
- **Index migration**: Copy-validate-delete with state tracking (9 tests)
- **Integration tests**: 5 end-to-end tests covering full daemon lifecycle

### Key Technical Decisions
- rmcp 1.2's `IntoTransport` blanket impl means `handler.serve(unix_stream)` just works
- Workspace header protocol: adapter sends `WORKSPACE:{path}\n` before MCP traffic
- tokio::sync::RwLock in WorkspacePool (not std) to avoid blocking async runtime
- `index_root_override` field on JulieWorkspace redirects to centralized storage

### Test Results
- Dev test tier: 6/7 buckets pass (1 timeout is pre-existing HuggingFace download)
- All new daemon/adapter/migration tests: 85+ tests passing

### Impact
Multi-client Julie is now structurally possible. The daemon owns indexes, a single VRAM footprint, and file watchers. Multiple adapters can connect simultaneously.
