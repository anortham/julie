## v6 Phase 1 Implementation Plan Complete

### What
Wrote 13-task implementation plan for the daemon + adapter architecture, plus a review findings document addressing critical gaps.

### Why
Need a concrete, executable plan before starting implementation. The plan went through a full review cycle that caught 3 critical issues, 7 major issues, and 7 minor issues.

### How
- Explored full codebase structure via Julie tools (handler, workspace, CLI, watcher, commands)
- Wrote detailed TDD plan with exact file paths, code sketches, and test commands
- Plan reviewer caught key issues: IPC-to-MCP bridge was hand-waved, RwLock in async context, missing spec coverage
- Created companion review-findings document with solutions for all critical/major issues

### Key Decisions in the Plan
- **Task 0 (new)**: rmcp API research spike before any code changes
- **Workspace header protocol**: Adapter sends `WORKSPACE:{path}\n` before MCP traffic so daemon knows which workspace to serve
- **rmcp transport-async-rw**: Used to serve MCP over IPC streams (not just stdio)
- **tokio::sync::RwLock**: In WorkspacePool instead of std::sync::RwLock for async safety
- **index_root_override field**: On JulieWorkspace to redirect index paths to central location
