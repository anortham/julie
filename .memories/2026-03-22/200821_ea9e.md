## v6 Phase 1 Complete: Daemon + Adapter Architecture Shipped

### What
Full implementation of Julie v6 Phase 1: persistent daemon with thin stdio adapters, centralized index storage, and per-project logging. Merged to main, rebuilt release, and dogfood-verified.

### Session Summary
This was a marathon session covering the entire v6 arc:

1. **Research phase**: Investigated Claude Code agent teams, MCP transport options, LSP multi-client patterns (ra-multiplex), rmcp crate capabilities. 3 parallel research agents.
2. **Design phase**: Wrote and twice-reviewed architecture spec (`docs/superpowers/specs/2026-03-22-v6-daemon-adapter-architecture-design.md`). Addressed 3 critical, 7 major issues from spec review.
3. **Planning phase**: Wrote 13-task implementation plan with review findings document.
4. **Implementation phase**: Subagent-driven execution across 13 tasks, many parallelized. 12 commits, 33 files, ~3,500 lines.
5. **Dogfood phase**: Merged to main, rebuilt release, restarted. Daemon auto-started successfully. Found project logging gap (daemon wrote all logs to `~/.julie/daemon.log` instead of also writing to project logs). Fixed it immediately.

### Architecture Delivered
- `julie-server` (no args) = adapter mode (auto-starts daemon, byte-forwards stdio to IPC)
- `julie-server daemon` = persistent daemon (IPC + future HTTP, owns all resources)
- `julie-server stop/status/restart` = lifecycle commands
- `~/.julie/indexes/{workspace_id}/` = centralized index storage
- `{project}/.julie/logs/` = per-project tool call logs (via ProjectLog)
- `~/.julie/daemon.log.*` = daemon lifecycle logs
- WorkspacePool with tokio RwLock for shared workspace state
- Auto-indexing guard prevents duplicate indexing across sessions
- Safe index migration (copy-validate-delete) from per-project to centralized
