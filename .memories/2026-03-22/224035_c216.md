## v6 Phase 2 Design + Implementation Plan Complete

### What
Brainstormed, designed, and planned v6 Phase 2: Shared Workspaces, Persistent Registry & Metrics. Two documents committed:
- **Spec:** `docs/superpowers/specs/2026-03-22-v6-phase2-shared-workspaces-design.md`
- **Plan:** `docs/superpowers/plans/2026-03-22-v6-phase2-shared-workspaces.md`

### Why
Phase 1 (daemon + adapter) shipped but workspace state was in-memory only, reference workspaces still duplicated indexes, file watchers multiplied per session, and metrics disappeared on reindex. Users couldn't track codehealth improvement over time or see evidence that Julie was delivering value.

### How
- **daemon.db** replaces planned `registry.json` with SQLite (WAL mode, ACID): workspaces table, workspace_references, codehealth_snapshots, tool_calls
- **WorkspacePool** backed by persistent storage; workspaces survive daemon restarts
- **WatcherPool** with ref-counted IncrementalIndexers and 5-minute grace period
- **Codehealth snapshots** auto-captured after indexing; trend comparison in query_metrics
- **Tool calls** redirected to daemon.db with workspace_id; 90-day retention
- **WorkspaceRegistryService** (916 lines) deprecated, replaced by DaemonDatabase
- Three-track agent team plan: Registry (A1-A8), Watchers (B1-B4), Routing (C1-C6), plus Integration (I1-I2)

### Key Decisions
- `daemon.db` over `registry.json` for ACID guarantees and natural metrics home
- Internal `Mutex<Connection>` pattern (matches existing SymbolDatabase usage)
- No non-daemon fallback for new features (Option on handler, silently skip)
- Codehealth snapshots are per-indexing-pass, automatic, never pruned
- Tool calls get 90-day rolling retention, pruned on daemon startup
- session_count reset to 0 on daemon startup (crash recovery)
