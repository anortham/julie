## Hermes-agent validates SQL perf fixes; next bottlenecks identified

### WHAT
Re-indexed `~/source/hermes-agent` cold (2,406 source files, 155,907 symbols, 235,981 identifiers, 16,112 types, 43,711 relationships) to validate the in-flight SQL perf fixes from the 21:05 checkpoint earlier today and identify the next bottlenecks. The 16-file unstaged diff (+1047/-162) is now battle-tested under 2x-the-symbols load.

### WHY
Yesterday this same workspace exhibited cartesian products, N+1 query loops, and a 32k-parameter IN clause that effectively crashed indexing. Today's fixes addressed those — needed real-world validation on the largest workspace Julie has ever indexed.

### HOW
- Verified daemon was running the post-fix binary (release built 18:00, daemon started 18:23)
- Set up Monitor on `~/.julie/daemon.log.2026-04-25` with phase/timing filter

### FOUR INVESTIGATIONS — ROOT CAUSES

**#1 Tantivy projection (29.3s)** — `src/search/projection.rs:199-264 project_documents` calls `load_symbol_contexts_from_database` (line 235) then `apply_documents_with_context` (line 236) with NO timing markers between them. The `apply_documents_with_context` (line 325-355) is a single-threaded loop of 156k `index.add_symbol_with_context` calls. Need split timing first; then likely Tantivy IndexWriter thread count is the lever.

**#2 Tree-sitter extraction (57s) — SERIAL** — `src/tools/workspace/indexing/pipeline.rs:157-236 extract_index_batch`: outer loop over 16 languages, inner loop calls `process_file_with_parser(...).await` one file at a time. Inside `processor.rs:187-198`, work IS spawn_blocking'd but immediately awaited. Net: 1 file at a time on what's effectively 1 core. Apple Silicon parallelism could take 57s → ~10-12s. **Single biggest win available.**

**#3 Long-line heuristic false positive on .md** — `src/tools/workspace/indexing/processor.rs:135-168` applies the minified/generated check to every language. Markdown prose with long lines triggers it (caught 2 SKILL.md files). Fix: skip the heuristic for `markdown` (and maybe `html`); existing `NON_EMBEDDABLE_LANGUAGES` const at `src/embeddings/metadata.rs:63` is the model.

**#4 Embedding selection log lies** — Math is correct. `src/embeddings/metadata.rs:257-320 select_budgeted_variables` applies per-language cap then a hard-coded `truncate(5000)` global cap (line 314). The log at `src/embeddings/pipeline.rs:317` reports `budget_cap=2042` (per-language) which makes selected_count=5000 look wrong. Two fixes: (a) rewrite log line to distinguish per_language_cap vs global_cap, (b) lift the magic 5000 into a named constant.
