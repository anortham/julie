## v7.4.1: Embedding Catch-up Fix

### WHAT
Fixed a bug where workspaces indexed before the embedding sidecar was ready would permanently have zero embeddings. Added catch-up detection in both the index and refresh paths, plus a guard against cancelling in-progress embedding tasks.

### WHY
The daemon starts indexing and initializing the sidecar in parallel. If indexing completes before the sidecar publishes "Ready", the embedding gate sees `db_mutated = false` on every subsequent incremental check and skips embedding forever. Discovered while testing with ~/source/hermes-agent (155K symbols, 0 embeddings).

### HOW
- Added `embedding_count()` check in `handle_index_command` and `refresh_workspace_internal` before skipping the embedding pipeline
- If symbols > 0 and embeddings == 0 and no embedding task is already running, schedule catch-up embedding
- Primary workspace uses the handler's open DB; non-primary opens a fresh connection to the target DB
- Codex review caught a race condition: catch-up could cancel an in-progress pipeline that hadn't stored its first batch yet. Added `embedding_tasks` guard to prevent this.

### IMPACT
Workspaces that missed initial embedding now self-heal on the next session connect or refresh. Also includes the test intelligence foundation changes from earlier today.
