## C# extractor identifier pattern analysis: only property was buggy

Investigated all 6 `.find(|c| c.kind() == "identifier")` patterns in `csharp/members.rs` to determine if they share the property name extraction bug fixed in v6.0.6.

### Findings: all other patterns are safe

- **extract_method (line 21):** Uses `.rev()` to search backwards from `parameter_list`, so it naturally finds the method name (last identifier before params), not the return type
- **extract_constructor (line 112):** No return type in constructors, first identifier IS the name
- **extract_destructor (line 176):** No return type, first identifier IS the class name
- **extract_field (line 326):** Searches inside `variable_declarator` (child node), not `field_declaration`. Type lives on the parent `variable_declaration` node.
- **extract_event (line 401):** Same as field, searches inside `variable_declarator`
- **extract_delegate (lines 449-454):** Explicitly handles 1 vs 2 identifier case

### Why property was uniquely vulnerable

Property declarations are the only C# member where the type and name are **sibling children** of the same AST node, both can be `identifier` nodes, and there's no structural landmark (like `parameter_list`) to anchor the search. Every other member type either lacks a return type, uses `.rev()`, or has the type on a different tree level.
