## v6.0.2 Release Binary Live on Windows

### What
Rebuilt release binary with v6.0.2 fixes and restarted Claude Code. Julie MCP server connects successfully. Forced reindex confirmed DirectML embeddings working at full GPU speed.

### Session Summary (3 critical fixes)

**1. Windows daemon connection (3 bugs)**
- `is_process_alive()` was a `false` stub on Windows, causing duplicate daemons
- `IpcConnector::connect()` didn't retry on `ERROR_PIPE_BUSY` (231)
- `wait_for_socket()` probe consumed pipe instances; timeout too short (10s vs 26s ORT init)
- Fixed with: `OpenProcess` API, exponential backoff retry, `WaitNamedPipeW` non-consuming probe, 45s timeout

**2. DirectML VRAM overflow (34x speedup)**
- `embed(texts, None)` sent 250 texts in one ORT call; long texts forced padding that exceeded 6GB VRAM
- DirectML silently fell back to CPU for oversized batches (115s vs 8s per batch)
- Fixed with `Some(32)` sub-batch size: all 20 batches consistent ~4s on GPU, 70s total vs ~40min

**3. Additional changes made by user between sessions**
- `accept()` reordered to create replacement pipe BEFORE `connect()` (state safety)
- `run_adapter()` simplified: removed reconnect logic, uses `block_in_place` for ensure_daemon_running
- `wait_for_socket` switched from `OpenOptions::open` to `WaitNamedPipeW` (non-consuming probe)
- `daemon_ipc_addr()` now uses FNV-1a hash of julie_home for pipe name isolation
- `create_exclusive()` gained retry limit (MAX_RETRIES=10)
- Workspace header normalized to forward slashes
- `drain_sessions` switched to `Notify`-based condvar pattern
- `is_transient_accept_error` added WSAEMFILE (10024) for Windows
