## Completed Task 5 for JS/TS pending relationship precision

Implemented Task 5 in the tree-sitter hardening worktree by plumbing structured unresolved relationship context through canonical extraction results and migrating JavaScript and TypeScript pending call extraction to emit structured targets.

- **What changed:** `ExtractionResults` now carries `structured_pending_relationships` alongside legacy `pending_relationships`
- **JS/TS migration:** `JavaScriptExtractor` and `TypeScriptExtractor` now build `UnresolvedTarget` values for direct imported calls and receiver-qualified member calls, then emit `StructuredPendingRelationship` records via `BaseExtractor::create_pending_relationship`
- **Compatibility:** Legacy pending relationships are still populated for downstream callers by degrading structured records into `PendingRelationship`
- **Registry path:** JS/TS/JSX/TSX now use explicit registry entrypoints so canonical extraction can return structured pending data without forcing a repo-wide extractor API migration
- **Verification:** Added regression assertions in JS/TS cross-file relationship tests for member-call receiver context and imported-call context; narrow cargo tests passed; `cargo xtask test dev` passed with all 9 buckets green

## Why

Task 4 introduced a structured unresolved model, but that information was being dropped before it escaped extraction. Without this plumbing, Task 5 would have been fake progress because JS/TS would still collapse precise unresolved targets down to flat names.

## How

- Added `structured_pending_relationships` to canonical results and normalization/rekey flows
- Kept legacy compatibility instead of breaking downstream consumers
- Used bounded file-level import scanning as a fallback for TypeScript import-context recovery, because TS import symbols do not currently preserve local binding names the same way JavaScript does
- Updated cross-file JS/TS tests to validate both degraded legacy names and structured target metadata

## Impact

- JS/TS pending call extraction now preserves receiver-qualified targets like `calc.double`
- Imported cross-file calls now retain structured import context in both JS and TS
- Canonical pipeline parity remains intact after the result-shape expansion
- `cargo xtask test dev` passed after the change set, so the Task 5 work is integrated into the broader branch safely
