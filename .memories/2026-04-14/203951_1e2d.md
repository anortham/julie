## Completed Task 6 OO-wave structured pending migration

Finished the Task 6 language wave in the tree-sitter hardening worktree by extending the structured unresolved relationship model beyond JS/TS to the OO extractor set.

- **Languages migrated:** Java, C#, Kotlin, Swift, PHP, Scala
- **What changed:** each migrated extractor now stores `structured_pending_relationships` alongside legacy pending records and exposes them to the canonical registry path
- **Call precision:** unresolved call edges now preserve receiver-qualified targets where the grammar exposes them, such as `Helper.process`, `Helper.Process`, `helper.compute`, `calc.double`, and similar forms
- **Inheritance precision:** unresolved extends/implements/conformance edges now preserve structured targets instead of flat strings only
- **Registry plumbing:** Java, C#, Kotlin, Swift, PHP, and Scala now use explicit registry entrypoints so canonical `ExtractionResults` returns structured pending data for those languages too
- **Tests:** updated the existing OO cross-file suites and added Scala-specific structured pending tests

## Why

Task 5 proved the structured unresolved model for JS/TS, but the next major risk cluster was the OO extractor wave. Those languages had many of the same false-resolution and information-loss patterns: member calls collapsed to terminal names and inheritance targets dropped precision before leaving extraction.

## How

- Added structured-pending storage/getters to each migrated extractor
- Switched unresolved relationship emission over to `BaseExtractor::create_pending_relationship`
- Kept the legacy `pending_relationships` list populated by degrading structured records for compatibility
- Added language-local unresolved-target builders rather than forcing a broad shared trait refactor in the middle of the migration
- Added Scala coverage with direct extractor tests because it did not already have a cross-file pending suite

## Impact

- The OO extractor wave now participates in the same structured unresolved relationship model as JS/TS
- Canonical extraction, public API parity, and the dev verification tier all stayed green after the migration
- Remaining warning noise in the branch is down to the pre-existing unused `warn` import in `crates/julie-extractors/src/base/extractor.rs`
