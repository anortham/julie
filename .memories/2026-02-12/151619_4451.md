## RTK Project Architecture & Integration Analysis

### 1. rtk init.rs - Initialization System

**Purpose**: Sets up RTK integration with Claude Code via hooks and instructions.

**Key Components**:
- `RTK_INSTRUCTIONS` constant: 137-line markdown block with full command reference (embedded in code)
- `RTK_SLIM` constant: 10-line slim reference from `hooks/rtk-awareness.md`
- `REWRITE_HOOK`: Hook script from `hooks/rtk-rewrite.sh` (embedded)

**Three Setup Modes**:

1. **Default mode** (Unix/macOS/Linux - recommended):
   - Installs hook to `~/.claude/hooks/rtk-rewrite.sh`
   - Writes slim `RTK.md` to `~/.claude/RTK.md`
   - Patches `~/.claude/CLAUDE.md` to add `@RTK.md` reference
   - Patches `~/.claude/settings.json` with PreToolUse hook entry

2. **Claude MD mode** (`--claude-md` flag):
   - Full injection of 137-line RTK_INSTRUCTIONS into CLAUDE.md
   - No hook, no settings.json patching
   - Legacy fallback for Windows

3. **Hook-only mode** (`--hook-only` flag):
   - Just the hook script, no RTK.md
   - Minimal overhead, Claude won't know about meta commands

**Hook Integration Mechanics**:
- Claude Code hook entry: `PreToolUse:Bash` matcher
- Runs `rtk-rewrite.sh` BEFORE each Bash command
- Hook reads JSON input with `tool_input.command` field
- Returns JSON with `hookSpecificOutput.updatedInput.command` (rewritten)
- Guards: checks `command -v rtk` and `command -v jq` before executing

**Key Functions**:
- `run_default_mode()`: Full setup (hook + RTK.md + @RTK.md + settings.json)
- `patch_settings_json()`: Deep-merges hook entry into JSON structure
- `ensure_hook_installed()`: Idempotent hook file write + chmod 755
- `atomic_write()`: Tempfile + rename pattern to prevent corruption
- `remove_hook_from_json()`: Uninstall support - removes RTK hook from PreToolUse array
- `uninstall()`: Complete cleanup (hook, RTK.md, CLAUDE.md, settings.json)

**Idempotency Features**:
- Checks if hook already present by matching `rtk-rewrite.sh` substring
- Skips if content unchanged (compare checksums)
- Handles migration from old RTK block to new @RTK.md format
- Detects incomplete migration states (missing end marker)

### 2. rtk-rewrite.sh - The Hook Script

**Location**: `/hooks/rtk-rewrite.sh`

**Functionality**:
- Claude Code hook that intercepts bash commands
- Transparently rewrites raw commands to `rtk` equivalents

**Workflow**:
1. Reads stdin as JSON with `jq`
2. Extracts `.tool_input.command` field
3. Matches against 40+ patterns (git, cargo, vitest, tsc, etc.)
4. Rewrites matching commands (e.g., `git status` → `rtk git status`)
5. Returns JSON with `updatedInput` (preserves all other fields)
6. Exits silently if rtk/jq missing (guards at top)

**Rewrite Examples**:
- `git status` → `rtk git status`
- `cargo test` → `rtk cargo test`
- `vitest` → `rtk vitest run`
- `pnpm list` → `rtk pnpm list`
- `curl URL` → `rtk curl URL`
- No rewrite if: command already uses rtk, contains heredocs

### 3. cc_economics.rs - Claude Code Spending vs RTK Savings Analysis

**Purpose**: Correlates Claude Code spending (via ccusage) with RTK token savings.

**Data Sources**:
- **Input**: ccusage binary (npm package or `which ccusage`)
- **Input**: RTK tracking database (sqlite)
- **Output**: Spending vs savings analysis

**Key Structures**:
- `PeriodEconomics`: Merges ccusage data + RTK tracking for a time period
  - ccusage metrics: cost, token breakdown (input, output, cache_create, cache_read)
  - rtk metrics: commands, saved tokens, savings %
  - Computed metrics: weighted_input_cpt, savings_weighted, blended_cpt, active_cpt

**Weighted Metrics Algorithm**:
```
weighted_units = input_tokens + (5.0 * output_tokens) + (1.25 * cache_create) + (0.1 * cache_read)
weighted_input_cpt = cost / weighted_units
savings_weighted = saved_tokens * weighted_input_cpt
```
Rationale: Output tokens cost 5x more, cache create costs 1.25x, cache read costs 0.1x (API pricing)

**Query Architecture**:
- `fetch()` calls ccusage binary with `--json` flag
- Parses JSON response by granularity (Daily, Weekly, Monthly)
- Merges with RTK tracker data using period keys (dates, weeks, months)
- Computes totals across all periods

**Display Modes**:
- Default: Summary table with Spent | Saved | Savings | RTK Cmds
- `--daily`: Daily breakdown
- `--weekly`: Weekly breakdown (converts Saturday→Monday for ISO alignment)
- `--monthly`: Monthly breakdown
- `--verbose`: Legacy metrics (active_cpt, blended_cpt) for reference

**Export Formats**:
- JSON: Full structure with daily/weekly/monthly arrays + totals
- CSV: Period rows with all metrics as columns
- Text: Formatted tables with separators

### 4. ccusage.rs - Claude Code Usage Data Integration

**Purpose**: Reads Claude Code spending data from ccusage binary.

**Integration Points**:
- Checks for `ccusage` binary via `which`
- Fallback: tries `npx ccusage`
- CLI: `ccusage [daily|weekly|monthly] --json --since 20250101`

**Data Structures**:
- `CcusageMetrics`: Input/output/cache tokens + total cost
- `CcusagePeriod`: Period key + metrics (date, week, or month)
- `DailyResponse`, `WeeklyResponse`, `MonthlyResponse`: JSON schemas

**Field Mapping** (camelCase from ccusage API):
- `inputTokens` → `input_tokens`
- `outputTokens` → `output_tokens`
- `cacheCreationTokens` → `cache_creation_tokens` (default 0)
- `cacheReadTokens` → `cache_read_tokens` (default 0)
- `totalTokens` → `total_tokens`
- `totalCost` → `total_cost`

**Error Handling**:
- Returns `Ok(None)` if ccusage not found (graceful degradation)
- Logs warning to stderr
- Parses JSON errors with serde context
- Validates required fields (inputs, outputs, cost)

### 5. display_helpers.rs - Formatting Utilities

**Purpose**: Unified formatting for tables and period statistics.

**Key Components**:
- `format_duration()`: Converts ms → "1h23m", "45.2s", "123ms"
- `PeriodStats` trait: Unified interface for Day/Week/Month stats
  - `icon()` → "📅", "📊", "📆"
  - `label()` → "Daily", "Weekly", "Monthly"
  - `period()` → formatted period string
  - Token/savings/timing getters

**Table Generation**:
- `print_period_table<T: PeriodStats>()`: Generic table printer
- Dynamically handles Day/Week/Month stats
- Outputs: Header | separator | rows | totals | separator
- Columns: Period | Cmds | Input | Output | Saved | Save% | Time
- Totals row: TOTAL | sum(cmds) | sum(input) | sum(output) | sum(saved) | avg% | avg_time

**Implementations**:
- `DayStats` impl: 50-char width, period="2026-01-20"
- `WeekStats` impl: 50-char width, period="01-20 → 01-26"
- `MonthStats` impl: 50-char width, period="2026-01"

### 6. gain.rs - Token Savings Analytics & Reporting

**Purpose**: Query and visualize RTK token savings statistics.

**Query Architecture**:
```
Tracker::get_summary() → GainSummary (aggregated)
Tracker::get_all_days() → Vec<DayStats> (SQL: grouped by date)
Tracker::get_by_week() → Vec<WeekStats> (SQL: grouped by ISO week)
Tracker::get_by_month() → Vec<MonthStats> (SQL: grouped by month)
Tracker::get_recent(N) → Vec<CommandRecord> (SQL: last N records)
```

**Display Modes**:
- Default: Summary with total_commands, tokens, savings %, breakdown by command, 30-day ASCII graph
- `--daily`: Full daily breakdown table
- `--weekly`: Weekly breakdown table
- `--monthly`: Monthly breakdown table
- `--history`: Recent command history (last 10 records)
- `--graph`: ASCII bar chart of last 30 days
- `--quota`: Pro tier quota savings estimate (44K tokens/5h baseline)

**Quota Calculation**:
- Pro: 6M tokens/month (~44K per 5 hours)
- 5x: 30M tokens
- 20x: 120M tokens
- Computes: (total_saved / quota) * 100 as "quota preserved %"

**Export Formats**:
- JSON: Summary + optional daily/weekly/monthly arrays
- CSV: Separate sections with headers (# Daily Data, # Weekly Data, etc.)
- Text: Formatted tables via `print_period_table()`

**SQL Queries** (derived from Tracker):
- Summary: COUNT, SUM(input/output/saved), AVG(savings_pct), AVG(exec_time_ms)
- By-day: GROUP BY date, ORDER BY date DESC
- By-week: GROUP BY strftime('%Y-W%W', timestamp), plus week_start/week_end
- By-month: GROUP BY strftime('%Y-%m', timestamp), ORDER BY month DESC
- Top commands: GROUP BY rtk_cmd, ORDER BY SUM(saved_tokens) DESC LIMIT 10

**ASCII Graph** (`print_ascii_graph()`):
- X-axis: Date (last 30 days)
- Y-axis: Tokens saved (proportional bar)
- Max 40 char bar width
- Shows tokens at end of bar

### 7. Database Schema (tracking.rs)

**Location**: `~/.local/share/rtk/tracking.db` (Linux), `~/Library/Application Support/rtk/tracking.db` (macOS)

**Table**: `commands`
```sql
id (INTEGER PRIMARY KEY)
timestamp (TEXT) -- ISO format
original_cmd (TEXT) -- raw command
rtk_cmd (TEXT) -- rewritten command
input_tokens (INTEGER) -- characters / 4
output_tokens (INTEGER) -- characters / 4
saved_tokens (INTEGER) -- input - output
savings_pct (REAL) -- (saved / input) * 100
exec_time_ms (INTEGER) -- elapsed milliseconds
```

**Index**: `idx_timestamp ON commands(timestamp)`

**Retention**: Auto-cleanup of records > 90 days old

**Token Calculation**:
- Uses heuristic: `char_count / 4` (approximates token count)
- Not exact (uses actual tokenizer for display)

### 8. Integration Flow Summary

**User runs command**:
```
$ git status
```

**Claude Code intercepts**:
1. Hook reads `.tool_input.command = "git status"`
2. `rtk-rewrite.sh` matches git pattern
3. Hook returns `updatedInput.command = "rtk git status"`
4. Command updated before execution

**RTK executes**:
```
$ rtk git status
# Compresses output, records stats
```

**Tracking Records**:
```
INSERT INTO commands VALUES (
  timestamp='2026-02-12T14:30:45Z',
  original_cmd='git status',
  rtk_cmd='rtk git status',
  input_tokens=800,
  output_tokens=50,
  saved_tokens=750,
  savings_pct=93.75,
  exec_time_ms=45
)
```

**Analytics**:
```
rtk gain              # shows summary
rtk gain --daily     # breakdown by day
rtk cc-economics     # correlates with ccusage spending
```

### 9. Key Design Decisions

1. **Hook + RTK.md split**: Reduces CLAUDE.md bloat (137→10 lines)
2. **Atomic writes**: Prevents settings.json corruption on crash
3. **Graceful degradation**: Works without ccusage (Option<T>)
4. **Weighted CPT**: Accounts for output/cache pricing ratios
5. **Heuristic tokenization**: char/4 approximation for quick tracking
6. **90-day retention**: Automatic cleanup prevents unbounded growth
7. **SQL aggregation**: Efficient period grouping (daily/weekly/monthly)
8. **Settings.json patching**: Deep merge preserves existing hooks

---

### 10. Open Questions & Notes

- ccusage is external (npm package) - optional, RTK works without it
- Token estimation (char/4) is heuristic, not exact
- Cache weights (1.25x create, 0.1x read) derived from API pricing
- Weekly format uses ISO Monday (converted from Saturday in merge)
- Saturday→Monday conversion adds complexity, could unify on ISO
- No disk size limits documented (SQLite can grow large over time)
