## Windows Named Pipe IPC Fix (v6.0.1)

### What
Implemented Windows named pipe IPC support for the v6 daemon architecture. The v6.0.0 release failed to build on Windows because the daemon's IPC module had a TODO stub for Windows named pipes, causing 16 compilation errors.

### Why
The v6 daemon architecture introduced Unix domain socket IPC between the adapter and daemon processes. The Windows `mod windows {}` in `ipc.rs` was left empty. The 5-agent code review didn't catch it because no agent tried cross-platform compilation.

### How
- Implemented `IpcListener` for Windows using `tokio::net::windows::named_pipe::ServerOptions/NamedPipeServer` with `Mutex`-based accept-and-replace pattern
- Implemented `IpcConnector` using `ClientOptions/NamedPipeClient`
- Added `IpcStream`/`IpcClientStream` type aliases to encapsulate platform differences
- Added `DaemonPaths::daemon_ipc_addr()` for platform-independent endpoint resolution
- Removed most `#[cfg(unix)]` gates from daemon/adapter code; platform logic now lives in `ipc.rs` and `paths.rs`
- Added Windows pipe probe in launcher, Windows process exit for stale-binary restart

### Impact
Fixes Windows CI build. v6.0.1 tagged and pushed, Release workflow running.
