## v6.0.1 Released: Windows Named Pipe IPC Fix

### What
Fixed Windows CI build failure for v6 daemon architecture. Implemented full Windows named pipe IPC support and released v6.0.1.

### Why
The v6.0.0 release tagged at `808763ce` failed the Windows build with 16 compilation errors. The daemon IPC module (`src/daemon/ipc.rs`) had a TODO stub for Windows named pipes, and multiple files used `tokio::net::UnixStream` without `#[cfg]` gates.

### How
- Implemented `IpcListener` for Windows using `tokio::net::windows::named_pipe::ServerOptions/NamedPipeServer` with `Mutex`-based accept-and-replace pattern
- Implemented `IpcConnector` using `ClientOptions/NamedPipeClient`
- Added `IpcStream`/`IpcClientStream` type aliases to encapsulate platform differences
- Added `DaemonPaths::daemon_ipc_addr()` for unified endpoint resolution
- Removed most `#[cfg(unix)]` gates from daemon/adapter code; platform logic now isolated in `ipc.rs` and `paths.rs`
- Added Windows pipe probe in launcher, Windows `process::exit(0)` for stale-binary restart

### Impact
- v6.0.1 Release workflow: **all 3 targets green** (Windows, Linux, macOS) + release created
- 5 files changed, +156/-43 lines
- 8/8 xtask dev buckets still green on macOS

### Lesson Learned
The 5-agent v6 code review missed this because no agent tried cross-platform compilation. Future reviews should include `cargo check --target x86_64-pc-windows-msvc`.
