## Phase 3 complete: SQL structured pending + JSON $ref + TOML domain relationships

### What
Phase 3.3 (TOML) closes the final Phase 3 task. End-of-phase status:

- **Phase 3.1 (SQL)**: cross-schema FK targets emit `StructuredPendingRelationship` (closed at a4beac3a / commit 8120d04e)
- **Phase 3.2 (JSON)**: JSON Schema `$ref` produces concrete relationships for local pointers and structured pending for external pointers (closed at 8120d04e / commit 2e4c50c7)
- **Phase 3.3 (TOML)**: Cargo `[dependencies]` emits Imports edges, pyproject `[tool.<x>.*]` emits References edges (closed at 2e4c50c7)

### Phase 3.3 specifics
- `crates/julie-extractors/src/toml/relationships.rs` (NEW, 167 lines): walks `table` nodes, dispatches by file basename
- `Cargo.toml` only: `[dependencies]`, `[dev-dependencies]`, `[build-dependencies]`, `[target.<triple>.dependencies]` → emit `RelationshipKind::Imports` from the table symbol to each child key (`serde`, `tokio`, etc.). Metadata carries `dependencyKind` (e.g., "dependencies", "dev-dependencies").
- `pyproject.toml` only: `[tool.<x>.*]` → emit `RelationshipKind::References` from the `[project]` table symbol to the tool table symbol, deduped by top-level tool name. `[tool.pytest]` and `[tool.pytest.ini_options]` collapse to a single `pytest` edge. Metadata carries `toolName`.
- All other tables (and all other files): no relationships emitted; the negative test `arbitrary_table_emits_no_relationship` proves this.

### Why
Cargo.toml and pyproject.toml are the load-bearing dependency manifests for Rust and Python codebases respectively. Without these edges, the call graph for any Rust or Python project loses the dependency layer entirely.

### Plumbing
- Language spec: TOML switches from `DATA_ONLY_CAPABILITIES` → `RELATIONSHIP_DATA_CAPABILITIES` (symbols+rels+identifiers true; pending+types false — TOML has no cross-file or static-type semantics)
- Registry: TOML moved out of `define_data_only_extractors!` into a hand-written `extract_toml`; the now-unused macro definition was removed
- `TomlExtractor` gained `extract_relationships`, `get_pending_relationships`, `get_structured_pending_relationships` methods
- `capabilities.json` updated: target_capabilities/capabilities for TOML `relationships` flipped true; `relationships` gap removed; `pending_relationships` and `types` gaps converted to `status: exception` with intrinsic-N/A reasons (TOML has no cross-doc construct; no static type system)
- New fixtures: `fixtures/extraction/toml/cargo_deps/Cargo.toml` + expected.json (4 Imports edges across 3 dependency table kinds); `fixtures/extraction/toml/pyproject/pyproject.toml` + expected.json (3 References edges, deduped on tool name)
