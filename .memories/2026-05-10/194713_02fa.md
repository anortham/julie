## Phase 3.2 closed: JSON Schema $ref relationships

### What
- JSON now emits relationships for JSON Schema `$ref` shapes:
  - **Local `#/path/to/Symbol`** → concrete `Relationship` (kind: References) from the containing object's parent symbol to the resolved target symbol; if the local pointer can't be resolved (malformed), no edge is emitted
  - **External `<file>#/path/to/Symbol`** → `StructuredPendingRelationship` with `target.import_context = Some("<file>")`, `target.terminal_name = "Symbol"`, `target.namespace_path` carrying preceding fragment segments, and `caller_scope_symbol_id` pointing at the containing object's parent
- Language spec switched JSON from `DATA_ONLY_CAPABILITIES` → `PENDING_NO_TYPES_CAPABILITIES` (symbols+rels+pending+identifiers true, types false)
- Registry: JSON moved out of `define_data_only_extractors!`; hand-written `extract_json` mirrors `extract_sql` and wires through pending + structured pending
- New module: `crates/julie-extractors/src/json/relationships.rs` (170 lines)
- `JsonExtractor` gained `extract_relationships`, `get_pending_relationships`, `get_structured_pending_relationships` methods
- New fixture `fixtures/extraction/json/refs/{source.json,expected.json}` proves both shapes (local resolved + external pending) in one file
- `capabilities.json` updated: target_capabilities/capabilities for JSON `relationships` and `pending_relationships` flipped true; the two open `$ref`-related gaps removed; `types` gap converted from `status: open` (with contradictory N/A required_closure) → `status: exception` with intrinsic-N/A reason

### Why
Pre-Phase-3.2, JSON went through `define_data_only_extractors!` so the relationship column was always empty. JSON Schema is the de-facto contract format for HTTP APIs and config validation; without `$ref` edges the call graph for any schema-driven codebase loses the document hierarchy.

### How
- Pair-node walk: detect `pair` nodes whose key string is `"$ref"` and whose value is a string literal
- Containing object's parent: `$ref pair → object → pair` (the named property's pair); the symbol whose `start_byte` matches the parent pair becomes both `from_symbol_id` and `caller_scope_symbol_id`
- Local resolution: split fragment by `/`, find a symbol whose name matches the terminal segment AND whose ancestor chain (via `parent_id`) matches the preceding segments in reverse order
- External detection: `value.find('#')` returns Some(non-zero) → split into `(file, fragment)`, build `UnresolvedTarget`, push via `base.add_structured_pending_relationship`
