## Phase 3.1 closed: SQL structured pending for cross-file FK targets

### What
- SQL extractor now emits `StructuredPendingRelationship` for cross-schema FK targets (e.g., `REFERENCES other_schema.users(id)`)
- Local-only FKs (e.g., `REFERENCES orders(id)` when both tables are in the same file) still resolve to concrete relationships
- Capability matrix: SQL `pending_relationships` flipped from `false` → `true` in both `target_capabilities` and `capabilities`; the open gap row was deleted (the schema disallows `status: closed` rows)
- Language spec (`crates/julie-extractors/src/language_spec/specs.rs`): SQL switched from `NO_PENDING_CAPABILITIES` → `FULL_CAPABILITIES`
- New fixture `fixtures/extraction/sql/cross_file/` with `source.sql` + regenerated `expected.json` (UPDATE_GOLDEN=1)
- Cross-file fixture row appended to SQL `fixtures` array in `capabilities.json`

### Why
Pre-Phase-3.1, SQL was in `define_no_pending_extractors!` and dropped any FK whose target was not in the same file. This dropped semantic edges that downstream tools needed.

### How
- Inline `REFERENCES` syntax in tree-sitter-sequel parses as `column_definition` containing `keyword_references` + `object_reference` directly — no `references_clause` or `foreign_key_constraint` wrapper. Confirmed by walking the AST for the cross_file fixture.
- Extended `extract_relationships_internal` (`crates/julie-extractors/src/sql/relationships.rs`) to dispatch on `column_definition` when it has a `keyword_references` child. Routes to existing `extract_foreign_key_relationship`.
- The `(Some, None)` arm of the (source_symbol, target_symbol) match builds an `UnresolvedTarget` (terminal_name/namespace_path split on `.`) and emits via `base.add_structured_pending_relationship`.
- Hand-written `extract_sql` in registry.rs already wired pending + structured_pending through the `ExtractionResults` shape.

### Tests
- New: `crates/julie-extractors/src/tests/sql/cross_file_pending.rs::test_sql_emits_structured_pending_for_cross_file_fk` — asserts namespace_path=["other_schema"], terminal_name="users", display_name="other_schema.users", receiver=None, import_context=None, RelationshipKind::References, line_number > 0, file_path="source.sql", caller_scope_symbol_id.is_some(), no structured pending for local "orders" reference, and a concrete relationship audit_events → orders still emits.
- All 30 SQL tests pass; 16 capability_matrix + golden + pending_shape_contract tests pass; extractors xtask bucket green (3.8s, 3/3 commands).

### Impact
- Phase 3.1 closure complete. Phase 3 has 2 tasks remaining: 3.2 (JSON $ref), 3.3 (TOML Cargo deps + pyproject tool tables). Phase 3 boundary gate (extractors bucket + ledger rows) follows.
