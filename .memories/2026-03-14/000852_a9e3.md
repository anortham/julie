## Released v5.0.5 — Fixed semantic fallback threshold for query-based KNN search

### What
Lowered `QUERY_SIMILARITY_THRESHOLD` from 0.3 to 0.2 in `fast_refs` semantic fallback. Cleaned up temporary diagnostic `warn!` logging from previous debugging session. Removed resolved "In Progress" item from TODO.md.

### Why
Semantic fallback was returning 0 results for reference workspaces (and would have failed on primary too for low-scoring queries). Root cause: bare symbol name queries embedded against rich metadata (kind + name + signature + docstring) produce naturally lower cosine similarity scores than symbol-to-symbol comparisons. The top KNN hit for "UserRepository" scored 0.275 — just below the 0.3 threshold — mapping to genuinely relevant symbols like `UserUpdateRequest`, `UsersController`, `UserService`.

### How
- Diagnosed via temporary `warn!` logging added in previous session — raw KNN scores confirmed results existed but were being filtered
- Lowered threshold from 0.3 → 0.2 (captures scores down to 0.20, top hits were 0.275/0.244/0.238)
- Added `debug!` logging to reference workspace error paths (previously silently swallowed)
- Removed diagnostic `warn!` traces from both `fast_refs.rs` and `similarity.rs`

### Verification
Thorough cross-contamination testing across both primary and reference workspaces:
- `fast_refs`, `fast_search`, `deep_dive` all tested with workspace-unique symbols
- Zero cross-workspace leakage detected
- Semantic fallback correctly scoped to queried workspace in all cases
- `PageController` → `PagesController` scored 0.40 (highest seen), confirming score distribution is healthy
