## Implemented Smart Test Filtering for fast_search

### What
Added `exclude_tests` parameter to `fast_search` that filters test symbols (those with `metadata["is_test"] = true`) from search results. Includes smart default resolution and explicit override.

### Why
When agents search for production code ("payment processing"), test functions clutter results. Phase 1 of Code Health Intelligence already tags test symbols during extraction — this feature leverages that metadata for precise, symbol-level filtering.

### How
- Added `exclude_tests: Option<bool>` to `FastSearchTool` struct
- Added `exclude_tests: bool` to `SearchFilter` struct
- Smart default in `text_search_impl`: definitions → include tests, NL content → exclude tests
- Extended `enrich_symbols_from_db` to also copy `metadata` (was only copying code_context + visibility)
- Added `filter_test_symbols()` helper called after enrichment in both hybrid and keyword paths of `definition_search_with_index`
- Updated all 17 files with callers/constructors

### Key Discovery
Layer A (test detection + 31 extractors), Layer B (test quality metrics), and deep_dive integration were already fully implemented from prior sessions. Only Chunk 1 (the `exclude_tests` parameter) was needed.

### Design Note
The NL auto-exclude smart default (`None` → `true` for NL non-definition queries) is effectively infrastructure — `FastSearchTool` routes content searches to `line_mode_search`, not `text_search_impl`, so the smart default only evaluates for definition searches (where it always resolves to `false`). Explicit `Some(true)`/`Some(false)` is the practical mechanism.

### Impact
- Agents can now call `fast_search(query="...", exclude_tests=true)` to get clean production-only results
- 4 new tests verify filtering, override, and default behaviors
- `text_search.rs` stays under 500-line limit (496 lines)
