## Code Health Intelligence — Design Complete (Phase 1 Plan Written)

### What
Designed a two-phase "Code Health Intelligence" feature set for Julie:
- **Phase 1** (planned): Test symbol detection across 31 languages + test quality scoring (assertion density, mock detection, quality tiers)
- **Phase 2** (preview): Test-to-code linkage, test risk scores, structural security risk signals (6 signals: exposure, input handling, sinks, blast radius, untested, flow depth)

### Why
AI coding agents are Julie's primary consumer. Two capabilities are critically underserved:
1. **Test quality** — not just "has tests" but "has GOOD tests" (assertion density, mock usage, error path coverage)
2. **Security risk** — structural risk profiles computed from existing data (public API + string params + DB calls + untested = HIGH risk)

Julie already has call graphs, centrality, types, visibility, and code bodies across 31 languages — we're assembling existing signals, not building new infrastructure.

### Key Decisions
- **Consumer**: AI agents (structured signals in existing tool output, not new tools)
- **Delivery**: Enrich `deep_dive` and `get_context` rather than new dedicated tools
- **Timing**: Index-time computation stored in metadata/columns
- **Phasing**: Phase 1 (test detection + quality) first, Phase 2 (linkage + risk + security) after validation
- **Cross-language**: Shared `is_test_symbol()` function with per-language attribute detection + generic name/path fallbacks; language-agnostic regex for body analysis

### Key Design Elements
- New `crates/julie-extractors/src/test_detection.rs` — shared detection logic
- New `src/analysis/` module — test_quality.rs, (future: test_linkage.rs, test_risk.rs, security_risk.rs)
- Quality tiers: thorough / adequate / thin / stub
- Stored in existing `metadata` JSON (no schema change for Phase 1)
- Phase 2 adds `test_coverage` table + `test_risk_score` and `risk_score` columns (migration 013)
