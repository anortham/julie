## Completed Task 4 code-quality review of query weighting changes

Reviewed `src/search/query.rs` and modified search tests for Task 4 scope, with targeted execution of query tests.

- **What I checked:**
  - Functional behavior and boolean-query semantics in weighted builders
  - Backward-compatibility path through legacy wrapper APIs
  - Test robustness and whether Task 3 test edits appear coupled to Task 4 implementation
- **Verification:** Ran `cargo test --lib tantivy_query_` (7 tests passed)
- **Key finding:** Weighted AND-mode semantics currently require alias/normalized groups as `Must`, which can over-constrain recall if weighted APIs are used in AND mode.
- **Key finding:** `NAME_FIELD_BOOST + 0.01` tie-break tweak in production query path appears test-driven and undocumented, creating maintainability/ranking-risk.
- **Test-risk finding:** Task 3-era weighting tests still rely on debug-string proximity heuristics and contain stale "RED tests" header comments; this increases brittleness and can incentivize implementation hacks.

**Impact:** Recommend "changes requested" until query semantics are clarified/fixed and test expectations are made less brittle.
