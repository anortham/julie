## Semantic Embedding Layer Design — Approved and Written

Full design document written to `docs/plans/2026-02-26-semantic-embedding-layer-design.md` after extensive brainstorming session.

### Key Decisions Made
- **Keep Julie's architecture** (SQLite + Tantivy + custom scoring) — don't swap for LanceDB, don't revive Miller
- **Add semantic embeddings** as a new layer alongside existing infrastructure
- **EmbeddingProvider trait** with ort/fastembed-rs as default, candle-coreml as future feature flag
- **sqlite-vec** for vector storage (must validate cross-platform again)
- **Symbol metadata only** (name + signature + doc_comment, NOT full code bodies)
- **Async background indexing** — keyword search immediate, semantic catches up
- **4-phase delivery**: Bug fixes → Foundation → Tool integration → Cross-language bridging

### Why Tantivy Alone Isn't Enough
- NL path prior was hardcoded to Julie's `src/` layout — no-op on C#/Python
- Tantivy can't bridge vocabulary mismatch (\"error handling\" → CircuitBreakerService)
- Cross-language bridging (UserDto ↔ IUser) is impossible with keywords alone

### Why Not LanceDB
- Would lose all custom Tantivy work (CodeTokenizer, field schema, weighted queries, NL pipeline)
- Julie is a structured code intelligence tool, not a vector DB with FTS on the side

### Why Not Miller
- Julie's tool layer (deep_dive, get_context, rename_symbol) is far ahead
- Python deployment story is painful
- Miller stays as-is for Python ML stack users

### Tantivy Vector Status
- Confirmed: Tantivy does NOT have native vector support (issue #815 open since 2020, still unmerged)
- Must use separate vector storage (sqlite-vec)
