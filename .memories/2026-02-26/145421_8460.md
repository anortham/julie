## Task 7 quality fixes applied and verified

Implemented focused quality fixes in search ranking/query token handling and added deterministic integration coverage.

- **What changed**
  - Updated `SearchIndex::search_symbols` to over-fetch NL query candidates before reranking, then truncate to requested `limit`.
  - Replaced adjacent-only dedup in tokenization with insertion-order dedup via `HashSet` tracking in `tokenize_query` and `tokenize_terms`.
  - Strengthened `test_nl_query_prefers_code_over_docs` to assert actual ranking preference (production code ranks above docs).
  - Added `test_nl_path_prior_can_change_top1_with_small_limit` to prove top-K rerank influence at `limit=1`.
  - Added `test_non_adjacent_duplicate_terms_do_not_change_relaxed_scores` to cover non-adjacent duplicate-term dedup behavior.

- **Why**
  - Fixes Task 7 gaps: weak preference assertion, non-adjacent duplicate leakage, and top-k/rerank blind spot for NL path prior.

- **How**
  - Introduced deterministic candidate over-fetch factor (`NL_RERANK_OVERFETCH_FACTOR = 4`) gated to NL-like queries, preserving bounded behavior.
  - Kept existing fallback semantics (AND first, OR fallback) and applied truncation only after post-search reranking.

- **Impact**
  - NL reranking can now influence final top-K even for small limits.
  - Query term dedup is deterministic and stable by first occurrence.
  - Integration tests now encode deterministic preference expectations rather than presence-only checks.
