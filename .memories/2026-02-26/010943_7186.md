## Phase 1 started: added code-first fallback guardrail for `get_context` pivot selection

### WHAT
- Added a new relevance guardrail function: `select_pivots_with_code_fallback(...)` in `src/tools/get_context/scoring.rs`
- Updated `run_pipeline` to use the new fallback selector instead of raw `select_pivots`
- Added a focused regression test module: `src/tests/tools/get_context_relevance_tests.rs`
- Registered the new test module in `src/tests/mod.rs`
- Added `Clone` derive to `SymbolSearchResult` in `src/search/index.rs` to support two-pass selection on the same result set

### WHY
- Natural-language queries can rank docs/memories/test structures above executable code.
- For agent workflows, a docs-only pivot set is often non-actionable and leads to extra round trips.

### HOW
- Primary pass: existing centrality-weighted `select_pivots`
- Guardrail check: if selected pivots are all non-actionable (non-code/test/structural/import), run second pass over code candidates only
- Candidate criteria: exclude `import`, test paths, non-code paths (`docs/`, `.memories/`, markdown/config), and structural kinds (`namespace/module/export`)
- Wrote a failing test first (unresolved symbol import), then implemented function and wired pipeline

### IMPACT
- `get_context` now has a deterministic recovery path when initial pivots are docs/memories-heavy.
- Keeps existing behavior when at least one actionable pivot is present.
- Adds coverage for this failure mode without touching external API shape.
