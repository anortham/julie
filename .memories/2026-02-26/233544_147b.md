## Task 5 Complete: Semantic Fallback for fast_search

Added a lightweight semantic fallback to `fast_search`'s definition search path — triggered only when the query is natural language AND keyword results are sparse (< 3).

### Changes Made (3 files, +75 lines)

1. **`src/search/hybrid.rs`**: Added `should_use_semantic_fallback(query, keyword_result_count)` gate function. Uses `is_nl_like_query` from scoring.rs + result count threshold.

2. **`src/tools/search/text_search.rs`**: Wired fallback into primary workspace definition search path:
   - Cloned `embedding_provider` before `spawn_blocking`
   - After enrichment, checks fallback gate → embeds query → KNN search → dedup by symbol ID → truncate to limit
   - All error paths are graceful (`if let Ok(...)` chains, no panics)
   - No deadlock risk: enrichment lock drops before fallback acquires its own lock

3. **`src/tests/tools/hybrid_search_tests.rs`**: Added 3 unit tests:
   - NL queries with sparse results trigger fallback
   - Identifier queries don't trigger fallback
   - NL queries with enough results don't trigger fallback

### Key Design Decisions
- **Definition search only** — content search doesn't benefit from semantic fallback
- **Primary workspace only** — reference workspaces don't have embedding providers
- **Owned HashSet for dedup** — avoids borrow checker issues with `&str` references into mutable `symbols` vec
- **Threshold = 3** — conservative: only kicks in when keyword search found almost nothing
