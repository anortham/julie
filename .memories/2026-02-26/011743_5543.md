## Phase 1 advanced: added actionable-coverage guardrail + run_pipeline integration test

### WHAT
- Extended `select_pivots_with_code_fallback` in `src/tools/get_context/scoring.rs` with an actionable-coverage threshold.
- Added new relevance unit test for low actionable coverage in `src/tests/tools/get_context_relevance_tests.rs`.
- Added new integration test file `src/tests/tools/get_context_pipeline_relevance_tests.rs` to verify fallback behavior through `run_pipeline`.
- Registered integration test module in `src/tests/mod.rs`.

### WHY
- Prior fallback only triggered when actionable pivots were zero.
- Mixed pivot sets like docs+docs+code could still leak non-actionable pivots despite having enough code alternatives.

### HOW
- New fallback conditions:
  - Trigger if actionable pivots == 0, OR
  - Trigger if actionable coverage < 50% and there are at least 2 actionable code alternatives.
- Added deterministic test data to force low-actionable primary ranking and validate code-only fallback.
- Added end-to-end pipeline test with indexed docs/memory/code symbols asserting code pivots survive and docs/memory pivots are dropped.

### IMPACT
- `get_context` now recovers not only from docs-only pivot sets, but also from low-quality mixed sets.
- Stronger regression safety: behavior is covered at both scoring-unit and pipeline-integration layers.
