## Hardened Task 1 cap test to require truncation behavior

Updated `src/tests/tools/search/tantivy_query_expansion_tests.rs` to strengthen `test_max_added_term_cap_is_respected` so it no longer passes on a naturally-small expansion set.

- **What changed:**
  - Replaced the lighter stress query with a broader multi-concept query intended to generate a large expansion fan-out.
  - Tightened assertion from `added_term_count <= MAX_ADDED_TERMS` to `added_term_count == MAX_ADDED_TERMS`.
  - Added guard assertion that `router` remains present, proving deterministic alias retention under capping.
- **Why:**
  - Previous assertion allowed false positives when the generated expansion set happened to be below the cap.
  - New assertion requires saturation of the cap, so missing cap enforcement should fail once expansion exists.
- **How:**
  - Kept test in RED-only mode by preserving references to `crate::search::expansion::*` without implementing Task 2.
  - Re-ran the requested targeted test command to verify compile failure remains due to missing expansion module.

**Impact:** Task 1 remains intentionally RED, but with a stronger regression guard for Task 2 implementation.
