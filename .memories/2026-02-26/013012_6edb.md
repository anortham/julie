## Added compact token-savings quality gate and started Phase 3 batch-query performance refactor

### WHAT
- Added a hard compact-mode token efficiency regression test in `src/tests/tools/get_context_formatting_tests.rs`:
  - `test_compact_reduces_estimated_tokens_by_at_least_20_percent`
  - Uses `TokenEstimator::estimate_string_hybrid` and enforces >=20% reduction for a representative fixture.
- Adjusted compact neighbor rendering in `src/tools/get_context/formatting.rs`:
  - In compact mode, `NeighborMode::SignatureAndDoc` now renders signatures but omits doc summaries to keep token footprint low.
- Began Phase 3 performance cleanup with batched relationship retrieval:
  - Added `SymbolDatabase::get_outgoing_relationships_for_symbols(...)` in `src/database/relationships.rs`.
  - Added database coverage test: `test_get_outgoing_relationships_for_symbols_batch` in `src/tests/core/database.rs`.
- Refactored `get_context` pivot entry pipeline to reduce N+1 queries in `src/tools/get_context/pipeline.rs`:
  - Batch fetch pivot symbols via `get_symbols_by_ids`
  - Batch fetch incoming/outgoing relationships for all pivots in one pass
  - Build precomputed maps for relation-name resolution instead of per-pivot DB calls

### WHY
- Needed an enforceable metric for Phase 2 compact-mode success.
- Needed to start Phase 3 by removing high-frequency per-pivot DB lookups that scale poorly.

### HOW
- Wrote failing tests first for new capabilities/constraints:
  - 20% compact token reduction threshold initially failed (16.3%), then fixed by further compacting neighbor output.
  - Batch outgoing relationships test failed due missing API, then implemented the new DB method.
- Updated pipeline internals to use batched lookups and fixed borrow semantics for cached symbol code contexts.

### IMPACT
- Compact mode now has a measurable and tested efficiency target.
- `get_context` pipeline now performs fewer DB round trips for multi-pivot contexts.
- Performance refactor is functionally covered and keeps existing output tests green.
