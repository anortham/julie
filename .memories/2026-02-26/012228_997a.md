## Completed Phase 1 and started Phase 2 for `get_context`

### WHAT
- **Phase 1 finalized** with stronger relevance guardrails in `src/tools/get_context/scoring.rs`:
  - Added auxiliary-path de-boost (`examples/`, `benches/`, `scripts/`)
  - Upgraded fallback trigger from only `0 actionable pivots` to:
    - `0 actionable pivots`, or
    - actionable coverage `< 50%` when >=2 actionable alternatives exist
  - Kept deterministic pivot ordering (`compare_pivots`)
- **Phase 2 started** with output UX improvements:
  - Added caller/callee deduplication in formatter output
  - Added `compact` rendering mode (`OutputFormat`) in `src/tools/get_context/formatting.rs`
  - Added tool-level `format` parameter in `src/tools/get_context/mod.rs` and pipeline wiring in `src/tools/get_context/pipeline.rs`

### WHY
- Natural-language queries still let docs/memory-heavy mixes crowd out code pivots in some cases.
- Readable output was useful but token-heavier than necessary; agent-facing workflows need a compact option.
- Duplicate caller/callee names created noisy outputs.

### HOW
- Added/expanded tests first, then implemented:
  - `src/tests/tools/get_context_relevance_tests.rs` now includes auxiliary-path and actionable-coverage cases
  - New integration file: `src/tests/tools/get_context_pipeline_relevance_tests.rs` verifies relevance behavior through `run_pipeline`
  - `src/tests/tools/get_context_formatting_tests.rs` now covers dedup and compact output
  - Added compact-format pipeline integration test in `src/tests/tools/get_context_tests.rs`
- Registered new test module in `src/tests/mod.rs`

### IMPACT
- `get_context` is now more robust at preferring actionable code pivots for ambiguous queries.
- Output is less noisy and can be rendered in a token-lean compact form when requested.
- Regression protection now covers unit + integration paths for relevance and formatting behavior.
