## Root cause analysis complete for cross-workspace dogfood issues

Investigated all 8 dogfood issues and identified 3 systemic root causes.

### Root Cause 1: NL path prior is Julie-specific
- `apply_nl_path_prior()` at `src/search/scoring.rs:127-147` checks hardcoded paths: `src/tests/`, `src/`, `docs/`, `fixtures/`
- These don't match C# (`COA.CodeSearch.McpServer.Tests/`) or Python (`python/tests/`) layouts
- Makes the prior a **no-op** for all non-Julie workspaces
- Miller survives because strong centrality (max 396) compensates; coa (max 168) doesn't

### Root Cause 2: C# centrality is weak
- coa has 1,786 relationships (vs 11,236 for julie)
- Important classes like `LuceneIndexService`, `TextSearchTool` have ref_score: 0
- Top symbols are generic: `GetFileName: 168`, `ToString: 117`, `Equals: 81`
- C# extractor likely misses DI constructor injection, interface implementations

### Root Cause 3: Definition search doesn't over-fetch
- `fast_search` definitions mode uses user's limit directly
- Definitions rank low in Tantivy (mentioned once in definition vs many times in references)
- `limit=5` excludes `LuceneIndexService` definition; `limit=10` finds it correctly
- Fix: over-fetch candidates, promote definitions, then trim

### Evidence
- SQLite queries confirmed centrality data across all 3 workspaces
- Code inspection confirmed NL path prior has no workspace-conditional logic
- Pipeline inspection confirmed scoring IS applied to reference workspaces (just ineffective)
- Reproducible: `LuceneIndexService` definition found with limit=10 but not limit=5
