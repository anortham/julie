## Julie startup delay under Claude Code — root cause narrowed to SessionStart hook

User reports julie tools take ~6.5 minutes to appear in Claude Code, while spawning instantly under Codex CLI. Same binary, same daemon. Investigation isolated the layer responsible.

### Evidence chain

Process tree from prior cold start (Claude Code launched 18:18:15, julie tools appeared after `/mcp` reconnect at 18:24:52):

```
27002  18:18:15  claude --dangerously-skip-permissions
 ↓ (6 min 37 sec gap)
28397  18:24:52  node .../julie-plugin/.../hooks/run.cjs
 ↓
28398  18:24:52  julie-server (adapter)
```

The julie binary did NOT execute before 18:24:52. Whatever caused the delay happened before any julie code ran for this session.

### Comparison: instant vs slow startup

| Harness | MCP server invocation | SessionStart hook | Result |
|---|---|---|---|
| Codex CLI | Direct exec of `/Users/murphy/source/julie/target/release/julie-server` | `codex-sessionstart.cjs` (no `async: false`) | Instant |
| Claude Code (fresh) | `node run.cjs` wrapper | `session-start.cjs` with `async: false` | 6.5 min delay |
| Claude Code (`--continue`) | `node run.cjs` wrapper | Hook NOT fired (matcher `startup|clear|compact` doesn't match resume) | 1.2 seconds |

The `--continue` data point is strong: same node wrapper, same daemon, only difference is the SessionStart hook NOT firing. Result: instant. That isolates the SessionStart hook with `async: false` as the prime suspect.

### Why the hook would block MCP spawn

`async: false` in julie-plugin's `hooks.json` tells Claude Code to wait synchronously for the hook to complete. Hypothesis: Claude Code delays plugin MCP server spawn until all `async: false` SessionStart hooks complete. The hook itself runs in ~60ms standalone, so the bug is in Claude Code's orchestration, not in the hook script.

Goldfish (no SessionStart hook) confirmed this pattern by spawning instantly at 18:18:13 alongside Claude Code launch.

### What's instrumented

Added timing instrumentation to the v7.9.3 plugin cache (NOT to julie-plugin source):
- `~/.claude/plugins/cache/julie-plugin/julie/7.9.3/hooks/session-start.cjs` — logs ppid, elapsedMs, stdinBytes, exitCode to `/tmp/julie-session-start-debug.log`
- `~/.claude/plugins/cache/julie-plugin/julie/7.9.3/hooks/run.cjs` — logs ppid, pid, argv to `/tmp/julie-run-cjs-debug.log`

First test (post `--continue` restart at 19:16:04): `run.cjs` fired 1.2s after Claude Code launch, `session-start.cjs` never fired. Tools instant.

### Cache vs source drift discovered

`run.cjs` in julie-plugin source uses `binaryName: 'julie-adapter'` + `daemonBinaryName: 'julie-daemon'` (the May 16 3-binary split). Cache v7.9.3 still uses `binaryName: 'julie-server'`. Functionally OK because julie-server is a compat shim, but next plugin release will fail under dev-link unless dev-link also symlinks julie-adapter and julie-daemon.

### Pending verification test

User needs to do a FRESH `claude --dangerously-skip-permissions` (not `--continue`) in `~/source/julie` and `cat /tmp/julie-session-start-debug.log /tmp/julie-run-cjs-debug.log` after tools appear. If the gap between SessionStart timestamp and run.cjs timestamp is multi-minute, confirmed — fix is to drop `"async": false` from `~/source/julie-plugin/hooks/hooks.json`.
