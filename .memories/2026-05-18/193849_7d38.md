## Fixed adapter terminal transport recovery for daemon startup

### WHAT
Added recovery behavior for Julie's HTTP adapter when daemon transport is lost after the adapter has already written MCP output. The adapter now re-runs daemon readiness before terminal exit on post-output transport loss, and the forwarder now treats post-output daemon EOF with outstanding requests as a transport error instead of a normal session end.

### WHY
Observed logs showed adapters exiting after `Failed to send JSON-RPC message to HTTP MCP daemon` without restarting the daemon; the daemon only started when a later adapter process appeared. That left Codex/Claude sessions able to spawn adapters but not promptly recover the shared daemon.

### HOW
- Extracted HTTP stdio forwarding into `src/adapter/forwarder.rs` to keep `http_stdio.rs` within file-size standards.
- Added `DaemonAdapterControl` and `run_http_adapter_inner` so tests can verify lifecycle calls through the adapter loop.
- On terminal post-output `AdapterError::Transport`, `run_http_adapter_inner` calls `ensure_daemon_ready` before returning.
- On `receive() == None` while requests remain in flight after output, the forwarder now returns a post-output transport error instead of `SessionEnded`.
- Added red/green regressions for close failure after output and daemon EOF after output with an outstanding request.

### IMPACT
Adapters still do not replay post-output sessions, preserving non-idempotent tool safety. They now proactively spawn/verify the daemon before exiting, reducing delayed daemon recovery after crashes or stale transport loss. Existing stale-binary/update replacement tests still pass.
