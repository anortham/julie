## Fixed deferred-session target workspace path resolution

### WHAT
- Added `JulieServerHandler::workspace_index_dir_for()` and routed DB/Tantivy path helpers through it.
- In daemon mode, target workspace path resolution now uses `WorkspacePool::indexes_dir()/workspace_id` directly instead of requiring a primary workspace storage anchor.
- Added a regression test for deferred CWD session -> explicit target refresh -> explicit target `fast_search` while primary remains unbound.

### WHY
A deferred daemon session can intentionally have no initialized primary. `manage_workspace(refresh, workspace_id=...)` succeeds through the path-backed indexing route, but later target search/embedding path probes called `workspace_db_file_path_for()`, which used `workspace_storage_anchor()` and errored with `Primary workspace not initialized`.

### HOW
- Kept strict pool-membership checks in connection-opening helpers.
- Moved shared path construction to a helper that uses the daemon workspace pool when present, matching `IndexRoute::path_backed_route` behavior.
- Preserved stdio/non-pool behavior through the existing storage-anchor fallback.

### IMPACT
Explicit target workspace search and embedding path resolution no longer depend on initializing the session primary first. Refreshing `julie-plugin` can be followed by `fast_search(workspace="julie-plugin_...")` without hitting the primary-anchor error.
