## Adapter logging shipped — diagnostic infrastructure for Claude Code cold-start delay

### What

Added file-based tracing to the Julie adapter, which previously ran with **zero `tracing_subscriber` init**. All `info!`/`warn!`/`error!` calls in `src/adapter/launcher.rs` and `src/adapter/http_stdio.rs` were silently dropped.

### Why

User reports Claude Code takes **minutes** to start Julie when the daemon is down. Codex (which points its MCP `command` directly at `target/release/julie-server`) starts fast in the same scenario. With no adapter logs, we have zero visibility into:
- Whether `spawn_daemon` succeeded
- Whether the launcher is in its 6× retry loop (1+2+4+8+16+16s backoff stacked on top of 60s readiness timeout per attempt = multi-minute worst case)
- Whether daemon liveness detection is failing

Daemon-side timing is already fast (`discovery.json` published in ~39ms from cold start), so the bottleneck is somewhere in the adapter path that we couldn't see.

### How

- Extracted `install_file_tracing(log_dir, prefix, env_filter_default)` in `src/logging.rs` — shared between daemon and adapter so the appender, time formatter, and `OnceLock` idempotency contract stay in lockstep.
- Refactored `DaemonRuntimeContext::install_tracing` to delegate (was 60 lines of inline tracing setup → 1-line wrapper). Existing daemon idempotency test still passes.
- Called `install_file_tracing(&paths.julie_home(), "adapter.log", "julie=info")` from BOTH entry points: `src/main.rs` no-args branch (julie-server shim) and `src/bin/julie-adapter.rs`.
- Errors from logging install go to stderr (not stdout — MCP protocol owns stdout).

### Impact

Adapter now writes to `~/.julie/adapter.log.YYYY-MM-DD` with the same format as `daemon.log`. Unblocks diagnosis of:
- This specific cold-start issue (next step: user reproduces, we read the timeline)
- Every future adapter-side bug (launcher errors, retry loops, transport failures)

Release binary rebuilt at 19:37 CDT (`target/release/julie-{server,daemon,adapter}`). Plugin's symlink at `~/.claude/plugins/cache/julie-plugin/julie/7.9.3/bin/aarch64-apple-darwin/julie-server` already points at the dev release binary so Claude Code picks it up on next cold start.

### Daemon-side finding (rules out one hypothesis)

`src/daemon/app.rs:345` calls `lifecycle.startup_complete()` (writes "ready") immediately after HTTP MCP transport binds, **before** embedding sidecar init (which runs as background task). Cold daemon → ready in ~39ms based on existing daemon log evidence. So the slow part of cold start is NOT the daemon ready signal.
