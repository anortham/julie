## Embedding Enrichment Brainstorm and Plan Complete

### What
Completed the full brainstorming and planning cycle for Julie's embedding enrichment project. Produced:
- Design spec: `docs/superpowers/specs/2026-04-03-embedding-enrichment-design.md`
- Implementation plan: `docs/superpowers/plans/2026-04-03-embedding-enrichment.md`

### Why
Julie's embeddings are built from thin metadata (kind + name + signature + doc). Richer tree-sitter data (callees, types, file paths, implementors) isn't being leveraged. This weakens semantic search, similar-symbol discovery, and context selection.

### Key Decisions
- **Approach:** Behavioral embeddings (Approach B) + selective query classification from Approach C
- **No model change:** CodeRankEmbed stays, sqlite-vec stays
- **Three-pronged improvement:**
  1. Enriched embedding text (file path, implementor names, field signatures)
  2. Query classification heuristic for dynamic keyword/semantic weighting
  3. Benchmark-driven threshold tuning and get_context pivot improvement
- **Evaluation lens:** Intelligence per token. Every change must reduce total tokens an agent burns to reach understanding.

### Plan Structure (11 tasks)
1. Benchmark baseline capture (4 workspaces: Julie, Zod, Flask, Cobra)
2. File path in format_symbol_metadata
3. Trait/interface implementor name enrichment
4. Wire implementor map into pipeline
5. Field signature enrichment (types, not just names)
6. Query classification (QueryIntent enum + classify_query heuristic)
7. Integrate classification into hybrid_search
8. Re-index and capture after benchmark
9. Threshold tuning (benchmark-driven)
10. Workflow instruction updates
11. Dev test regression check

### Impact
Should improve: conceptual search quality, similar-symbol relevance, get_context pivot selection. All measured by before/after benchmark on real queries across 4 languages.
