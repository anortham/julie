## v6.5.7: 6 Indexing/Embedding Pipeline Fixes + Adapter Stale-Binary Retry

### What
7 commits fixing 4 bugs, 2 enhancements, and 1 adapter reliability issue:

1. **fix(search):** `matches_filter` now checks `exclude_tests` via `is_test_path()`, filtering test symbols from semantic candidates before RRF merge
2. **fix(embeddings):** `spawn_workspace_embedding` reads `embedding_count()` from workspace DB instead of using `stats.symbols_embedded` (run count)
3. **fix(index):** Gated embedding spawn on `db_mutated || force`; routed force-clear to reference workspace DB via `workspace_db_path()` instead of handler's primary DB
4. **fix(watcher):** `IncrementalIndexer.embedding_provider` changed to `SharedEmbeddingProvider` (`Arc<RwLock<...>>`), propagated at both lazy-init write-back sites (`embeddings.rs:81`, `nl_embeddings.rs:132`). `dispatch_file_event` now passes `LanguageConfigs` to `reembed_symbols_for_file`.
5. **fix(adapter):** Added retry loop (up to 2 retries, 2s pause) around initial connection so stale-binary daemon shutdowns are transparent instead of requiring manual `/mcp` reconnect

### Key Discovery
Code review caught that `initialize_embedding_provider()` propagation in `workspace/mod.rs` was dead code: always called on workspace clones (which have `watcher: None`). Had to add propagation at the actual write-back sites in `embeddings.rs` and `nl_embeddings.rs`.

### Verification
- 8 xtask dev buckets green
- Bug 4 verified live: `exclude_tests=true` filters test results from semantic path
- Enh 5 verified live: second `index` call skips embedding pipeline
- Adapter fix verified live: 9-second transparent restart vs manual reconnect
- Bug 3 vector_count verified via characterization test (can't query vec0 from CLI sqlite3)

### Impact
- No more data loss on force-indexing reference workspaces
- Incremental file edits now get semantic embeddings updated (previously always None)

- Stale-binary restarts are transparent (~9s vs manual reconnect)
