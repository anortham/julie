## WHAT
Implemented the GDScript lane for Task 13: function signatures now use synthetic declaration text, identifier extraction now records the rightmost method name for attribute/member calls, and relationship extraction now mirrors that call target handling.

## WHY
The previous extractor behavior leaked full function bodies into signatures and mis-modeled attribute-style calls, which polluted the symbol/index output and broke the intended call-name semantics.

## HOW
- Added a synthetic signature helper in `crates/julie-extractors/src/gdscript/functions.rs` and reused it for constructors.
- Updated `crates/julie-extractors/src/gdscript/identifiers.rs` to detect attribute-call suffixes and use the method identifier node instead of the dotted chain or argument identifier.
- Updated `crates/julie-extractors/src/gdscript/relationships.rs` to resolve attribute-style calls using the same rightmost method name and receiver text.
- Added/updated GDScript regressions for function signatures, attribute-call identifiers, relationship target naming, and static function type inference.

## IMPACT
The GDScript extractor now models declarations and call targets more sanely, and the new tests pin the expected behavior. Verification is currently blocked by unrelated Bash compile errors from a parallel worker, so the GDScript tests could not be run to a clean green finish in this session.
