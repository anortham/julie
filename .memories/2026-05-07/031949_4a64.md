## Task 5 amended: removed catchup_in_progress dedup flag

Lead pushed back on initial keep-decision for `catchup_in_progress`. Walked through the three-session scenario and confirmed CAS dedup silently drops legitimate catch-up work windows.

### Scenario the gate fixes (CAS could not)
- Sessions 1, 2, 3 connect within 1 second
- S1 wins CAS, snapshots workspace mtime, runs 30s catch-up
- S2/S3 lose CAS, log "skipping", return immediately
- User creates `foo.rs` mid-run after S1's snapshot
- S1 finishes; flag clears; S2/S3 already returned
- `foo.rs` is missing from index until next external trigger

With the gate alone, S2/S3 block on `acquire_gate(workspace_id)` instead of bailing. When each acquires, `filter_changed_files` short-circuits via mtime+blake3 if nothing changed, runs the incremental update otherwise. Redundant invocations are cheap; silently-dropped invocations are bugs.

### Changes
- Removed `catchup_in_progress` field + 3 initializers in handler.rs
- Removed CAS guard + flag-clear in `run_auto_indexing`; replaced doc comment to explain gate serialization
- Removed local `use std::sync::atomic::Ordering` (now unused in run_auto_indexing)
- Deleted tautological `test_catchup_dedup_flag_prevents_concurrent_scans` (tested AtomicBool::compare_exchange, not the handler)
