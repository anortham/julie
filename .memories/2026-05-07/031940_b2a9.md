## Task 6 Complete: INFO-level watcher observability

### WHAT
Implemented INFO-level event observability for the file watcher on the `filewatcher-pause-architecture` branch.

### WHY
Daemon log had 16,531 INFO lines and 0 DEBUG lines visible at operator log level. After content changes, no file-level indexing activity was visible. Gate contention from catch-up indexing was completely silent.

### HOW
Created `src/watcher/observability.rs` with:
- `RateLimiter`: token-bucket using `AtomicU64` (window_start_nanos + window_count), no new crates. Configurable window duration for testing.
- `timed_acquire_gate`: wraps `acquire_gate`, logs INFO with wait_ms if elapsed >= threshold (100ms default).
- `LogCapture` + `LogCaptureLayer`: tracing_subscriber::Layer for tests. Captures INFO+ events into shared Arc<Mutex<VecDeque>>.

Promoted 2 key `debug!` calls in `handlers.rs` to `info!`:
- Hash-match skip: "Watcher: {relative_path} unchanged (hash match), skipping re-index"
- Extraction result: "Watcher: extracted N symbols, M identifiers, K relationships from {relative_path} ({language})"

### Key discovery
Task 5 (pause-removal) ran in parallel and already committed the scaffolding:
- `pub mod observability` in `mod.rs` (commit 23f49344)
- `timed_acquire_gate` wiring in `runtime.rs` replacing all 5 `acquire_gate` calls (commit c5397413)
- `pub mod watcher_observability` in `tests/mod.rs`

So this task only needed to create the 2 new files + modify handlers.rs.

### Tests
Fixed test structure: used `tracing::subscriber::set_default` instead of `with_default(..., || block_on(...))` to avoid "cannot start runtime from within runtime" panic in `#[tokio::test]` context.

### IMPACT
6/6 tests pass. Operators now see file path + symbol count in daemon.log for every indexed file, "unchanged" for hash matches, and gate wait times when catch-up holds the lock > 100ms.
