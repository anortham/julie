## WHAT
Updated the Lua extractor so function symbols get synthetic declaration signatures, and bare `require(...)` calls become import relationships with structured pending metadata.

- Function definitions now use one-line signatures instead of the full body text.
- Bare `require` calls now emit `RelationshipKind::Imports` pending relationships when unresolved.
- Resolved local import symbols are preferred when the target name already exists in the file.

## WHY
Lua signatures were leaking bodies into the symbol index, which makes search noisy and produces ugly symbol cards. Bare `require` calls were also disappearing from the relationship layer, so imports were not tracked unless they happened to be assigned to a local variable.

## HOW
- Added a small signature builder in `crates/julie-extractors/src/lua/functions.rs` that assembles `function` or `local function` headers from the parsed name and `parameters` field, then normalizes whitespace.
- Extended `crates/julie-extractors/src/lua/relationships.rs` to detect `require` calls, parse the target string or expression, and emit import relationships or structured pending imports with `import_context` metadata.
- Added regression tests for the Lua function signature and bare require path in `src/tests/lua`.

## IMPACT
- Lua function symbols are now searchable without dragging their bodies around.
- Bare module loads are visible as import relationships instead of being swallowed as ordinary calls or ignored entirely.
- Exact worker tests passed for both regression cases, so the lane is green at the narrow test level.
