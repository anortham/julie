## Julie tree-sitter findings remediation plan

### WHAT
I split `docs/TREE_SITTER_REVIEW_FINDINGS_STATUS.md` into a gated remediation graph instead of trying to solve the whole review in one lump.

- Parent planning task: `t_a60f28db` — Fred drafts the remediation order and assigns owners.
- Child workstreams:
  - `t_ab5cbe7d` — TS-RF-001/002: resolved relationship ID collisions + embedded span rekeying
  - `t_1998f4c5` — TS-RF-003/004: Vue SFC identifier spans + exact `<script setup>` line assertions
  - `t_ed069b93` — TS-RF-005: HTML embedded body offset detection
  - `t_62b607c6` — TS-RF-006: SQL view/trigger relationship coverage
  - `t_6b3dcadf` — TS-RF-007: regex fallback symbol spans
  - `t_676924d0` — TS-RF-008 review: capability matrix / certification gap alignment

### WHY
The findings are mixed: some are hard correctness bugs, some are test gaps, and some are certification/reporting debt. Grouping them by mechanism lets the crew work in parallel without overlapping file ownership or muddying the review chain.

### HOW
I used the findings doc plus direct code inspection to bucket the work into:
- core relationship identity / normalization,
- Vue SFC span correctness,
- HTML embedded offset detection,
- SQL extractor coverage,
- regex fallback span creation,
- capability/report validation.

The parent planning task is the gate; the implementation tasks stay blocked until Fred sets the exact sequence.

### IMPACT
This gives Julie a durable remediation map without starting code changes prematurely. It also keeps the review loop clean: first sequence and ownership, then TDD-based implementation, then a final coverage check on the capability/report side.
