## Completed Task 14 extractor idiom remediation

### WHAT
Implemented Task 14 from `docs/plans/2026-05-06-tree-sitter-extractor-audit-remediation.md` across PHP, C#, Java, Scala, Swift, Kotlin, and Dart extractor paths.

Key changes:
- PHP constructor property promotion now emits class-parented property symbols with visibility, signatures, and `propertyType` metadata.
- PHP namespace-aware pending targets now keep qualified display/callee names plus namespace path metadata.
- C# local functions and stable-name lambdas are extracted, and partial class declarations get full-name plus `partial` marker linkage.
- Java record components use parser `type`/`name` fields and emit record-parented property symbols for qualified, generic, and array component types.
- Scala case-class constructor fields are property symbols, `val` kind depends on scope, and non-function annotations attach to class/object/property/type-alias symbols.
- Swift extensions are no longer plain class symbols; they use `SymbolKind::Module` plus `symbol_role = extension` metadata. Non-function annotation metadata now covers extensions, type aliases, enum cases, and properties.
- Kotlin and Dart non-function annotation coverage now includes class/object/property or field/type-alias paths.

### WHY
Task 14 addressed audit findings around language idioms that were previously modeled incompletely or misleadingly, especially constructs that affect symbol search, parent-child navigation, and relationship resolution.

- Refactoring new C# helpers into `local_callables.rs` and `partial_classes.rs` to keep touched implementation files under the 500-line limit.
- Moving new Task 14 tests into focused per-language `task14.rs` files instead of expanding already oversized legacy test modules.
- Consolidating duplicated Swift `annotation_keys_csv` helpers into one `SwiftExtractor` helper.
- Fixing the Dart annotation path after exact verification exposed that class fields are extracted through `dart/members.rs`, not `dart/functions.rs`.
