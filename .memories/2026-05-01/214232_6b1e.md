## Fixed CLI daemon MCP initialization

- **WHAT:** Updated the CLI daemon client so it performs the MCP `initialize` request and `notifications/initialized` notification before sending `tools/call` over the daemon IPC stream.
- **WHY:** The daemon session is served by rmcp, which expects `initialize` as the first MCP request. The CLI client sent `tools/call` first, so rmcp rejected the session and the CLI fell back to standalone mode.
- **HOW:** Added an MCP initialization step to `DaemonClient`, switched response reading to byte-wise newline reads so buffered data is preserved, and added a Unix stream-pair regression test that proves the JSON-RPC order is `initialize`, `notifications/initialized`, then `tools/call`.
- **IMPACT:** CLI tool calls can use daemon mode again instead of burning time on standalone fallback.
