## Fixed Julie Daemon Health And Startup Repair

### WHAT
- Rebuilt and restarted Julie after fixing daemon-facing regressions in CLI MCP initialization, health reporting under SQLite lock contention, and startup catch-up repair.
- Added regression coverage for CLI MCP initialize-before-tool-call, health cached counts when the symbol DB is busy, and deleted-file startup cleanup.
- Started the rebuilt release daemon through launchd because Codex's exec harness cleans child processes spawned by ordinary CLI commands.

### WHY
- Dashboard showed runtime/data degradation after the prior release and the CLI daemon tool path was not performing an MCP initialize handshake.
- Health reported fake `1 symbols across 0 files` while SQLite was busy, which made the dashboard misleading during catch-up.
- Startup repair detected deleted files but could race active embedding work and leave stale rows, causing repeat catch-up churn.

### HOW
- `DaemonClient` now performs MCP `initialize`, sends `notifications/initialized`, and reads newline JSON without `BufReader` over-read.
- Health attaches daemon registry stats to primary workspace state and uses them when the SQLite mutex is busy.
- Startup repair cancels active primary embedding work, runs index with embeddings skipped, and `run_auto_indexing` no longer runs a duplicate index after repair.
- Verified with targeted nextest cases, `cargo xtask test changed`, `cargo fmt --check`, `cargo build --release`, CLI daemon `fast_search`, and dashboard `/status/live`.

### IMPACT
- Dashboard live status is green: overall ready, data plane ready, runtime plane ready, MPS sidecar initialized.
- CLI daemon calls work against the rebuilt daemon.
- Current Codex MCP Julie transport in this thread still reports closed, so this session may need an MCP reconnect/restart to use Julie tools directly again.
