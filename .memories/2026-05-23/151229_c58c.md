## Implemented fast_search backend phase 1

### WHAT
Added the first backend-param phase for `fast_search`:
- New shared `SearchBackend` enum and resolved backend state.
- `FastSearchTool.backend: Option<SearchBackend>` wired through serde, defaults, direct callers, dashboard/search matrix callers, and `SearchExecutionParams`.
- Explicit `backend=semantic` runs KNN symbol search.
- Explicit `backend=hybrid` runs existing hybrid RRF search.
- Explicit semantic/hybrid fallback to lexical when provider is unavailable or the workspace has zero stored embeddings.
- `SearchTrace.backend_fallback` and fast_search telemetry serialization.
- Extracted `wait_for_embedding_provider_settled()` from NL embedding lazy init for reuse.
- Added backend fixture tests and registered the new test module in xtask search buckets.

### WHY
The plan called for exposing embeddings through existing tools without adding another MCP tool. This implements the fast_search slice while preserving omitted-backend lexical behavior.

### HOW
- Kept lexical path in `execute_search_unified` as the fallback/default path.
- Added an early semantic/hybrid dispatch only when resolved backend is not lexical.
- Semantic/hybrid results are symbol-backed only and preserve actual symbol kind via `SymbolSearchResult` -> `SearchHit::from_symbol` conversion.
- Reused daemon settlement behavior for explicit semantic/hybrid requests; stdio init behavior remains unchanged.
- Added a user-visible fallback note in rendered fast_search output only when fallback was caused by explicit backend request.

### IMPACT
Phase 1 fast_search backend opt-in is implemented and verified. `deep_dive` and `get_context` backend parameters are still not implemented in this checkpoint.
