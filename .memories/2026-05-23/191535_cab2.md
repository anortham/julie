## Fixed fast_search locations dogfood defects and backend docs

### WHAT
- Fixed `fast_search return_format="locations"` so exact file/path and structured symbol-style queries keep unified ranking instead of being replaced by line-mode content hits.
- Added locations scope-rescue header handling so scoped misses clearly say when results are outside `file_pattern`.
- Prevented docs-role candidates from receiving the reranker's source-language phrase bonus when their language looks source-like (for example docs HTML).
- Updated implemented `fast_search backend` descriptions in MCP tool docs, `JULIE_AGENT_INSTRUCTIONS.md`, Julie source skills, plugin mirror skills, and Razorback docs.

### WHY
Dogfooding showed full-format lexical ranking was mostly correct, while locations mode corrupted it for file/path and symbol-shaped queries such as `application.js`, `projection_policy.py`, and `AuthenticationInterceptor RefreshWindow`. The backend param also existed in code without enough user-facing/tool-facing guidance.

### HOW
- Added failing tests for exact filename locations ranking and scope-rescue note preservation in `src/tests/tools/search/file_mode_tests.rs`.
- Kept line-mode locations only for content-like prose or marker queries; path-like and structured lookup queries now render unified locations.
- Added a reranker test proving docs-role HTML does not receive `PHRASE_SOURCE_LANG_BOOST`.
- Ran `cargo xtask sync-plugin` to mirror updated Julie skills into `~/source/julie-plugin`.

### IMPACT
Locations output now matches the high-quality unified ranking for file/path and symbol-like queries while preserving line-level locations for content markers. Agents now get explicit guidance that only `fast_search` currently has `backend`, with lexical for file/path and semantic/hybrid for concept-to-symbol discovery.
