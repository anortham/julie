## Fixed semantic backend locations-mode dogfood regression

### WHAT
Dogfooding the rebuilt MCP server showed `fast_search(..., backend="semantic", return_format="locations")` returned lexical line-mode locations instead of semantic symbol locations. Added a regression test and fixed `FastSearchTool::execute_with_trace` so line-mode snippet/location supplementation only runs for lexical executions, not `fast_search_semantic` or `fast_search_hybrid` executions.

### WHY
Semantic/hybrid fast_search backends are explicitly symbol-only. Locations mode was violating that contract by replacing semantic results with lexical `try_line_mode_locations()` output whenever there was no exact symbol-name match.

### HOW
- Added `semantic_backend_locations_render_semantic_hits_not_lexical_line_mode` to `src/tests/tools/search/backend_param_tests.rs`.
- Fixture includes a lexical-only line match in `src/notes.rs` and a semantically nearest symbol in `src/lib.rs`.
- The RED test failed with output from `src/notes.rs:1`.
- Added `symbol_backend_active` guard in `src/tools/search/mod.rs` to skip line-mode enrichment/locations for `fast_search_semantic` and `fast_search_hybrid` trace strategies.

### IMPACT
Current source preserves semantic/hybrid backend output in locations mode. The live MCP server still needs another rebuild/restart before this dogfood fix is active there.
