## Took down broken julie v7.13.3 release + rolled back julie-plugin

The v7.13.3 release pushed to GitHub tonight was broken. User asked to take it down until it can be investigated.

- **julie release:** Converted the `v7.13.3` GitHub release to a **draft** (`gh release edit v7.13.3 --draft`). Hidden from users, no longer "latest"; tag, 4 binary assets, and notes all preserved. `v7.13.2` is latest again. Git tag `v7.13.3` (203339b7) left intact.
- **julie-plugin rollback:** Remote `main` was shipping v7.13.3 via orphan commit `fdedd01`. Restored to last-good v7.13.2 state (`4165fcb`, "launch julie-server directly") with a non-destructive roll-forward commit `7ca54d07` (tree of 4165fcb, parent = fdedd01) via the GitHub API — no force-push, fully reversible. Now: package.json + .claude-plugin/marketplace.json → 7.13.2, bin/archives → v7.13.2 binaries.

### Key discovery
- The julie-plugin **release workflow force-pushes a fresh ORPHAN commit each release** (no shared history). Confirmed via GitHub API: `fdedd013` has `parents: []`. That's why a plain `git revert` couldn't apply and each release wipes the previous tree. Makes rollbacks fragile — worth fixing.

### Not yet done
- Root cause of *why* v7.13.3 is broken — NOT investigated. User wanted takedown first.

### Gotcha
- `gh release view --json` rejects `tag`/`isLatest` fields (use `tagName`; latest is implicit via no-arg `gh release view`).
- zsh glob-expands `parents[]=` in `gh api -f` — use `jq -nc | gh api --input -` with a JSON body instead.
