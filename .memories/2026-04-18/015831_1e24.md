## Wrote agent tool surface implementation plan

### WHAT
Created `docs/plans/2026-04-17-agent-tool-surface-implementation.md` as the execution plan for the approved tool-surface redesign.

### WHY
The design doc settled the product decisions, but the branch still needed concrete implementation slices with real file targets and acceptance criteria. Without that, the work would drift across three subsystems and the docs cleanup would get dropped.

### HOW
Mapped the codebase before writing the plan:
- `src/handler.rs` ranges for `query_metrics`, `fast_refs`, and `edit_symbol`
- `src/tools/mod.rs`, `src/tools/metrics/mod.rs`, `src/tools/editing/mod.rs`, and `src/tools/navigation/mod.rs`
- dashboard metrics routes and templates
- existing dashboard, metrics, editing, and tracing tests

The plan stays light because execution is expected in-session. It breaks the work into four tasks:
1. remove `query_metrics` and shift visibility to the dashboard
2. replace `edit_symbol` with AST-backed `rewrite_symbol`
3. add bounded shortest-path `call_path`
4. align agent instructions and site examples with the live tool surface

### IMPACT
The branch now has a concrete execution path with narrow task boundaries, exact file paths, and acceptance criteria. The next step is execution choice, not more design reconstruction.
