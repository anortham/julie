## Completed Task 1: replace `add` with `register`

Implemented the first slice of the workspace-management hard-C plan in the worktree branch.

- Replaced the public `manage_workspace(operation="add")` command surface with `register`
- Moved the registry command file from `add_remove.rs` to `register_remove.rs`
- Added `handle_register_command()` that canonicalizes the path, reuses an existing workspace row when present, indexes or refreshes the target workspace, persists stats/status, and leaves the workspace inactive for the current session
- Updated request routing/comments so `register` is not treated as a primary-targeting operation during deferred-session preflight
- Updated README and workspace architecture docs to advertise `register` instead of `add`
- Reworked global-targeting tests to prove the new semantics and to reject the legacy `add` operation

## WHY

The old `add` semantics encoded a dead pairing model and forced confusing primary-binding behavior. The hard-C design calls for a clean split where `register` means "track and index this workspace" and `open` remains the only activation boundary.

## HOW

- Renamed the command variant and operation dispatch in `src/tools/workspace/commands/mod.rs`
- Added a new register path in `src/tools/workspace/commands/registry/register_remove.rs`
- Removed pairing-oriented request-targeting commentary in `src/handler.rs`
- Updated docs and focused tests in `src/tests/tools/workspace/global_targeting.rs`
- Verified behavior with narrow `cargo nextest run --lib <exact_test_name>` loops

## IMPACT

- Deferred sessions can register a workspace without silently binding a primary workspace
- Legacy `add` calls now fail fast as unknown operations instead of steering users into stale behavior
- The public command surface now matches the hard-C design direction, which clears the way for pairing-table removal and cleanup-event work in Task 2
