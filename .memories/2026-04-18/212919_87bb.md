## Workspace management implementation plan written

### WHAT
Created a light implementation plan at `docs/plans/2026-04-18-workspace-management-hard-c-implementation-plan.md`.

### WHY
The design was approved and the next step needed concrete execution slices, exact file ownership, and testable acceptance gates. The cut spans command surface, daemon storage, cleanup behavior, dashboard mutations, and lingering terminology debt, so an implementation plan was needed before autonomous execution.

### HOW
- Chose a light plan because the work is likely to execute in-session with subagents rather than as an async handoff.
- Oriented on the codebase with Julie tools before planning:
  - `get_context` for workspace, cleanup, and dashboard seams
  - `deep_dive` on `JulieServerHandler::manage_workspace`, `DaemonDatabase::add_reference`, and cleanup-adjacent symbols
  - `get_symbols` and `rg -n` for exact files and line anchors
  - `fast_refs` to map pairing-storage callers
- Decomposed the work into four tasks:
  1. Replace `add` with `register`
  2. Remove pairing storage and add shared cleanup
  3. Turn Projects into the workspace control surface
  4. Purge stale terminology and update cross-workspace tests
- Pulled `workspace/registry.rs` into scope after finding old primary/reference TTL semantics still encoded there.
- Kept the dashboard route split explicit by planning a new `projects_actions.rs` file to avoid blowing past the repo's 500-line file target.

### IMPACT
- The next execution pass has concrete file paths, acceptance criteria, and module-shape decisions.
- The plan now covers both user-facing cleanup and the internal stale terminology that would otherwise keep reviving the old model.
- Review can happen on one focused implementation plan instead of an ad hoc pile of changes.
