## Workspace management design approved

### WHAT
Wrote and approved a design spec for the next workspace-management pass at `docs/plans/2026-04-18-workspace-management-hard-c-design.md`.

### WHY
The codebase already moved to global workspace targeting, but commands, dashboard copy, storage, and tests still carry `reference workspace` semantics. That drift makes workspace cleanup, dashboard actions, and watcher reasoning harder than they need to be.

### HOW
- Defined the post-cut model in terms of **current**, **known**, **active**, and **target** workspaces only.
- Chose a clean command split:
  - `register` for non-activating index creation or refresh
  - `open` for path or ID based activation into the current session
- Chose a clean storage cut:
  - remove `workspace_references`
  - no compatibility alias for `add`
  - no favorites or recents in this pass
- Defined stale-workspace pruning rules tied to missing paths, inactive session state, watcher state, and in-flight indexing state.
- Scoped the dashboard around workspace management actions: add, open, refresh, delete, plus a compact cleanup log.

### IMPACT
- Gives the implementation pass one honest workspace model instead of a split-brain one.
- Unblocks dashboard management work without inventing dashboard-only semantics.
- Makes deleted worktree cleanup part of the product model instead of a manual chore.
- Deletes stale lore from commands, docs, and tests before it spreads further.
