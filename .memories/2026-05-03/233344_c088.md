## HTTP daemon startup and admission gates

### WHAT
- Wired `run_daemon` to start the real Streamable HTTP MCP transport using `HttpJulieService`.
- Added per-daemon bearer token generation and discovery publication from daemon startup.
- Added `HttpSessionAdmission` so HTTP initialize requests reuse the existing IPC stale-binary and version gate decisions.
- Deferred Julie session tracker registration until HTTP initialize passes admission and workspace-header parsing.
- Added focused tests for daemon HTTP discovery, version mismatch rejection, stale binary rejection, and rejected-session dashboard silence.
- Updated the HTTP transport plan ledger and Task 4a notes.

### WHY
The HTTP transport module existed, but production daemon startup did not expose it. More importantly, HTTP sessions would have bypassed the restart/version rules that keep adapter and daemon binaries in sync.

### HOW
- Reused `stale_binary_accept_action` and `version_gate_action` from daemon lifecycle.
- Kept rmcp as the transport/session envelope but moved Julie-owned session tracker registration behind admission.
- Verified with narrow HTTP tests first, then `changed`, then one `dev` gate.

### IMPACT
- The daemon now publishes a live localhost Streamable HTTP MCP endpoint with private token discovery.
- Rejected HTTP initialize requests do not start workspace sessions or publish dashboard session changes.
- IPC compatibility remains covered through the affected `transport`, `lifecycle`, and `daemon` buckets.
