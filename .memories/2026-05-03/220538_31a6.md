## HTTP Transport Task 0-1 Batch

### WHAT
- Locked Julie's `rmcp` dependency to the intentional 1.6 line and enabled the Streamable HTTP client feature set needed for the stdio shim plan.
- Added a structured daemon transport endpoint contract that supports legacy IPC and future Streamable HTTP discovery.
- Added dedicated MCP transport discovery/token paths separate from the dashboard `daemon.port` file.
- Updated `DaemonLauncher` readiness fallback to probe the discovered canonical transport when `daemon.state` is unavailable, while preserving IPC fallback.
- Updated the HTTP transport plan with SDK audit notes, completed Task 0/1 criteria, and verification evidence.

### WHY
The HTTP daemon transport plan needs a safe discovery/probe seam before wiring the actual Streamable HTTP MCP server. Reusing the dashboard port file or treating stale HTTP state as ready would recreate the same lifecycle fragility this architecture cleanup is trying to remove.

### HOW
- Used TDD: added failing tests for HTTP discovery serialization, live/stale HTTP readiness probing, MCP path separation, and adapter readiness through HTTP discovery.
- Kept existing IPC methods working by making `TransportEndpoint::new` construct the legacy IPC variant and preserving IPC bind/connect behavior.
- Stored auth material as a token path in discovery JSON, not as the bearer token value.
- Verified with exact tests first, then lead-owned `changed` and `system` gates.

### IMPACT
This gives the next plan slice a clean place to publish the localhost Streamable HTTP MCP server without port scanning, dashboard coupling, or changing stdio tool output formats.
