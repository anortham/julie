## Session Detach Moved Out Of Runtime Pool

### WHAT
Moved session detach ownership from `WorkspacePool` to `WorkspaceSessionAttachment`.

- Added `SessionWorkspaceState::mark_workspace_detached`.
- Added `WorkspaceSessionAttachment::detach_workspace_once` and `detach_workspace_resources`.
- Removed `WorkspacePool::disconnect_session` and the watcher pool field from `WorkspacePool`.
- Rewired IPC session cleanup and dashboard workspace cleanup to detach through the attachment service.
- Updated daemon, roots, dashboard, and integration tests that still encoded the old pool-owned detach behavior.
- Recorded new verification rows in the workspace service split plan.

### WHY
The previous slice moved attach side effects out of the runtime pool, but detach still lived there. That left the architecture split in half: sessions attached via one service and detached via a runtime cache. That is confusing and leak-prone.

### HOW
Used TDD around service-owned detach:

- Wrote a failing test for idempotent detach decrementing session count and watcher refs once.
- Implemented detach on `WorkspaceSessionAttachment` and session-state removal.
- Replaced remaining `WorkspacePool::disconnect_session` call sites with service detach calls or service resource cleanup.
- Kept `WorkspacePool` focused on runtime cache operations.

### IMPACT
Session attach and detach side effects now have one owner. Watcher refs and daemon session counts no longer depend on `WorkspacePool` as a hidden lifecycle proxy.
