## Runtime Pool Side Effects Moved To Session Attachment

### WHAT
Moved the session attach side effects out of `WorkspacePool::get_or_init` and into `WorkspaceSessionAttachment`.

- `WorkspacePool` now initializes and returns `JulieWorkspace` runtimes without incrementing daemon `session_count` or attaching watcher refs.
- `WorkspaceSessionAttachment` now owns daemon `session_count` increments and watcher `attach` calls.
- `JulieServerHandler` uses the attachment service for initial shared workspace construction and deferred primary rebinds.
- Updated daemon tests that still treated raw runtime lookup as a session attach.
- Added verification ledger rows to `docs/plans/2026-05-03-workspace-service-split.md`.

### WHY
The workspace service split plan needs a clean boundary: runtime lookup should not also mean session ownership. The old coupling made tests and production paths depend on hidden side effects, which is exactly how watcher/session leaks sneak in.

### HOW
Used TDD around the boundary:

- Added/kept failing coverage proving `get_or_init` has no session-count or watcher side effects.
- Routed initial handler attach and deferred rebind attach through `WorkspaceSessionAttachment`.
- Converted stale watcher and IPC tests to attach through the service before verifying disconnect behavior.
- Kept `WorkspacePool::disconnect_session` as the temporary detach compatibility bridge.

### IMPACT
The code now has one clearer owner for session attach side effects. This also exposed and cleaned up old tests that were accidentally specifying the previous coupling instead of the intended behavior.
