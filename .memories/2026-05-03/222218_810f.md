## HTTP Transport Server Module

### WHAT
- Added `src/daemon/http_transport.rs`, a daemon-owned Streamable HTTP MCP transport module using `rmcp::StreamableHttpService` and `LocalSessionManager`.
- Added tests for 127.0.0.1-only binding, discovery publication after readiness, cleanup on shutdown, explicit SDK session policy, and a real MCP `initialize` POST through the mounted rmcp service.
- Registered `src/tests/daemon/http_transport.rs` and updated the xtask `transport` bucket so HTTP transport tests run with the transport specialist gate.

### WHY
Task 2 needed a tested HTTP server module before touching the stdio shim or deleting IPC. The key risk was accidentally publishing a port before the server could answer, or thinking the transport bucket covered HTTP tests when it did not.

### HOW
- Used a generic `Service<RoleServer>` factory so the module can mount rmcp now without prematurely duplicating Julie session construction.
- Bound only `127.0.0.1` for this batch; `::1` remains rejected until separately tested.
- Used a readiness route `/mcp/ready` and waited for it before writing `daemon-mcp-transport.json`.
- Made SDK session defaults explicit via `HttpTransportConfig::session_config()`.
- Updated `xtask/test_tiers.toml` and manifest contract expectations to include `tests::daemon::http_transport` in the `transport` bucket.

### IMPACT
The HTTP transport server shape is now real and testable, but real `JulieServerHandler` wiring still needs the Task 4/5 workspace-startup header/service-wrapper contract because rmcp's service factory is synchronous while Julie handler construction is async and workspace-specific.
