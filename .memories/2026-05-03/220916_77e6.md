## Fixed Stale HTTP Discovery IPC Fallback

### WHAT
During inline review of the Task 0-1 HTTP transport batch, I found and fixed a migration bug: a stale `daemon-mcp-transport.json` file could prevent readiness fallback to a live legacy IPC endpoint when `daemon.state` was missing.

### WHY
Old/no-state daemon compatibility is one of the hidden invariants in the launcher. If stale HTTP discovery blocks IPC fallback, an adapter can sit in `Starting` even though the daemon is actually accepting IPC connections.

### HOW
- Added `test_readiness_ready_via_ipc_when_http_discovery_is_stale` as a RED test.
- Updated `DaemonLauncher::probe_transport_endpoint` to try the discovered endpoint first, then fall back to IPC when the discovered endpoint is not IPC and is not ready.
- Verified the new test, the stale-without-IPC test, `cargo check`, and the `transport` xtask bucket.

### IMPACT
Keeps the HTTP transport discovery migration compatible with legacy IPC fallback and avoids a false-starting daemon state caused by stale HTTP discovery files.
