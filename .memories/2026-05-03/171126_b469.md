## Implemented test runtime gate reliability

### WHAT
- Added a dedicated `xtask-runner` bucket and routed xtask plus harness-doc changes to it.
- Split coarse `tools-search` and `tools-misc` buckets into smaller, named buckets with command-level timing.
- Added `cargo xtask test inventory --bucket <name>` and `--tier <name>` as a report-only nextest-list audit.
- Updated AGENTS, CLAUDE, and the Testing Guide to document bucket execution, inventory, and worker test ownership.
- Added contract tests for manifest bucket specs, docs synchronization, inventory reporting, and changed-file routing.

### WHY
The previous test workflow made small changes expensive because broad buckets hid where time went and agents could rerun costly suites unnecessarily. The branch needed smaller gates, better diagnostics, and docs that tell agents exactly what they can and cannot run.

### HOW
- Kept `cargo xtask test` as the canonical runner.
- Replaced `tools-search` and `tools-misc` in `dev` and `full` with split buckets based on existing module boundaries.
- Added command timing lines in the existing runner flow in an earlier commit, then used those timings to calibrate expected seconds for the split buckets.
- Implemented inventory with a fake-executor-tested trait so tests do not invoke real cargo.
- Fixed a dev-gate failure where `tests::tools::get_symbols_target_filtering::` selected zero tests because that module only contains ignored slow tests; dogfood keeps the non-ignored target-filtering coverage isolated.

### IMPACT
- `cargo xtask test changed` for this batch selected only `xtask-runner` and passed in 1.6s.
- `cargo xtask test dev` passed in 371.4s and now prints per-command timings for every bucket.
- Future agents can run a focused bucket or inventory audit without turning a tiny edit into a multi-hour loop.
