## Cleanup Activity Facade Added

### WHAT
Centralized workspace cleanup activity checks behind `WorkspaceCleanupActivity`.

- Added `WorkspaceCleanupActivity` in registry cleanup code to own watcher ref checks, live indexing checks, and runtime eviction during cleanup.
- Changed cleanup helpers to accept the activity facade instead of raw `WorkspacePool` and `WatcherPool` options.
- Added registry module helpers for handler-based registry store and cleanup activity construction.
- Updated open, register, remove, list, clean, daemon cleanup sweep, dashboard workspace state, and cleanup tests to use the new facade.
- Recorded verification ledger rows for the Task 5 slice.

### WHY
Task 5 was still leaking lifecycle resource wiring through every registry command. Even though behavior was correct, command code was manually threading runtime and watcher pools into cleanup policy. That kept the architecture muddy.

### HOW
Used compile-red refactoring:

- Updated an integration cleanup test to call `run_cleanup_sweep` with a not-yet-existing `WorkspaceCleanupActivity`.
- Implemented the facade and moved watcher/runtime activity checks into it.
- Replaced raw command wiring with `registry_store_for_handler` and `cleanup_activity_for_handler`.
- Ran narrow command, cleanup, dashboard, and daemon buckets.

### IMPACT
Registry commands now depend on store and cleanup activity boundaries instead of directly composing low-level watcher/runtime cleanup policy at each call site. Behavior stayed stable in the targeted gates.
