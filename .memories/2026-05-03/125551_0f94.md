## Updated Razorback Model Routing For Architecture Cleanup

### WHAT

Updated `RAZORBACK.md` so coupled Julie architecture work defaults to stronger routing when shared invariants are involved. Also normalized the May 3 plan docs so they point back to `RAZORBACK.md` instead of copying the global model table.

### WHY

The upcoming plans touch daemon lifecycle, HTTP transport/security, projection readiness, workspace session/watcher ownership, dead-code classification, and xtask verification routing. Those are exactly the areas where cheap-by-default workers can produce passing narrow tests while missing hidden invariants.

### HOW

- Changed the coupled implementation policy from `gpt-5.4-mini xhigh` by default to `gpt-5.3-codex high/xhigh` for shared-invariant work.
- Added explicit strategy/gate-review routing for verification policy, xtask routing, SDK capability audits, transport security, auth/Origin/Host validation, and protocol compatibility.
- Added a rule that plan docs should not copy the global routing table and that `RAZORBACK.md` wins on conflicts.
- Updated the six May 3 plan docs with plan-specific overrides only.

### IMPACT

Future Razorback execution should delegate low-risk boxed-in work to cheaper models, but route daemon, projection, transport, workspace lifecycle, verification-policy, and ambiguous cleanup work to stronger Codex lanes before implementation starts.
