## v7.9.2 release — sidecar bootstrap race fix + deferred workspace open

### WHAT
Cut patch release v7.9.2 covering two reliability fixes already in the working tree:

1. **Sidecar bootstrap single-flight lock** (`src/embeddings/sidecar_bootstrap.rs`, `sidecar_supervisor.rs`). Cross-process `fs2` advisory lock at `<sidecar-cache>/.julie-sidecar-bootstrap.lock` (parent of venv, so `remove_dir_all(venv_path)` during rebuild can't invalidate it). Lock spans venv create + sidecar editable install + CUDA torch reinstall + version probe.
2. **`IndexRoute::path_backed_route` deferred-open fast path** (`src/tools/workspace/indexing/route.rs`). When `handler.workspace_pool` is `Some`, compute `<pool.indexes_dir()>/<workspace_id>/{db,tantivy}` directly instead of routing through `workspace_storage_anchor`, which requires a bound primary workspace.

### WHY
- Operator logged 14+ parallel `uv pip install torch==2.11.0` and 10+ parallel `python -c "import torch"` probes from one machine on first-run. Same venv, multiple processes, no serialization. Bootstrap was racing itself.
- `manage_workspace(operation="open", path=...)` failed in deferred-session daemons (no primary bound) because path resolution went through the anchor helper, which insists on a current workspace ID.

### Review verdict
Reviewed the diff before agreeing to release. Subagent flagged route.rs as a behavior change vs slow path — verified manually: `pool.indexes_dir()` equals `anchor_override.parent()` (workspace/mod.rs:555), so the fast path is equivalent for the pool case, just doesn't require a bound primary. No regression.
