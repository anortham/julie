## Serialized sidecar bootstrap provisioning

### WHAT
- Added a process-wide file lock for managed embedding sidecar bootstrap in `src/embeddings/sidecar_bootstrap.rs`.
- `build_sidecar_launch_config()` now acquires the lock before embedded source extraction, venv validation/creation, package install, and CUDA torch reinstall.
- Added `test_managed_sidecar_bootstrap_is_serialized_across_concurrent_callers` in `src/tests/core/sidecar_supervisor_tests.rs`.

### WHY
A Linux user log snippet showed many concurrent `uv pip install --reinstall-package torch ... cu124` and `python -c import torch` processes at Claude Code MCP startup. Investigation showed these were sidecar provisioning processes, not proof of multiple live Julie daemons. The managed sidecar install marker was checked with plain read/write and had no cross-process lock, so concurrent provider initialization could all enter pip install against the same venv.

### HOW
The new bootstrap lock uses `fs2::FileExt::lock_exclusive()` on `.julie-sidecar-bootstrap.lock` beside the managed venv. The lock covers the whole managed bootstrap critical section so late callers re-check the install marker after the first caller finishes.

The regression test uses fake `uv`, `nvidia-smi`, and venv Python binaries, starts 8 concurrent `build_sidecar_launch_config()` callers, and asserts only one base install runs. On Linux it also expects exactly one CUDA torch reinstall.

### IMPACT
Prevents MCP startup or concurrent stdio/daemon provider initialization from spawning multiple sidecar provisioning jobs for the same cache/venv. Daemon singleton code was separately verified as already guarded by adapter lock plus exclusive PID creation.
