## A1.1: Add julie-adapter and julie-daemon bin targets

### WHAT
Added two new binary entry points to the julie codebase as the bootstrap step for the daemon-split plan:
- `src/bin/julie-adapter.rs` — thin adapter binary (~20 lines)
- `src/bin/julie-daemon.rs` — daemon lifecycle binary (~10 lines)
- `src/daemon/cli.rs` — `pub async fn run()` parsing `start|stop|status`, dispatching to existing lifecycle functions
- `Cargo.toml` — two new `[[bin]]` entries

### WHY
First step (A1.1) of the daemon split plan: establish the binary entry points so subsequent tasks can fill in real logic.

### HOW
- `julie-adapter` reuses `julie::cli::Cli` (which has `command: Option<Command>`) and ignores the command field — always runs adapter mode. Passes `--workspace` through `resolve_workspace_startup_hint` into `run_adapter`.
- `julie-daemon` delegates entirely to `julie::daemon::cli::run()`.
- `src/daemon/cli.rs` has its own minimal `DaemonCli` / `DaemonCommand` clap derive (`start|stop|status`). Logging setup for `start` is verbatim copy from `main.rs` `Command::Daemon` branch.
- `pub mod cli;` added to `src/daemon/mod.rs`.

### IMPACT
- `cargo build --release --bins` now produces three binaries: `julie-server` (73 MB), `julie-daemon` (72 MB), `julie-adapter` (2.7 MB — thin, no daemon stack).
- `julie-server` behavior unchanged.
- `julie-adapter` and `julie-daemon start` delegate to the same code paths as `julie-server` today.
- Unblocks A1.2–A1.8.
