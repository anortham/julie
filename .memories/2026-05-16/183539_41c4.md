## A2.2c-followup: pool-migration of structural sites (partial)

Continued Task #4 — migrating sites blocked by shared `Arc<Mutex<>>` type with the Primary branch. Found a hard structural ceiling around `SymbolDatabase`'s `Sync` requirement when stored in cross-await structs.

### Migrated successfully

**1. Handler accessor.** Added `JulieServerHandler::primary_pooled_database()` — returns an owned `SymbolDatabase` for the primary workspace via `get_pooled_database_for_workspace(require_primary_workspace_identity()?)`. Legacy `primary_database()` retained for the ~22 remaining callers.

**2. fast_refs (whole module).**
- `try_semantic_fallback`: dropped `primary_db` param, acquires its own pooled DB per branch
- `resolve_source_names`: dropped `primary_db` param, acquires its own pooled DB per branch
- `find_references_and_definitions`: dropped `primary_db` param. **Consolidated the Primary path's 5 separate `spawn_blocking` calls into 1**, since owned pooled `SymbolDatabase` can't be cloned across `spawn_blocking` boundaries. Same 4-strategy logic preserved (exact lookup, variant matching, relationships, identifiers); now runs in a single blocking task with no inter-task lock contention.
- `call_tool`: simplified — no longer fetches/passes `primary_db`.

**3. rename.rs (2 callers).** Both `metrics_metadata` and `handle_rename_symbol` updated: dropped the `primary_db` acquire boilerplate, just pass `workspace_target`.

**4. search/line_mode.**
- `run_line_mode_workspace_fetch` and `run_line_mode_with_scope_rescue` changed from `Arc<Mutex<SymbolDatabase>>` → `&SymbolDatabase` (no internal locks needed)
- `line_mode_matches`: both Primary and Target branches use pooled DB now (Primary via `primary_pooled_database`, Target via `get_pooled_database_for_workspace`). SearchIndex still on Arc<Mutex<>> — that's a separate concern.

### Structural ceiling discovered (DEFERRED)

**3 sites cannot migrate without a wrapper:** PrimaryWorkspaceState, WorkspaceEditTarget, WorkspaceQueryTarget all store `Arc<Mutex<SymbolDatabase>>` in a struct that crosses `.await` points. The MCP tool macro requires `Send` futures, which requires captured `&Struct` to be `Sync`.

**SymbolDatabase is `Send` but NOT `Sync`** — rusqlite's `Connection` contains a `RefCell<LruCache<...>>`. Owned SymbolDatabase works fine in local async fns (where pool migration succeeded). Stored in a struct accessed by reference across awaits, it breaks `Sync`.

The only way forward for those 3 sites is wrapping each pooled DB in `Arc<Mutex<SymbolDatabase>>` again — which restores `Send + Sync` and still gets per-snapshot pool acquisition. But that loses some of the migration's value (each snapshot's reads serialize through the Mutex, even though only one consumer holds the connection).

Sites deferred:
- `health/checker::primary_workspace_health` (PrimaryWorkspaceState::database)
- `tools/editing/rewrite_symbol::resolve_workspace_target` (WorkspaceEditTarget)
- `tools/navigation/call_path::resolve_workspace_target` (WorkspaceQueryTarget)
