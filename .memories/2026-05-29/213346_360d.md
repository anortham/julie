## Task #18 COMPLETE: Construct parity for Vue, Razor, VB.NET, C#, Dart

### WHAT
All 5 languages in task #18 now capture generic type arguments at CONSTRUCTION (`new X<T>()`) and HERITAGE (`class C extends Base<T>`) use-sites.

### HOW (per language)

**C# and Razor** — pre-existing: the `identifier` arm + `is_csharp_type_usage_identifier` parent-climbing already fires on `generic_name` in both construction and heritage contexts. Confirmed with new tests.

**VB.NET** — completed in task #8 this session: `generic_type` arm with outermost guard (`parent != type_argument_list`). Covers field annotations, `New List(Of User)()`, and `Inherits List(Of User)`. All 6 tests pass.

**Dart** — root cause found and fixed. Dart uses TWO different AST structures:
- Annotations: `type { type_identifier + type_arguments }` (standard)
- Construction/heritage: TWO sibling `type` nodes — first is name wrapper, SECOND is the `<...>` args container (NO `type_arguments` node here)

Fix in `record_outermost_dart_type_arguments`: climb to grandparent; if `new_expression`/`superclass`/`interfaces`/`mixins`/`mixin_application`, call `parent.next_named_sibling()` to get the args container `type` node, pass it directly to `extract_type_arguments`. `decompose_dart_type_arg` works unchanged because the container's named children ARE the expected `type` arg-wrappers. Removed 5 debug tests after fix verified.

**Vue** — added two new arms to `extract_identifier_from_node_with_content`:
- `extends_clause`: `child_by_field_name("value")` → terminal identifier → `child_by_field_name("type_arguments")` → `extract_vue_type_arguments`
- `new_expression`: `child_by_field_name("constructor")` → terminal identifier → `named_children().find("type_arguments")` → `extract_vue_type_arguments`
- Added `terminal_identifier_from_content` helper (mirrors TS extractor's `terminal_identifier`)
