## Phase 2 Step 0 complete — `CanonicalWriteSet` parameter-object refactor

Behavior-preserving refactor that replaces the 5 positional `&[...]` slices threaded through the atomic write functions with a single borrowed `CanonicalWriteSet<'a>` struct. This is plan cross-cutting Rule 3: it converts "silently drop a new collection on one of the parallel write paths" from a runtime bug into a compile error, ahead of Phase 2/3 adding `type_arguments` / `literals`.

- **WHAT:** New `pub(crate) struct CanonicalWriteSet<'a> { files, symbols, relationships, identifiers, types }` in `src/database/bulk/atomic.rs`. Internal fns (`incremental_update_atomic_with_metadata`, `bulk_store_fresh_atomic_with_metadata`, `replace_workspace_data_atomic`, `fresh_insert_atomic`, `insert_batch_tx`) now take `&CanonicalWriteSet`. Added `ExtractedBatch::canonical_write_set()` in `src/indexing_core/batch.rs` as the single batch→write-set mapping.
- **KEY DISCOVERY:** The plan's Rule 2 named 2 production write paths (pipeline, persistence) + orphan cleanup. Step 0 found a **4th: the live single-file watcher** at `src/watcher/handlers.rs:283`, which builds slices ad-hoc from a single-file `ExtractionResults` (no `ExtractedBatch`). It's now routed through an explicit `CanonicalWriteSet { .. }` literal. Updated the plan's Rule 2 table to document all 4 paths + the 2 compile-forced struct-construction choke points.
- **HOW (no test churn):** Retained the public positional wrappers `incremental_update_atomic`/`bulk_store_fresh_atomic` as thin struct-building delegators, so the ~90 test call sites + orphan cleanup (`workspace.rs:17`) stayed untouched and keep defaulting absent collections to empty. The 4 production data-write paths all construct the struct explicitly → compile error if a future field isn't wired.
- **IMPACT:** Phase 2/3 now add a collection by editing the struct + `canonical_write_set()` + the watcher literal; the compiler flags the last two until wired. No production path can silently drop the new data.
