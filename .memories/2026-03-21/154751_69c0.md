## CodeRankEmbed sidecar working on Windows with DirectML

### What
Fixed three cascading issues preventing CodeRankEmbed (768d) from working via the Python sidecar on Windows:

1. **Missing `einops` dependency** — CodeRankEmbed's custom HuggingFace model code (`modeling_hf_nomic_bert.py`) imports `from einops import rearrange, repeat`, but `einops` wasn't in `pyproject.toml`
2. **Sidecar timeout too tight** — `DEFAULT_SIDECAR_TIMEOUT_MS` was 5s, but 500 texts through CodeRankEmbed on DirectML + first-batch GPU graph compilation took >5s
3. **No VRAM-aware GPU batch sizing** — Internal Python batch_size was hardcoded at 32

### How
- Added `einops` + `wmi` (Windows GPU detection) to `pyproject.toml` runtime deps
- Reduced Rust pipeline batch from 500→250 texts per request, increased timeout from 5s→30s
- Ported Miller's VRAM detection + batch formula `(VRAM_GB / 6.0) * 30` clamped to [16, 128] into `runtime.py`
- Bumped install marker to v9 to force venv reinstall

### Also fixed
- **README docs overhaul** — All three harness examples (Claude Code, VS Code, Cursor) now show `JULIE_WORKSPACE`, `JULIE_EMBEDDING_PROVIDER`, `JULIE_EMBEDDING_SIDECAR_MODEL_ID`. Fixed `auto`→`sidecar` since examples show CodeRankEmbed. Added `JULIE_WORKSPACE` to env options table.
- **`contains_blacklisted_directory` Windows bug** — Function checked full absolute path including `AppData`, false-positive on all Windows temp paths. Fixed to relativize against workspace root.
- **Windows-only test skip** — `test_workspace_init_sidecar_bootstrap_failure_falls_back_to_ort` skipped on Windows where `auto` resolves directly to ORT
- **52 build warnings eliminated** — Unused imports, variables, `mut`, dead code across ~20 test files
- **CLAUDE.md Windows binary lock note** — Explicit warning not to attempt `cargo build --release` while session is active

### Result
30/30 batches completed, 7,468 symbols embedded with CodeRankEmbed 768d on DirectML at ~75 sym/s. Zero timeouts.
