## Embedding Enrichment: Complete with Resolver Bonus Fix

### What
Completed the full embedding enrichment pipeline plus a bonus resolver fix:

1. **Multi-sentence doc extraction** (`extract_doc_excerpt`, 300 chars)
2. **Callee name enrichment** (`build_callee_map` + `callees_by_symbol` param)
3. **MAX_METADATA_CHARS 600 -> 1200** + Enum force-re-embed fix
4. **Resolver: hard rejection -> penalty** (`return 0` -> `saturating_sub(75)`)

### Why
Dogfooding showed `get_context("metrics save database persist")` couldn't find `record_tool_call`. Root cause chain:
- Doc truncation dropped "SQLite write" (fixed by #1)
- No callee names in embedding text (fixed by #2)
- 600 char budget too tight (fixed by #3)
- Resolver hard-rejected `insert_tool_call` as callee target because `handler.rs` doesn't explicitly reference `SymbolDatabase` by name (fixed by #4)

### How
- Subagent-driven TDD for tasks 1-3 (plan + 3 implementation agents + reviews)
- Manual TDD for resolver fix (diagnosed via search-debug skill, confirmed with goldfish recall that original design intended soft boost not hard filter)
- Resolution rate: 17% -> 28% (6,794 additional relationships recovered)

### Impact
- Enrichment verified working: queries with vocabulary overlap find enriched symbols correctly
- `record_tool_call -> insert_tool_call` relationship now exists in graph
- Remaining gap: code embedding models can't bridge English synonyms ("save" != "record"). This is the NL query expansion problem, not solvable by enrichment alone
