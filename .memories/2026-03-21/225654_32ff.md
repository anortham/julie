## Embedding Pipeline Cancellation Fix

### What
Added proper cancellation to the embedding pipeline so force reindex doesn't race with a concurrent auto-index pipeline. Previous abort-only approach killed the outer async task but left the inner `spawn_blocking` thread running on the GPU, causing 10-30x slower batches from GPU contention.

### Why
Race condition discovered via log analysis: auto-index spawns embedding pipeline at startup. When force reindex comes in seconds later, `JoinHandle::abort()` only cancels the outer `tokio::spawn` task. The inner `spawn_blocking(|| run_embedding_pipeline(...))` keeps running because `spawn_blocking` tasks are not cancelled when their parent is aborted. This caused:
1. Two pipelines doing GPU inference simultaneously (choppy 20-30s batches vs normal 2s)
2. The aborted pipeline writing embeddings back after `clear_all_embeddings` ran

### How
- Added `Arc<AtomicBool>` cancellation flag to `run_embedding_pipeline_cancellable()`
- Flag is checked between batches; pipeline breaks cleanly when set
- `handler.embedding_task` now stores `(cancel_flag, JoinHandle)` tuple
- Before starting a new pipeline: set flag to true, then abort handle
- Force reindex in both `handle_index_command` and `handle_refresh_command` uses the same cancel+abort+clear sequence
