## v5.5.3 Released — Windows Jina-code-v2 via ORT + DirectML

### What
Released v5.5.3 fixing Windows embedding stack end-to-end. Tagged and pushed to `origin/main`.

### Key Changes
1. **Jina-code-v2 as default ORT model on Windows** — 768d code-optimized embeddings via DirectML GPU. No Python sidecar needed. `JULIE_EMBEDDING_ORT_MODEL_ID` env var for model selection.
2. **Sidecar probe encode** — detects GPU inference failures after model load (CodeRankEmbed RoPE on DirectML produces zero vectors). Auto-falls back: patch DirectML → CPU → fail.
3. **Rust macro noise filter** — `vec!`, `format!`, `matches!`, `bail!`, `info!` etc. no longer extracted as symbols. Blocklist + item-position check eliminates ~2800 junk symbols.
4. **`contains_blacklisted_directory` Windows fix** — was checking full absolute path including `AppData`, breaking all temp dir tests on Windows.
5. **Sidecar timeout 5s→30s, batch 500→250** — needed for larger models.
6. **VRAM-aware batch sizing** — ported from Miller, formula `(VRAM_GB/6)*30` clamped [16,128].

### Investigation Trail (for future reference)
- CodeRankEmbed on DirectML: loads fine, but inference fails in `apply_rotary_emb` — `torch.cat` with non-contiguous tensors from `einops.repeat`. `.contiguous()` patch fixes isolated ops but not the full model pipeline.
- The sidecar was silently storing **zero vectors** for every symbol (4,671 zero vectors). KNN cosine between zeros is degenerate → all similarities display as 1.00 → HTML attribute functions appear as semantic neighbors for everything.
- Jina-code-v2 via fastembed ORT is the clean Windows path — pre-packaged ONNX, DirectML EP, no Python.
