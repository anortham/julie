## Codex Review + Dogfooding Fixes for Edit Tools

Had Codex (GPT-5.4) review the editing tools implementation. Found 5 issues, fixed 3 immediately.

### Codex Findings + Actions

| Finding | Severity | Action |
|---------|----------|--------|
| `file_path` fallback in edit_symbol | HIGH | **Fixed** -- re-apply filter strictly after find_symbol returns |
| CRLF normalization in replace_symbol_body/insert_near_symbol | MEDIUM | **Fixed** -- detect_line_ending() preserves original line endings |
| format_unified_diff cascades on insertions | Dogfood | **Fixed** -- replaced naive index-comparison with LCS-based diff |
| Bracket validation false-rejects (string literals) | Dogfood | **Fixed** -- delta-based comparison (before vs after) instead of absolute check |
| Fuzzy match splice width for DMP hits | HIGH | Documented as known limitation |
| Line-level editing loses byte precision | HIGH | Accepted, Phase 2 |

### Dogfooding Discoveries

Two bugs found in under 5 minutes of real use:

1. **Diff cascade** -- `format_unified_diff` compared lines by index, so an insertion shifted every subsequent line into the diff output. Fixed with LCS alignment + prefix/suffix trimming for performance.

2. **Bracket false rejection** -- `edit_file` rejected a legitimate edit to a test file because string literals contained intentionally unbalanced brackets (e.g., `\"foo())\"`). Ironic: the safety feature blocked us from writing tests for its own bugs. Fixed by comparing bracket balance delta (before vs after) instead of absolute counts.

### Key Insight: check_bracket_balance API Change

Changed from `check_bracket_balance(content) -> Result` (absolute check) to `check_bracket_balance(before, after) -> Result` (delta check). Updated both callers in edit_file.rs and edit_symbol.rs.
