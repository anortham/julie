## Edit Tool Improvements: Hunk Headers + Trimmed-Line Fuzzy Matching

### Unified Diff Hunk Headers
- **What:** Added standard `@@ -start,count +start,count @@` hunk headers to `format_unified_diff()`
- **Why:** Diff output had no location info. LLMs couldn't tell if a change was at line 7 or line 700
- **How:** Precompute `(old_lineno, new_lineno)` for each diff op, then emit hunk headers per visible range
- **Impact:** Both `edit_file` and `edit_symbol` now return standard unified diffs with line numbers. Multiple disjoint hunks are naturally separated by their headers

### Trimmed-Line Fuzzy Matching (Phase 2)
- **What:** Added whitespace-tolerant matching to `find_all_matches()` as Phase 2 (between exact and DMP bitap)
- **Why:** DMP's bitap algorithm has a 32-char limit, making fuzzy matching useless for multi-line patterns (the common case). Agents guessing wrong indentation would get "No match found"
- **How:** Split old_text into lines, trim each, find consecutive trimmed-line matches in file. Return `MatchSpan { start, end }` covering the actual file content (with original whitespace)
- **Impact:** Handles tabs-vs-spaces, 2-vs-4-vs-8-space indent, trailing whitespace. All without reading the file first

### Key Design Decisions
- `MatchSpan` struct replaces bare `usize` positions. Carries both start and end, so fuzzy matches (where actual content length differs from old_text) splice correctly
- Exact substring match stays as Phase 1 (authoritative when text is literally present)
- Trimmed-line matching rejects all-blank-line patterns (too ambiguous)
- DMP bitap stays as Phase 3 for short single-line typo tolerance

### Dogfooding Discovery
- DMP bitap has an existing length-mismatch bug: when fuzzy match finds content that's a different length than old_text, the splice uses old_text's length, garbling the result. The MatchSpan refactor fixes this for trimmed matches but the DMP path still has the issue (deferred)
