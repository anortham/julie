## Token-Saving Edit Tools: Design + Implementation Plan Complete

Brainstormed and designed two new DMP-powered MCP editing tools for Julie that let agents edit files without reading them first (bypassing the mandatory Read-before-Edit tax in Claude Code).

### What

- **Design spec:** `docs/superpowers/specs/2026-04-03-token-saving-edit-tools-design.md`
- **Implementation plan:** `docs/superpowers/plans/2026-04-03-token-saving-edit-tools.md`

### Tools Designed

1. **`edit_file`** - DMP fuzzy find-and-replace for any file. Agent provides old_text/new_text, DMP handles fuzzy matching. Supports first/last/all occurrence modes.
2. **`edit_symbol`** - Symbol-aware editing using Julie's indexed boundaries. Agent references a symbol by name, Julie looks up location. Operations: replace, insert_after, insert_before.
3. **Markdown section fix** - Nearly free: expand section line ranges in the extractor so get_symbols returns full section content.

### Key Decisions

- **DMP-based**: `diff-match-patch-rs 0.5.1` already in Cargo.toml, never removed
- **Progressive rollout** (Phase 1 = these 2 tools, Phase 2 = read_file replacement, hook escalation)
- **Reuse `deep_dive::data::find_symbol`** for edit_symbol's symbol lookup
- **Golden master testing** (SOURCE/CONTROL fixtures, byte-for-byte verification)
- **Four-layer adoption**: tool descriptions, SessionStart hook, PreToolUse nudge, editing skill

### Research Conducted

- Analyzed Serena's tool adoption strategy (system prompt persuasion, tool markers, context-specific exclusion)
- Examined Julie's old DMP tools (removed in v2.2.0): EditingTransaction, fuzzy_replace, edit_lines
- Verified markdown extractor stores section content as doc_comment but heading node's line range is too narrow
- Confirmed file watcher keeps index live (no stale boundary concern)
