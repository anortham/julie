## PR 2 (Tasks 2.1+2.2) — PID-reuse defense + create_exclusive backoff

Extended PID file format from single-integer to three-field and hardened the retry loop.

### What was done
- Format changed to `<pid> <creation_time_unix_micros> <binary_mtime_unix_micros>\n`
- PID-reuse defense: `check_running` and `create_exclusive` now verify creation_time matches the running process before accepting a PID file
- Exponential backoff in `create_exclusive`: `50ms * 2^min(n,7)`, cap 5000ms, fires AFTER first failure
- Non-NotFound `remove_file` errors propagated via `bail!` (was silently swallowed with `let _`)
- Legacy single-int files rejected by parser (treated as stale, removed by `check_running`)

### Key implementation decisions
- `PidFileContents` struct with `parse()` + `Display`
- macOS: `sysctl(KERN_PROC/KERN_PROC_PID)` raw 648-byte buffer; p_starttime at offset 0 (verified with clang — sizeof(kinfo_proc)=648, offset=0 on arm64/x86_64). No libproc needed.
- Linux: `/proc/<pid>/stat` field 21 ticks-since-boot + `/proc/uptime` for absolute wall-clock
- Windows: `OpenProcess` + `GetProcessTimes` with FILETIME→UNIX epoch conversion (subtract 116_444_736_000_000_000 100-ns intervals, divide by 10)
- `check_running` also removes legacy/unreadable files (not just parse-successful stale ones)
