## Added content-aware `.h` parser routing

### WHAT

Canonical extraction now routes ambiguous `.h` headers by content instead of always treating them as C.

- Added `detect_language_for_source` for source-aware parser selection
- Kept extension-only `detect_language_for_path` for fixture inventory and static extension checks
- Added a `.h` C++ syntax detector that ignores comments and quoted strings before looking for C++ tokens
- Updated manager debug labels to use the same source-aware language selection as extraction
- Added tests proving C++ `.h` syntax routes to `cpp` and plain C `.h` syntax stays `c`

### WHY

The audit found that C++ projects using `.h` headers lost C++ semantics because the language spec maps `.h` to C first. Changing all `.h` files to C++ would break plain C projects, so the fix needs a narrow content-aware fallback.

### HOW

Only `.h` gets special handling. The detector strips C-family comments and strings, then recognizes strong C++ signals like `namespace`, `class`, access labels, templates, concepts, `::`, `requires`, and related C++ keywords. Otherwise detection falls back to the existing extension table.

### IMPACT

C++ headers now extract class and method symbols through the C++ parser, while normal C headers continue through the C parser. This fixes H32 without turning every header into C++ by default.
