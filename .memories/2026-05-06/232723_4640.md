## PR1 Task 1.4: Daemon shutdown LIFO reorder + explicit pool teardown

### What
Reordered the `run_daemon` shutdown sequence from broken order to correct LIFO dependency order, and extracted it into a testable `perform_shutdown_sequence` helper.

**Old (broken) order:**
1. `reaper_handle.abort()` + `cleanup_sweep_handle.abort()`
2. `embedding_service.shutdown().await` (wrong - HTTP still live)
3. `http_transport.shutdown().await` (wrong - pools not shut down explicitly)
4. Housekeeping (port file, pid file, state file)
5. (WorkspacePool and WatcherPool dropped implicitly on stack unwind)

**New (correct) LIFO order:**
1. `reaper_handle.abort()` + `cleanup_sweep_handle.abort()` (background tasks first)
2. `http_transport.shutdown().await` (stops new requests)
3. `embedding_service.shutdown().await` (safe now that transport gate is closed)
4. `workspace_pool.shutdown().await` (commits Tantivy writes, releases file locks)
5. `watcher_pool.shutdown().await` (releases OS file-watcher handles)
6. Housekeeping (port file, pid file, state file)

### Why
In-flight HTTP requests could observe a torn-down embedding service. `WorkspacePool` and `WatcherPool` were dropped implicitly (stack unwind) instead of being shut down explicitly via their `shutdown()` methods added by Tasks 1.1 and 1.2.

### How
- Added `pub(crate) async fn perform_shutdown_sequence(...)` in `src/daemon/mod.rs`
- Function takes `call_log: Option<Arc<Mutex<Vec<&'static str>>>>` as a test hook; `None` in production (zero overhead)
- Tests use the call_log to observe step order; assert position in log vector
- Two ordering tests in `src/tests/daemon/shutdown_ordering.rs`:
  - `test_shutdown_calls_pools_after_transport`
  - `test_shutdown_calls_workspace_pool_before_watcher_pool`
- Both tests spin up real `HttpTransportServer` (loopback bind) + lightweight pool instances

### Impact
- Closes the window where in-flight HTTP requests could observe torn-down embedding service
- WorkspacePool and WatcherPool now shut down explicitly and in correct order
- Ordering is verified by tests (not just code review)
