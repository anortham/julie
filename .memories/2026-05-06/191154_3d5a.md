## Completed Task 7 visibility and doc-comment extractor remediation

### WHAT
- Fixed base visibility extraction so method and function bodies cannot accidentally supply visibility words.
- Made doc-comment recognition language-specific, including explicit slash-star doc support only for languages that declare it.
- Added richer visibility values for Swift and database round trips: `Internal`, `FilePrivate`, and `Open`.
- Added Rust inner doc extraction and fixed Rust return-type parsing so `where` clauses are not folded into return types.
- Fixed a TypeScript visibility regression revealed by golden refresh: constructor parameter properties no longer make the class or constructor itself `Private`.

### WHY
The review findings called out false-positive visibility, overly broad doc-comment matching, missing Rust doc/return-type cases, and lost Swift visibility detail. The old behavior produced plausible but wrong metadata, which is worse than loudly failing because search and navigation silently inherit bad facts.

### HOW
- Reworked base visibility extraction to inspect only explicit modifier children.
- Added start/continuation doc-comment style matching so broad comment syntax can remain for languages that intend it while `/**` is no longer accepted globally.
- Stored new visibility variants through dedicated conversion helpers instead of lossy `Display` strings.
- Added targeted extractor tests first, confirmed the key regressions failed, then fixed production code and refreshed only the affected goldens.
- Used Julie blast radius after edits to confirm the affected call paths and test scope.

### IMPACT
Task 7 is implemented and verified. The extractor metadata is now more precise across base fallback behavior, doc comments, Rust, Swift, and TypeScript. The TypeScript golden change is intentional: prior `Private` visibility came from constructor parameter text, not from the class or constructor declaration itself.
