## PR 1 Task 1.1: WorkspacePool::shutdown() implemented

Added explicit pool-wide shutdown to prevent Tantivy write loss and Windows file lock races on daemon restart.

### What

- Added `pub async fn shutdown(&self)` to `WorkspacePool` in `src/daemon/workspace_pool.rs`
- Created `src/tests/daemon/workspace_pool_shutdown.rs` with two TDD tests
- Registered new test module in `src/tests/daemon/mod.rs`

### Why

Daemon shutdown previously dropped `WorkspacePool` implicitly, which does not commit uncommitted Tantivy segments. On Windows, the next daemon spawned 50-500ms later could race kernel handle cleanup and fail to acquire the Tantivy lock with `LockBusy`.

### How

- `shutdown()` takes a write lock, drains the HashMap into a Vec, drops the lock, then iterates entries
- For each workspace's `search_index`, locks the mutex and calls `SearchIndex::shutdown()` (commits + drops writer)
- Poisoned mutexes recovered via `e.into_inner()`, matching the handler.rs:626-641 pattern exactly
- Per-workspace failures are logged (`warn!`) but do not propagate — API is infallible

### Tests

- `test_workspace_pool_shutdown_calls_search_index_shutdown`: 3-workspace pool, all report `is_shutdown() == true` after `pool.shutdown().await`
- `test_workspace_pool_shutdown_recovers_from_poisoned_mutex`: poison one mutex via `catch_unwind`, assert shutdown does not panic and both workspaces report `is_shutdown() == true`
- Both RED then GREEN confirmed
