## Fixed C++ relationship resolution with duplicate names

### WHAT

C++ relationship extraction now resolves relationships through the scoped symbol index without dropping every duplicated name case.

- Added grouped candidate access to `ScopedSymbolIndex` while leaving `unique_symbol_map` strict for existing precision checks
- Resolved C++ inheritance from class/struct symbols by span and base-type candidates
- Resolved C++ call callers by containing symbol span instead of caller-name uniqueness
- Added regression coverage for duplicate method names, overloaded free functions, constructor/class name collisions, and same-name base classes in different namespaces
- Refreshed the C++ basic golden because it now correctly emits recovered `extends` and `helper` call relationships

### WHY

The audit found that C++ cross-reference extraction silently skipped valid relationships whenever symbol names were duplicated. That meant common code shapes like constructors, overloads, repeated method names, and namespace-local class names lost edges.

### HOW

Kept the general duplicate-name guard intact and made the C++ extractor opt into grouped candidates only where it has enough syntax context to disambiguate. Caller resolution uses byte containment, and inheritance prefers a base type with the same parent namespace/class when duplicates exist.

### IMPACT

C++ extraction now preserves real call and inheritance edges in duplicated-name cases without weakening the shared relationship precision contract.
