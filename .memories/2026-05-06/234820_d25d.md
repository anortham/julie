## Task 3.3 (PR 3) — admit_initialize short-circuit verification

### What
Added test coverage verifying that `admit_initialize` short-circuits on the first failed
admission gate. Found the short-circuit was **already in place** via Rust's `?` operator at
line 472 of `src/daemon/mcp_session.rs`.

### Why
The task description identified a scenario where stale-binary gate AND version gate could
both call `mark_restart_pending` for the same admit attempt, producing duplicate log lines.
The concern was whether the second gate was actually guarded by the first gate's error
propagation.

### How
- Added `pub(crate) apply_action_call_count: Arc<AtomicUsize>` to `HttpSessionAdmission`
- Counter is incremented at the top of every `apply_admission_action` call
- Two integration tests in `src/tests/daemon/admit_initialize_short_circuit.rs`:
  1. `test_admit_initialize_short_circuits_on_stale_binary_reject` — stale binary (0 sessions) +
     mismatched version header: asserts count == 1 (version gate never reached)
  2. `test_admit_initialize_runs_version_gate_when_stale_passes` — binary NOT stale +
     mismatched version: asserts count == 2 (both gates ran)
- Verified RED by removing `?` from first call: test correctly failed with count=2 vs expected 1
- Restored `?` — both tests GREEN

### Impact
- Short-circuit invariant is now tested and will catch any future regression
- The `?`-based short-circuit already guaranteed correct behavior; tests document the contract
- No behavioral change to production code (only instrumentation added)
