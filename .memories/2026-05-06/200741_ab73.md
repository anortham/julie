## Task 9 TypeScript And JavaScript Structure

### WHAT
Implemented Task 9 from `docs/plans/2026-05-06-tree-sitter-extractor-audit-remediation.md`. Confidence: 93/100.

- Threaded TypeScript parent IDs through the symbol visitor instead of regenerating parent IDs from ancestor coordinates.
- Added `internal_module` namespace extraction and namespace parent propagation.
- Converted TypeScript imports to one `Import` symbol per local binding with source metadata.
- Added TypeScript TSX component identifier extraction from opening and self-closing JSX tags.
- Added JavaScript and TypeScript constructor call identifiers for `new` expressions.

### WHY
The audit findings showed TypeScript structure was inconsistent: classes and members used fragile parent ID recomputation, imports were named after modules instead of bindings, JSX component usages were invisible to identifier extraction, and constructor calls were not represented as calls.

### HOW
Used Julie `get_context`, `deep_dive`, `fast_refs`, and `blast_radius` to orient on extractor entry points and callers. Added RED tests first for namespace/class/interface parent threading, TypeScript import aliases, TSX component identifiers, and JS/TS `new` expressions. Then updated the TypeScript visitor to pass `parent_id` through extraction and ported binding-level import behavior from JavaScript while keeping TypeScript-specific metadata.

### IMPACT
Task 9 behavior is covered by targeted regressions and the batch gate. Parent scope extraction no longer depends on regenerated row/column IDs, TypeScript imports are searchable by local name, JSX component usages appear as call identifiers, and constructor calls participate in identifier data.
