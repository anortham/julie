## Completed remaining C/C++ Task 6 extraction fixes

### WHAT

Finished the remaining C/C++ relationship and symbol-modeling pieces from Task 6.

- Added C++20 concept extraction as `SymbolKind::Type` with concept metadata
- Made nested C++ class/struct/union/enum visibility use the existing C++ access-specifier logic instead of hardcoding public
- Added C and C++ `Uses` relationships for user-defined type references in declarations, fields, return types, and parameters
- Added low-confidence structured pending calls for indirect C calls such as `(*callback)()`
- Moved the C++ type-usage declaration/noise filters into shared helpers so identifiers and relationships use the same rules
- Refreshed C/C++ goldens to include the new type-use edges

### WHY

Task 6 still had H16, H34, M14, and M36 open after the earlier C++ call/overload and `.h` routing work. The missing pieces meant C/C++ type coupling was invisible, indirect C callbacks vanished, nested C++ types lied about visibility, and C++20 concepts were not indexed at all.

### HOW

Relationship walkers now carry one `ScopedSymbolIndex` through traversal. Type-use edges resolve local user types by exact candidate, with a top-level alias preference for `typedef struct Name Name` shapes. Missing type targets become structured pending `Uses`. Indirect C call extraction normalizes pointer/parenthesized/subscript/member call expressions into unresolved call targets with lower confidence.

### IMPACT

C and C++ graph output now carries useful type coupling and callback evidence without weakening direct call resolution. C++ symbol extraction also handles modern concepts and nested access semantics correctly.
