## Fixed H35 Total Parse Failure Handling

### WHAT
- Changed canonical extraction so `parser.parse(...) == None` returns an empty degraded `ExtractionResults` with one parse diagnostic instead of returning `Err`.
- Added a small parser injection seam, `extract_canonical_with_parse`, to test total parser failure without relying on huge inputs or timeouts.
- Applied the same degraded-result behavior to JSONL records when a record parse returns `None`.
- Updated the JSONL identity test to assert the span-based ID contract introduced in Task 3.

### WHY
The compiled audit finding `H35` called out an all-or-nothing failure mode: a total tree-sitter parser failure lost the file entirely and gave callers no structured diagnostic payload. Recovered parse errors were already handled, but parser `None` was not.

### HOW
- Wrote RED test `test_extract_canonical_parse_none_returns_degraded_result_with_diagnostic`; it first failed because the pipeline had no injection seam and still modeled parser `None` as an error.
- Changed `parse_for_language` and the internal parser helper to return `Result<Option<Tree>>`.
- When the option is `None`, built an empty `ExtractionResults` and attached a `ParseDiagnosticKind::Error` spanning the input content.
- Root-caused a follow-on failing JSONL pipeline test to an old row/column-only ID expectation and updated it to the current full-span hash.

### IMPACT
Callers can now keep file-level extraction flow alive and inspect a structured parse diagnostic when tree-sitter gives no tree at all. This closes H35 without changing successful parse behavior.
