## Deep Dive Symbol Lookup Issue Analysis

### Problem Statement
`deep_dive("Phoenix.Channel", workspace="phoenix_ac16deb4")` returns JS Channel results from priv/static/phoenix.js instead of the Elixir module at lib/phoenix/channel.ex, even when `context_file="lib/phoenix/channel.ex"` is specified.

### Root Cause Identified: Qualified Name Resolution Mismatch for Elixir Modules

#### Why The Bug Occurs

1. **Elixir Module Naming:** Elixir modules are stored with their full qualified name as a **single symbol name**, not as parent/child relationships. Example:
   - Stored as: `Symbol { name: "Phoenix.Channel", kind: Module, ... }`
   - NOT stored as: `Symbol { name: "Channel", parent_id: "Phoenix", ... }`

2. **Qualified Name Resolution Code Flow (data.rs:52-111):**
   When calling `find_symbol("Phoenix.Channel", context_file)`:
   - Line 58: `parse_qualified_name("Phoenix.Channel")` succeeds, returns `("Phoenix", "Channel")`
   - Line 59: Calls `db.find_symbols_by_name("Channel")` ← **SEARCHING FOR WRONG NAME**
   - This finds JS Channel from priv/static/phoenix.js (exact match for name="Channel")
   - Line 63: Calls `db.find_symbols_by_name("Phoenix")` - likely returns 0 results (Phoenix itself isn't a top-level module)
   - Lines 67-88: The qualified match filter finds NO matches because:
     - JS Channel has no parent_id (it's top-level), so it's not filtered in
     - Even if it were, there's no Phoenix parent to match against
   - Line 90: Falls through because qualified match is empty

3. **Fallback Lookup (line 93):**
   - `db.find_symbols_by_name("Phoenix.Channel")` ← **CORRECT QUERY**
   - This should find the Elixir module with exact name="Phoenix.Channel"
   - **BUT**: The results from the failed qualified match above were NOT discarded - they persist in scope

#### Critical Code Location

#### Why Results Are Wrong When `context_file` is Specified

The user reported using `context_file="lib/phoenix/channel.ex"`, but the JS results are still returned. This suggests:

1. **The fallback is working**: `find_symbols_by_name("Phoenix.Channel")` correctly finds only the Elixir module (no match for JS)
2. **context_file filtering happens** at lines 99-108: it filters to only symbols whose `file_path.contains(context_file)`
3. **The Elixir module matches**: "lib/phoenix/channel.ex".contains("lib/phoenix/channel.ex") = true
4. **Return should be the Elixir module**

But if JS results are being returned instead, one of these must be true:
- The fallback is somehow returning wrong results
- The context_file filter is not being applied
- The context_file string doesn't match the stored file_path (path normalization issue?)

#### Database Query
File: `src/database/symbols/queries.rs`, function `find_symbols_by_name()` (lines 65-82)

```sql
SELECT ... FROM symbols WHERE name = ?1 ORDER BY language, file_path
```

Orders by `language` (alphabetically), then `file_path`. "elixir" < "javascript" alphabetically, so if both existed with the same name, Elixir would come first. But they have DIFFERENT names:
- Elixir: name="Phoenix.Channel"
- JS: name="Channel"

#### Test Coverage Gap
Test `test_find_symbol_qualified_name()` in deep_dive_tests.rs tests parent/child relationships but NOT the case where qualified names are single symbol names (Elixir pattern).

### Key Insights

1. **Qualified name resolution assumes parent/child relationships** - this works for Rust (Engine::process), Python (MyClass.method), etc., but breaks for Elixir's dot-notation namespace modules

2. **The fallback search should work** - it searches for the full "Phoenix.Channel" name, which should find only the Elixir module

3. **If JS results are being returned, either:**
   - The fallback search is not being used (qualified match succeeds wrongly)
   - Path normalization is causing context_file filter to fail
   - Query results are being reordered somewhere
