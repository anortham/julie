## Fix: Batch Tantivy commits in file watcher

**WHAT**: Moved Tantivy `commit()` calls from per-file handlers to batch commits in the queue processor loop.

**WHY**: When a git worktree is cleaned up, dozens of file deletion events arrive at once. Each `handle_file_deleted_static` was calling `idx.commit()` individually, which triggers Tantivy's segment merge policy. The merge asynchronously deletes old `.term` segment files, causing subsequent commits to fail with `FileDoesNotExist`.

**HOW**:
- Removed `idx.commit()` from `handle_file_created_or_modified_static` and `handle_file_deleted_static` in `handlers.rs`
- Added a single batch `commit()` after the while loop in both `start_watching` (background processor) and `process_pending_changes` in `mod.rs`
- Tantivy's `IndexWriter` is designed to accumulate operations — only `commit()` makes them durable and triggers merges

**IMPACT**: Eliminates `FileDoesNotExist` WARN spam during worktree cleanup. Also slightly improves throughput since one commit per tick is cheaper than N commits.
