## GDScript Language Verification Complete — 2 Bugs Fixed

### Bug 1: GDScript Identifier Extraction Missing Type Annotations
- **WHAT**: The GDScript `identifiers.rs` only handled `call`, `get_node`, `attribute`, and `subscript` AST nodes. Type annotations (`var x: Type`, `func f(a: Type) -> ReturnType`, `extends BaseClass`) were completely missed.
- **WHY**: `PandoraEntity` (core model class in pandora reference project) had **0 incoming references** despite being used in every file. `fast_refs`, `deep_dive` "Used by", and centrality were all broken for GDScript.
- **HOW**: Added `"type"` node handler to `extract_identifier_from_node()` in `identifiers.rs`. Tree-sitter's GDScript grammar uses uniform `type` → `identifier` structure across extends, variable annotations, parameter types, and return types — one 13-line handler catches them all.
- **IMPACT**: PandoraEntity went from 0 → 70 dependents. 4 new tests, all 28 GDScript extractor tests pass.

### Bug 2: Centrality 0.00 for Type-Referenced Symbols (Systemic Fix)
- **WHAT**: `compute_reference_scores()` only read from the `relationships` table (call graph). Symbols referenced primarily via type annotations had zero centrality even with many identifier references.
- **WHY**: GDScript classes are referenced via type annotations, not function calls. But this affects ALL languages where type annotations are common.
- **HOW**: Added Step 1b to `compute_reference_scores()` — SQL query counting cross-file `type_usage` identifiers by name match for type-defining symbols (class, struct, enum, interface, trait, type, module, namespace). Weight 1.0 per reference.
- **IMPACT**: PandoraEntity centrality: 0.00 → 1.00. PandoraCategory: 0.00 → 1.00. Change Risk now accurately reflects reality. **Benefits all 33 languages.**
