## Complete Analysis: Deep Dive Symbol Lookup Bug for Elixir Modules

### Summary
Deep dive fails for Elixir `Phoenix.Channel` because the qualified name resolution strategy (`find_symbol`) attempts to parse dot-notation module names as parent/child relationships, which breaks for Elixir's flat module namespace design.

### Root Cause: Architecture Mismatch

**Elixir Module Design:**
- Elixir stores modules with dot-notation as **single symbol names**: `name="Phoenix.Channel"`
- These are top-level module definitions, NOT parent/child relationships
- Example: `defmodule Phoenix.Channel do ... end` → `Symbol { name: "Phoenix.Channel", parent_id: None }`

**find_symbol() Design (src/tools/deep_dive/data.rs:52-111):**
- Prioritizes qualified name resolution for languages WITH parent/child hierarchies
- Assumes `.` or `::` separators indicate parent/child: `"Engine::process"` → parent=Engine, child=process
- This works for: Rust (struct::method), Python (class.method), JavaScript (object.method)
- This BREAKS for: Elixir dot-notation modules

### Bug Execution Flow

**When `deep_dive("Phoenix.Channel", context_file="lib/phoenix/channel.ex")`:**

1. **Qualified name parsing (line 58):**
   - `parse_qualified_name("Phoenix.Channel")` → Returns `Some(("Phoenix", "Channel"))`

2. **Wrong first lookup (line 59):**
   - Calls `db.find_symbols_by_name("Channel")`
   - Returns: JavaScript `Channel` from priv/static/phoenix.js (if it exists as a function/class)
   - Does NOT return: Elixir `Phoenix.Channel` (exact name doesn't match)

3. **Failed parent matching (lines 63-88):**
   - Calls `db.find_symbols_by_name("Phoenix")` → Returns 0 results (Phoenix itself isn't a module)
   - Parent filter fails: no Phoenix parent to match against
   - Qualified candidates list is empty
   - Falls through (line 90)

4. **Correct fallback (line 93):**
   - Calls `db.find_symbols_by_name("Phoenix.Channel")`
   - Should find: Elixir module `Phoenix.Channel` with `parent_id=None, file_path="lib/phoenix/channel.ex"`

5. **context_file filtering (lines 99-108):**
   - Input symbols: `[Phoenix.Channel from lib/phoenix/channel.ex]`
   - Filter: `symbols.filter(|s| s.file_path.contains("lib/phoenix/channel.ex"))`
   - Match: `"lib/phoenix/channel.ex".contains("lib/phoenix/channel.ex")` → true
   - Should return: `[Phoenix.Channel]` ✓

**BUT** - if JS Channel results are being returned instead, this suggests:
- Either the fallback lookup is NOT finding Phoenix.Channel (unlikely if Tantivy finds it)
- OR the fallback results are incorrect somehow
- OR the qualified match is succeeding when it shouldn't (parent matching bug)

### Code Paths & SQL Queries

**Database Query (src/database/symbols/queries.rs:65-82):**
```sql
SELECT ... FROM symbols WHERE name = ?1 ORDER BY language, file_path
```

Returns results ordered by language (elixir < javascript), then file_path. This should prioritize Elixir results if they exist with the same name. But:
- "Phoenix.Channel" (Elixir) ≠ "Channel" (JavaScript) - different names
- First query finds "Channel" only
- Fallback query should find "Phoenix.Channel" only

### Path Normalization (src/base/extractor.rs:35-96)

Paths are canonicalized AND converted to relative Unix-style:
1. Canonicalize: resolves symlinks (/var → /private/var on macOS)
2. Convert to relative: canonical_path relative to workspace_root
3. Normalize to Unix: replace '\\'  with '/'
4. Store in `symbol.file_path` (relative Unix-style)

This should be consistent across all extractions, so context_file filtering should work.

### Why JS Results Appear

**If JS Channel is being returned, possible causes:**

1. **Qualified match succeeds wrongly:**
   - JS Channel somehow has parent_id pointing to a Phoenix symbol (very unlikely)
   - OR the parent matching filter is inverted (bug)

2. **Fallback returns wrong results:**
   - `find_symbols_by_name("Phoenix.Channel")` somehow returns JS Channel
   - This would only happen if stored as name="Channel" with parent/class="Phoenix"

3. **Result ordering issue:**
   - If fallback returns [JS Channel, Phoenix.Channel], and we return first result
   - But deep_dive returns ALL results, not just first

4. **context_file filtering fails:**
   - "lib/phoenix/channel.ex".contains("lib/phoenix/channel.ex") should always be true
   - Unless there's Windows path handling: "lib\\phoenix\\channel.ex" vs "lib/phoenix/channel.ex"
   - But BaseExtractor normalizes to Unix-style, so shouldn't happen

### Architecture Recommendation

The qualified name resolution strategy should detect Elixir modules (or any language where qualified names are single symbols) and skip the parent/child matching for those cases. Options:

1. **Check language first:** If language=="elixir", skip qualified parsing, use full name directly
2. **Use metadata:** Mark symbols that store qualified names differently
3. **Fallback-first:** Try full name first, only attempt parent/child if full name returns nothing
4. **Smart parsing:** Only parse qualified names for languages known to use parent/child structure
