## Planned the Phase 2C watcher runtime boundary

Grounded the approved watcher split against the live 1,099-line runtime and wrote a single-task implementation plan.

- **Boundary:** `runtime.rs` keeps `QueueRuntime`, all fields/constants, constructors, gate/cancellation logic, cycle ordering, watcher-facing methods, and the test-only commit-failure seam.
- **Children:** `processing.rs` owns queue batching and shutdown draining; `repairs.rs` owns persisted and overflow repair; `projection.rs` owns Tantivy retry, commit, failure accounting, and durable freshness publication.
- **Concurrency:** The plan explicitly preserves gate acquisition/drop scopes, dirty-retry ordering, and the shutdown rule that releases the drain guard before reacquiring the workspace gate.
- **Doubt pass:** Claude found the original sibling/privacy contradiction; the reconciled child-module design avoids field widening and keeps effective visibility stable.
- **Verification:** Baseline `cargo nextest run -p julie-runtime` passes 90/90. The plan adds structural RED/GREEN, runtime crate, changed-scope, reliability, and final dev gates.
