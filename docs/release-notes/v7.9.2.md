# Julie v7.9.2

Sidecar bootstrap reliability + deferred-session workspace open fix. This
patch eliminates a thundering-herd race in the embedding sidecar bootstrap
and lets `manage_workspace(operation="open", path=...)` succeed when no
primary workspace is bound yet.

## Highlights

- **Single-flight sidecar bootstrap.** Concurrent first-run sessions on
  the same machine no longer race each other through `uv pip install
  torch` and the torch version probe. A cross-process file lock at
  `<sidecar-cache>/.julie-sidecar-bootstrap.lock` serializes the slow
  bootstrap path while leaving the fast steady-state path uncontended.
- **Deferred-session open-by-path.** When the daemon has a workspace
  pool but no primary workspace is currently bound, opening a workspace
  by path now resolves its on-disk index location directly from the
  pool's shared indexes directory instead of routing through the
  primary-anchor helper. This unblocks the deferred-session open flow.

## Sidecar Bootstrap Lock

Before this fix, every concurrent process that started bootstrap on a
machine with an unprovisioned sidecar venv could each spawn its own
`uv pip install torch==<ver>` and `python -c "import torch; ..."`
probe, producing the symptom captured in the operator log: 14+ parallel
torch installs and 10+ parallel version probes pinned to 30-40% CPU each.

The fix:

- `acquire_bootstrap_lock(&venv_path)` opens (creating if needed) a
  `.julie-sidecar-bootstrap.lock` file in the **parent** of the venv,
  then takes an exclusive `fs2` advisory lock on it.
- Placing the lock file outside the venv means `ensure_venv_exists` can
  `remove_dir_all(venv_path)` to rebuild without invalidating the lock.
- The lock spans the entire managed-bootstrap path: venv creation,
  install-marker check, sidecar editable install, optional CUDA torch
  reinstall, and torch version verification.
- Drop releases the OS lock; on process crash the kernel releases the
  lock at handle close, so a crashed bootstrap doesn't strand future
  starts.

Override paths (`JULIE_EMBEDDING_SIDECAR_PROGRAM`,
`JULIE_EMBEDDING_SIDECAR_VENV`) skip the lock and continue to behave
exactly as before.

## Deferred Workspace Open Routing

`IndexRoute::path_backed_route` now checks `handler.workspace_pool`
first. When the pool is present, on-disk paths are computed directly:

- `<pool.indexes_dir()>/<workspace_id>/db/symbols.db`
- `<pool.indexes_dir()>/<workspace_id>/tantivy`

This matches the layout that `workspace_storage_anchor`-routed lookups
produce in the pool case, but does not require a bound `current
workspace`. Stdio-mode (no pool) and override-based layouts still flow
through the existing helper and are unchanged.
