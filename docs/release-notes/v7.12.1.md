# Julie v7.12.1

Patch release that finishes the daemon cold-start fix from v7.12.0 (commit
`1c45455a`) and makes `discovery.json` the single, authoritative lifecycle
source for adapter readiness on the modern daemon.

No user-facing tool semantics changed. Existing MCP clients and CLI invocations
keep working unchanged.

## Highlights

- **Mandatory discovery handoff on daemon startup.** `run_daemon` now treats
  the early `phase="starting"` `discovery.json` publish as a hard requirement;
  if it fails, startup aborts instead of running a daemon adapters cannot
  observe. `DaemonApp::serve` likewise treats the late `phase="running"`
  publish as a hard requirement, and on failure tears down the HTTP transport,
  bearer token, port file, legacy state file, and the prior
  `phase="starting"` discovery record before returning an error. Closes the
  "headless / stuck-starting" failure mode introduced by v7.12.0's cold-start
  race fix — a daemon could be HTTP-ready but permanently invisible to
  adapters if the running overwrite silently failed.
- **`discovery.json` is the only new-daemon lifecycle source for adapters.**
  `DaemonLauncher::daemon_readiness`, `is_daemon_running`, and the polling
  loops no longer consult `daemon.pid` or `daemon.state`. The legacy
  attach path (`legacy_migration::detect_and_attach`) remains the only
  channel through which the launcher will recognize an old-style daemon, and
  it is invoked explicitly. Stale `daemon-mcp-transport.json` is no longer
  used as a fallback unless an explicit legacy PID is live.
- **Adapter readiness is now non-mutating.** The launcher no longer unlinks
  legacy `daemon.pid` / `daemon.state` files as a side effect of readiness
  checks. Two competing lifecycle contracts had been alive in parallel; the
  new-daemon path is now strictly read-only against legacy artifacts.

## Bug Fixes

- `fix(daemon): mandatory phase="starting" discovery publish in run_daemon` —
  failure now aborts startup with context, instead of warning and continuing
  into a daemon adapters cannot see.
- `fix(daemon): mandatory phase="running" discovery publish in DaemonApp::serve` —
  failure now removes the prior `phase="starting"` discovery record, the
  bearer token, the port file, and the daemon state file, then shuts down
  the HTTP transport and returns a typed startup error. Prevents stuck
  `phase="starting"` daemons.
- `fix(adapter): drop legacy PID/state fallback from new-daemon readiness` —
  legacy artifacts no longer influence `DaemonReadiness` for modern daemons.

## Tests

- New: `test_run_daemon_aborts_when_starting_discovery_publish_fails` —
  proves early publish failure aborts before the HTTP transport starts.
- New: `test_daemon_app_serve_aborts_when_running_discovery_publish_fails` —
  proves late publish failure removes discovery, state, token, and transport
  artifacts.
- New: `test_transport_endpoint_refuses_stale_transport_discovery_without_live_legacy_pid` —
  proves stale `daemon-mcp-transport.json` is not used as a silent fallback.
- Renamed/rewrote launcher readiness tests to assert that legacy PID/state
  files are ignored *and not mutated* by new-daemon readiness checks.

## Upgrade Notes

- No configuration changes required. No CLI flags removed or renamed.
- No MCP tool signatures changed.
- Workspace indexes from v7.12.0 do not require rebuild.
- Adapters running v7.12.0 against v7.12.1 daemons (or vice versa) keep
  working: `discovery.json` is the documented contract on both sides. The
  removed reads are of files the modern daemon no longer writes.
- Operators relying on `daemon.pid` / `daemon.state` for external monitoring
  should switch to `discovery.json` (already the documented contract since
  v7.10.x; this release removes the last fallbacks that masked the switch).
