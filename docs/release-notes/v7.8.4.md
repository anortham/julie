# Julie v7.8.4

Workspace-root safety hardening, faster per-tool test routing, and a docs fix
for the Codex MCP install snippet.

## Highlights

- Sensitive system paths (filesystem root, home directories, `/Users`,
  `/home`, `/root`, Windows system/user containers) are now refused at every
  workspace registration entry point — not only the process-cwd auto-binding
  path covered in v7.8.3. Daemon registry rows can no longer be created for
  these roots.
- MCP handler split into per-tool files (`src/handler/tools/<name>.rs`) so a
  one-tool edit no longer drags `cargo xtask test changed` into a full dev-tier
  fallback. Test bucket count grew 14 → 22 with finer-grained per-tool routing.
- Corrected the Codex MCP install instructions on the marketing site. The
  panel referenced a non-existent `codex mcp add julie -- node ...` command;
  it now shows the actual `~/.codex/config.toml` block users need to paste.

## Workspace-Root Safety Hardening

v7.8.3 added a sensitive-root guard for the process-cwd startup path, but
`WorkspacePool::get_or_init`, `JulieServerHandler::primary_binding_for_root`,
and the `manage_workspace` `index` / `open` / `register` commands could still
upsert sensitive roots into the daemon registry. In practice this surfaced as
`/Users/murphy` appearing as a pending registered workspace.

This release:

- Extracts the guard into `src/workspace/root_safety.rs` (single source of
  truth, separate CWD vs. generic error messages).
- Inserts the guard before any upsert/bind/index path:
  `WorkspacePool::add_workspace_root`, primary-binding canonicalization, the
  index force-reindex flow, the open-target add and existing-row branches, and
  the register/remove command.
- Adds regression test
  `test_get_or_init_rejects_sensitive_root_without_registering_row` proving the
  daemon pool refuses `/` without leaving a registry row.

## Per-Tool Test Routing

The MCP handler previously lived in a single `src/handler.rs` that the changed
selector treated as shared infrastructure — every edit to it forced
`cargo xtask test changed` to fall back to the full dev tier.

- Each of the 12 MCP tools now lives in its own
  `src/handler/tools/<name>.rs`, with a dedicated `#[tool_router]` impl block
  composed in `handler.rs`. `handler.rs` shrunk 3,254 → 2,500 lines.
- `xtask/src/changed.rs` removed `src/handler/` from `DEV_FALLBACK_PREFIXES`,
  added cross-cutting handler files to `DEV_FALLBACK_FILES` to preserve their
  fallback safety, and added 1:1 tool→bucket routing (e.g.
  `src/handler/tools/fast_search.rs` → `tools-search-*`,
  `src/handler/tools/edit_file.rs` → `tools-editing`).
- The monolithic `tools-search` and `tools-get-context` buckets were split
  into per-tool buckets. Buckets: 14 → 22; dev tier entries: 22 → 32.
- New routing tests cover handler tool dispatch, telemetry, and shared-
  infrastructure fallback (`changed_tests.rs`, `manifest_contract_tests.rs`).

The user-visible effect: tool-only edits now run a small targeted bucket set
instead of the full dev tier.

## Codex MCP Install Snippet Fix

The marketing site's Codex install panel previously showed
`codex mcp add julie -- node /path/to/julie-plugin/hooks/run.cjs`. That
command does not exist — Codex configures MCP servers via
`~/.codex/config.toml`. The panel and the copy-to-clipboard payload now show:

```toml
[mcp_servers.julie]
command = "node"
args = ["/path/to/julie-plugin/hooks/run.cjs"]
```

(The same incorrect command appeared in v7.8.3's release notes; treat the
TOML block above as authoritative.)
