# Julie v7.10.1

`JULIE_HOME` env override for daemon storage relocation, a wave of `fast_search`
ranking and dogfood-quality fixes, a dashboard fallback fix, and hardening from
an adversarial code review.

## Upgrade Safety

**No action required for existing installs.** When `JULIE_HOME` is unset (the
default), Julie continues to use `~/.julie` exactly as v7.10.0 did. Existing
daemon state, indexes, registry, and tokens stay in place. Set `JULIE_HOME`
only if you want to move Julie's storage to another drive or path.

## Highlights

- **`JULIE_HOME` storage override.** Operators can relocate daemon state and
  workspace indexes off `~/.julie` (e.g. to a fast SSD or shared volume) by
  setting `JULIE_HOME=/abs/path`. The override is direct — no `.julie` is
  appended. `daemon.db`, lock/pid/port/token/discovery files, adapter and
  daemon logs, migration state, and `indexes/<workspace_id>/...` all live
  under the new home. Empty and relative values are rejected at startup so
  misconfigurations surface immediately. See `docs/OPERATIONS.md` for the
  full migration checklist.
- **`fast_search` definition quality.** A series of ranking and presentation
  fixes for the dogfood corpus: exact definition matches no longer get out-shouted
  by imports, related-definition matches surface their signatures, locations
  output now aligns with the full-format result, and the long-tail noise on
  exact symbol lookups is reduced.
- **Dashboard intelligence fallback respects `JULIE_HOME`.** The
  `open_workspace_db` fallback in `src/dashboard/routes/intelligence.rs` used
  to bypass `DaemonPaths` and open `~/.julie/indexes/...` directly. It now
  routes through `DaemonPaths::workspace_db_path()`, so the dashboard sees the
  same data the daemon writes regardless of `JULIE_HOME`.
- **Adversarial review hardening.** Five findings from a Codex deep review
  fixed in one batch: discovery walker + watcher now exclude anything under
  the configured `JULIE_HOME`; relative `JULIE_HOME` is rejected with a clear
  error; the workspace walker's "skip global Julie home" guard holds even
  when the env is misconfigured; the macOS path-identity check no longer
  unconditionally lowercases (case-sensitive APFS users were getting wrong
  answers); daemon-state filename blacklist tightened.

## What's New

### `JULIE_HOME` Override

`DaemonPaths::try_new()` (`src/paths.rs`) resolves the daemon home in this
order:

1. `JULIE_HOME` env var, if set, non-empty, and an absolute path.
2. `dirs::home_dir().join(".julie")` (the legacy default).

Empty `JULIE_HOME` returns `InvalidInput`. Relative `JULIE_HOME` returns
`InvalidInput` with the actual value in the message — operators see the
mistake instead of silently getting cwd-dependent paths.

The two legacy workspace walkers (`src/tools/workspace/paths.rs` and
`src/workspace/mod.rs`) used to compute the global Julie home with their own
`HOME`/`USERPROFILE` lookups. Those duplicates are gone; both walkers now
share `DaemonPaths::is_any_known_julie_home()`, which consults the configured
override AND always treats the conventional `~/.julie` as a known home — so
even a broken env can't trick the walker into capturing `~/.julie` as a
workspace.

The discovery walker (`src/tools/workspace/discovery.rs`) and file watcher
(`src/watcher/filtering.rs`) consult `DaemonPaths::is_under_julie_home()` and
refuse to index any file under the daemon home, defending against the
operator footgun of pointing `JULIE_HOME` inside a workspace tree.

Daemon-specific filenames (`daemon.token`, `daemon-mcp.token`,
`daemon-mcp-transport.json`, `daemon.state`) joined the existing blacklist
(`daemon.lock`, `daemon.pid`, `daemon.log`, `daemon.db`) as belt-and-suspenders.

### Search Ranking and Dogfood Fixes

Six commits land on the `fast_search` quality bar:

- Exact definition lookups no longer return import statements above the
  actual definitions.
- Related-definition matches show their signature lines instead of stripping
  to bare names.
- `return_format="locations"` and `return_format="full"` now agree on
  exact-definition output ordering.
- Long-tail noise reduction for exact symbol lookups.
- Tighter dogfood ranking for the `fast_search` definition path.
- General dogfood result improvements across the same area.

The `search_quality` dogfood bucket (the 100MB SQLite-fixture regression
gate) was used to validate each change.

### Dashboard Fallback Bug Fix

The intelligence page's `open_workspace_db` fallback path joined
`dirs::home_dir()` with `.julie/indexes` directly, bypassing
`DaemonPaths`. Under `JULIE_HOME`, the dashboard would have opened a
stale or empty index. The fallback now uses `DaemonPaths::try_new()` so
both the workspace-pool path and the no-pool fallback share one source
of truth.

## Adversarial Review Hardening (Codex)

Five findings from a Codex adversarial review of the JULIE_HOME work were
all fixed in commit `fe39bd4a` before tagging this release.

1. **Workspace-contained `JULIE_HOME` could leak daemon state into the
   index.** Token files, transport metadata, and discovery JSON were not
   in the blacklist. Two-layer fix: `DaemonPaths::is_under_julie_home()`
   short-circuits the discovery walker and the watcher, and explicit
   daemon filenames joined the blacklist.
2. **Relative `JULIE_HOME` yielded cwd-dependent daemon identities.**
   `try_new()` now rejects non-absolute values with a clear error.
3. **Misconfigured `JULIE_HOME` disabled the walker's global-home
   guard.** `DaemonPaths::is_any_known_julie_home()` always checks the
   conventional `~/.julie` as a fallback, even when the env is invalid.
4. **macOS path comparison case-folded unconditionally.** The
   `is_julie_home()` check used `to_string_lossy().to_lowercase()` on
   all macOS, which is wrong on case-sensitive APFS. Now canonicalizes
   both sides and compares; falls back to raw `PathBuf` equality only
   when canonicalization fails.
5. **Path tests raced with HOME-mutating tests in other modules.** The
   path-derivation tests now use `DaemonPaths::with_home(<tmp>)` and
   no longer depend on `dirs::home_dir()`. The two remaining HOME-readers
   use multi-key `serial(julie_home_env, home_env)`.

14 new tests were added across `tests::daemon::paths`, `watcher::filtering`,
and `tests::tools::workspace::discovery`. Polluted-env regression
(`JULIE_HOME=/tmp/x cargo nextest run --lib tests::daemon::paths`) is now
30/30 green.

## Upgrade Operational Notes

- **`JULIE_HOME` is unrelated to `JULIE_WORKSPACE`.** `JULIE_WORKSPACE`
  chooses the current workspace root; it does not move daemon state.
- **Per-workspace project logs at `<project>/.julie/logs` are unaffected.**
  They remain project-local regardless of `JULIE_HOME`.
- **All Julie processes must agree on `JULIE_HOME`.** Adapter, daemon,
  CLI tools, and MCP clients need the same value. Mismatched values give
  you two independent daemons writing to different homes.
- **Setting `JULIE_HOME` does not auto-migrate existing data.** The new
  home starts empty. To preserve indexes, move the contents of `~/.julie`
  yourself before starting the new daemon.
- **Empty `JULIE_HOME` is a hard error at startup.** Either set a non-empty
  absolute path or unset the variable.
- **Relative `JULIE_HOME` is also a hard error.** Use an absolute path.
