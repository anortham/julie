# Julie v7.13.3

Rescue cleanup release: Julie returns to a single `julie-server` MCP runtime,
removes the split adapter/daemon launch path, adds a dashboard launcher through
`manage_workspace`, and keeps shrinking the expensive test buckets that made
ordinary rescue work too slow.

## Why This Release Exists

The v7.13.x daemon split introduced too much operational complexity for the
actual user: coding agents running Julie as an MCP server. This release removes
the stale process surfaces and makes the default runtime simpler again:
`julie-server` serves MCP in-process over stdio and elects a per-workspace
leader for writes.

## Runtime And Config Changes

- Manual MCP configs that point at a Julie binary should point at
  `julie-server` directly. Local configs or wrappers that still reference
  `julie-adapter` or `julie-daemon` must be updated.
- Plugin users should keep using the plugin launcher; the matching
  `julie-plugin` update changes that launcher to exec `julie-server`.
- Removed runtime surfaces include the forked daemon, stdio adapter, daemon PID
  and discovery files, daemon HTTP dashboard host, daemon migration helpers, and
  `julie daemon` process-management commands.
- Existing workspace data remains in the normal Julie index locations. The
  runtime change is about process ownership, not source extraction semantics.

## Dashboard

- `manage_workspace` now supports `operation: "dashboard"` from an MCP session.
  It starts the dashboard server in the background, opens the default browser,
  and returns the local URL.
- Loaded primary workspaces are now registered in the workspace registry used by
  the dashboard, so a healthy in-process MCP session no longer leaves the
  dashboard project list showing zero workspaces.
- One-shot CLI wrappers reject dashboard launch because the background server
  would be tied to a short-lived CLI process. Use `julie-server dashboard` for
  direct CLI dashboard launches.

## Test Economy

- The `xtask` test runner now splits the broad workspace and line-search
  buckets into smaller buckets so `changed` can select narrower slices.
- The rescue status docs now track the remaining runtime pressure explicitly:
  workspace indexing is still the biggest slow bucket and remains the next
  target.

## Plugin Distribution

The matching `julie-plugin` update must launch `julie-server` directly from
`hooks/run.cjs`. During the first launch after upgrade, that launcher should
also stop an old extracted `julie-daemon` if one exists so Windows users do not
hit stale binary locks during re-extraction.

## Compatibility

No schema or external extract-contract change in this release.

## Known Issues and Caveats

- **Multi-session leader election is one-shot (no live follower promotion).**
  Per workspace, the first session to acquire the leader lock owns writes and the
  file watcher; other sessions on the same workspace are read-only followers and
  cleanly refuse write tools with a clear message ("another session owns writes
  for this workspace; this is a read-only follower"). If the leader session
  exits, an already-running follower stays read-only until it is restarted —
  leadership is not re-elected live inside a running process. A freshly started
  session re-acquires the lock and reconciles. This is bounded, not a data race:
  the only effect is degraded (read-only) write availability for an existing
  follower until restart.
- **Requests are bounded, not unbounded.** Every in-process MCP read request is
  wrapped in a timeout (default 120s, override
  `JULIE_INPROCESS_REQUEST_TIMEOUT_SECS`, set `0` to disable) and returns a
  bounded error instead of hanging. This closes the earlier failure mode where a
  request against a freshly-opened workspace could stall a session. Repair runs
  in a non-cancellable background task so a timed-out request cannot corrupt an
  in-flight index.
- **Dashboard recovery markers are read-only legacy data.** The unclean-shutdown
  recovery-marker file is no longer written by the in-process server (recovery is
  handled by the startup reconcile). The dashboard only surfaces markers left
  behind by a pre-upgrade daemon build; on a clean in-process install the panel
  is empty by design.
- **Registry visibility is best-effort.** If the workspace registry DB cannot be
  opened (e.g. a filesystem or permission error under `~/.julie/`), the session
  logs a warning and continues with core search/navigation/edit intact; only
  dashboard/list visibility is degraded for that session.

## Operational Knobs

- `JULIE_INPROCESS_REQUEST_TIMEOUT_SECS` — per-request in-process deadline in
  seconds (default 120; `0` disables the ceiling).
- `JULIE_INPROCESS_EMBED_WAIT_SECS` — startup wait for the embedding provider
  before degrading to keyword-only search (default 5); the host keeps warming in
  the background for later sessions.
