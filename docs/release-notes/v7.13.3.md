# Julie v7.13.3

Rescue cleanup release: Julie returns to a single `julie-server` MCP runtime,
removes the split adapter/daemon launch path, adds a dashboard launcher through
`manage_workspace`, and keeps shrinking the expensive test buckets that made
ordinary rescue work too slow.

## Why This Release Exists

The v7.13.x daemon split introduced too much operational complexity for the
actual user: coding agents running Julie as an MCP server. This release removes
the stale process surfaces and makes the default runtime simpler again:
`julie-server` serves MCP in-process over stdio and elects a per-workspace
leader for writes.

## Runtime And Config Changes

- Manual MCP configs that point at a Julie binary should point at
  `julie-server` directly. Local configs or wrappers that still reference
  `julie-adapter` or `julie-daemon` must be updated.
- Plugin users should keep using the plugin launcher; the matching
  `julie-plugin` update changes that launcher to exec `julie-server`.
- Removed runtime surfaces include the forked daemon, stdio adapter, daemon PID
  and discovery files, daemon HTTP dashboard host, daemon migration helpers, and
  `julie daemon` process-management commands.
- Existing workspace data remains in the normal Julie index locations. The
  runtime change is about process ownership, not source extraction semantics.

## Dashboard

- `manage_workspace` now supports `operation: "dashboard"` from an MCP session.
  It starts the dashboard server in the background, opens the default browser,
  and returns the local URL.
- One-shot CLI wrappers reject dashboard launch because the background server
  would be tied to a short-lived CLI process. Use `julie-server dashboard` for
  direct CLI dashboard launches.

## Test Economy

- The `xtask` test runner now splits the broad workspace and line-search
  buckets into smaller buckets so `changed` can select narrower slices.
- The rescue status docs now track the remaining runtime pressure explicitly:
  workspace indexing is still the biggest slow bucket and remains the next
  target.

## Plugin Distribution

The matching `julie-plugin` update must launch `julie-server` directly from
`hooks/run.cjs`. During the first launch after upgrade, that launcher should
also stop an old extracted `julie-daemon` if one exists so Windows users do not
hit stale binary locks during re-extraction.

## Compatibility

No schema or external extract-contract change in this release.

| Signal | Value |
|--------|-------|
| `LATEST_SCHEMA_VERSION` | **28** |
| `EXTRACT_CONTRACT_VERSION` | **3** |
| `hash_algorithm` | **blake3** |

Consumers already compatible with v7.13.2's `(28, 3, blake3)` external extract
metadata do not need a schema re-pin for this release.
