# Julie v7.14.0

Maintenance release that supersedes the pulled v7.13.3 rollout. It keeps the
single `julie-server` runtime direction from v7.13.3, fixes the indexing
performance regression found after that release was drafted back, and hardens
the matching `julie-plugin` launcher so old extracted binaries do not survive a
version upgrade.

## Why This Release Exists

v7.13.3 was pulled after OpenClaw indexing exposed a scaling regression and a
plugin-cache issue made it possible for the dashboard to show an older Julie
version even when the plugin manifests had advanced. v7.14.0 is the clean
replacement release line for existing Julie users.

## Indexing Performance

- Fresh full indexing now defers secondary identifier index maintenance while
  loading large identifier batches, then restores the indexes before commit.
- Relationship-resolution disambiguation now uses exact scoped identifier
  presence instead of scanning qualified identifier prefixes and filtering the
  result set in memory.
- The indexing projection path now moves the extracted batch into projection
  instead of cloning large symbol/file/cleanup vectors.
- Projection rebuild logging now records major phase timings, which gives
  future release checks a clearer view of where large-workspace time is spent.

The OpenClaw regression that previously ran past the local 3-minute budget was
rechecked after the fix as a forced standalone index with embeddings disabled:
`13,494` files, `565,029` symbols, and `185,010` relationships completed in
`164.4s`.

## Plugin Launcher

The matching `julie-plugin` launcher now records the archive basename that
produced the extracted binary. When the plugin updates from one
`julie-vX.Y.Z-*` archive to another, the launcher re-extracts even if a cached
`bin/<target>/julie-server` file has a newer mtime.

This fixes the stale-binary failure mode seen during the pulled v7.13.3 rollout:
an old extracted `julie-server 7.13.2` could keep running and make the dashboard
show `7.13.2` while the plugin package had moved forward.

## Runtime And Compatibility

- No schema or external extract-contract change.
- Manual MCP configs should still point at `julie-server`.
- Plugin users should continue to use the plugin launcher; the launcher handles
  archive extraction and cache invalidation.

| Signal | Value |
|--------|-------|
| `LATEST_SCHEMA_VERSION` | **28** |
| `EXTRACT_CONTRACT_VERSION` | **3** |
| `hash_algorithm` | **blake3** |

Consumers already compatible with v7.13.2's `(28, 3, blake3)` external extract
metadata do not need a schema re-pin for this release.
