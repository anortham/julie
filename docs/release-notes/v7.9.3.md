# Julie v7.9.3

Daemon singleton hardening + auto-embedding catch-up on session connect.
This patch closes a Linux-only daemon respawn cascade that produced 577
processes and hundreds of browser tabs in a 29-second window, and fixes
a related gap where a workspace indexed before the embedding sidecar
finished bootstrapping would stay at 0 vectors indefinitely.

## Highlights

- **Stable Linux process identity.** `linux_process_creation_time` no
  longer derives its value from wall-clock arithmetic. The identity is
  now an FNV-1a hash of `/proc/sys/kernel/random/boot_id` plus the raw
  `/proc/<pid>/stat` starttime ticks. Both inputs are kernel-managed
  and immune to `settimeofday`, NTP step adjustments, suspend/resume,
  and VM live-migration time-warp.
- **Singleton daemon enforcement via flock.** A new
  `daemon.singleton.lock` file (never unlinked) is exclusively flocked
  for the daemon's lifetime. Acquired before PID-file creation, so any
  racer that slips past `O_CREAT|O_EXCL` is caught by a kernel-enforced
  lock and bails cleanly with "another daemon is already running for
  this JULIE_HOME".
- **Three-state PID file classifier.** `PidFile::check_running` is now a
  thin back-compat wrapper around `check_status -> PidFileStatus
  { Alive(u32) | Dead | Indeterminate }`. The launcher distinguishes
  "no daemon" from "daemon mid-write" so it never unlinks the state
  file or respawns into a racing daemon.
- **MissingEmbeddings repair reason.** When a workspace has indexed
  symbols but zero embeddings (e.g. indexed before the Python sidecar
  finished bootstrapping), `plan_primary_workspace_repair` now surfaces
  `MissingEmbeddings` instead of reporting "Index is up-to-date".
  `spawn_workspace_embedding` runs in the background once the provider
  settles, no full re-index required.

## The 577-Daemon Cascade

On Linux, `linux_process_creation_time` previously derived an absolute
epoch value from `SystemTime::now() − /proc/uptime + ticks/HZ` in f64.
`now` and `/proc/uptime` are separate syscalls and `/proc/uptime` is
only hundredths-of-a-second precise, so two calls for the same live PID
returned values that drifted by µs–ms. `check_running` did an exact
`==` comparison and **unlinked the running daemon's PID file** on
mismatch. The adapter's poll loop saw the file disappear, declared the
daemon dead, spawned another, repeated every ~50ms. In one user
report, 577 daemons spawned in 29 seconds, each auto-opening a
dashboard browser tab.

A subsequent attempt to derive boot time from `/proc/stat btime`
(`getboottime64` = `offs_real − offs_boot`) was still vulnerable to
`settimeofday`-style wall-clock steps — flagged by an adversarial Codex
review. The shipped fix uses `boot_id`, which is a UUID generated at
boot and never recomputed from the real-time clock.

Three concurrent defenses now apply:

1. **Stable identity** — `check_running` cannot misclassify a live
   daemon as PID-recycled.
2. **Singleton lock** — even if a future identity regression sneaks in,
   only one daemon can hold `daemon.singleton.lock` per JULIE_HOME.
3. **Indeterminate PID-file state** — empty PID files within a small
   freshness window are no longer treated as Dead; the launcher waits
   instead of unlinking a daemon mid-`create_exclusive`.

The singleton lock file lives at `~/.julie/daemon.singleton.lock` and
is never deleted (so the inode stays stable across process restarts).
The PID file path and format are unchanged.

## Auto-Embedding Catch-Up

`run_primary_workspace_repair` previously returned `None` ("Index is
up-to-date — no indexing needed") for any workspace whose symbols were
already on disk and whose files hadn't changed, regardless of whether
embeddings existed. If the embedding sidecar bootstrapped *after* the
initial index ran (cold-start sidecar takes 30–60s, indexing takes <5s),
the workspace would stay at 0 vectors forever — even after later
sessions saw the provider go Ready.

The fix adds an `IndexingRepairReason::MissingEmbeddings` variant.
After the existing nine repair-reason checks, when no other reason
fires AND `embedding_count == 0`, the planner pushes
`MissingEmbeddings` (unless a catch-up embedding task is already
in-flight for this workspace — checked via `handler.embedding_tasks` so
concurrent session connects don't cancel each other's work).
`run_primary_workspace_repair_body` treats `MissingEmbeddings` as a
reason that requires the embedding pipeline, so the existing catch-up
branch in `commands/index.rs` fires `spawn_workspace_embedding`.

## Adversarial Review Findings Addressed

Codex (gpt-5.5 xhigh, fresh reviewer with `--ignore-user-config
--ignore-rules`) flagged four issues during this branch's review.
Three were addressed before tagging:

- **Finding #1 (high):** `/proc/stat btime` is not invariant for the
  boot session; `settimeofday` affects it. **Fixed** — identity now
  derives from `boot_id` + raw start_ticks, no wall-clock value
  involved.
- **Finding #2 (high):** `MissingEmbeddings` could cancel its own
  in-flight repair under concurrent session connects (the second
  session sees `embedding_count == 0` because the first session's
  embedding task hasn't stored its first batch yet, then
  `cancel_primary_embedding_task` kills the running task and restarts
  from scratch). **Fixed** — planner checks
  `handler.embedding_tasks` before pushing `MissingEmbeddings`.
- **Finding #3 (medium):** A corrupt PID file with a far-future mtime
  could pin the launcher in `Starting` forever. **Fixed** —
  `is_pid_file_fresh` only accepts future mtimes within a small skew
  tolerance (5s), and `classify_legacy_or_unparseable` requires the
  content to be empty for the Indeterminate path.

**Finding #4 (medium, open):** Workspaces where every symbol is
filtered out by embedding-eligibility policy (docs/config-only) will
loop `MissingEmbeddings` every session connect, wasting ~2s per
connect. Not catastrophic; a clean fix requires a persisted
"embedding-pipeline-completed-with-N-eligibles" marker in
`embedding_config`. Tracked as a follow-up.

## Upgrade Notes

- No schema or contract changes.
- No reindex required.
- A new file `~/.julie/daemon.singleton.lock` will be created on first
  daemon start; it is never removed and holds the singleton invariant.
  Safe to delete only while no daemon is running.
- Linux operators who saw the daemon-respawn cascade (a flood of
  `Starting Julie daemon` log lines and dashboard browser tabs) should
  see steady-state one-daemon behavior after upgrading. The underlying
  identity-drift bug is Linux-specific and was visible across multiple
  kernel versions; macOS and Windows were not affected.
- A workspace previously stuck at 0 vectors because the embedding
  sidecar wasn't ready at first-index time will catch up automatically
  on the next session connect once the provider goes Ready (no manual
  `force=true` re-index required).

## Caveats

- Catch-up will re-run on every session connect for workspaces whose
  symbols are all filtered out of embedding eligibility (documentation-
  / config-only repos). Wasted work is bounded to one pipeline pass
  per connect; tracked as a follow-up that needs a persisted
  "pipeline-completed-with-N-eligibles" marker.
- The Python 3.10–3.13 sidecar requirement is unchanged. Hosts
  without a compatible Python interpreter will still see
  `Embedding provider unavailable` and run keyword-only.
