# Julie v7.9.0

Extractor trust-contract release. This version makes tree-sitter extractor
quality more explicit, adds certified symbol body spans and body hashes, and
persists those fields through Julie's storage/search surfaces for downstream
tools that need safer code-edit coordinates.

## Highlights

- New extractor contract version: `2026-05-11.body-span-v1`.
- Every applicable language now participates in the same `body_span` /
  `body_hash` contract. No language-specific edit allowlist is introduced.
- `fixtures/extraction/capabilities.json` now records
  `kind_coverage.body_spans` with `supported`, `not_applicable`, and
  `open_gaps` entries. Current body-span open gaps: `0`.
- Julie's symbol database now stores optional body span byte/line/column
  coordinates plus `body_hash`, with automatic schema migration.
- The tree-sitter certification report now includes body-span coverage and is
  backed by release-profile real-world evidence: 22 verified repos, 0 skipped.

## Body Span And Hash Contract

Julie extractors now emit optional `body_span` and `body_hash` fields for
symbols with a meaningful body.

- `body_span` is a normalized span in original source coordinates:
  1-based lines, 0-based columns, and byte offsets.
- For grammars with explicit body nodes, the span follows the tree-sitter body
  node. That means brace-delimited languages include the body delimiters when
  the grammar body node includes them; indentation-suite languages use the
  suite range.
- Leaf symbols and symbols without a replaceable body report no body span.
- `body_hash` is derived from a normalized token stream inside `body_span`, so
  whitespace-only edits do not change the hash while identifiers, literals,
  operators, comments, punctuation, and delimiters do.

This is intentionally additive. Existing consumers can ignore the new fields.
Consumers that use them should gate on `EXTRACTION_CONTRACT_VERSION` or the
capability snapshot rather than hard-coding language names.

## Extractor Capability Model

The capability matrix now distinguishes:

- `supported`: the language fixture has evidence for that kind and capability.
- `not_applicable`: the kind exists but the capability does not make sense for
  that kind, such as leaf symbols with no body.
- `open_gaps`: the kind is applicable but not yet supported.

Public body-edit support is blocked unless every applicable language has zero
body-span open gaps. This keeps all supported languages on the same functional
bar.

## Storage And Search

The SQLite schema moves to version 25 and adds nullable columns:

- `body_start_line`, `body_start_col`, `body_end_line`, `body_end_col`
- `body_start_byte`, `body_end_byte`
- `body_hash`

Symbol insertion, bulk insertion, query hydration, lightweight symbol queries,
and text-search enrichment now preserve the new fields. The semantic index
engine version includes the new extractor contract, so workspaces should be
refreshed/reindexed before depending on body-span data.

## Documentation Cleanup

Historical raw extractor review dumps under `docs/findings/` were removed.
Current truth now lives in generated capability fixtures, the certification
report, real-world evidence, and the extractor contract docs.

- Existing databases migrate automatically on open.
- Reindex/refresh workspaces before relying on `body_span` or `body_hash`.
- Downstream extractor consumers should update expected contract version to
  `2026-05-11.body-span-v1` and read `kind_coverage.body_spans` from the
  capability snapshot instead of using language allowlists.
