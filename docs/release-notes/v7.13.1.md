# Julie v7.13.1

Windows daemon and adapter hardening. This patch release fixes the
overnight adapter hang, converges the adapter↔daemon version gate so two
client versions can no longer flap the shared daemon, repairs Windows
path normalization and singleton-lock detection, fixes a workspace
implicit-index regression that could fragment the index, and corrects a
line-mode scoped-filter miss. It also batches Tantivy commits and makes
the `xtask` test harness behave on Windows.

**No contract change.** Schema stays at **28**, extract contract at **2**
(unchanged from v7.13.0). Miller and other consumers pinned to `(28, 2)`
need no re-pin. This release is purely runtime/correctness fixes — no
extraction-output changes.

## Highlights

- **The all-night hang is fixed.** When the adapter lost its connection to
  the daemon mid-request (stale-binary restart, daemon crash, transport
  drop), in-flight tool calls could hang forever — a tool call left running
  overnight would still be waiting in the morning. The adapter now answers
  **every** orphaned in-flight request: idempotent calls are requeued and
  replayed against a fresh daemon, and non-idempotent calls get a synthesized
  JSON-RPC lost-connection error so the client unblocks instead of hanging.
  (`src/adapter/forwarder.rs`, `answer_orphaned_in_flight`.)
- **Version-gate convergence (no more daemon flap).** When a connecting
  adapter reports a different version than the running daemon, the gate now
  converges on a single winner instead of letting each side restart the daemon
  into its own version (the v7.12.1-adapter ↔ v7.12.2-daemon flap). A **newer**
  adapter restarts the stale daemon (converges upward); an **older** adapter is
  rejected via the new `RejectStaleAdapter` action — the daemon keeps running
  at its newer version and the stale client must be relaunched to upgrade.
  (`src/daemon/lifecycle.rs`.)
- **Windows singleton-lock detection.** The daemon's single-instance guard now
  recognizes `ERROR_LOCK_VIOLATION` (raw OS error **33**) as a lock conflict,
  not just `WouldBlock` — Windows reports `LockFileEx` contention with code 33,
  so the previous check could miss a live daemon and mis-handle startup.
  Shutdown-event signaling tightened alongside it. (`src/daemon/singleton.rs`.)
- **Windows path normalization fallback.** `to_relative_unix_style` now falls
  back to a case-insensitive, separator-agnostic string comparison when
  `Path::strip_prefix` fails on a drive-letter/case mismatch or a `\\?\` UNC
  vs. plain-path mismatch. Previously such mismatches dropped a raw absolute
  path into the database instead of a workspace-relative one. The same fallback
  is mirrored in the `julie-extractors` crate, and `base/span.rs` now routes
  through `to_relative_unix_style` with explicit `Ok`/`Err` handling instead of
  an absolute-path shortcut. (`src/utils/paths.rs`,
  `crates/julie-extractors/src/utils/paths.rs`.)
- **Implicit-index subdirectory regression fixed.** A daemon or handler rooted
  at a project **subdirectory** (e.g. started from `<project>/src/...`) running
  an implicit `index` (no explicit path) now walks up to the marked project
  root via workspace markers, instead of indexing the subdir as if it were the
  whole workspace — which fragmented the index and broke cross-file navigation
  for the rest of the project. Explicit-path and non-primary-target routing are
  unchanged; only the implicit case was affected. (`src/tools/workspace/commands/index.rs`.)
- **Line-mode scoped-filter miss fixed.** Scoped line-mode searches
  (`file_pattern` / language filters) could miss in-scope results hiding behind
  many high-scoring out-of-scope files. The candidate hard cap is raised to
  2000 and the fetch loop now tracks true candidate-window saturation
  (`LineModeCandidateWindow`) rather than inferring it from result-count, so the
  second-pass widening fires when it should. (`src/tools/search/line_mode.rs`.)

## Fixes

- **Tantivy commit batching.** Projection writes now use `commit=false` and let
  a single `release_writer()` perform one commit, instead of committing on each
  projected document — fewer fsyncs per projection. The file watcher continues
  to commit per file so live edits remain immediately searchable.
  (`src/search/projection.rs`, `src/watcher/runtime.rs`.)
- **xtask test harness on Windows.**
  - New `xtask/src/process.rs` `manifest_command` replaces `cmd /C` for
    invoking nextest with filter expressions, so filterset syntax no longer gets
    mangled by `cmd.exe` quoting.
  - Buckets whose tests are all `cfg(unix)` no longer fail the Windows run with
    nextest's empty-selection exit code 4 — they pass via `--no-tests pass`.
  - Per-bucket timeouts and concurrency (`-j 1` serialization) recalibrated for
    Windows so legitimate runs stop timing out. (`xtask/test_tiers.toml`,
    `xtask/src/runner.rs`, `xtask/src/changed.rs`.)
- **Test hermeticity.** `test_resolve_workspace_root_canonicalizes` now uses a
  real `tempfile::tempdir()` instead of the hardcoded `/tmp` (which resolved to
  `C:\tmp` on Windows and panicked in `canonicalize().unwrap()` on a clean box
  or CI). Added Windows path-fallback unit tests (case-mismatch and
  path-equals-root-modulo-case) in both `src/utils/paths.rs` and the extractor
  crate. (`src/tests/cli_tests.rs`.)

## Test evidence

- **Implicit-index regression (TDD):**
  `test_manage_workspace_implicit_index_from_subdir_resolves_to_project_root`
  written first and confirmed **RED** — pre-fix it resolved
  `…\workspace\src\nested` (the subdir) instead of `…\workspace` (the project
  root). After removing the `!explicit_path_requested` short-circuit it is
  **GREEN**.
- **No regression in workspace routing:** full
  `tests::tools::workspace::refresh_routing` module **10/10 passed**, including
  the explicit-subdir guard
  (`test_manage_workspace_index_subdirectory_under_current_root_stays_primary`)
  and all rebound/reference force/non-force cases.
- **Branch gate:** `cargo xtask test changed` mapped the diff to 17 buckets
  (cli, core, extractors, projection, get-context, and the search family).
  **16 passed**; the lone red was
  `content_scoring_tests::content_hits_have_nonzero_scores_sorted_by_score` in
  `tools-search-format-quality`. Root-caused as **pre-existing, load-induced
  flakiness — not a branch regression**: the test file and the `search_ready`
  reader-readiness gate (B-I6) are unchanged on this branch, the only
  `src/search/` change is the gate-neutral projection commit-batching, and the
  test **passes in isolation (1/1)** and passed in the prior full dev tier. The
  proximate cause is the test's fixed 500 ms sleep against a search path that
  does not yet block on `search_ready`; under heavy parallel load the
  Tantivy reader-reload window is exceeded.
