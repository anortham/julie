# Julie v7.12.2

Patch release with two correctness fixes: the stale-binary daemon restart no
longer force-aborts live MCP sessions, and the file watcher no longer silently
drops delete/rename events when the workspace lives under a symlinked path.

No user-facing tool semantics changed. No configuration changes required.
Workspace indexes from v7.12.x do not require rebuild.

## Bug Fixes

### Daemon: stale-binary restart stops force-aborting live sessions (`fb8d5797`)

The stale-binary auto-restart fired its shutdown the instant `restart_pending`
was armed — `mark_restart_pending` signaled the restart channel on the first
request regardless of how many sessions were connected. With a live session,
that bridged straight into `shutdown()`'s 60s drain, which waits for the session
count to reach zero (i.e. for clients to disconnect). Long-lived MCP sessions
don't disconnect during a daemon-initiated restart, so the drain always timed
out and **force-aborted the live session, writing a recovery marker every time**
(the observed `count=18` recovery-marker storm).

The fix is *defer + reap*:

- `mark_restart_pending` now signals the restart channel only when
  `active_sessions == 0`. Arming with live sessions just sets the latch; the
  disconnect handler re-fires the restart at `remaining == 0` — the documented
  "exit after the last session disconnects" moment.
- A new **restart-scoped session idle-reaper** bounds the leaked-session case the
  prior 2026-05-17 hang-guard worried about: while a restart is pending, sessions
  idle past `session_idle_timeout()` (default 300s, override with
  `JULIE_DAEMON_SESSION_IDLE_TIMEOUT_SECS`) are evicted, and the restart fires
  once the count hits zero — without force-aborting.
- Each session now tracks `last_activity`, touched on every request and
  notification, so an actively-used session is never reaped.

`SIGTERM` / `julie-daemon stop` semantics are unchanged — an operator-initiated
stop still drains and (on timeout) force-aborts as before. Only the
*self-initiated stale-binary restart* defers instead of force-aborting.

### Watcher: stop dropping delete/rename events under symlinked roots (`e1072f20`)

The watcher's path filtering relativized event paths with a naive
`path.strip_prefix(workspace_root)`. On macOS, `notify`/FSEvents reports the
**canonical** event path (e.g. `/private/tmp/proj/...`) even when the workspace
is registered under a **symlinked** root (`/tmp/proj/...`, since `/tmp` →
`/private/tmp`). `strip_prefix` then fails, and the directory-blacklist check
fell back to inspecting the *absolute* path — whose ancestor component `tmp`
matches the blacklist — so delete/rename events for now-gone files were silently
dropped. The result: orphaned symbols and embeddings lingering in the index for
files that were actually deleted or moved. This affects any workspace located
under a symlinked path, not only temp dirs.

The fix adds `paths::relative_within_workspace(path, root)`, a symlink-tolerant
strip (direct strip → canonicalize the root and retry → canonicalize the
candidate and retry). Canonicalizing the *root* recovers the relative path even
for a deleted leaf, because only the root needs to still exist. It is wired into
all three filtering sites (directory blacklist, gitignore, deletion gitignore).
The duplicated Windows `\\?\` UNC-prefix stripping was lifted to a shared
module-level helper.

## Tests

- New unit tests in `src/tests/core/paths.rs` for `relative_within_workspace`,
  using an **explicit symlink** so the symlinked-root coverage is deterministic
  across platforms (independent of whether `/tmp` happens to be a symlink).
- The three previously-failing watcher event-queue tests are now green and serve
  as the end-to-end regression guard:
  `test_remove_event_queued_for_deleted_file`,
  `test_rename_any_nonexistent_path_emits_deleted`,
  `test_queue_atomic_save_coalesces_delete_then_modify`.
- Daemon lifecycle/session/restart tests revised to assert **clean reaping** (no
  recovery marker) instead of force-abort.

## Upgrade Notes

- No configuration changes required. No CLI flags removed or renamed. No MCP tool
  signatures changed.
- Workspace indexes from v7.12.x do not require rebuild. If a v7.12.1-or-earlier
  daemon ran against a workspace under a symlinked path, a deleted/renamed file
  may have left a stale index entry; it will be reconciled on the next catch-up
  scan, or you can force a rebuild with `manage_workspace operation=index
  force=true`.
- New optional env var: `JULIE_DAEMON_SESSION_IDLE_TIMEOUT_SECS` (default 300)
  controls how long an idle session is kept while a stale-binary restart is
  pending before it is reaped.
